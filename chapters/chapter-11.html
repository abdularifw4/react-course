React Internals — Bongkar Mesinnya 🔬

Selama 10 bab lo udah jago pakai React. Sekarang kita buka kap mesinnya: apa yang beneran terjadi pas lo nulis JSX, manggil setState, atau ngasih key di list. Ini materi yang misahin developer yang "bisa React" sama developer yang ngerti React — dan ini yang ditanyain di interview senior.

01

KENAPA SENIOR HARUS PAHAM INTERNALS 🧭

Mindset · 8 menit baca

Pertanyaan jujur dulu: kalau app lo jalan, ngapain repot-repot belajar isi perut React? Jawabannya simpel — lo cuma bisa nge-debug sesuatu yang lo pahami. Selama semuanya lancar, pemahaman internals emang nggak kepake. Tapi begitu ada bug aneh — state yang "nyasar" ke item lain, effect yang jalan dua kali, komponen yang re-render 500 kali per detik — developer yang nggak paham internals cuma bisa nebak-nebak sambil nambahin console.log di mana-mana. Developer yang paham internals langsung tahu ke mana harus lihat.

Analoginya kayak montir vs orang yang cuma bisa nyetir. 🚗 Dua-duanya bisa bawa mobil dari titik A ke titik B. Tapi pas mobil ngeluarin bunyi "tek-tek-tek" dari mesin, yang bisa nyetir doang cuma bisa bilang "kayaknya rusak deh". Montir langsung tahu: itu bunyi dari klep, bukan dari transmisi. Paham internals React bikin lo jadi montirnya — lo bisa "denger" bug dan langsung tahu asalnya dari fase render, dari reconciliation, atau dari scheduling.

Alasan kedua: interview. Pertanyaan interview level junior itu "apa itu useState?". Pertanyaan level senior itu "apa yang terjadi di dalam React pas lo manggil setState?", "kenapa key nggak boleh pakai index?", "apa bedanya render phase sama commit phase?". Semua pertanyaan itu nggak bisa dijawab dengan hafalan API — harus paham mekanismenya. Dan kabar baiknya: jawaban semua pertanyaan itu ada di bab ini.

Alasan ketiga yang sering dilupain: keputusan arsitektur. Kapan perlu React.memo? Kapan useTransition beneran ngebantu dan kapan cuma gaya-gayaan? Kenapa fetch di dalam render itu dosa besar? Tanpa paham internals, lo cuma ngikutin "kata orang di Twitter". Dengan paham internals, lo bisa reasoning sendiri — dan itu definisi senior yang sebenarnya.

Satu hal penting sebelum mulai: bab ini bukan bedah source code React (itu ratusan ribu baris dan berubah tiap versi). Kita fokus ke model mental yang akurat: konsep-konsep yang stabil dari versi ke versi dan yang beneran kepake buat debugging sama interview. Jadi santai, nggak akan ada baris kode dari repo facebook/react di sini. 😄

02

DARI JSX KE ELEMENT OBJECT 📦

Fondasi · 12 menit baca

Rahasia pertama yang harus lo tahu: browser nggak ngerti JSX sama sekali. JSX itu bukan bagian dari JavaScript — dia sintaks tambahan yang harus di-compile dulu (biasanya sama Babel atau SWC di dalam Vite/Next.js) sebelum jalan di browser. Dan hasil compile-nya bukan HTML, bukan string, tapi... panggilan fungsi biasa.

Setiap tag JSX yang lo tulis diubah jadi panggilan React.createElement(type, props, ...children) (atau fungsi jsx() di "automatic runtime" yang dipakai tooling modern — sama aja idenya). Dan fungsi itu return apa? Sebuah object JavaScript polos. Bukan DOM node. Bukan instance komponen. Cuma object dengan beberapa property: type, props, key, dan ref. Object ini yang disebut React element.

Analoginya: element itu cetak biru (blueprint) rumah, bukan rumahnya. 🏗️ Arsitek bisa gambar ulang blueprint seratus kali sehari — murah banget, cuma kertas. Tapi bangun ulang rumah beneran? Mahal dan lama. React element = blueprint (object murah, dibuat ulang tiap render), DOM node = rumah beneran (mahal untuk dibuat dan diubah). Seluruh strategi React dibangun di atas fakta ini: bikin blueprint baru itu murah, jadi bikin aja terus tiap render, lalu bandingin blueprint lama vs baru untuk menentukan renovasi minimal.

Ini juga menjawab kebingungan klasik: bedanya component sama element. Component itu fungsi — pabriknya. Element itu object hasil pabrik itu — deskripsi "gue pengen ada Tombol dengan label X di sini". Waktu lo nulis <Tombol label="Beli" />, fungsi Tombol belum dipanggil. Yang kejadian baru pembuatan element dengan type: Tombol. Fungsinya baru dipanggil nanti sama React pas render. Makanya lo nggak boleh manggil komponen kayak fungsi biasa (Tombol()) — itu ngelewatin React dan bikin hooks-nya kacau.

Karena children juga element, hasil akhirnya adalah pohon object bersarang — element tree. Object ini juga immutable: sekali dibuat, nggak boleh diubah. Kalau UI perlu berubah, React nggak ngedit element lama — dia bikin tree baru dari nol. Kedengeran boros? Tunggu sampai lesson 3, ini justru kuncinya.

JSX vs hasil compile-nya — dua kode ini identik
// Yang lo tulis (JSX):
const ui = (
  <div className="card">
    <h1>Halo</h1>
    <Tombol label="Beli" />
  </div>
);

// Yang dilihat browser setelah compile (classic runtime):
const ui = React.createElement(
  "div",
  { className: "card" },
  React.createElement("h1", null, "Halo"),
  React.createElement(Tombol, { label: "Beli" })
);

// Perhatikan: "div" itu string, tapi Tombol itu REFERENSI FUNGSI.
// Makanya nama komponen wajib huruf besar — biar compiler bisa bedain
// tag HTML (string) dari komponen (variabel).
console.log sebuah element — coba sendiri di app lo!
console.log(<h1 className="judul">Halo</h1>);

// Output di console (disederhanakan):
// {
//   $$typeof: Symbol(react.element),   ← "stempel" keamanan dari React
//   type: "h1",                        ← mau render apa
//   key: null,                         ← identitas di dalam list (lesson 5!)
//   ref: null,
//   props: {
//     className: "judul",
//     children: "Halo"                 ← children juga bagian dari props
//   }
// }
//
// Cuma object biasa! Bukan DOM, bukan HTML, bukan sihir.
// ($$typeof pakai Symbol supaya JSON dari server nggak bisa
//  nyamar jadi element — proteksi injection.)

💡 OH TERNYATA GITU! "Render" di React artinya React manggil fungsi komponen lo untuk dapet element tree baru — bukan "menggambar ke layar". Menggambar ke layarnya kejadian belakangan (commit phase, lesson 4). Mulai sekarang, tiap denger kata "re-render", bayangin "fungsi gue dipanggil lagi dan return object baru", bukan "browser ngegambar ulang". Model mental ini bakal nyelametin lo berkali-kali.

⚠️ JANGAN TERJERAT! Karena element dibuat ulang tiap render, jangan pernah nyimpen element di state lalu mutasi (el.props.children = ... → di development React nge-freeze element & props: mutasi bakal error atau diem-diem diabaikan; di production behavior-nya undefined, tetep JANGAN) dan jangan bandingin element dengan === antar render — selalu beda object walau isinya sama.

03

VIRTUAL DOM: MITOS VS FAKTA 🕵️

Konsep Inti · 12 menit baca

Oke, sekarang kita bongkar mitos paling awet di dunia React: "React itu cepat karena pakai Virtual DOM". Kalimat ini ada di ribuan tutorial, dan... menyesatkan. Kalau lo ngomong gini di interview senior, siap-siap di-follow-up sampai keringetan. Mari kita luruskan.

Faktanya #1: operasi VDOM itu kerja TAMBAHAN, bukan penghemat. Pikirin logikanya: React bikin element tree baru, bandingin dengan yang lama, baru nyentuh DOM. Itu tiga langkah. Manipulasi DOM langsung yang ditulis dengan presisi (kayak yang dilakukan jQuery master jaman dulu, atau yang di-generate compiler Svelte) cuma satu langkah — dan pasti lebih cepat. VDOM nggak pernah bisa lebih cepat dari operasi DOM minimal yang ditulis tangan. Nggak mungkin, secara matematika aja udah kalah.

Faktanya #2: yang mahal itu DOM, dan VDOM adalah strategi untuk MENGHINDARINYA. Nyentuh DOM itu mahal karena bisa men-trigger layout, style recalculation, dan paint di browser. Nulis dan bandingin object JavaScript itu murah banget. Jadi trade-off-nya React: "gue rela bayar biaya kecil (bikin + diff object tree) supaya nggak bayar biaya besar (nyentuh DOM sembarangan)". Hasilnya bukan yang tercepat, tapi cukup cepat — sambil ngasih lo sesuatu yang jauh lebih berharga.

Faktanya #3: nilai jual sebenarnya adalah DEKLARATIF, bukan kecepatan. Tanpa React, lo harus mikir transisi: "kalau user login, hapus tombol login, tambah avatar, update menu...". Dengan React, lo cuma deklarasiin hasil akhir: "kalau login, UI-nya gini". Yang mikirin transisinya (elemen mana yang harus ditambah/dihapus/diubah) adalah proses diffing React. Analoginya kayak nulis ulang draft skripsi di Google Docs vs dibandingin fitur "compare documents" — lo tulis aja versi baru sesuka hati, biar mesin yang nyari bedanya dan cuma nge-highlight bagian yang berubah. 📝

Terus yang di-diff itu apa? Bukan DOM asli, dan bukan "screenshot" DOM. Yang dibandingin adalah element tree hasil render sekarang vs data hasil render sebelumnya yang React simpan (di fiber tree — sabar, lesson 6). DOM asli baru disentuh di akhir, itupun cuma bagian yang ketahuan beda. Fun fact: tim React sendiri udah jarang pakai istilah "Virtual DOM" karena sering disalahartikan — dokumentasi baru lebih suka nyebut proses ini "reconciliation".

Jadi jawaban interview yang bener: "VDOM itu bukan soal jadi lebih cepat dari DOM manipulation manual — dia trade-off: biaya diffing yang murah ditukar dengan programming model deklaratif, plus jaminan update DOM yang minimal dan terprediksi." Itu jawaban level senior. 🎓

Apa yang sebenarnya di-diff — dan hasil akhirnya di DOM
// Render #1 menghasilkan tree:            Render #2 menghasilkan tree:
// { type: 'div', props: {                 { type: 'div', props: {
//     children: [                             children: [
//       { type: 'h1',                           { type: 'h1',
//         props: { children: 'Keranjang: 2' }     props: { children: 'Keranjang: 3' }
//       },                                      },
//       { type: 'button', ... }                 { type: 'button', ... }
//     ]                                       ]
// }}                                       }}

// React membandingkan kedua tree ini (object vs object, murah):
// - div   → sama type-nya? ya → pertahankan DOM node-nya
// - h1    → sama type-nya? ya → cek props → children berubah!
// - button→ sama persis → skip
//
// Operasi DOM yang benar-benar dijalankan CUMA SATU:
h1DomNode.textContent = 'Keranjang: 3';
// Bukan innerHTML ulang, bukan bangun ulang div. Sebiji itu doang.

💡 OH TERNYATA GITU! Ini juga menjelaskan kenapa re-render itu biasanya bukan masalah performa: yang keulang cuma pembuatan object + diffing (murah). App baru lemot kalau (a) fungsi komponen lo sendiri berat (loop/filter data gede tiap render), atau (b) hasil diff-nya memang banyak perubahan DOM. Makanya diagnosa dulu pakai Profiler sebelum nyebar React.memo ke mana-mana (pelajaran Bab 10!).

⚠️ JANGAN TERJERAT! Jangan bilang "React paling cepat karena VDOM" di interview. Svelte dan SolidJS nggak pakai VDOM dan di banyak benchmark justru lebih cepat. Kekuatan React itu ekosistem + model deklaratif + concurrent rendering — bukan klaim "tercepat".

04

RENDER PHASE & COMMIT PHASE 🎬

Konsep Inti · 12 menit baca

Setiap update di React melewati dua fase yang tugasnya beda total. Paham pemisahan ini = paham setengah dari internals React. Analoginya: syuting film. 🎬 Fase pertama, sutradara boleh ngambil take berulang-ulang, gonta-ganti angle, bahkan buang semua footage dan mulai ulang — penonton nggak lihat apa-apa. Fase kedua, film final ditayangin di bioskop — ini publik, nggak bisa ditarik lagi, dan harus utuh (nggak mungkin nayangin setengah film).

Render phase = proses syutingnya. React manggil fungsi-fungsi komponen lo, ngumpulin element tree baru, lalu diff dengan yang lama buat nandain apa aja yang perlu berubah. Sifat pentingnya: fase ini harus murni (pure) — nggak boleh ada efek samping. Kenapa aturannya seketat itu? Karena di React modern, render phase bisa di-pause, diulang dari awal, atau dibuang sama sekali tanpa pernah sampai ke layar (detailnya di lesson 6). Kalau lo naruh side effect di sini — fetch, mutasi variabel global, setState tanpa kondisi — efek itu bisa kejalan dua kali, atau kejalan untuk render yang nggak pernah tayang. Kacau.

Commit phase = penayangannya. React ngambil daftar perubahan hasil diffing dan menuliskannya ke DOM beneran — insert node, update attribute, hapus node. Fase ini sinkron dan nggak bisa diinterupsi — harus selesai dalam satu tarikan napas, supaya user nggak pernah lihat UI setengah jadi (misal: header udah baru tapi body masih lama). Habis DOM di-update, React jalanin useLayoutEffect secara sinkron (sebelum browser sempet nge-paint), lalu browser paint, lalu useEffect jalan belakangan tanpa ngeblokir tampilan.

Dari sini lahir aturan main yang selama ini lo hafal tanpa tahu alasannya: side effect ditaruh di useEffect — karena useEffect dijamin cuma jalan setelah commit, alias setelah render-nya beneran tayang. Nggak akan pernah jalan untuk render yang dibuang. Refs jangan dibaca/ditulis saat render — karena ref.current untuk DOM baru diisi saat commit; saat render dia masih nunjuk ke yang lama (atau null). Ukur layout di useLayoutEffect — karena dia jalan setelah DOM berubah tapi sebelum paint, jadi lo bisa ukur + koreksi posisi tanpa kelihatan "loncat".

Cheat sheet-nya: render phase = mikir (boleh dibatalin), commit phase = eksekusi (sekali jalan, tuntas). Semua yang "boleh dibatalin" harus pure; semua yang "nyata" (DOM, subscription, timer, fetch) hidupnya setelah commit.

Boleh vs nggak boleh di masing-masing fase
function ProfilUser({ userId }) {
  // ===== RENDER PHASE — semua di body fungsi ini =====
  // ✅ boleh: hitung-hitungan dari props/state (pure)
  const inisial = userId.slice(0, 2).toUpperCase();

  // ❌ HARAM di render phase:
  // fetch('/api/user/' + userId);        → fetch tiap render + bisa dobel
  // window.title = userId;               → side effect ke luar
  // ref.current.focus();                 → DOM belum tentu ada/final
  // setCount(count + 1);                 → infinite render loop!
  // localStorage.setItem('last', userId) → efek untuk render yang mungkin dibuang

  // ===== SETELAH COMMIT — tempat yang benar untuk efek =====
  React.useEffect(() => {
    // ✅ DOM udah tayang, render ini "resmi" — aman ngefek
    document.title = 'Profil ' + userId;
    const ctrl = new AbortController();
    fetch('/api/user/' + userId, { signal: ctrl.signal });
    return () => ctrl.abort(); // cleanup (penting buat lesson 9!)
  }, [userId]);

  React.useLayoutEffect(() => {
    // ✅ DOM udah berubah TAPI belum di-paint → pas buat ukur layout
    // const tinggi = boxRef.current.offsetHeight;
  });

  return <div>{inisial}</div>;
}

// │ 2. Dapet element tree baru                                  │
// │ 3. Diff dengan tree lama → tandai efek: UPDATE/PLACE/DELETE │
// └──────────────────────────────────────────────────────────────┘

// │ 6. Isi/update refs (ref.current = node baru)                │

💡 OH TERNYATA GITU! Sekarang lo tahu kenapa console.log di body komponen bisa muncul padahal UI-nya nggak berubah: log itu bagian dari render phase — dan render bisa terjadi lalu hasilnya dibuang (atau di-StrictMode-in, lesson 9). Log di useEffect = "render ini beneran tayang". Log di body = "React lagi mikir doang".

⚠️ JANGAN TERJERAT! useLayoutEffect itu sinkron dan ngeblokir paint — kalau isinya berat, UI kerasa nge-freeze. Default-nya selalu useEffect; pakai useLayoutEffect CUMA kalau lo perlu ukur/koreksi layout sebelum user sempet lihat.

05

RECONCILIATION — ALGORITMA DIFFING & RAHASIA KEY 🔑

Wajib Interview · 15 menit baca

Sekarang bagian paling penting: gimana persisnya React membandingkan dua tree? Secara teori, algoritma optimal untuk diff dua pohon itu kompleksitasnya O(n³) — untuk 1.000 elemen artinya semiliar operasi. Nggak mungkin dipakai. Jadi React ambil jalan pintas: dua heuristik (asumsi praktis) yang bikin diffing jadi O(n), dengan syarat lo sebagai developer bantu dikit. Dua heuristik itu adalah inti dari reconciliation.

Heuristik #1: kalau type-nya beda, buang seluruh subtree. <div> jadi <span>? React nggak akan repot-repot ngecek dalemnya — DOM node lama dihancurkan, semua komponen di dalamnya di-unmount (state-nya hilang!), lalu dibangun ulang dari nol. Ini juga berlaku untuk komponen: <FormLogin /> diganti <FormDaftar /> = unmount + mount, walaupun dua-duanya sama-sama ngerender form yang mirip. Kalau type-nya sama, React pertahankan DOM node-nya dan cuma update attribute yang berubah — state anak-anaknya selamat.

Heuristik #2: untuk list, React butuh identitas — itulah key. Tanpa key, React bandingin anak-anak list berdasarkan posisi: anak ke-0 lama vs anak ke-0 baru, dst. Buat list yang cuma nambah di ekor, itu oke. Tapi begitu lo hapus item pertama, semuanya bergeser: item ke-1 lama dibandingin sama item ke-0 baru (beda data!), dan React mikir "oh, SEMUA item berubah" — padahal cuma satu yang dihapus. Dengan key yang stabil, React bisa mencocokkan item lama dan baru berdasarkan identitas: "key 'b' masih ada, cuma pindah posisi — pertahankan".

Nah, ini alasan kenapa index sebagai key itu bom waktu: index = posisi. Ngasih key={i} itu sama aja kayak nggak ngasih key — lo cuma bikin linter diem, perilakunya tetap perbandingan posisi. Analoginya: loker gym yang labelnya "urutan antrian", bukan nama orang. 🏋️ Lo naruh barang di loker #1 karena lo antrian pertama. Besoknya orang di depan lo pulang, antrian geser, dan tiba-tiba barang di loker #1 "jadi milik" orang lain. Barang (state DOM: isi input, posisi scroll, fokus) nempel di loker (posisi), bukan di orang (data). Key berbasis id = loker berlabel nama — mau antrian geser kayak gimana pun, barang lo tetap barang lo.

Yang "nyasar" pas key salah itu semua state yang nempel ke posisi: isi uncontrolled input, state internal komponen (useState), fokus, scroll, bahkan animasi CSS. Bug-nya sering nggak kelihatan di demo (list statis), lalu meledak di production pas user mulai hapus/sort/insert. Buktiin sendiri di playground bawah ini — ini salah satu bug paling klasik di dunia React. 👇

Heuristik #1 — type menentukan hidup-mati subtree
// Kasus A: type SAMA → DOM node dipertahankan, cuma props di-update
// render lama: <div className="merah" />
// render baru: <div className="biru" />
// aksi React : node.className = 'biru'   (murah, state anak aman)

// Kasus B: type BEDA → subtree lama dihancurkan total
// render lama: <div><Counter /></div>
// render baru: <section><Counter /></section>
// aksi React : unmount div + Counter (state Counter HILANG),
//              mount section + Counter baru (state mulai dari nol)

// Makanya pola ini bahaya diam-diam:
function Layout({ mobile, children }) {
  // tiap `mobile` berubah, SEMUA anak di-remount & kehilangan state!
  return mobile ? <div>{children}</div> : <main>{children}</main>;
}
Heuristik #2 — diffing list: tanpa key vs dengan key
// Data lama: [Ayam, Bakso, Cilok] → hapus "Ayam" → [Bakso, Cilok]

// ── key = index (perbandingan posisi) ──────────────────────
// posisi 0: "Ayam"  vs "Bakso" → dianggap item yang SAMA, teksnya di-update
// posisi 1: "Bakso" vs "Cilok" → dianggap item yang SAMA, teksnya di-update
// posisi 2: "Cilok" vs (kosong) → dihapus
// → 2 update + 1 delete, dan state DOM baris 0 (input/fokus) "diwariskan"
//   ke data yang salah. BUG. 💥

// ── key = id (perbandingan identitas) ──────────────────────
// key "ayam"  → nggak ada di tree baru → unmount bersih
// key "bakso" → masih ada → dipertahankan utuh, nggak disentuh
// key "cilok" → masih ada → dipertahankan utuh, nggak disentuh
// → 1 delete doang. Cepat DAN benar. ✅

{menu.map(item => <MenuRow key={item.id} item={item} />)}

💡 OH TERNYATA GITU! Key juga bisa jadi senjata, bukan cuma aturan: ganti key = paksa remount. Mau reset seluruh state form pas user ganti akun? <FormProfil key={userId} /> — tiap userId berubah, React unmount form lama dan mount yang baru, semua state balik ke awal. Satu atribut, nol useEffect reset manual. Ini trik favorit para senior (dan resmi direkomendasikan docs React).

⚠️ JANGAN TERJERAT! Kapan index boleh jadi key? CUMA kalau list-nya memenuhi semua ini: nggak pernah di-reorder, nggak pernah di-filter/insert/hapus dari tengah, dan item-nya nggak punya state (input, checkbox, animasi). Ragu? Pakai id. Dan JANGAN pakai Math.random() sebagai key — itu lebih parah dari index: semua item remount tiap render.

06

FIBER — KENAPA REACT DITULIS ULANG TOTAL 🧵

Arsitektur · 13 menit baca

Tahun 2017, React 16 rilis dengan sesuatu yang gila: seluruh mesin internalnya ditulis ulang dari nol, tapi API-nya nggak berubah sama sekali. Proyek rahasia bertahun-tahun ini namanya Fiber. Pertanyaannya: kenapa repot banget? Apa yang salah sama mesin lama?

Masalahnya: mesin lama (disebut "stack reconciler") memproses seluruh tree secara rekursif — sekali mulai render, dia harus jalan sampai habis, nggak bisa berhenti di tengah. Buat tree kecil, nggak masalah. Tapi bayangin tree dengan ribuan komponen: render bisa makan 200ms, dan selama 200ms itu main thread ke-lock — browser nggak bisa ngerespon ketikan, klik, atau animasi. Itulah rasa "UI nge-freeze" yang bikin app kerasa murahan. JavaScript itu single-threaded: kalau React lagi kerja, nggak ada yang lain bisa kerja.

Analoginya begini: mesin lama itu kayak kasir minimarket yang lagi ngelayanin borongan 500 barang dan nggak bisa di-interrupt. 🛒 Lo cuma mau bayar sebotol air, tapi harus nunggu 500 barang itu di-scan semua. Fiber ngubah kasirnya jadi lebih cerdas: tiap beberapa detik dia nengok ke antrian — "ada yang urgent nggak?" Kalau ada yang cuma bawa sebotol air (ketikan user = prioritas tinggi!), dia pause scanning borongan, layani yang urgent sampai selesai, baru lanjut borongannya dari posisi terakhir. Total kerjanya sama, tapi nggak ada yang merasa ditelantarkan.

Gimana caranya secara teknis (tetap tanpa source code, tenang)? Kuncinya ganti struktur data. Rekursi lama pakai call stack JavaScript — dan call stack nggak bisa di-pause. Fiber menggantinya dengan linked list dari object-object yang disebut fiber node — satu fiber per komponen/elemen, masing-masing punya penunjuk ke anak, saudara, dan induknya. Merender jadi kayak jalan-jalan menyusuri rantai: kerjakan satu fiber ("unit of work" — panggil komponennya, diff anak-anaknya), simpan posisi, cek jam. Udah kerja ~5ms? Serahkan dulu main thread ke browser, nanti lanjut dari fiber terakhir. Karena posisinya disimpan di object (bukan di call stack), render jadi interruptible — bisa di-pause, dilanjut, bahkan dibuang kalau ada update yang bikin kerjaan tadi basi.

Fiber juga punya trik double buffering kayak game: ada dua tree — current (yang lagi tayang di layar) dan workInProgress (draft yang lagi dibangun di belakang layar). Semua kerja render dilakukan di draft; layar nggak disentuh sama sekali. Begitu draft selesai, commit phase tinggal "tukar pointer" + terapkan mutasi — makanya commit bisa cepat dan atomik. Fiber node ini juga rumah bagi state lo: hooks disimpan sebagai linked list di dalam fiber komponennya, diurutkan berdasarkan urutan pemanggilan — dan ini alasan asli kenapa hooks nggak boleh dipanggil di dalam if: urutannya harus sama persis tiap render, karena React nyocokinnya pakai urutan, bukan nama.

Penting dicatat: Fiber (v16) baru nyiapin kemampuan pause-resume ini. Fitur yang beneran memanfaatkannya — concurrent rendering — baru dinyalain penuh di React 18 lewat createRoot. Itu cerita lesson 7 dan 8.

Gambaran konseptual sebuah fiber node (BUKAN source code asli)
// Bayangan sederhana isi satu fiber node — cukup buat model mental:
const fiber = {
  type: FormLogin,        // komponen/tag yang diwakili
  memoizedState: hooks,   // linked list hooks: useState #1 → useState #2 → ...
  memoizedProps: props,   // props dari render terakhir yang tayang

  child: fiberAnak,       // ↓ anak pertama       (pengganti rekursi:
  sibling: fiberSebelah,  // → saudara berikutnya   tree jadi linked list
  return: fiberInduk,     // ↑ balik ke induk       yang bisa di-pause)

  alternate: fiberKembar, // pasangan di tree satunya (current ⇄ workInProgress)
  flags: 'Update',        // hasil diff: Placement / Update / Deletion
  lanes: prioritas,       // seberapa urgent update di fiber ini
};

💡 OH TERNYATA GITU! "Kenapa hooks nggak boleh di dalam if/loop" akhirnya kejawab tuntas: hooks disimpan sebagai linked list berurutan di fiber. Render pertama: useState #1 → #2 → #3. Kalau render kedua ada if yang bikin #2 ke-skip, React nyocokin state #3 ke slot #2 — state ketuker semua. Bukan aturan gaya-gayaan, tapi konsekuensi langsung struktur datanya.

⚠️ JANGAN TERJERAT! Interruptible ≠ "React nyicil ngegambar ke layar". Yang bisa di-pause cuma render phase (fase mikir). Commit ke DOM tetap sinkron dan atomik — user nggak akan pernah lihat UI setengah jadi. Kalau lo jawab "React nge-render DOM sedikit-sedikit" di interview, itu salah.

BATCHING & SCHEDULING — SATU RENDER UNTUK SEMUA ⚡

React 18 · 11 menit baca

Kuis cepat: di dalam satu event handler lo manggil setState tiga kali berturut-turut. Berapa kali komponen re-render? Kalau jawaban lo "tiga"... salah. Jawabannya satu. Ini namanya batching: React ngumpulin semua update state yang terjadi berdekatan, lalu memprosesnya dalam satu render. Kayak nitip belanja ke teman yang mau ke warung 🏪 — tiga orang nitip, temannya tetap jalan sekali, bawa satu daftar belanjaan. Bukan bolak-balik tiga kali.

Ini juga alasan kenapa setState kerasa "nggak langsung": manggil setCount(count + 1) itu nggak mengubah variabel count saat itu juga — dia cuma menjadwalkan update ke antrian React. Variabel count di scope lo tetap nilai lama sampai render berikutnya (ingat: dia cuma const biasa!). Makanya setCount(count + 1) tiga kali cuma naik 1 (tiga-tiganya baca count lama yang sama), sementara setCount(c => c + 1) tiga kali naik 3 — updater function dieksekusi berurutan terhadap nilai antrian terbaru pas render.

Yang berubah besar di React 18: automatic batching. Dulu (React 17 ke bawah), batching cuma jalan di dalam event handler React. Begitu update terjadi di setTimeout, promise .then(), atau async/await setelah fetch — tiap setState langsung render sendiri-sendiri. Tiga setState setelah fetch = tiga render, lengkap dengan tiga kemungkinan UI "nanggung" di antaranya. React 18 (dengan createRoot) membereskan ini: di mana pun update terjadi, tetap di-batch. Kalau lo beneran butuh DOM ke-update sinkron di tengah-tengah (misal mau ukur posisi buat scroll), ada pintu darurat: flushSync dari react-dom — tapi ini jarang banget dibutuhkan.

Bagian kedua judul lesson ini: scheduling alias prioritas. Nggak semua update itu setara. React 18 mengelompokkan update ke dalam "lanes" (jalur) berdasarkan urgensinya: input diskrit kayak klik dan ketikan = prioritas tinggi (harus kerasa instan), transisi dan update yang ditandai nggak urgent = prioritas rendah (boleh nunggu, boleh di-pause, boleh dibuang kalau keburu basi). Di sinilah investasi arsitektur Fiber (lesson 6) balik modal: karena render bisa diinterupsi, React bisa lagi ngerjain update prioritas rendah, terus user ngetik → pause, kerjain ketikan dulu, baru balik. Gimana cara lo sebagai developer menandai update sebagai "nggak urgent"? Itu useTransition — lesson berikutnya.

Demo: berapa kali render kalau setState 3x?
function Demo() {
  const [a, setA] = React.useState(0);
  const [b, setB] = React.useState(0);
  const [c, setC] = React.useState(0);
  console.log('RENDER!'); // hitung berapa kali muncul

  function handleClick() {
    setA(a + 1);
    setB(b + 1);
    setC(c + 1);
    // React 17 & 18: "RENDER!" muncul 1x — di-batch (event handler)
    console.log(a, b, c); // masih nilai LAMA! setState = jadwal, bukan assignment
  }

  async function handleFetch() {
    const res = await fetch('/api/data');
    setA(1);
    setB(2);
    setC(3);
    // React 17 (setelah await): "RENDER!" 3x — nggak di-batch 😱
    // React 18 (createRoot)   : "RENDER!" 1x — automatic batching ✅
  }

  return <button onClick={handleClick}>{a}-{b}-{c}</button>;
}
Updater function & flushSync — dua senjata seputar batching

💡 OH TERNYATA GITU! Batching juga menjelaskan kenapa dua setState dengan object yang "kelihatan sama" nggak bikin dua render nanggung — dan kenapa React sengaja bikin setState asinkron: kalau tiap setState langsung render sinkron, komponen dengan 5 state yang di-update bareng bakal render 5x dengan 4 di antaranya menampilkan kombinasi state yang nggak konsisten (setengah baru setengah lama). Batching = konsistensi, bukan cuma performa.

⚠️ JANGAN TERJERAT! Jangan "nunggu" state baru dengan setTimeout atau baca state langsung setelah setState — itu race condition yang nunggu meledak. Butuh nilai terbaru buat kalkulasi lanjutan? Hitung dulu di variabel lokal (const next = count + 1), pakai variabel itu, baru setCount(next). Butuh bereaksi terhadap perubahan state? Itu kerjaan useEffect.

08

CONCURRENT FEATURES — USETRANSITION, USEDEFERREDVALUE, SUSPENSE 🚦

React 18 · 13 menit baca

Sekarang saatnya panen dari semua teori Fiber + scheduling: fitur concurrent yang bisa lo pakai langsung. Intinya satu kalimat: lo ngasih tahu React mana update yang urgent dan mana yang boleh telat, dan React bakal mengatur supaya yang urgent nggak pernah ke-blok sama yang berat.

useTransition — buat kasus "satu aksi user memicu dua update: satu ringan, satu berat". Contoh nyata: search box dengan hasil filter dari 10.000 produk. Update ringan = teks di input (harus instan!), update berat = ngerender ulang list hasil filter. Tanpa transition, dua-duanya jadi satu render besar → tiap ketikan nunggu list selesai → input nge-lag. Dengan startTransition(() => setFilter(text)), update list ditandai prioritas rendah: input di-render duluan (cepat), list nyusul di belakang — dan kalau user ngetik lagi sebelum list selesai, render list yang setengah jadi dibuang dan mulai ulang dengan teks terbaru. Bonusnya isPending buat nampilin indikator "lagi mikir...".

useDeferredValue — sama tujuannya, beda pintu masuk. Kadang lo nggak pegang setState-nya (misal value datang dari props, atau dari library). const deferredQuery = useDeferredValue(query) ngasih lo "versi telat" dari sebuah nilai: pas query berubah, deferredQuery masih nilai lama dulu, lalu nyusul di render prioritas rendah. Pasangan sejatinya React.memo di komponen berat — supaya komponen berat itu cuma re-render pas nilai telatnya beneran berubah. Rule of thumb: pegang setter-nya → useTransition; cuma pegang nilainya → useDeferredValue.

Suspense — declarative loading state: "kalau ada anak yang belum siap, tampilkan fallback". Yang perlu jujur diomongin: Suspense untuk data fetching di client itu bukan barang DIY — komponen lo nggak bisa "suspend" sendiri dengan fetch biasa; butuh integrasi dari framework/library (Next.js App Router, Relay, TanStack Query lewat useSuspenseQuery). Yang udah dipakai semua orang hari ini: React.lazy + Suspense buat code splitting, dan Suspense boundary di Next.js buat streaming SSR.

Kapan BENERAN dipakai di production? Jujur-jujuran: jarang tapi menentukan. 90% interaksi (toggle modal, submit form, update kecil) nggak butuh transition — malah jadi noise. Pakai useTransition/useDeferredValue pas ada gejala spesifik: input atau interaksi terasa nge-lag karena satu render berat yang nggak bisa dihindari — filter/search list gede, ganti tab dengan konten berat, re-render chart/visualisasi, navigasi antar halaman berat. Urutannya tetap: ukur dulu (Profiler), coba kecilin rendernya (memo, virtualisasi — Bab 10), baru concurrent features sebagai lapisan terakhir. Rasain sendiri bedanya di playground ini 👇 (ketik cepat di dua mode).

Pola useTransition & useDeferredValue
// ── useTransition: lo pegang setState-nya ───────────────
function Search({ products }) {
  const [query, setQuery] = React.useState('');
  const [filter, setFilter] = React.useState('');
  const [isPending, startTransition] = React.useTransition();

  function handleChange(e) {
    setQuery(e.target.value);                       // urgent: input instan
    startTransition(() => setFilter(e.target.value)); // santai: list nyusul
  }

  const hasil = products.filter(p => p.nama.includes(filter));
  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending && <span>⏳ menyaring...</span>}
      <DaftarProduk items={hasil} />
    </>
  );
}

// ── useDeferredValue: lo cuma pegang nilainya ───────────
function Hasil({ query }) {                // query datang dari luar
  const deferredQuery = React.useDeferredValue(query);
  const stale = query !== deferredQuery;   // lagi nyusul? kasih tanda
  return (
    <div style={{ opacity: stale ? 0.5 : 1 }}>
      <ListBerat query={deferredQuery} />  {/* wajib React.memo! */}
    </div>
  );
}

💡 OH TERNYATA GITU! startTransition itu bukan debounce/throttle. Debounce = nunda mulai kerja (nunggu user diem). Transition = kerja langsung mulai, tapi bisa diinterupsi dan dibuang kalau ada yang lebih urgent. Di production keduanya sering dipakai barengan: debounce buat ngirit network request, transition buat ngejaga rendering tetap responsif. Beda layer, beda musuh.

⚠️ JANGAN TERJERAT! Update di dalam startTransition harus sinkron di dalam callback-nya — startTransition(async () => { await ...; setState(...) }) nggak nge-mark setState-nya sebagai transition (setState-nya jalan setelah await, di luar jendela transition). Dan jangan bungkus update input yang harus instan pakai transition — nanti ketikan user yang malah ditunda. 😅

STRICTMODE & DOUBLE RENDER — FITUR, BUKAN BUG 👮

Debugging · 10 menit baca

Pertanyaan yang muncul di StackOverflow ribuan kali: "kenapa useEffect gue jalan 2x?? React bug ya??". Bukan bug. Itu StrictMode — dan setelah 8 lesson internals, lo sekarang punya semua konteks buat ngerti kenapa dia sengaja nyiksa kode lo.

Ingat lesson 4 dan 6: render phase harus pure karena bisa diulang/dibuang, dan effect harus punya cleanup karena komponen bisa unmount-remount kapan aja. Masalahnya: kode yang melanggar aturan ini kelihatan baik-baik aja di development — bug-nya baru muncul di production, di kondisi yang susah direproduksi. StrictMode adalah jawabannya: "daripada nunggu meledak di production, gue ledakkin sekarang di depan mata lo". Analoginya kayak simulasi gempa buat ngetes gedung 🏗️ — goyangannya sengaja, supaya kelemahan struktur ketahuan pas masih bisa diperbaiki.

Yang dilakukan StrictMode di development (dan HANYA di development — di production build dia nggak melakukan apa-apa): (1) Double render — fungsi komponen, updater function, dan initializer useState/useReducer dipanggil dua kali. Kalau render lo pure, hasil dua panggilan identik dan nggak ada yang berubah. Kalau ada side effect nyempil di render (mutasi variabel luar, push ke array global) — efeknya dobel, langsung kelihatan. (2) Double effect — setiap effect dijalankan dengan siklus mount → cleanup → mount lagi. Ini simulasi "komponen bisa unmount dan remount" (yang beneran terjadi dengan fitur kayak Fast Refresh, atau fitur reuse state yang disiapkan React). Kalau cleanup lo bener, siklus ekstra ini nggak berbekas. Kalau nggak ada cleanup — koneksi dobel, listener dobel, fetch dobel: ketahuan.

Jadi apa yang harus lo lakukan pas lihat efek jalan 2x? JANGAN matiin StrictMode (itu kayak nyopot alarm kebakaran karena bunyinya berisik), dan jangan juga pakai hack useRef penanda "udah jalan sekali" buat nge-skip run kedua — itu nyembunyiin masalah yang sama. Solusinya selalu: tulis cleanup yang bener, supaya efek lo aman dijalankan berapa kali pun. Fetch → AbortController atau flag ignore. Listener → removeEventListener. Interval → clearInterval. Koneksi → disconnect. Pola umumnya: setiap efek harus bisa "membatalkan dirinya sendiri".

Bonus deteksi StrictMode lainnya: warning buat API deprecated (legacy lifecycle, string ref, findDOMNode). Intinya, StrictMode itu kontrak masa depan: kode yang lolos StrictMode dijamin siap buat fitur concurrent hari ini dan fitur React berikutnya.

Efek yang gagal vs lulus StrictMode
// ❌ GAGAL StrictMode: nggak ada cleanup
React.useEffect(() => {
  fetch('/api/user/' + id).then(r => r.json()).then(setUser);
  // mount → cleanup(kosong) → mount lagi = DUA fetch jalan.
  // Kalau yang PERTAMA selesai terakhir → UI nampilin data basi.
  // (Ini juga persis race condition di production pas `id` ganti cepat!)
}, [id]);

// ✅ LULUS StrictMode: efek bisa membatalkan dirinya sendiri
React.useEffect(() => {
  const ctrl = new AbortController();
  fetch('/api/user/' + id, { signal: ctrl.signal })
    .then(r => r.json())
    .then(setUser)
    .catch(e => { if (e.name !== 'AbortError') setError(e); });
  return () => ctrl.abort(); // run pertama dibatalkan bersih ✨
}, [id]);

// ✅ Pola yang sama untuk listener/interval/koneksi:
React.useEffect(() => {
  const t = setInterval(tick, 1000);
  return () => clearInterval(t); // tanpa ini: StrictMode = timer dobel
}, []);

💡 OH TERNYATA GITU! StrictMode double-fetch itu sebenarnya simulasi race condition gratis. Efek fetch yang "kena masalah" di StrictMode hampir pasti punya bug race condition beneran di production (user klik profil A lalu cepat-cepat klik profil B — respons A datang terakhir → nama B, data A 💀). Benerin buat StrictMode = benerin buat production. Dua burung, satu AbortController.

⚠️ JANGAN TERJERAT! Ketemu tutorial yang nyaranin hapus <StrictMode> dari main.jsx "biar nggak double render"? Kabur. Double render itu gejala; yang sakit itu kode efeknya. Dan inget: di production StrictMode nol overhead — nggak ada alasan performa buat nyopotnya.

KESALAHAN UMUM — JANGAN SAMPAI KEJADIAN DI KODE LO 🚨

Empat dosa internals yang paling sering kejadian di code review (dan interview)

1. Index sebagai key di list yang dinamis. Udah dibedah tuntas di lesson 5: index = posisi, bukan identitas. List yang bisa di-sort/filter/hapus + index key = state nyasar. Linter diem bukan berarti kode bener.

Kesalahan #1 — index key
// ❌ SALAH — lolos linter, tapi bug nunggu di production
{todos.map((todo, i) => <TodoRow key={i} todo={todo} />)}

// ✅ BENAR — identitas stabil dari data
{todos.map(todo => <TodoRow key={todo.id} todo={todo} />)}

// Nggak punya id? Bikin pas datanya DIBUAT (bukan pas render!):
const tambah = (teks) =>
  setTodos(t => [...t, { id: crypto.randomUUID(), teks }]);

2. Mutasi state langsung. Sekarang lo tahu alasan teknisnya: React menentukan "perlu render ulang nggak?" dengan membandingkan referensi object (pakai Object.is). Lo push ke array lama = isinya berubah tapi referensinya sama = React bilang "nggak ada perubahan" = UI diem. Dan bahkan kalau render kepicu dari state lain, React.memo dan dependency array useEffect juga ketipu referensi yang sama.

Kesalahan #2 — mutasi state
// ❌ SALAH — referensi sama, React nggak lihat perubahan
todos.push(baru);        setTodos(todos);   // UI diem 😶
todo.done = true;        setTodos(todos);   // UI diem
todos.sort(bandingkan);  setTodos(todos);   // sort() itu MUTASI in-place!

// ✅ BENAR — selalu bikin referensi baru
setTodos([...todos, baru]);
setTodos(todos.map(t => t.id === id ? { ...t, done: true } : t));
setTodos([...todos].sort(bandingkan)); // copy dulu, baru sort
// atau pakai method baru yang non-mutating: todos.toSorted(bandingkan)

3. Ngulang mitos "VDOM bikin React paling cepat". Di interview, ini bedanya kandidat yang ngafal tutorial vs yang paham. Versi salah: "React cepat karena Virtual DOM lebih cepat dari DOM". Versi benar: "diffing element tree itu kerja tambahan yang murah; imbalannya model deklaratif + update DOM minimal. Diffing-nya O(n) berkat heuristik type & key, dan framework tanpa VDOM bisa saja lebih cepat." Bilang gitu, pewawancara langsung duduk tegak. 😎

4. Side effect di render phase. Fetch/subscribe/mutasi global di body komponen = efek bisa jalan dobel (StrictMode, render yang dibuang) dan bikin render loop kalau nyentuh setState. Kasus paling sering: setState langsung di body "biar sinkron sama props" — itu resep infinite loop atau minimal render dobel yang nggak perlu.

Kesalahan #4 — side effect saat render
function Keranjang({ items }) {
  // ❌ SALAH — side effect + setState telanjang di render phase
  fetch('/api/track?n=' + items.length);      // kepanggil tiap render (dobel di StrictMode)
  const [total, setTotal] = React.useState(0);
  setTotal(hitungTotal(items));               // 💥 infinite render loop!

  // ✅ BENAR #1 — nilai turunan nggak butuh state SAMA SEKALI
  const total2 = hitungTotal(items);          // hitung aja tiap render, murah

  // ✅ BENAR #2 — efek beneran (network) hidup di useEffect + cleanup
  React.useEffect(() => {
    const ctrl = new AbortController();
    fetch('/api/track?n=' + items.length, { signal: ctrl.signal });
    return () => ctrl.abort();
  }, [items.length]);

  return <p>Total: {total2}</p>;
}

// dan berapa nilai akhir n?

Jawaban: 2 render tambahan, n = 4. Dua setN pertama jalan sinkron di event handler → di-batch jadi 1 render (n: 0→2). Lalu await motong eksekusi — dua setN berikutnya jalan di microtask setelah render pertama selesai. Berkat automatic batching React 18, dua-duanya tetap digabung jadi 1 render lagi (n: 2→4). Total 2 render. Bonus: di React 17, dua setN setelah await TIDAK di-batch → 3 render total. Kalau updater-nya setN(n + 1) (bukan function), nilai akhirnya cuma 1 — keempat panggilan baca n=0 dari closure handler yang sama (re-render bikin closure baru, tapi async function yang lagi jalan tetap pegang closure lama). Paham soal ini = paham lesson 7 tuntas.

✏️ Latihan 2: Cari bug-nya — tombol sort yang "kadang jalan kadang nggak" klik untuk lihat solusi
Soal — user lapor: klik "Urutkan" kadang nggak ngefek
function DaftarHarga({ awal }) {
  const [produk, setProduk] = React.useState(awal);

  function urutkan() {
    setProduk(produk.sort((a, b) => a.harga - b.harga));
  }

  return (
    <>
      <button onClick={urutkan}>Urutkan termurah</button>
      {produk.map((p, i) => <Baris key={i} produk={p} />)}
    </>
  );
}

Ada DUA dosa internals di sini. Pertama: .sort() memutasi array in-place dan me-return array yang sama — setProduk menerima referensi lama, Object.is bilang "sama", render di-skip. UI-nya baru "kebetulan" update kalau ada state lain yang berubah — makanya laporan bug-nya "kadang jalan". Kedua: key={i} — begitu sort beneran jalan, semua baris "ganti isi tapi tetap di posisinya", jadi state internal tiap Baris (misal tombol "favorit" yang lagi aktif) nggak ikut pindah.

Solusi
function urutkan() {
  // referensi BARU → React lihat perubahan → render jalan
  setProduk(prev => [...prev].sort((a, b) => a.harga - b.harga));
  // (atau: prev.toSorted((a, b) => a.harga - b.harga))
}

// dan key pakai identitas, bukan posisi:
{produk.map(p => <Baris key={p.id} produk={p} />)}
✏️ Latihan 3: Jelasin ke junior — "kenapa fetch gue jalan 2x, hapus StrictMode aja ya?" klik untuk lihat solusi

Tulis (beneran, di kepala atau di catatan) jawaban 4-5 kalimat seolah lo lagi code review junior yang mau ngehapus StrictMode gara-gara fetch dobel. Kalau jawaban lo mencakup poin-poin ini, lo lulus:

Model jawaban: "Fetch-nya jalan 2x karena StrictMode sengaja nge-mount → cleanup → mount ulang setiap komponen di development, buat mendeteksi efek yang nggak punya cleanup. Ini dev-only — user di production nggak akan ngalamin double fetch dari StrictMode. TAPI, kalau efek kita 'rusak' karena dijalankan dua kali, artinya dia juga rusak pas user pindah halaman cepat atau ganti filter cepat — respons lama bisa datang belakangan dan nimpa data baru (race condition). Jadi yang kita perbaiki bukan StrictMode-nya, tapi efeknya: tambahin AbortController di cleanup. Setelah itu run kedua otomatis membatalkan run pertama, dan kode kita sekaligus kebal race condition di production."

Kode pendukung jawaban
React.useEffect(() => {
  const ctrl = new AbortController();
  fetch(url, { signal: ctrl.signal })
    .then(r => r.json())
    .then(setData)
    .catch(e => { if (e.name !== 'AbortError') setError(e); });
  return () => ctrl.abort();
}, [url]);

RECAP — MODEL MENTAL LENGKAP LO SEKARANG 📝

Dari JSX sampai layar — lo sekarang tahu seluruh perjalanannya.

  • ✓JSX → element object — JSX di-compile jadi createElement yang me-return object polos {type, props, key, ref}. Element = blueprint murah; component = pabriknya; DOM = rumah beneran yang mahal.
  • ✓Virtual DOM yang sebenarnya — bukan "bikin React tercepat"; diffing itu biaya tambahan yang murah, ditukar dengan model deklaratif + mutasi DOM minimal. Istilah kerennya sekarang: reconciliation.
  • ✓Render vs commit phase — render = mikir (pure, boleh dibuang/diulang), commit = eksekusi ke DOM (sinkron, atomik). useEffect hidup setelah commit; makanya side effect dilarang di render.
  • ✓Reconciliation O(n) — dua heuristik: type beda = remount subtree (state hilang), list dicocokkan pakai key. Index sebagai key = perbandingan posisi = state nyasar. Key ganti = remount paksa (trik reset form!).
  • ✓Fiber — rewrite total React 16: rekursi diganti linked list fiber node, kerja dicicil per unit of work, dua tree (current ⇄ workInProgress). Hasilnya: render yang bisa di-pause — fondasi semua fitur concurrent. Hooks disimpan berurutan di fiber → makanya nggak boleh di dalam if.
  • ✓Batching & scheduling — setState = jadwal, bukan assignment; React 18 nge-batch di mana pun (1 render untuk 3 setState, bahkan setelah await). Updater function (c => c + 1) buat update beruntun; flushSync cuma buat darurat.
  • ✓Concurrent features — useTransition (lo pegang setter) & useDeferredValue (lo pegang nilai) menandai update sebagai non-urgent yang boleh diinterupsi; Suspense buat lazy loading + streaming (data fetching-nya lewat framework). Dipakai pas ada render berat yang nggak bisa dihindari — bukan di semua tempat.
  • ✓StrictMode — double render & double effect di dev itu disengaja: mendeteksi render yang nggak pure dan efek tanpa cleanup. Solusinya selalu cleanup yang benar (AbortController!), bukan nyopot StrictMode.
