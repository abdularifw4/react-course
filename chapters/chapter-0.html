Belum pernah ngoding JavaScript serius? Santai. Bab ini adalah jembatan lo dari nol ke React — cuma materi yang beneran kepake, tanpa teori yang bikin ngantuk. Selesai bab ini, lo siap masuk Bab 1 dengan pede.

01

Kenapa Harus Jago JS Dulu?

Mindset · 7 menit baca

Sebelum kita ngomongin React, gue mau jujur dulu sama lo: 90% error yang bikin pemula frustrasi di React itu sebenarnya bukan error React — itu error JavaScript. Serius. Lo bakal ketemu pesan kayak undefined is not a function atau Cannot read properties of undefined, dan itu semua masalah JS murni, bukan React.

Analoginya gini: React itu kayak motor balap. Kenceng, keren, semua orang pengen naik. Tapi kalau lo belum bisa naik sepeda — belum ngerti keseimbangan, ngerem, belok — naik motor balap itu bukan keren, itu bahaya. JavaScript itu sepedanya. Nggak perlu jadi pembalap dulu, tapi minimal lo harus lancar kayuh, lancar belok, lancar berhenti.

Kabar baiknya: lo nggak perlu belajar SEMUA JavaScript. JS itu bahasa yang luas banget — ada prototype, generator, proxy, dan segala macam fitur yang bahkan developer senior jarang sentuh. Yang lo butuhin buat React itu cuma sekitar 20% dari JS, tapi 20% yang paling sering dipakai. Dan 20% itulah isi bab ini: variabel, function, array method, object, destructuring, module, dan async. Itu doang. Beneran.

Nah, sebelum mulai, lo butuh satu alat yang udah ada di laptop lo sekarang juga: browser console. Ini "taman bermain" gratis buat nyobain JavaScript tanpa install apa pun. Cara bukanya:

Cara Buka Browser Console — Playground Gratis Lo
// Chrome / Edge / Brave:
//   Windows : tekan F12, lalu klik tab "Console"
//   Mac     : Cmd + Option + J
// Firefox:
//   Windows : Ctrl + Shift + K
//   Mac     : Cmd + Option + K

// Terus ketik ini dan tekan Enter:
console.log('Halo, gue calon React developer! 🚀');

// Console juga bisa jadi kalkulator:
5 + 3           // 8
100 * 0.5       // 50

// Dan bisa ngecek tipe data:
typeof 'halo'   // 'string'
typeof 42       // 'number'
typeof true     // 'boolean'

Kebiasaan yang gue saranin banget selama ngikutin bab ini: jangan cuma baca — ketik ulang setiap contoh kode di console. Ilmu ngoding itu masuknya lewat jari, bukan lewat mata. Orang yang cuma baca tutorial 10 jam kalah sama orang yang praktik 1 jam. Ini bukan motivasi kosong, ini pengalaman semua developer di dunia.

Satu lagi: di bab ini juga ada playground interaktif yang bisa lo edit langsung di halaman ini, plus challenge yang ngetes kode lo otomatis. Jadi lo nggak harus bolak-balik ke console terus — tapi console tetap skill wajib karena nanti pas debugging React, lo bakal hidup di sana.

OH TERNYATA GITU! console.log() itu bakal jadi sahabat terbaik lo selama karir ngoding. Developer senior dengan gaji 2 digit pun masih pakai console.log buat debugging tiap hari. Nggak usah malu — itu alat sah dan powerful!

JANGAN TERJERAT! Jangan skip bab ini karena "pengen cepet ke React". Pemula yang loncat langsung ke React biasanya balik lagi ke sini setelah 2 minggu frustrasi. Investasi beberapa jam sekarang = hemat berminggu-minggu kebingungan nanti.

02

Variabel & Tipe Data — Toples Berlabel

Dasar · 10 menit baca

Oke, materi pertama: variabel. Bayangin dapur nenek lo yang penuh toples berlabel — ada toples bertuliskan "Gula", "Kopi", "Kerupuk". Label itu nama variabelnya, isi toplesnya itu datanya. Kapan pun lo butuh gula, lo nggak nyari "benda putih manis butiran kecil" — lo langsung ambil toples berlabel "Gula". Variabel itu ya gitu: nama untuk menyimpan data biar gampang dipanggil lagi.

Di JavaScript modern, ada dua cara bikin variabel: let dan const. Bedanya simpel: let boleh diganti isinya, const nggak boleh.

let vs const — Kapan Pakai yang Mana
// let = isinya BOLEH diganti
let umur = 25;
umur = 26;            // ✅ boleh, ulang tahun!

// const = isinya NGGAK BOLEH diganti (constant)
const nama = 'Budi';
nama = 'Andi';        // ❌ ERROR! Assignment to constant variable

// RULE OF THUMB (dipake semua developer React):
// 1. Selalu mulai dengan const
// 2. Ganti ke let CUMA kalau nilainya memang perlu berubah
// Kenyataannya: 95% variabel di kode React itu const!

// Oh iya, ada juga `var` (cara lama sebelum 2015):
var lama = 'jangan dipakai';  // ⛔ lupakan var, ini warisan masa lalu

Terus, data yang bisa disimpan di toples itu ada beberapa tipe. Kenalan dulu sama lima tipe dasar yang bakal lo temuin tiap hari di React:

5 Tipe Data yang Wajib Lo Kenal
// 1. STRING — teks, diapit kutip
const namaProduk = 'Kopi Susu Gula Aren';
const merk = "Kopi Kenangan";          // kutip satu/dua sama aja

// 2. NUMBER — angka (bulat & desimal, satu tipe yang sama)
const harga = 18000;
const rating = 4.8;

// 3. BOOLEAN — cuma dua nilai: true atau false
const sedangPromo = true;
const sudahHabis = false;
// Di React: nentuin komponen mana yang tampil, loading atau nggak, dll.

// 4. UNDEFINED — variabel ada, tapi BELUM diisi
let alamat;
console.log(alamat);   // undefined ("toplesnya ada, isinya belum ada")

// 5. NULL — sengaja dikosongin ("toplesnya SENGAJA gue kosongin")
let pacar = null;      // sengaja dikosongin... eh maksudnya 😅

// Cek tipe pakai typeof:
typeof namaProduk;     // 'string'
typeof harga;          // 'number'
typeof sedangPromo;    // 'boolean'

Bedanya undefined dan null sering bikin bingung, jadi gue kasih analogi: lo buka kulkas nyari sambel. Undefined = rak sambelnya bahkan nggak ada di kulkas itu. Null = raknya ada, tapi ada notes "sambel sengaja nggak dibeli minggu ini". Dua-duanya "kosong", tapi yang satu kelupaan, yang satu disengaja. Di React, lo bakal sering ketemu dua-duanya — misalnya data user yang belum di-fetch (undefined) vs user yang memang nggak login (null).

Kenapa ini penting banget buat React? Karena state di React itu ya variabel-variabel ini. Jumlah item di keranjang belanja? Number. Nama user? String. Modal lagi kebuka atau nggak? Boolean. Data yang belum datang dari server? Undefined atau null. Nguasain tipe data = nguasain setengah dari state management. Beneran deh.

Satu kebiasaan kecil yang bakal bikin kode lo kelihatan profesional dari hari pertama: penamaan variabel. Konvensi di dunia JavaScript itu camelCase — kata pertama huruf kecil, kata berikutnya huruf besar: totalHarga, isLoggedIn, daftarProduk. Dan tolong, kasih nama yang jujur: variabel x atau data2 itu kayak toples tanpa label — dua minggu lagi lo sendiri lupa isinya apa. Khusus boolean, awali dengan is, has, atau sudah biar kebaca kayak pertanyaan: isLoading, hasError. Kode yang enak dibaca itu bukan gaya-gayaan — itu sopan santun ke rekan tim (dan ke diri lo sendiri di masa depan).

OH TERNYATA GITU! const itu ngunci label toplesnya, bukan isinya! Kalau isinya object atau array, isi di dalamnya masih bisa diubah: const keranjang = []; keranjang.push('kopi') itu sah. Yang dilarang cuma nunjuk toples lain: keranjang = []. Konsep ini bakal penting banget pas bahas state React nanti!

03

Function — Mesin Blender Lo

Dasar · 12 menit baca

Kalau variabel itu toples, function itu mesin blender: lo masukin bahan (input), mesinnya muter (proses), keluar jus (output). Masukin mangga → keluar jus mangga. Masukin alpukat → keluar jus alpukat. Mesinnya sama, hasilnya tergantung input. Dan yang paling penting: sekali beli blender, bisa dipakai berkali-kali — nggak perlu bikin mesin baru tiap mau ngejus.

Ini konsep paling penting di seluruh bab ini, karena — spoiler — komponen React itu literally cuma function. Function yang nerima data dan return tampilan. Jadi kalau lo paham function, lo udah paham 70% cara kerja komponen React.

Function Declaration — Cara Klasik
// Bikin "mesin blender"-nya:
function buatJus(buah) {
  return 'Jus ' + buah + ' siap! 🥤';
}
//        ↑ parameter = slot input bahan
// return = hasil yang keluar dari mesin

// Pakai mesinnya (bisa berkali-kali!):
buatJus('Mangga');    // 'Jus Mangga siap! 🥤'
buatJus('Alpukat');   // 'Jus Alpukat siap! 🥤'

// Parameter bisa lebih dari satu:
function hitungTotal(harga, jumlah) {
  return harga * jumlah;
}
hitungTotal(18000, 3);   // 54000

// DEFAULT PARAMETER — nilai cadangan kalau input nggak dikasih:
function sapa(nama = 'Kawan') {
  return 'Halo, ' + nama + '!';
}
sapa('Budi');   // 'Halo, Budi!'
sapa();         // 'Halo, Kawan!'  ← pakai default
// Di React: default props pakai teknik yang PERSIS sama!

Nah, sekarang bagian yang krusial. JavaScript modern punya cara penulisan function yang lebih ringkas, namanya arrow function. Dan begitu lo buka kode React mana pun di dunia, lo bakal lihat arrow function di mana-mana — di event handler, di dalam map, di useEffect, semuanya. Jadi lo wajib lancar bacanya:

Arrow Function — Dialek Sehari-hari React
// Evolusi dari klasik ke arrow — fungsinya SAMA:
function dobelKlasik(n) {
  return n * 2;
}

// 1. Arrow function versi lengkap:
const dobel = (n) => {
  return n * 2;
};

// 2. Kalau parameternya cuma satu, kurungnya boleh dibuang:
const dobel2 = n => {
  return n * 2;
};

// 3. Kalau isinya cuma satu ekspresi, {} dan return boleh dibuang:
const dobel3 = n => n * 2;    // ⚡ IMPLICIT RETURN — paling sering dipakai!

dobel3(5);   // 10

// Kenapa React penuh arrow function? Karena ringkas buat "fungsi sekali pakai":
// tombol.addEventListener('click', () => console.log('diklik!'));
//
// Nanti di React lo bakal nulis ginian tiap hari:
// <button onClick={() => setCount(count + 1)}>Tambah</button>
// angka.map(n => n * 2)
//
// ⚠️ HATI-HATI: kalau pakai {} , return WAJIB ditulis!
const salah = n => { n * 2 };   // ❌ return undefined! ({} = body biasa)
const benar = n => n * 2;       // ✅ implicit return
const benar2 = n => { return n * 2; };  // ✅ explicit return

Perhatiin baik-baik contoh salah di atas — itu salah satu bug paling sering di dunia React. Begitu lo pakai kurung kurawal {}, JavaScript nganggep itu "badan function biasa" dan nunggu kata return. Nggak ada return? Hasilnya undefined. Komponen lo bakal blank tanpa error yang jelas, dan lo bakal garuk-garuk kepala setengah jam. Gue pernah. Semua orang pernah.

Terus, kenapa sih React "penuh arrow function"? Karena di React, lo sering banget butuh function kecil yang dipakai sekali di tempat — misalnya "kalau tombol ini diklik, jalankan ini". Nulis function handleClick() {...} lengkap tiap kali itu bertele-tele. Arrow function bikin lo bisa nulis logika langsung di tempat dia dipakai, singkat dan jelas. Anggap function declaration itu masak besar buat hajatan, arrow function itu masak mie instan — dua-duanya masak, beda occasion.

OH TERNYATA GITU! Komponen React itu cuma function biasa yang namanya diawali huruf kapital dan return tampilan: function Tombol() { return ... }. Nggak ada sihir. Kalau lo udah bisa bikin buatJus(), lo secara teknis udah bisa bikin komponen.

Array Methods — Ban Berjalan Pabrik

WAJIB KUASAI · 15 menit baca

Oke, ini dia lesson terpenting di seluruh bab ini. Kalau lo cuma bisa hafal satu hal dari Bab 0, hafal ini: array method, terutama map(). Kenapa? Karena hampir semua UI di dunia itu list: feed Instagram = list post, marketplace = list produk, WhatsApp = list chat. Dan di React, semua list itu di-render pakai map(). Titik. Map itu jantungnya render list di React.

Array sendiri itu simpel: daftar belanjaan. Satu kertas, banyak item, ada urutannya. Nah, array method itu kayak ban berjalan di pabrik: barang-barang lewat satu per satu, dan di tiap pos ada mesin yang ngapa-ngapain — ada pos yang ngubah tiap barang (map), pos yang nyortir buang yang jelek (filter), pos yang nyari satu barang spesifik (find), dan pos yang ngerangkum semuanya jadi satu angka (reduce).

map & filter — Duo Paling Sering Dipakai di React
const harga = [3500, 15000, 8000, 18000];

// MAP — ubah SETIAP item, hasilnya array BARU dengan panjang sama
const hargaDiskon = harga.map(h => h * 0.9);
// [3150, 13500, 7200, 16200]
// Analogi: ban berjalan yang ngecat semua barang. 4 masuk, 4 keluar.

// FILTER — saring item yang lolos kondisi, hasilnya array BARU (bisa lebih pendek)
const mahal = harga.filter(h => h > 10000);
// [15000, 18000]
// Analogi: satpam yang cuma ngizinin masuk yang memenuhi syarat. 4 masuk, 2 keluar.

// PENTING: array aslinya NGGAK BERUBAH!
console.log(harga);   // [3500, 15000, 8000, 18000] — masih utuh ✅

// Bisa juga array of object (ini bentuk data paling umum di React):
const produk = [
  { nama: 'Indomie', harga: 3500 },
  { nama: 'Kopi Susu', harga: 15000 },
];
const namaProduk = produk.map(p => p.nama);
// ['Indomie', 'Kopi Susu']
find, reduce, some, every — Pelengkap yang Sering Nongol
const keranjang = [
  { nama: 'Indomie Goreng', harga: 3500, stok: 10 },
  { nama: 'Kopi Susu',      harga: 15000, stok: 0 },
  { nama: 'Ayam Geprek',    harga: 18000, stok: 5 },
];

// FIND — cari SATU item pertama yang cocok (bukan array, tapi itemnya langsung)
const kopi = keranjang.find(item => item.nama === 'Kopi Susu');
// { nama: 'Kopi Susu', harga: 15000, stok: 0 }
// Kalau nggak ketemu? Hasilnya undefined (bukan error!)

// REDUCE — rangkum semua item jadi SATU nilai
const total = keranjang.reduce((akumulator, item) => {
  return akumulator + item.harga;
}, 0);   // ← 0 = nilai awal akumulator
// 36500
// Analogi: kasir yang nge-scan barang satu-satu sambil nambahin ke total

// SOME — "ada MINIMAL SATU yang memenuhi?" → true/false
const adaYangHabis = keranjang.some(item => item.stok === 0);
// true (Kopi Susu stoknya 0)

// EVERY — "SEMUANYA memenuhi?" → true/false
const semuaTersedia = keranjang.every(item => item.stok > 0);
// false (karena Kopi Susu habis)

Sekarang bagian serunya. Di React, render list itu polanya selalu: punya array data → map jadi elemen tampilan. Cobain playground di bawah ini — edit kodenya langsung (tambah item belanjaan, ganti emoji, apa aja) terus klik Jalankan:

Lihat kan? belanjaan.map(...) ngubah tiap object jadi satu baris <li>. Data 4 item → tampilan 4 baris. Nambah data → tampilan otomatis nambah. Ini persis cara Tokopedia nampilin ribuan produk — bukan nulis HTML ribuan kali, tapi satu map() yang jalanin data dari server.

Sekarang giliran lo. Challenge pertama di course ini: gabungin filter() + map(). Ubah kodenya sampai semua test hijau.

OH TERNYATA GITU! Cara ngebedain method: map = "ubah semua" (panjang sama), filter = "saring" (bisa lebih pendek), find = "cari satu" (bukan array!), reduce = "rangkum jadi satu", some/every = "cek kondisi" (true/false). Kelimanya nggak pernah ngubah array asli — selalu bikin nilai baru. Konsep "bikin baru, jangan ubah yang lama" ini namanya immutability, dan itu aturan emas React!

JANGAN TERJERAT! map() butuh lo me-return sesuatu di dalam callback-nya. arr.map(x => { x * 2 }) menghasilkan array penuh undefined — inget pelajaran arrow function tadi! Di React, ini wujudnya jadi list yang tiba-tiba kosong padahal datanya ada.

Object & Destructuring — Bongkar Paket

WAJIB KUASAI · 13 menit baca

Kalau array itu daftar belanjaan (urutan penting), object itu KTP: satu entitas dengan banyak keterangan berlabel — nama, alamat, tanggal lahir. Nggak ada urutan "item ke-3", yang ada "field bernama alamat". Hampir semua data di aplikasi nyata bentuknya object: data user, data produk, response API — semuanya object (atau array berisi object).

Object — Bikin, Akses, Ubah
// Object = kumpulan pasangan "kunci: nilai"
const user = {
  nama: 'Budi Santoso',
  umur: 25,
  kota: 'Bandung',
  hobi: ['ngoding', 'ngopi'],   // nilai bisa array...
  alamat: {                      // ...bahkan object lagi (nested)
    jalan: 'Jl. Braga No. 10',
    kodePos: '40111',
  },
};

// Akses pakai titik:
user.nama;             // 'Budi Santoso'
user.alamat.jalan;     // 'Jl. Braga No. 10'
user.hobi[0];          // 'ngoding'

// Akses pakai bracket (kalau nama key-nya dinamis):
const key = 'kota';
user[key];             // 'Bandung'

// ⭐ OPTIONAL CHAINING (?.) — akses aman tanpa takut error:
user.kantor?.nama;     // undefined (bukan error, padahal kantor nggak ada!)
// Tanpa ?. : user.kantor.nama → 💥 TypeError!
// Di React ini penyelamat pas data dari API belum datang.

Sekarang, materi yang bikin kode React kelihatan "aneh" buat pemula: destructuring. Analoginya: lo terima paket dari kurir. Cara lama: lo bawa dus utuh ke mana-mana, tiap butuh sesuatu lo buka dus dulu (user.nama, user.kota...). Destructuring: langsung bongkar dusnya di depan pintu, ambil barang-barang yang lo butuhin, taruh masing-masing di meja dengan namanya sendiri. Sekali bongkar, semua siap pakai.

Destructuring, Spread & Rest — Trio Andalan React
const user = { nama: 'Budi', umur: 25, kota: 'Bandung' };

// DESTRUCTURING — bongkar object jadi variabel:
const { nama, kota } = user;
nama;   // 'Budi'
kota;   // 'Bandung'
// Dulu harus: const nama = user.nama; const kota = user.kota; — capek!

// Bisa dikasih nilai default & ganti nama:
const { pekerjaan = 'belum diisi', nama: namaLengkap } = user;

// Destructuring juga bisa buat ARRAY (berdasarkan posisi):
const [pertama, kedua] = ['merah', 'kuning', 'hijau'];
pertama;   // 'merah'
// 👀 Kenal pola ini? const [count, setCount] = useState(0) — itu dia!

// SPREAD (...) — "tuang semua isi" ke wadah baru:
const userBaru = { ...user, kota: 'Jakarta' };
// { nama: 'Budi', umur: 25, kota: 'Jakarta' } ← copy + update, aslinya utuh!

const angka = [1, 2, 3];
const angkaPlus = [...angka, 4];    // [1, 2, 3, 4] — array baru!

// REST (...) — kebalikan spread: "sisanya tampung di sini":
const { nama: n, ...sisanya } = user;
sisanya;   // { umur: 25, kota: 'Bandung' }

function jumlahkan(...semuaAngka) {          // rest di parameter
  return semuaAngka.reduce((a, b) => a + b, 0);
}
jumlahkan(1, 2, 3, 4);   // 10

Yang paling penting buat React: destructuring bisa dilakukan langsung di parameter function. Dan ini — perhatiin baik-baik — adalah PERSIS cara komponen React nerima props. Kalau lo paham blok kode di playground bawah ini, lo udah paham props sebelum ketemu React:

Coba edit playground-nya: ganti nama, tambah hobi, atau bikin KartuUser ketiga. Perhatiin juga {...userPindahan} — itu spread yang "nuang" semua isi object jadi props. Trik ini bakal lo pakai terus-terusan di React.

OH TERNYATA GITU! Pas nanti lo lihat kode React function Profil({ nama, foto }) — itu bukan sintaks khusus React. Itu destructuring parameter JavaScript biasa yang barusan lo pelajarin! React cuma manggil function lo dengan satu argumen object (props), dan lo bongkar di tempat. Mind = blown?

JANGAN TERJERAT! Spread itu shallow copy — cuma nyalin lapisan terluar. Kalau object lo nested (object di dalam object), lapisan dalamnya masih "nyambung" ke yang asli. Buat Bab 0 cukup tahu aja dulu; nanti pas state React makin kompleks, ini bakal relevan lagi.

Template Literals & Conditional — Kalau… Maka…

Dasar · 10 menit baca

Dua materi ringan tapi supersering dipakai. Pertama, template literals — cara modern nyambungin teks dengan variabel. Dulu kita nyambung string pakai tanda plus yang ribet dan rawan typo. Sekarang tinggal pakai backtick (tanda ` — di kiri atas keyboard, sebelah angka 1) dan "lubang" ${...}:

Template Literals — String dengan "Lubang"
const nama = 'Budi';
const total = 36500;

// Cara lama (plus-plus, rawan lupa spasi):
const pesanLama = 'Halo ' + nama + ', total belanja: Rp' + total;

// ✨ Template literal (pakai backtick ` , bukan kutip biasa!):
const pesan = `Halo ${nama}, total belanja: Rp${total}`;
// 'Halo Budi, total belanja: Rp36500'

// Di dalam ${} boleh ekspresi apa aja:
const info = `Dengan pajak: Rp${total * 1.11}`;
const status = `Kamu ${total > 50000 ? 'dapat gratis ongkir 🎉' : 'belum dapat gratis ongkir'}`;

// Bonus: bisa multi-baris tanpa ribet
const alamatLabel = `Penerima: ${nama}
Kota: Bandung`;

Kedua, dan ini yang gede: conditional alias percabangan. Aplikasi nyata itu penuh keputusan: kalau user login tampilkan dashboard, kalau belum tampilkan tombol login. Kalau keranjang kosong tampilkan "yuk belanja", kalau ada isinya tampilkan daftarnya. Di React, pola ini namanya conditional rendering, dan bahan bakunya tiga: ternary, &&, dan ||. Anggap aja ternary itu lampu lalu lintas: satu kondisi, dua jalur — hijau lewat sini, merah lewat situ.

Ternary, && dan || — Bahan Baku Conditional Rendering
const isLoggedIn = true;
const stok = 0;
const namaPanggilan = '';

// TERNARY — if/else versi satu baris: kondisi ? kalauBenar : kalauSalah
const sapaan = isLoggedIn ? 'Selamat datang kembali!' : 'Silakan login dulu';

// Nanti di React (hafalin bentuknya, ini tiap hari muncul):
// {isLoggedIn ? <Dashboard /> : <TombolLogin />}

// && — "kalau kiri benar, hasilkan yang kanan; kalau salah, stop"
isLoggedIn && console.log('User aktif!');   // jalan, karena true
// Di React: {adaPromo && <BannerPromo />} → banner cuma muncul kalau ada promo

// || — "pakai yang kiri; kalau 'kosong', pakai cadangan kanan"
const tampilkanNama = namaPanggilan || 'Tamu';   // 'Tamu' ('' dianggap falsy)

// ?? (nullish coalescing) — kayak || tapi cuma peduli null/undefined:
const jumlah = 0;
jumlah || 10;    // 10  ← ups, 0 dianggap "kosong" padahal valid!
jumlah ?? 10;    // 0   ← ✅ 0 tetap dipakai, cuma null/undefined yang diganti

// FALSY values (dianggap "salah" oleh JS) — HAFALKAN:
// false, 0, '' (string kosong), null, undefined, NaN
// Selain itu semua TRUTHY (termasuk '0', [], {} — hati-hati!)

Kenapa nggak pakai if/else biasa aja? Boleh banget — dan di logika biasa, silakan pakai. Tapi di dalam JSX React nanti, lo cuma bisa naruh ekspresi (sesuatu yang menghasilkan nilai), bukan statement kayak if/else. Ternary dan && itu ekspresi — makanya mereka jadi senjata utama conditional rendering. Ini alasannya lo latihan sekarang, biar nanti nggak kaget.

OH TERNYATA GITU! Baca ternary itu kayak nanya: kondisi ? jawabanYa : jawabanTidak — "Udah login? Dashboard : kalau nggak, halaman login". Tanda tanya = pertanyaan, titik dua = "kalau nggak". Sekali kebaca gini, lo nggak bakal kebalik lagi.

JANGAN TERJERAT! jumlahItem && <Keranjang /> dengan jumlahItem = 0 bakal me-render angka 0 di layar (bukan "nggak render apa-apa")! Karena 0 && apapun menghasilkan 0, dan React nampilin angka 0. Solusi: jumlahItem > 0 && .... Ini bug legendaris yang nongol di app production beneran!

Modules — Lego yang Bisa Dibagi-bagi

Dasar · 8 menit baca

Bayangin lo nulis semua kode aplikasi dalam SATU file berisi 10.000 baris. Nyari satu function kayak nyari kaos kaki di kamar yang nggak pernah diberesin. Makanya JavaScript modern punya module system: tiap file = satu module, dan antar-file bisa saling "minjemin" kode pakai export dan import.

Analoginya: tiap file itu kotak lego dengan label. Kotak "roda", kotak "jendela", kotak "pintu". Mau bangun rumah lego? Ambil (import) jendela dari kotak jendela, pintu dari kotak pintu. Nggak perlu semua lego ditumpuk di satu kotak raksasa. Di project React nyata, tiap komponen biasanya hidup di file sendiri: Navbar.jsx, ProductCard.jsx, dst.

Export — Dua Rasa: Default & Named
// ===== file: matematika.js =====

// NAMED EXPORT — boleh banyak per file, masing-masing punya nama:
export const PI = 3.14159;

export function tambah(a, b) {
  return a + b;
}

export function kali(a, b) {
  return a * b;
}

// DEFAULT EXPORT — cuma boleh SATU per file, si "bintang utama":
export default function hitungLuasLingkaran(r) {
  return PI * r * r;
}
Import — Perhatiin Kurung Kurawalnya!
// ===== file: app.js =====

// Import DEFAULT export → TANPA kurung kurawal, nama BEBAS:
import hitungLuas from './matematika.js';
import terserahMauNamainApa from './matematika.js';   // sah! tetap dapat yang default

// Import NAMED export → PAKAI kurung kurawal, nama HARUS SAMA:
import { tambah, kali, PI } from './matematika.js';

// Campuran keduanya sekaligus:
import hitungLuasLingkaran, { tambah as jumlahkan } from './matematika.js';
//                                    ↑ "as" = kasih alias

// Yang bakal sering lo tulis di React:
// import React, { useState, useEffect } from 'react';
//        ↑ default     ↑ named exports
// SEKARANG lo tahu kenapa useState pakai kurung kurawal! 🎉

// Import dari package (tanpa ./ berarti dari node_modules):
// import axios from 'axios';
// Import dari file sendiri (pakai ./ atau ../):
// import Navbar from './components/Navbar';

Aturan mainnya gampang diinget: default export itu kayak juara umum sekolah — cuma satu, semua orang tahu, mau lo panggil dengan nama apa pun orangnya tetap itu. Named export itu kayak absen kelas — banyak nama, dan lo harus manggil sesuai nama di absen (atau pakai as buat kasih nama panggilan).

Di dunia React, konvensi yang paling umum: satu komponen per file, di-export sebagai default. Sementara utility function, konstanta, dan custom hooks biasanya named export. Lo bakal lihat pola ini di hampir semua codebase React yang lo buka.

Pertanyaan yang mungkin muncul di kepala lo: "kalau kode gue kepecah jadi puluhan file, gimana browser ngegabunginnya?" Jawabannya: ada alat bernama bundler (di course ini kita bakal pakai Vite) yang tugasnya kayak koki catering — dia baca semua "resep" import/export lo, ngumpulin bahan dari semua file, terus nyajiin satu paket rapi yang siap disantap browser. Lo nggak perlu mikirin detailnya sekarang; yang penting lo paham bahwa import/export itu cara lo menyusun kode, dan bundler yang ngurus sisanya. Makanya di project React nanti lo nggak bakal nulis puluhan tag script di HTML — cukup satu entry point, sisanya keurus otomatis.

OH TERNYATA GITU! Error The requested module does not provide an export named 'X' itu 99% karena ketuker default vs named — lo pakai kurung kurawal padahal harusnya nggak, atau sebaliknya. Sekarang lo udah tahu obatnya sebelum kena penyakitnya.

Async — Pesen Kopi Sambil Nunggu

Penting · 14 menit baca

Ini materi terakhir yang "berat" — tapi tenang, gue bawain pelan-pelan. Pertanyaan pembuka: pas lo buka Tokopedia, dari mana datangnya data produk? Dari server, lewat internet. Dan internet itu... butuh waktu. Bisa 100 milidetik, bisa 3 detik kalau sinyal lagi drama. Pertanyaannya: selama nunggu data, aplikasi lo ngapain?

Analoginya: lo ke kedai kopi. Sistem synchronous (serba nunggu): lo pesen, terus berdiri mematung di depan kasir sampai kopinya jadi — nggak boleh duduk, nggak boleh main HP, antrian di belakang lo ikut macet. Absurd kan? Sistem asynchronous: lo pesen, dikasih nomor antrian, lo duduk santai scroll HP, dan pas kopinya jadi, barista manggil nomor lo. Semua orang happy. JavaScript (dan semua app modern) kerjanya pakai sistem kedua — dan "nomor antrian" itu di JS namanya Promise.

Evolusi Async: Callback → Promise → Async/Await
// === ERA 1: CALLBACK (cara lama) ===
// "Nanti kalau udah selesai, panggil function ini ya"
ambilUser(function (user) {
  ambilPesanan(user.id, function (pesanan) {
    ambilDetail(pesanan[0], function (detail) {
      // makin dalam makin miring... 😵
      // ini dijuluki "CALLBACK HELL" / piramida kiamat
    });
  });
});

// === ERA 2: PROMISE (nomor antrian) ===
// Promise = objek "janji" yang punya 3 kondisi:
// pending (nunggu) → fulfilled (berhasil ✅) atau rejected (gagal ❌)
ambilUser()
  .then(user => ambilPesanan(user.id))     // rantai, nggak nested!
  .then(pesanan => console.log(pesanan))
  .catch(error => console.log('Gagal:', error.message));

// === ERA 3: ASYNC/AWAIT (cara modern, paling enak dibaca) ===
// Kode async yang KELIHATANNYA kayak kode biasa:
async function tampilkanPesanan() {
  try {
    const user = await ambilUser();            // "tunggu sampai jadi"
    const pesanan = await ambilPesanan(user.id);
    console.log(pesanan);
  } catch (error) {
    console.log('Gagal:', error.message);      // semua error nyangkut di sini
  }
}
// Aturan: di dalam function, await wajib function-nya ditandai `async`.
// (Pengecualian: di console browser modern dan ES modules,
//  await level atas alias top-level await juga jalan.)

Tiga era di atas itu bukan sekadar sejarah — lo bakal ketemu ketiganya di kode orang. Tapi yang bakal lo tulis sehari-hari adalah async/await, karena paling gampang dibaca: "await = tungguin ini dulu, baru lanjut baris berikutnya". Di belakang layar dia tetap Promise — cuma bajunya lebih rapi.

Sekarang alat paling penting: fetch() — cara JavaScript ngambil data dari server/API. Ini yang bakal jadi tulang punggung hampir semua app React yang lo bikin:

fetch() — Ngambil Data dari API (coba di console!)
// Coba PASTE ini di browser console lo sekarang — beneran jalan!
async function ambilProduk() {
  try {
    // 1. Kirim request ke API (ini API publik gratis buat latihan)
    const response = await fetch('https://fakestoreapi.com/products/1');

    // 2. Cek dulu: server balasnya oke nggak?
    if (!response.ok) {
      throw new Error(`Server error: ${response.status}`);
    }

    // 3. Ubah response jadi object JavaScript (ini juga async → await!)
    const produk = await response.json();

    console.log(produk.title);   // nama produknya
    console.log(produk.price);   // harganya
    return produk;
  } catch (error) {
    // Internet mati? Server down? Semua ketangkap di sini.
    console.log('Waduh, gagal ambil data:', error.message);
  }
}

ambilProduk();

// Perhatiin ada DUA await:
// await fetch(...)        → nunggu server bales
// await response.json()   → nunggu body-nya selesai dibaca & di-parse
// Dua-duanya "nomor antrian" alias Promise!

Di React nanti, pola ini persis sama, cuma dibungkus: data hasil fetch disimpan ke state, dan selama nunggu lo nampilin loading spinner. Konsep "pending → berhasil / gagal" bakal jadi tiga state UI klasik: loading, data, error. Lo bakal ketemu trio ini ratusan kali sepanjang karir. Kenalan dari sekarang = untung besar.

Satu insight terakhir biar mental model lo bener: await itu nggak bikin seluruh aplikasi berhenti. Yang "nunggu" cuma function async itu doang — sisanya (animasi, klik tombol, scroll) tetap jalan normal. Balik ke analogi kedai kopi: pas lo nunggu nomor antrian dipanggil, kasir tetap ngelayanin pelanggan lain, mesin espresso tetap ngebul, musik tetap muter. Itulah kenapa aplikasi web modern kerasa responsif walau lagi ngambil data: nunggu-nya nggak macetin yang lain. Kalau konsep ini udah klik di kepala lo, selamat — lo udah paham sesuatu yang banyak lulusan bootcamp aja masih suka kebalik.

OH TERNYATA GITU! async function itu selalu return Promise, apa pun isinya. Makanya lo nggak bisa "ngambil nilai langsung" dari function async — lo harus await atau .then() dulu. Kalau lo pernah bingung kenapa console.log(ambilProduk()) munculnya Promise {<pending>} — sekarang lo tahu alasannya!

JANGAN TERJERAT! fetch() itu nggak nganggep HTTP error (404, 500) sebagai error — dia cuma reject kalau koneksinya gagal total. Makanya wajib cek response.ok manual kayak contoh di atas. Banyak developer (termasuk yang udah kerja!) kelewat ini dan bingung kenapa error 404-nya "nggak ketangkap".

DOM & Events — Apa yang React Otomatisasi

Konsep · 9 menit baca

Lesson terakhir! Ini singkat, tapi penting banget buat ngerti kenapa React itu ada. Jadi gini: halaman web yang lo lihat di browser itu direpresentasikan sebagai DOM (Document Object Model) — semacam pohon keluarga untuk elemen HTML. <body> itu kakek, di bawahnya ada anak-anak <div>, yang punya anak lagi <h1>, <p>, dan seterusnya. JavaScript bisa manjat pohon ini dan ngubah-ubah isinya — itulah yang bikin halaman web "hidup".

Vanilla DOM — Cara Manual (yang Bakal React Gantikan)
// HTML-nya: <button id="tombol">Klik: 0</button>

// 1. CARI elemennya di pohon DOM:
const tombol = document.querySelector('#tombol');

// 2. PASANG "kuping" (event listener) — nungguin kejadian:
let jumlahKlik = 0;
tombol.addEventListener('click', () => {
  jumlahKlik = jumlahKlik + 1;

  // 3. UPDATE tampilannya MANUAL:
  tombol.textContent = `Klik: ${jumlahKlik}`;
});

// Event lain yang sering dipakai:
// 'input'  → user ngetik di form
// 'submit' → form dikirim
// 'change' → dropdown/checkbox berubah

// Kelihatan simpel? Iya, KALAU cuma satu tombol.
// Sekarang bayangin keranjang belanja:
// - klik "tambah" → update angka di tombol, DI badge navbar,
//   DI total harga, DI tombol checkout (disabled kalau kosong)...
// - 4+ tempat harus di-update MANUAL, satu kelupaan = BUG.
// Ini yang bikin app vanilla JS gede jadi spaghetti. 🍝

Nah, di sinilah React masuk sebagai pahlawan. Filosofinya: lo nggak perlu lagi nyentuh DOM manual. Lo cukup deklarasiin "kalau datanya begini, tampilannya begitu" — dan React yang ngurus semua update-nya. Analoginya: vanilla DOM itu kayak ngecat rumah sendiri — lo yang mikir tembok mana yang perlu dicat ulang, ngamplas, nutup lantai pakai koran. React itu kayak nyewa kontraktor: lo kasih desain akhirnya ("temboknya biru, ada rak di sini"), kontraktor yang mikir cara paling efisien buat ngerjainnya.

Konkretnya, ini perbandingan yang sama persis — counter tadi versi React (belum perlu lo pahami detailnya, cukup rasain bedanya):

Sneak Peek — Counter yang Sama, Versi React
function Tombol() {
  const [jumlahKlik, setJumlahKlik] = React.useState(0);

  return (
    <button onClick={() => setJumlahKlik(jumlahKlik + 1)}>
      Klik: {jumlahKlik}
    </button>
  );
}

// Nggak ada querySelector. Nggak ada addEventListener.
// Nggak ada textContent = ... manual.
// Lo cuma bilang: "tampilan tombol = 'Klik: ' + jumlahKlik".
// Data berubah → React OTOMATIS update DOM-nya. Itu aja intinya!

// Dan coba perhatiin, SEMUA yang ada di situ udah lo pelajarin:
// - function          → Lesson 3
// - const             → Lesson 2
// - destructuring []  → Lesson 5
// - arrow function    → Lesson 3
// - template-ish {}   → Lesson 6
// Lo UDAH SIAP. 🎓

Jadi kenapa lo tetap perlu ngerti DOM dan event walau React ngurusin semuanya? Karena: (1) pas debugging, DevTools nunjukin DOM asli, bukan kode React lo; (2) event React (onClick, onChange) itu pembungkus tipis dari event DOM asli yang barusan lo pelajarin; dan (3) sesekali lo tetap butuh akses DOM langsung di React (namanya ref — nanti di Bab 4). Paham fondasinya = nggak pernah merasa React itu "sihir".

OH TERNYATA GITU! React itu bukan pengganti JavaScript — dia justru JavaScript banget. Semua yang lo pelajarin di bab ini (map, destructuring, arrow function, ternary, async) bakal muncul mentah-mentah di kode React. Itulah kenapa developer yang JS-nya kuat belajar React-nya ngebut. Dan sekarang, itu lo.

KESALAHAN UMUM — Jebakan Betmen Pemula

Tiga bug klasik yang bakal nyelamatin lo berjam-jam debugging kalau lo kenal dari sekarang.

1. Mutasi array — push vs spread. Di JavaScript murni, push() itu sah. Tapi push() mengubah array asli (mutasi) — dan di React, mutasi state itu dosa besar: React nggak sadar datanya berubah, jadi tampilan nggak ke-update padahal datanya udah nambah. Biasain dari sekarang: bikin array baru pakai spread.

Mutasi vs Immutable
const keranjang = ['Indomie', 'Kopi'];

// ❌ SALAH (kebiasaan yang bakal nyakitin lo di React):
keranjang.push('Telur');          // ngubah array ASLI (mutasi!)
// Di React: state berubah diam-diam → UI nggak re-render → bug misterius

// ✅ BENAR (kebiasaan React-ready):
const keranjangBaru = [...keranjang, 'Telur'];   // array BARU
// Array lama utuh, array baru punya item tambahan.
// React lihat "oh, arraynya beda" → re-render → UI update ✨

// Hal yang sama buat hapus & ubah item:
const tanpaKopi = keranjang.filter(item => item !== 'Kopi');      // hapus
const upperCase = keranjang.map(item => item.toUpperCase());       // ubah
// filter & map emang udah return array baru — makanya mereka darling-nya React!

2. == vs ===. Dua sama dengan itu terlalu baik hati: dia maksa-maksain dua nilai beda tipe biar "cocok" (type coercion), dan hasilnya sering absurd. Tiga sama dengan itu tegas: nilai DAN tipe harus sama. Pakai === selalu, tanpa kecuali.

== yang Baperan vs === yang Tegas
// ❌ == : "yang penting mirip-mirip lah"
'5' == 5        // true  😱 (string dipaksa jadi number)
0 == ''         // true  😱
null == undefined  // true 😱
0 == false      // true  😱

// ✅ === : "harus sama nilai DAN tipenya"
'5' === 5       // false ✅ (string ≠ number)
0 === ''        // false ✅
5 === 5         // true  ✅

// Kasus nyata yang sering kejadian:
const userId = '7';        // dari URL/form, selalu STRING!
const targetId = 7;        // dari database, NUMBER
userId == targetId;        // true — "jalan" tapi nyembunyiin bug tipe data
userId === targetId;       // false — kelihatan langsung ada yang salah
// Solusi bener: Number(userId) === targetId

// RULE: pakai === dan !== SELALU. Lupakan == ada. Titik.

3. Lupa return di arrow function ber-body {}. Udah disinggung di Lesson 3, tapi saking seringnya bug ini muncul (apalagi di dalam map() pas render list React!), dia layak diulang dengan contoh React-flavored:

List Kosong Misterius vs Tiga Cara Benar
const menu = ['Nasi Goreng', 'Mie Ayam', 'Sate'];

// ❌ SALAH — {} tanpa return = semua item jadi undefined:
const list1 = menu.map(item => { `<li>${item}</li>` });
// [undefined, undefined, undefined] → di React: list BLANK, nggak ada error!

// ✅ BENAR cara 1 — implicit return (tanpa kurung kurawal):
const list2 = menu.map(item => `<li>${item}</li>`);

// ✅ BENAR cara 2 — pakai {} tapi TULIS return-nya:
const list3 = menu.map(item => {
  return `<li>${item}</li>`;
});

// ✅ BENAR cara 3 — return object? Bungkus pakai KURUNG BIASA:
const list4 = menu.map(item => ({ nama: item, tersedia: true }));
//                              ↑ tanpa (), JS ngira { itu awal body function!

// Cara ngenalin bug ini di React nanti:
// "datanya ada di console.log, tapi listnya nggak muncul di layar"
// → 90% kasusnya: lupa return di dalam map. Cek arrow function lo!

JANGAN TERJERAT! Ketiga jebakan ini punya satu kesamaan: nggak ngasih error message. Kode jalan, nggak merah-merah, tapi hasilnya salah diam-diam. Bug paling berbahaya bukan yang bikin app crash — tapi yang senyap. Kenali polanya sekarang, dan lo bakal jadi debugger yang jauh lebih cepat dari rata-rata pemula.

LATIHAN — Saatnya Gerakin Jari

Kerjain dulu di browser console atau playground di atas, BARU buka solusinya. Serius, jangan langsung ngintip — otot ngoding cuma tumbuh kalau dipakai!

Latihan 1: Total Belanja dengan Diskon klik untuk lihat solusi

Soal: dari array [12000, 5000, 30000, 8000], hitung total harga setelah semua item didiskon 10%, tapi hanya item yang harganya di atas 7.000 yang dihitung. Pakai kombinasi filter + map + reduce.

Solusi Latihan 1
const harga = [12000, 5000, 30000, 8000];

const total = harga
  .filter(h => h > 7000)          // [12000, 30000, 8000] — 5000 tersingkir
  .map(h => h * 0.9)              // [10800, 27000, 7200] — diskon 10%
  .reduce((sum, h) => sum + h, 0); // 45000 — dijumlah semua

console.log(total);   // 45000

// Perhatiin CHAINING-nya: filter return array → langsung .map()
// → return array lagi → langsung .reduce(). Rantai kayak gini
// bakal sering banget lo tulis di React. Elegan kan? ✨
Latihan 2: Bongkar Object User API klik untuk lihat solusi

Soal: diberi object const res = { data: { user: { nama: 'Sari', role: 'admin' } }, status: 200 }. (a) Ambil nama dan role pakai destructuring nested dalam SATU baris. (b) Bikin function sapa yang nerima object user via destructuring parameter dan return string template literal "Halo Sari, role kamu: admin".

Solusi Latihan 2
const res = { data: { user: { nama: 'Sari', role: 'admin' } }, status: 200 };

// (a) Destructuring nested — "bongkar dus di dalam dus":
const { data: { user: { nama, role } } } = res;
console.log(nama, role);   // 'Sari' 'admin'
// Catatan: `data` dan `user`-nya sendiri NGGAK jadi variabel,
// mereka cuma "jalur" menuju nama & role.

// (b) Destructuring di parameter + default value biar aman:
const sapa = ({ nama, role = 'member' }) => `Halo ${nama}, role kamu: ${role}`;

sapa(res.data.user);            // 'Halo Sari, role kamu: admin'
sapa({ nama: 'Joko' });         // 'Halo Joko, role kamu: member' ← default jalan!

// Bentuk (b) ini adalah 100% pola komponen React nerima props.
// Selamat, lo baru aja nulis "komponen" pertama lo. 😎
Latihan 3: Fetch + Filter Data API Beneran klik untuk lihat solusi

Soal (level bos!): pakai async/await, fetch daftar produk dari https://fakestoreapi.com/products, lalu tampilkan judul produk yang harganya di bawah $20 saja, dengan format "NamaProduk — $harga". Jangan lupa error handling! (Kerjain di browser console — ini API publik gratis.)

Solusi Latihan 3
async function produkMurah() {
  try {
    const response = await fetch('https://fakestoreapi.com/products');
    if (!response.ok) throw new Error(`HTTP ${response.status}`);

    const produk = await response.json();   // array of object

    const murah = produk
      .filter(p => p.price < 20)                    // saring di bawah $20
      .map(p => `${p.title} — $${p.price}`);        // format string

    murah.forEach(baris => console.log(baris));
    return murah;
  } catch (error) {
    console.log('Gagal:', error.message);
  }
}

produkMurah();

RECAP — BEKAL LO KE BAB 1

Semua ini bakal muncul lagi di React. Bukan "mungkin muncul" — PASTI muncul.

  • ✓Console = playground gratis lo — F12, tab Console, dan console.log() bakal jadi sahabat debugging seumur karir.
  • ✓const dulu, let kalau perlu berubah — dan const ngunci labelnya, bukan isinya. Lima tipe dasar: string, number, boolean, null, undefined.
  • ✓Function = blender, komponen React = function — arrow function itu dialek sehari-hari React; hati-hati implicit return vs body {} yang butuh return.
  • ✓map/filter/find/reduce/some/every — map = jantung render list React. Semuanya bikin nilai baru, nggak ngubah array asli (immutability!).
  • ✓Destructuring & spread — bongkar object/array jadi variabel; destructuring di parameter = PERSIS cara komponen React nerima props. Spread = copy + update tanpa mutasi.
  • ✓Template literal, ternary, &&, || — bahan baku conditional rendering. Awas jebakan angka 0 dengan &&!
  • ✓import/export — default (tanpa kurung kurawal, satu per file) vs named (pakai kurung kurawal, nama harus sama). Kunci baca kode React mana pun.
  • ✓async/await + fetch — Promise = nomor antrian kedai kopi; try/catch buat error; wajib cek response.ok. Trio loading/data/error nunggu lo di React.
  • ✓DOM & events — cara manualnya udah lo lihat, dan sekarang lo ngerti PERSIS apa yang React otomatisasi. React bukan sihir, dia cuma JavaScript yang terorganisir.
