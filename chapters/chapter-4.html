1. 💰 useMemo — Ngumpetin Hasil Kalkulasi

Pernah ngalamin app kamu lag pas ngetik di search bar? Padahal cuma filter data doang. Nah, itu biasanya karena kamu nge-rekalkulasi sesuatu yang sebenarnya nggak perlu di setiap render.

Bayangin gini: kamu lagi masak nasi. Tiap kali kamu buka rice cooker buat cek, nasi-nya belum matang — tapi buka-tutup terus bikin steam keluar dan proses masaknya makin lama. useMemo itu kayak timer di rice cooker-nya. Biar kamu nggak usah cek berkali-kali, dia cuma kasih tahu kalau emang udah berubah aja. 😄

import { useState, useMemo } from 'react';

function ProductList({ products }) {
  const [search, setSearch] = useState('');
  const [darkMode, setDarkMode] = useState(false);

  // ❌ TANPA useMemo — filter jalan TIAP KALI render, padahal
  //    products sama search-nya nggak berubah saat toggle darkMode
  const filteredProducts = products.filter(p =>
    p.name.toLowerCase().includes(search.toLowerCase())
  );

  // ✅ DENGAN useMemo — cuma re-kalkulasi kalau search atau products berubah
  const filteredProductsOptimized = useMemo(() => {
    console.log('Filtering... (cuma jalan kalau perlu)');
    return products.filter(p =>
      p.name.toLowerCase().includes(search.toLowerCase())
    );
  }, [products, search]); // ← dependencies-nya

  // darkMode berubah? filteredProductsOptimized NGGAK di-rekalkulasi.
  // Mantap kan? 😎
}

💡 Pro tip: Jangan pake useMemo buat semuanya ya! Kalau kalkulasinya cuma simple kayak arr.length + 1, malah makin lambat karena overhead useMemo-nya sendiri. Pakai kalau kalkulasinya berat — filter data besar, sorting, atau mapping kompleks.

⚠️ Peringatan: useMemo itu bukan semantik guarantee — React bisa aja "lupa" hasil memoized-nya buat free up memory. Jangan depends on useMemo buat correctness, cuma buat performance aja.

2. 📦 useCallback — Nggak Perlu Bikin Function Baru Terus

Okay, ini temennya useMemo tapi khusus buat function. Kamu tau nggak, setiap kali komponen kamu re-render, semua function yang didefinisikan di dalamnya itu di-buat ulang? Ya, walaupun isinya sama persis.

Analoginya gini: kamu punya resep masakan yang sama, tapi tiap hari kamu nulis ulang di kertas baru. Padahal kan bisa di-fotocopy dan dipakai berulang kali. Itu yang useCallback lakukan — dia nge-"fotocopy" function kamu biar reference-nya tetap sama antar render. 📋

import { useState, useCallback, memo } from 'react';

// Komponen anak yang di-memo biar nggak re-render kalau props-nya sama
const HeavyButton = memo(function HeavyButton({ onClick, label }) {
  console.log(`Rendering: ${label}`);
  return (
    <button onClick={onClick}>{label}</button>
  );
});

function ParentComponent() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState('');

  // ❌ TANPA useCallback — function baru TIAP RENDER
  // HeavyButton akan re-render terus karena onClick reference berubah
  const handleClick = () => {
    setCount(c => c + 1);
  };

  // ✅ DENGAN useCallback — reference tetap sama selama deps nggak berubah
  const handleClickMemoized = useCallback(() => {
    setCount(c => c + 1);
  }, []); // ← kosong karena setCount itu stabil (nggak pernah berubah)

  // Sekarang HeavyButton nggak re-render pas kamu ngetik di input!
  return (
    <div>
      <input value={text} onChange={e => setText(e.target.value)} />
      <HeavyButton onClick={handleClickMemoized} label="Count" />
      <p>Count: {count}</p>
    </div>
  );
}

💡 Kapan pakai useCallback? Sesungguhnya, kebanyakan kasus kamu nggak perlu ini. Hanya pakai kalau kamu ngirim function sebagai props ke komponen yang udah di-React.memo, atau function itu jadi dependency di useEffect/useMemo lain. Jangan over-engineer!

3. 🎯 useRef — Memory Rahasia yang Nggak Bikin Re-render

Ini hooks yang paling sering dipahami salah. Banyak orang kepikiran useRef cuma buat akses DOM element (kayak document.getElementById). Tapi sebenarnya dia jauh lebih powerful dari itu!

Bayangin useRef itu kayak kotak penyimpanan rahasia 📦. Isinya bisa kamu ubah kapan aja, tapi bedanya sama useState — nggak akan pernah bikin komponen re-render. Nilainya tetap ada dari render ke render, persis kayak variabel global tapi aman karena scoped ke komponen.

import { useState, useRef, useEffect } from 'react';

// 📌 KEGUNAAN 1: Fokus ke input secara otomatis
function AutoFocusInput() {
  const inputRef = useRef(null);

  useEffect(() => {
    inputRef.current?.focus(); // Boom! Langsung fokus
  }, []);

  return <input ref={inputRef} placeholder="Aku auto-focus!" />;
}

// 📌 KEGUNAAN 2: Ngitung berapa kali render (tanpa bikin re-render!)
function RenderCounter() {
  const renderCount = useRef(0);
  renderCount.current += 1; // Nggak trigger re-render sama sekali

  return <p>Aku udah render {renderCount.current} kali</p>;
}

// 📌 KEGUNAAN 3: Simpan previous value
function PreviousValueTracker() {
  const [value, setValue] = useState('');
  const prevValue = useRef('');

  useEffect(() => {
    prevValue.current = value; // Simpan sebelum re-render
  }, [value]);

  return (
    <div>
      <input onChange={e => setValue(e.target.value)} />
      <p>Sekarang: {value}</p>
      <p>Sebelumnya: {prevValue.current}</p>
    </div>
  );
}

// 📌 KEGUNAAN 4: Timer / interval ID (biar bisa di-clear)
function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  const intervalRef = useRef(null);

  const start = () => {
    intervalRef.current = setInterval(() => {
      setSeconds(s => s + 1);
    }, 1000);
  };

  const stop = () => clearInterval(intervalRef.current);

  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}

⚠️ Jangan lupa: .current adalah mutable — kamu bisa langsung assign nilainya. Tapi jangan baca/tulis ref.current pas proses rendering ya (di body function), karena nilainya belum tentu final. Pakai di useEffect atau event handler aja.

4. 🌍 useContext — Tamatnya Prop Drilling

Udah pernah ngalamin yang namanya prop drilling belum? Jadi gini — kamu punya data theme (dark/light), terus kamu kirim lewat props dari App → Header → Navbar → ThemeToggle. Nah, Header dan Navbar nggak butuh data theme sama sekali, tapi mereka jadi "korban" harus nerima dan meneruskan props-nya. Capek kan? 😩

useContext itu solusinya. Bayangin dia kayak pengumuman sekolah lewat speaker 📢. Daripada info harus disampaikan dari satu orang ke orang ke orang (prop drilling), speaker bisa langsung nyampai ke semua kelas yang butuh. Siapapun yang punya speaker (Context), bisa denger pengumumannya.

import { createContext, useContext, useState } from 'react';

// 1. Bikin Context dulu
const ThemeContext = createContext();

// 2. Bikin Provider komponen (yang ngasih data)
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('dark');

  // Wrap semua value yang mau di-share dalam satu object
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// 3. Custom hook biar lebih clean (opsional tapi recommended ✨)
function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error('useTheme harus dipakai di dalam ThemeProvider!');
  }
  return context;
}

// 4. Pakai di komponen manapun — tanpa props!
function ThemeToggle() {
  const { theme, setTheme } = useTheme();
  // Tiba-tiba, komponen deep di hierarki bisa akses langsung
  // Nggak perlu props dari atas lagi. Oh ternyata gitu! 🤯

  return (
    <button onClick={() => setTheme(t => t === 'dark' ? 'light' : 'dark')}>
      Sekarang: {theme}
    </button>
  );
}

// 5. Wrap App dengan Provider
function App() {
  return (
    <ThemeProvider>
      <Header />
      <Main />
      <Footer />
    </ThemeProvider>
  );
}

💡 Best practice: Selalu bikin custom hook untuk context kamu (kayak useTheme() di atas). Kenapa? (1) Nggak perlu import Context-nya di tiap komponen, (2) Bisa kasih error message kalau lupa wrap Provider, (3) Lebih clean dan readable.

⚠️ Jangan kebablasan: Context itu bukan replacement buat state management library (Redux, Zustand, dll). Setiap kali context value berubah, SEMUA komponen yang consume itu context bakal re-render. Buat data yang sering berubah (kayak input teks), lebih baik pake state management library atau local state aja.

5. 🎛️ useReducer — Kalau useState Udah Nggak Kuat

Pernah punya komponen yang state-nya kayak spaghetti? Lima useState bertumpuk, dan logic-nya saling ketergantungan. Pas kamu update satu state, harus update state lainnya juga. Kodenya jadi rame dan susah di-trace. 😵‍💫

useReducer itu kayak dashboard kontrol di pesawat ✈️. Daripada ada banyak tombol independen yang masing-masing ngatur bagian berbeda, semua kontrol masuk ke satu panel sentral. Pilot (reducer) yang ngatur semua — tinggal terima perintah (action) dan tentukan hasilnya.

import { useReducer } from 'react';

// Reducer = "otak" yang nentuin state baru dari action
function formReducer(state, action) {
  switch (action.type) {
    case 'SET_FIELD':
      return { ...state, [action.field]: action.value };
    case 'RESET':
      return action.initialState;
    case 'SUBMIT':
      return { ...state, isSubmitting: true };
    case 'SUBMIT_SUCCESS':
      return { ...state, isSubmitting: false, success: true };
    case 'SUBMIT_ERROR':
      return { ...state, isSubmitting: false, error: action.error };
    default:
      return state;
  }
}

function ComplexForm() {
  const initialState = {
    name: '',
    email: '',
    password: '',
    isSubmitting: false,
    success: false,
    error: null,
  };

  const [state, dispatch] = useReducer(formReducer, initialState);

  // Sekarang semua logic terpusat di satu reducer
  // Ngirim data? dispatch. Reset form? dispatch. Error handling? dispatch.
  // Clean banget! 🧹

  return (
    <form onSubmit={(e) => {
      e.preventDefault();
      dispatch({ type: 'SUBMIT' });
    }}>
      <input
        value={state.name}
        onChange={e => dispatch({ type: 'SET_FIELD', field: 'name', value: e.target.value })}
      />
      <input
        value={state.email}
        onChange={e => dispatch({ type: 'SET_FIELD', field: 'email', value: e.target.value })}
      />
      <button disabled={state.isSubmitting}>
        {state.isSubmitting ? 'Loading...' : 'Submit'}
      </button>
      <button type="button" onClick={() => dispatch({ type: 'RESET', initialState })}>
        Reset
      </button>
    </form>
  );
}

💡 Kapan pilih useReducer vs useState? Kalau state kamu saling terkait dan update-nya kompleks — pake useReducer. Kalau cuma simple kayak toggle atau counter — useState udah cukup. Jangan force gunain useReducer buat hal yang simple, nanti malah over-engineered. Intinya: pake yang paling mudah dipahami untuk kasusmu.

6. 🧩 Custom Hooks — Superpower React yang Sebenarnya

Okay, ini bagian yang paling "oh ternyata gitu!" dari semua hooks. Custom hooks itu bukan feature yang separate — dia lebih kayak cara berpikir. Semua hooks yang udah kita bahas? Itu bisa kamu gabung jadi satu "super hook" yang reusable di mana aja.

Bayangin kamu punya 3 komponen yang butuh logic fetch data + loading + error. Tanpa custom hook, kamu nulis logic yang sama 3 kali. Dengan custom hook? Tulis sekali, pakai sepuasnya. Itu kayak punya resep masakan favorit — sekali tulis di buku resep, tinggal buka-buka lagi kalau mau masak. 📖🍳

import { useState, useEffect, useCallback } from 'react';

// 🔥 Custom hook: useFetch — buat fetch API yang clean
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  const fetchData = useCallback(async () => {
    try {
      setLoading(true);
      setError(null);
      const res = await fetch(url);
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      const json = await res.json();
      setData(json);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }, [url]);

  useEffect(() => { fetchData(); }, [fetchData]);

  return { data, loading, error, refetch: fetchData };
}

// 🔥 Custom hook: useLocalStorage — persist state ke localStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const saved = localStorage.getItem(key);
      return saved ? JSON.parse(saved) : initialValue;
    } catch {
      return initialValue;
    }
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

// 🔥 Custom hook: useClickOutside — detect klik di luar element
function useClickOutside(ref, handler) {
  useEffect(() => {
    const listener = (event) => {
      if (!ref.current || ref.current.contains(event.target)) return;
      handler(event);
    };
    document.addEventListener('mousedown', listener);
    document.addEventListener('touchstart', listener);
    return () => {
      document.removeEventListener('mousedown', listener);
      document.removeEventListener('touchstart', listener);
    };
  }, [ref, handler]);
}

// Pakai semuanya! 🎉
function UserProfile() {
  const { data: user, loading, error, refetch } = useFetch('/api/user');
  const [theme, setTheme] = useLocalStorage('theme', 'dark');
  const dropdownRef = useRef(null);

  useClickOutside(dropdownRef, () => {
    setIsDropdownOpen(false);
  });

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return <div>Hello, {user.name}!</div>;
}

💡 Naming convention: Custom hook WAJIB namanya diawali use — misalnya useFetch, useLocalStorage, useWindowSize. Ini bukan sekadar konvensi, tapi ini yang bikin React bisa detect dan enforce Rules of Hooks.

7. 📏 Rules of Hooks — Dua Aturan Emas

React punya hanya 2 aturan soal hooks. Dua doang! Tapi kalau dilanggar, siap-siap chaos. Bug aneh yang nggak masuk akal, state yang ketuker, atau malah app crash. Yuk kita bahas biar kamu nggak pernah melanggarnya. 🙅‍♂️

// ❌ ATURAN 1: JANGAN panggil hooks di dalam condition, loop, atau nested function
function BadComponent({ showCounter }) {
  if (showCounter) {
    const [count, setCount] = useState(0); // 💀 KALAU showCounter = false,
  }                                      //    hook ini hilang! React bingung.
  const [theme, setTheme] = useState('dark'); // Kadang jadi hook pertama,
}                                         // kadang jadi hook kedua. CHAOS!

// ✅ BENAR: Semua hooks DI TIER paling atas komponen
function GoodComponent({ showCounter }) {
  const [count, setCount] = useState(0);      // Selalu hook ke-1
  const [theme, setTheme] = useState('dark'); // Selalu hook ke-2

  if (!showCounter) return null; // Condition-nya boleh di bawah
}


// ❌ ATURAN 2: JANGAN panggil hooks dari regular JavaScript function
function badHelper() {
  const [value, setValue] = useState(''); // 💀 Ini bukan React component!
}

// ✅ BENAR: Hooks cuma boleh di:
// (1) React function components
// (2) Custom hooks (function yang namanya diawali "use")
function useMyHook() {  // ✅ Nama diawali "use" → custom hook
  const [value, setValue] = useState('');
  return [value, setValue];
}

⚠️ Kenapa sih aturan ini penting? React mengandalkan urutan pemanggilan hooks buat tau state mana yang mana. Hook ke-1 = state pertama, hook ke-2 = state kedua, dst. Kalau urutannya bisa berubah-ubah (karena condition), React nggak bisa matching state dengan benar. Makanya, hooks harus selalu dipanggil dalam urutan yang sama di setiap render.

💡 Auto-enforcement: Pasang eslint-plugin-react-hooks di project kamu. Plugin ini otomatis nge-catch kalau kamu melanggar Rules of Hooks. Nggak perlu hafal aturannya — biar ESLint yang jaga. 🤖

8. 🏎️ useTransition & useDeferredValue — UI Tetap Smooth

Pernah bikin filter list yang pas kamu ngetik, seluruh halaman jadi lag dan nge-freeze? Itu karena filter 10.000 item itu jalan di main thread yang sama dengan rendering UI. Hasilnya? Typing-nya jadi choppy dan user kesel. 😤

useTransition dan useDeferredValue itu kayak bilang ke React: "Eh, update yang ini nggak terlalu penting, kerjain nanti aja yang penting UI tetap responsif." Mirip kayak ngasih priority lane di jalan tol — kendaraan VIP (user input) duluan, yang berat-berat (filter data) belakangan. 🛣️

import { useState, useTransition, useDeferredValue } from 'react';

// useTransition: manual marking update sebagai "non-urgent"
function SearchWithTransition() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    const value = e.target.value;
    setQuery(value); // ⚡ URGENT — input langsung update, UI tetap smooth
    startTransition(() => {
      setResults(filterHeavyList(value)); // 🐢 NON-URGENT — boleh ditunda
    });
  };

  return (
    <div>
      <input value={query} onChange={handleChange} />
      {isPending && <p className="opacity-50">Filtering...</p>}
      <ul>{results.map(item => <li key={item.id}>{item.name}</li>)}</ul>
    </div>
  );
}

// useDeferredValue: auto-defer sebuah value
function SearchWithDeferred({ items }) {
  const [filter, setFilter] = useState('');
  const deferredFilter = useDeferredValue(filter); // ⏳ Otomatis ditunda

  const filtered = items.filter(item =>
    item.name.toLowerCase().includes(deferredFilter.toLowerCase())
  );

  return (
    <div>
      <input value={filter} onChange={e => setFilter(e.target.value)} />
      <ul>{filtered.map(item => <li key={item.id}>{item.name}</li>)}</ul>
    </div>
  );
}

💡 useTransition vs useDeferredValue? Pake useTransition kalau kamu sendiri yang trigger state update (kayak di event handler). Pake useDeferredValue kalau valuenya udah ada dan kamu cuma mau "slow down" penggunaannya. Two different tools, one same goal: keep UI responsive! 🎯

📝 Ringkasan Chapter 4

Wah, kamu udah sampai akhir! 🎉 Ini ringkasan cepetnya:

💰
useMemo

Cache hasil kalkulasi berat. Biar nggak re-kalkulasi di tiap render.

📦
useCallback

Cache function reference. Biar child component yang di-memo nggak re-render.

🎯
useRef

Kotak penyimpanan rahasia. Nggak trigger re-render, bisa buat akses DOM atau simpan value.

🌍
useContext

Stop prop drilling. Share data ke seluruh komponen tree tanpa kirim props berkali-kali.

🎛️
useReducer

State management buat logic yang kompleks. Semua update logic terpusat di satu reducer function.

🧩
Custom Hooks

Gabung hooks jadi super hook yang reusable. Ini nyata-nyata superpower React!

🏎️
useTransition & useDeferredValue

Biar UI tetap smooth pas ngerjain update berat. Non-urgent updates ditunda.

Sekarang kamu udah punya arsenal hooks yang lengkap! Dari yang basic sampe advanced. Ingat ya — jangan over-engineer. Pakai hooks yang kamu butuhkan, bukan yang kelihatan keren. Happy coding! 💚
