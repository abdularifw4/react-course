USEMEMO — CACHE HASIL KALKULASI BIAR NGGAK MAKSA RE-RENDER 🧠

Optimasi · 10 menit baca

Pernah ngalamin app kamu lag pas ngetik di search bar? Padahal cuma filter data doang. Nah, itu biasanya karena kamu nge-rekalkulasi sesuatu yang sebenarnya nggak perlu di setiap render. Setiap kali kamu ketik satu huruf, React re-render komponen, dan semua kalkulasi diulang dari nol — padahal data yang difilter mungkin sama aja.

Di dunia nyata, ini problem yang sering banget. Misalnya kamu punya e-commerce dengan 5,000 produk. User ngetik "se" di search bar — React filter 5,000 item, re-render list, user ngetik "se" lagi jadi "sep", React filter lagi 5,000 item... terus berulang. Dalam satu detik, bisa jadi 5-10 filter yang nggak perlu. CPU usage naik, battery berkurang, user kesel. 😤

Bayangin gini: kamu lagi masak nasi. Tiap kali kamu buka rice cooker buat cek, nasi-nya belum matang — tapi buka-tutup terus bikin steam keluar dan proses masaknya makin lama. useMemo itu kayak timer di rice cooker-nya. Biar kamu nggak usah cek berkali-kali, dia cuma kasih tahu kalau emang udah berubah aja. Hasil kalkulasi disimpen di memory, dan baru dihitung ulang kalau dependency-nya berubah.

Sintaksnya: useMemo(() => kalkulasiBerat(), [dependencies]). Parameter pertama fungsi yang return nilai, parameter kedua array dependency. Selama dependency nggak berubah, React kasih hasil cache-nya langsung. Simplenya: "hitung sekali, simpen, pakai berkali-kali sampai ada yang berubah."

Tapi hati-hati — useMemo bukan magic bullet. Dia punya overhead sendiri: React harus nyimpen hasilnya di memory dan ngecek dependencies setiap render. Jadi kalau kalkulasinya murah (kayak arr.length + 1 atau name.split(' ')[0]), malah makin lambat! Oh ternyata gitu! 🤯

React juga nggak menjamin hasil memoized-nya bakal selalu tersimpan. Di situasi memory pressure, React bisa aja "lupa" hasilnya buat free up memory. Jadi jangan pernah depends on useMemo buat correctness logic — cuma buat performance optimization. Kalau kamu hilangkan useMemo, app kamu harusnya tetap jalan dengan benar, cuma mungkin lebih lambat.

Best practice: pakai useMemo di situasi spesifik aja — (1) filter/sort data besar (1000+ item), (2) kalkulasi yang jadi dependency useEffect lain, (3) bikin object/array yang jadi props buat komponen yang di-memo. Di luar itu? Cukup tulis langsung, biar kodenya lebih simple dan readable. 💡

useMemo — Kapan Perlu & Kapan Tidak
// ✅ Cocok: kalkulasi berat yang sering dipanggil
function SearchResults({ query, items }) {
  const results = useMemo(() => {
    console.log('Filtering...'); // cuma jalan kalau query/items berubah
    return items.filter(item =>
      item.name.toLowerCase().includes(query.toLowerCase())
    );
  }, [query, items]);
  return results.map(r => <li key={r.id}>{r.name}</li>);
}

// ❌ JANGAN gunakan useMemo untuk kalkulasi murah
const firstName = useMemo(() => user.name.split(' ')[0], [user.name]);
// Overkill! Cukup: const firstName = user.name.split(' ')[0];
Perbandingan: Dengan vs Tanpa useMemo
import { useState, useMemo } from 'react';

function ProductList({ products }) {
  const [search, setSearch] = useState('');
  const [darkMode, setDarkMode] = useState(false);

  // ❌ TANPA useMemo — filter jalan TIAP KALI render
  const filteredProducts = products.filter(p =>
    p.name.toLowerCase().includes(search.toLowerCase())
  );

  // ✅ DENGAN useMemo — cuma re-kalkulasi kalau search/products berubah
  const filteredProductsOptimized = useMemo(() => {
    return products.filter(p =>
      p.name.toLowerCase().includes(search.toLowerCase())
    );
  }, [products, search]);

}

💡 Pro tip: Jangan pake useMemo buat semuanya! Kalau kalkulasinya simple kayak arr.length + 1, malah makin lambat karena overhead-nya sendiri. Pakai kalau kalkulasinya berat — filter data besar (1000+ item), sorting kompleks, atau bikin object yang jadi dependency useEffect lain.

⚠️ Peringatan: useMemo itu bukan semantik guarantee — React bisa aja "lupa" hasil memoized-nya buat free up memory. Jangan depends on useMemo buat correctness logic, cuma buat performance aja.

02

USECALLBACK — STABILKAN FUNCTION REFERENCE 📦

Optimasi · 9 menit baca

Ini temennya useMemo tapi khusus buat function. Setiap kali komponen re-render, semua function yang didefinisikan di dalamnya itu di-buat ulang dari nol — walaupun isinya sama persis. JavaScript bikin function baru yang reference-nya beda setiap render.

Analoginya: kamu punya resep masakan yang sama, tapi tiap hari nulis ulang di kertas baru. Padahal bisa di-fotocopy dan dipakai berulang kali. Itu yang useCallback lakukan — dia nge-"fotocopy" function kamu biar reference-nya tetap sama antar render.

Formatnya: useCallback(fn, [dependencies]). Fakta seru: secara internal, useCallback(fn, deps) itu ekivalen dengan useMemo(() => fn, deps). Sama konsep, beda target — useMemo cache value, useCallback cache function reference.

Nah pertanyaannya: kapan sih ini beneran bantu? Jawabannya: hanya pas kamu ngirim function ke child component yang udah di-wrap React.memo(). Tanpa memo, child bakal re-render anyway walau function-nya sama. Dengan memo + stable reference, child bisa skip re-render. Duo yang nggak bisa dipisah! 🔗

Use case lain: pas function kamu jadi dependency di useEffect atau useMemo lain. Kalau function reference berubah setiap render, effect kamu bakal jalan ulang terus — padahal sebenarnya nggak perlu. useCallback ngunci reference-nya biar stabil. Oh ternyata gitu! 🤯

Tapi ingat — useCallback juga punya overhead. Dia harus ngecek dependencies setiap render, dan nyimpen function reference di memory. Jangan over-engineer! Pakai hanya di situasi yang emang butuh. Kalau nggak yakin, measurable dulu pake React DevTools Profiler — liat apakah child component yang di-memo itu beneran skip re-render. Data nggak bohong. 📊

useCallback + React.memo — Combo Optimasi
// Problem: Parent re-render bikin Child ikut re-render
function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [filter, setFilter] = useState('all');

  // useCallback mencegah pembuatan fungsi baru setiap render
  const addTodo = useCallback((text) => {
    setTodos(prev => [...prev, { id: Date.now(), text, done: false }]);
  }, []);

  return (
    <>
      <FilterBar filter={filter} setFilter={setFilter} />
      <TodoInput onAdd={addTodo} />
    </>
  );
}

const TodoInput = React.memo(({ onAdd }) => {
  console.log('TodoInput render'); // cuma jalan sekali!
  return <input />;
});
useCallback — Dengan vs Tanpa
import { useState, useCallback, memo } from 'react';

const HeavyButton = memo(function HeavyButton({ onClick, label }) {
  console.log(`Rendering: ${label}`);
  return <button onClick={onClick}>{label}</button>;
});

function ParentComponent() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState('');

  // ❌ TANPA useCallback — function baru TIAP RENDER
  const handleClick = () => { setCount(c => c + 1); };

  // ✅ DENGAN useCallback — reference tetap sama
  const handleClickMemoized = useCallback(() => {
    setCount(c => c + 1);
  }, []);

  return (
    <div>
      <input value={text} onChange={e => setText(e.target.value)} />
      <HeavyButton onClick={handleClickMemoized} label="Count" />
      <p>Count: {count}</p>
    </div>
  );
}

💡 Kapan pakai useCallback? Hanya pakai kalau: (1) ngirim function ke komponen yang udah di-React.memo(), (2) function jadi dependency di useEffect/useMemo lain. Jangan over-engineer!

⚠️ useCallback bukan magic bullet: Dia juga punya overhead — harus cek dependencies setiap render. Kalau child-nya nggak di-memo, useCallback nggak bantu apa-apa.

03

USEREF — MEMORY RAHASIA YANG NGGAK BIKIN RE-RENDER 📦

Fundamental · 10 menit baca

Ini hooks yang paling sering dipahami salah. Banyak orang kepikiran useRef cuma buat akses DOM element. Tapi sebenarnya dia jauh lebih powerful! useRef itu kayak kotak penyimpanan rahasia — isinya bisa kamu ubah kapan aja, tapi nggak akan pernah bikin komponen re-render.

Nilainya tetap ada dari render ke render. Dia punya satu properti: .current. Kamu bisa baca dan tulis langsung tanpa setter function — ini mutabel, beda sama state yang harus pake updater.

Empat use case utama: (1) Akses DOM — fokus input, scroll ke element. (2) Simpan mutable value — timer ID, render count. (3) Track previous value. (4) Interval timer yang bisa di-clear. Mari liat! 🔥

Banyak orang bingung: kapan pakai useRef vs useState? Rule of thumb-nya gini: kalau perubahannya harus keliatan di UI, pakai useState. Kalau perubahannya cuma "di belakang layar" (DOM manipulation, timer ID, flag internal), pakai useRef. useRef itu backend, useState itu frontend. Jangan kebalik! 🔀

Hal penting lain: ref.current itu bisa langsung kamu baca dan tulis — nggak perlu setter function kayak useState. Tapi ini juga berarti React nggak track perubahannya. Jadi kalau kamu update ref.current, komponen nggak bakal re-render. Itu sengaja — itulah gunanya! 💡

useRef — Empat Kegunaan Utama
import { useState, useRef, useEffect } from 'react';

// 📌 1: Auto-focus input
function AutoFocusInput() {
  const inputRef = useRef(null);
  useEffect(() => { inputRef.current?.focus(); }, []);
  return <input ref={inputRef} placeholder="Aku auto-focus!" />;
}

// 📌 2: Render counter (tanpa bikin re-render!)
function RenderCounter() {
  const renderCount = useRef(0);
  renderCount.current += 1;
  return <p>Rendered {renderCount.current} times</p>;
}

// 📌 3: Track previous value
function PreviousValueTracker() {
  const [value, setValue] = useState('');
  const prevValue = useRef('');
  useEffect(() => { prevValue.current = value; }, [value]);
  return (
    <div>
      <input onChange={e => setValue(e.target.value)} />
      <p>Now: {value} | Before: {prevValue.current}</p>
    </div>
  );
}

// 📌 4: Timer / interval ID
function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  const intervalRef = useRef(null);
  const start = () => {
    intervalRef.current = setInterval(() => setSeconds(s => s + 1), 1000);
  };
  const stop = () => clearInterval(intervalRef.current);
  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}
Custom Hook: usePrevious
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => { ref.current = value; });
  return ref.current;
}

function SearchBar() {
  const [query, setQuery] = useState('');
  const prevQuery = usePrevious(query);

  useEffect(() => {
    if (prevQuery && query !== prevQuery) {
      console.log(`Query: "${prevQuery}" → "${query}"`);
    }
  }, [query, prevQuery]);

  return <input value={query} onChange={e => setQuery(e.target.value)} />;
}

⚠️ Jangan lupa: .current mutabel — bisa langsung assign. Tapi jangan baca/tulis pas proses rendering (di body function). Pakai di useEffect atau event handler aja.

Satu gotcha yang sering bikin pusing: pas kamu render list dengan useRef, ref-nya bisa overlap kalau list-nya bisa berubah. Misal kamu punya list 5 item, terus filter jadi 3 — ref index 3 dan 4 masih nge-hold element yang udah nggak ada. Solusinya? Pakai callback ref pattern atau simpan refs di Map/Object. Ini teknik advanced, tapi penting buat ditahu supaya nggak kena bug misterius. 🐛

Pro tip: useRef juga bisa dipake buat bikin custom "instance variable" buat component kamu. Misalnya kamu mau nyimpen scroll position, WebSocket connection, atau ID animation frame — semua ini nggak perlu trigger re-render. useRef jadi wadahnya. Ini pola yang dipake banyak library populer kayak React Query dan SWR. 🔥

USECONTEXT — TAMATNYA PROP DRILLING 🌍

State Global · 10 menit baca

Udah pernah ngalamin prop drilling? Kamu punya data theme (dark/light), kirim lewat props dari App → Header → Navbar → ThemeToggle. Header dan Navbar nggak butuh data theme, tapi mereka jadi "korban" harus nerima dan meneruskan props. Capek kan? 😩

useContext itu solusinya — kayak pengumuman sekolah lewat speaker 📢. Daripada info disampaikan dari satu orang ke orang (prop drilling), speaker bisa langsung nyampai ke semua kelas yang butuh.

Pola recommended: bikin Context → bikin Provider → bikin custom hook. Dengan custom hook, kamu bisa kasih error kalau lupa wrap Provider — nggak akan ada bug misterius.

Satu hal yang sering lupa: setiap kali context value berubah, SEMUA komponen yang consume context itu bakal re-render. Bukan cuma yang pakai value yang berubah — SEMUA. Ini bedanya sama Redux yang bisa selective subscribe ke slice tertentu. Jadi kalau context kamu punya banyak data (theme + user + settings + cart), pertimbangkan buat pisah context per concern. Theme berubah? Komponen yang cuma butuh user data nggak ikut re-render. Oh ternyata gitu! 🤯

Best practice: selalu bikin custom hook untuk setiap context (kayak useTheme(), useAuth()). Tiga alasan: (1) nggak perlu import Context di tiap komponen, (2) bisa throw error kalau lupa wrap Provider, (3) bisa tambah logic tambahan di hook (kayak default values, derived state). Ini pola yang dipake hampir semua production React app.

useContext — Full Pattern: Theme
import { createContext, useContext, useState } from 'react';

// 1. Bikin Context
const ThemeContext = createContext();

// 2. Bikin Provider
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('dark');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// 3. Custom hook — recommended! ✨
function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error('useTheme harus dipakai di dalam ThemeProvider!');
  }
  return context;
}

// 4. Pakai di mana saja — tanpa props!
function ThemeToggle() {
  const { theme, setTheme } = useTheme();
  return (
    <button onClick={() => setTheme(t => t === 'dark' ? 'light' : 'dark')}>
      Sekarang: {theme}
    </button>
  );
}

// 5. Wrap App
function App() {
  return (
    <ThemeProvider>
      <Header />
      <Main />
      <Footer />
    </ThemeProvider>
  );
}
useContext — Multiple Contexts & Auth
// Pisah context untuk performance — kalau theme berubah,
// komponen yang cuma butuh user data NGGAK ikut re-render
const UserContext = createContext();
const ThemeContext = createContext();

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <UserContext.Provider value={{ name: 'Arif', role: 'admin' }}>
        <Dashboard />
      </UserContext.Provider>
    </ThemeContext.Provider>
  );
}

function Dashboard() {
  const { name, role } = useContext(UserContext);
  const theme = useContext(ThemeContext);
  return (
    <div className={theme}>
      <h1>Welcome, {name} ({role})</h1>
    </div>
  );
}

💡 Best practice: Selalu bikin custom hook untuk context kamu (kayak useTheme()). Kenapa? (1) Nggak perlu import Context-nya di tiap komponen, (2) Bisa kasih error kalau lupa wrap Provider, (3) Lebih clean dan readable.

⚠️ Jangan kebablasan: Context bukan replacement buat state management library (Redux, Zustand). Setiap kali context value berubah, SEMUA komponen yang consume itu context bakal re-render. Buat data yang sering berubah (kayak input teks), lebih baik pake state management library atau local state aja.

05

USEREDUCER — KALAU USESTATE UDAH NGGAK KUAT 🎛️

State Management · 10 menit baca

Pernah punya komponen yang state-nya kayak spaghetti? Lima useState bertumpuk, logic-nya saling ketergantungan. Pas update satu state, harus update state lainnya juga. Kodenya jadi rame dan susah di-trace. 😵‍💫

useReducer itu kayak dashboard kontrol di pesawat ✈️. Daripada ada banyak tombol independen, semua kontrol masuk ke satu panel sentral. Pilot (reducer) ngatur semua — tinggal terima perintah (action) dan tentukan hasilnya.

Sintaksnya: const [state, dispatch] = useReducer(reducer, initialState). Reducer adalah fungsi murni yang nerima state sekarang + action, lalu return state baru. Dispatch adalah fungsi buat ngirim action ke reducer. Simple, predictable, dan easy to test.

Kelebihan useReducer dibanding useState: (1) semua update logic terpusat di satu tempat, nggak tersebar di event handler mana-mana. (2) reducer itu pure function — bisa di-test tanpa render komponen. (3) memudahkan implementasi fitur kayak undo/redo (cukup simpan history of actions). (4) bikin state transition predictable — selalu dari action tertentu ke state tertentu.

Tapi jangan kebablasan juga — buat simple toggle atau counter, useState jauh lebih readable. useReducer cocoknya buat state yang punya banyak tipe update, atau pas state-nya kompleks dan saling tergantung. Intinya: pake yang paling gampang dipahami untuk kasusmu. Jangan force complexity! 🎯

useReducer — Complex State Logic
const initialState = { count: 0, loading: false };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + 1 };
    case 'decrement':
      return { ...state, count: state.count - 1 };
    case 'setLoading':
      return { ...state, loading: action.payload };
    case 'reset':
      return initialState;
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);
  return (
    <div>
      <p>Count: {state.count}</p>
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
      <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
    </div>
  );
}
useReducer — Form dengan Banyak State
import { useReducer } from 'react';

function formReducer(state, action) {
  switch (action.type) {
    case 'SET_FIELD':
      return { ...state, [action.field]: action.value };
    case 'RESET':
      return action.initialState;
    case 'SUBMIT':
      return { ...state, isSubmitting: true };
    case 'SUBMIT_SUCCESS':
      return { ...state, isSubmitting: false, success: true };
    case 'SUBMIT_ERROR':
      return { ...state, isSubmitting: false, error: action.error };
    default:
      return state;
  }
}

function ComplexForm() {
  const initialState = {
    name: '', email: '', password: '',
    isSubmitting: false, success: false, error: null,
  };
  const [state, dispatch] = useReducer(formReducer, initialState);

  return (
    <form onSubmit={(e) => { e.preventDefault(); dispatch({ type: 'SUBMIT' }); }}>
      <input value={state.name}
        onChange={e => dispatch({ type: 'SET_FIELD', field: 'name', value: e.target.value })} />
      <input value={state.email}
        onChange={e => dispatch({ type: 'SET_FIELD', field: 'email', value: e.target.value })} />
      <button disabled={state.isSubmitting}>
        {state.isSubmitting ? 'Loading...' : 'Submit'}
      </button>
    </form>
  );
}

💡 Kapan pilih useReducer vs useState? Kalau state kamu saling terkait dan update-nya kompleks — pake useReducer. Kalau cuma simple kayak toggle atau counter — useState udah cukup. Intinya: pake yang paling mudah dipahami untuk kasusmu.

Satu pattern advanced yang wajib kamu tahu: useReducer bisa dispatch dari context. Ini pattern yang dipake Redux sendiri! Kamu bisa bikin Provider yang ngasih state + dispatch ke semua child, tanpa harus props drilling. Ini cocok buat app-level state management tanpa library external. Power duo: useReducer + useContext = lightweight Redux! 💪

Contoh nyata: form dengan 10 field, loading state, validation errors, dan conditional rendering. Dengan useState kamu butuh 10+ setter functions yang saling terkait. Dengan useReducer? Satu reducer function, semua logic terpusat, dan state transition-nya predictable. Code-nya lebih clean, lebih mudah di-debug, dan lebih gampang ditest. 🧪

CUSTOM HOOKS — SUPERPOWER REACT YANG SEBENARNYA 🧩

Abstraksi · 12 menit baca

Okay, ini bagian yang paling "oh ternyata gitu!" dari semua hooks. Custom hooks itu bukan feature yang separate — dia lebih kayak cara berpikir. Semua hooks yang udah kita bahas bisa kamu gabung jadi satu "super hook" yang reusable di mana aja.

Bayangin kamu punya 3 komponen yang butuh logic fetch data + loading + error. Tanpa custom hook, kamu nulis logic yang sama 3 kali. Dengan custom hook? Tulis sekali, pakai sepuasnya. Itu kayak punya resep masakan favorit — sekali tulis di buku resep, tinggal buka lagi kalau mau masak. 📖🍳

Aturan emas: custom hook WAJIB namanya diawali use — misalnya useFetch, useLocalStorage, useWindowSize. Ini bukan sekadar konvensi — ini yang bikin React bisa detect dan enforce Rules of Hooks.

Kenapa custom hooks itu powerful? Karena dia bikin logic yang reusable tanpa share state antar komponen. Setiap komponen yang pakai custom hook punya state-nya sendiri — nggak shared. Ini bedanya sama Context (shared state) vs Custom Hook (shared logic, independent state). Oh ternyata gitu! 🤯

Custom hooks juga bikin kode kamu jauh lebih testable. Karena reducer dan hook logic terpisah dari component JSX, kamu bisa test-nya unit test tanpa render apapun. Test useFetch? Cukup test bahwa dia return data, loading, error yang bener. Gampang! ✅

Pattern yang sering dipake di production: extract semua business logic ke custom hook, biar component cuma fokus ke rendering UI. Ini bikin component kamu tipis (thin components) dan logic kamu terorganisir. Contoh: useAuth, useDebounce, useMediaQuery, useIntersectionObserver. Semua ini bisa kamu tulis sekali dan pakai di banyak proyek!

useFetch & useLocalStorage — Custom Hooks
import { useState, useEffect, useCallback } from 'react';

// 🔥 Custom Hook: useFetch
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  const fetchData = useCallback(async () => {
    try {
      setLoading(true); setError(null);
      const res = await fetch(url);
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      const json = await res.json();
      setData(json);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }, [url]);

  useEffect(() => { fetchData(); }, [fetchData]);
  return { data, loading, error, refetch: fetchData };
}

// 🔥 Custom Hook: useLocalStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const saved = localStorage.getItem(key);
      return saved ? JSON.parse(saved) : initialValue;
    } catch { return initialValue; }
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

// 🔥 Custom Hook: useClickOutside
function useClickOutside(ref, handler) {
  useEffect(() => {
    const listener = (event) => {
      if (!ref.current || ref.current.contains(event.target)) return;
      handler(event);
    };
    document.addEventListener('mousedown', listener);
    document.addEventListener('touchstart', listener);
    return () => {
      document.removeEventListener('mousedown', listener);
      document.removeEventListener('touchstart', listener);
    };
  }, [ref, handler]);
}

function UserProfile() {

  if (error) return <p>Error: {error}</p>;

  return (
    <div>
      <p>Hello, {user.name}! Theme: {theme}</p>
      <button onClick={() => setTheme(t => t === 'dark' ? 'light' : 'dark')}>
        Toggle Theme
      </button>
      <button onClick={refetch}>Refresh</button>
    </div>
  );
}

USETRANSITION & USEDEFERREDVALUE — UI TETAP SMOOTH 🏎️

Concurrent Features · 9 menit baca

Pernah bikin filter list yang pas kamu ngetik, seluruh halaman jadi lag dan nge-freeze? Itu karena filter 10.000 item itu jalan di main thread yang sama dengan rendering UI. Hasilnya? Typing-nya jadi choppy dan user kesel. 😤

useTransition dan useDeferredValue itu kayak bilang ke React: "Eh, update yang ini nggak terlalu penting, kerjain nanti aja yang penting UI tetap responsif." Mirip kayak ngasih priority lane di jalan tol — kendaraan VIP (user input) duluan, yang berat-berat (filter data) belakangan. 🛣️

Bedanya: useTransition untuk manual marking update sebagai "non-urgent". useDeferredValue untuk auto-defer sebuah value. Two different tools, one same goal: keep UI responsive! 🎯

Mekanismenya keren: pas kamu panggil startTransition(), React nge-prioritaskan update yang urgent (input, klik) dulu, baru ngerjain update non-urgent (filter list). Kalau user masih ngetik pas filter belum selesai, React bisa nggak jadiin filter yang lama dan langsung kerjain yang baru. Ini namanya interruptible rendering — dan ini beda mendasar dari React 17 ke React 18.

Tapi ingat, fitur ini bukan solusi untuk semua performance problem. Kalau list kamu punya 100.000 item, yang kamu butuhkan mungkin virtualization (windowing) kayak react-window, bukan defer. useTransition cocok buat update yang berat tapi manageable (filter 1000-5000 item, render chart yang kompleks). Pake tools yang tepat untuk masalah yang tepat! 🎯

useTransition — Manual Non-Urgent Update
import { useState, useTransition } from 'react';

function SearchWithTransition() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    const value = e.target.value;
    setQuery(value); // ⚡ URGENT — input langsung update
    startTransition(() => {
      setResults(filterHeavyList(value)); // 🐢 NON-URGENT — boleh ditunda
    });
  };

  return (
    <div>
      <input value={query} onChange={handleChange} />
      {isPending && <p className="opacity-50">Filtering...</p>}
      <ul>{results.map(item => <li key={item.id}>{item.name}</li>)}</ul>
    </div>
  );
}
useDeferredValue — Auto-Deferred Version
import { useState, useDeferredValue } from 'react';

function SearchWithDeferred({ items }) {
  const [filter, setFilter] = useState('');
  const deferredFilter = useDeferredValue(filter); // ⏳ Otomatis ditunda

  const filtered = items.filter(item =>
    item.name.toLowerCase().includes(deferredFilter.toLowerCase())
  );

  return (
    <div>
      <input value={filter} onChange={e => setFilter(e.target.value)} />
      <ul>{filtered.map(item => <li key={item.id}>{item.name}</li>)}</ul>
    </div>
  );
}

// useDeferredValue cocok kalau kamu udah punya value dan
// cuma mau "slow down" penggunaannya — nggak perlu ubah cara kamu setState

💡 Kapan mana? Pake useTransition kalau kamu sendiri yang trigger state update (kayak di event handler). Pake useDeferredValue kalau valuenya udah ada dan kamu cuma mau "slow down" penggunaannya. Keduanya bagian dari React Concurrent Features yang bikin app tetap responsif.

⚠️ Catatan: Fitur ini cuma bisa jalan di React 18+. Dan ingat, ini bukan solusi untuk semua performance problem. Kalau list kamu punya 100.000 item, yang kamu butuhkan mungkin virtualization (windowing), bukan defer. Pake tools yang tepat untuk masalah yang tepat!

Bonus: isPending dari useTransition itu super useful buat UX. Kamu bisa nunjukin loading skeleton atau blur pas update non-urgent lagi dikerjain. User tau kalau app-nya masih hidup dan respond, cuma lagi proses data. Ini bedanya sama app yang freeze total — user bisa tetap interaksi sambil nunggu. Pengalaman yang jauh lebih baik! ✨

MINI PROJECT — TASK MANAGER APP 🚀

Praktik · 20 menit

Sekarang saatnya gabungin semua hooks yang udah kita pelajari jadi satu project nyata! Kita bakal bikin Task Manager App yang pake useReducer buat state management, useRef buat auto-focus input, useMemo buat filter, dan useLocalStorage custom hook buat persist data. Keren banget kan? 🔥

Project ini nge-gabungin 4 hooks sekaligus dalam satu komponen. Ini contoh nyata gimana hooks advanced itu bekerja bareng di app sungguhan — bukan cuma teori terpisah-pisah.

Langkah 1: Reducer & Custom Hook
import { useReducer, useRef, useMemo, useEffect } from 'react';

// ===== CUSTOM HOOK: useLocalStorage =====
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const saved = localStorage.getItem(key);
      return saved ? JSON.parse(saved) : initialValue;
    } catch { return initialValue; }
  });
  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);
  return [value, setValue];
}

// ===== REDUCER: Task Logic =====
function taskReducer(state, action) {
  switch (action.type) {
    case 'ADD_TASK':
      return {
        ...state,
        tasks: [...state.tasks, {
          id: Date.now(),
          text: action.payload,
          done: false,
          createdAt: new Date().toISOString()
        }]
      };
    case 'TOGGLE_TASK':
      return {
        ...state,
        tasks: state.tasks.map(t =>
          t.id === action.payload ? { ...t, done: !t.done } : t
        )
      };
    case 'DELETE_TASK':
      return {
        ...state,
        tasks: state.tasks.filter(t => t.id !== action.payload)
      };
    case 'SET_FILTER':
      return { ...state, filter: action.payload };
    default:
      return state;
  }
}

const initialState = { tasks: [], filter: 'all' };
Langkah 2: Komponen Utama
function TaskManager() {
  const saved = useLocalStorage('tasks', initialState);
  const [state, dispatch] = useReducer(taskReducer, saved[0]);
  const inputRef = useRef(null);

  // Sync ke localStorage setiap state berubah
  useEffect(() => {
    saved[1](state);
  }, [state]);

  // useMemo untuk filter — cuma hitung ulang kalau tasks/filter berubah
  const filteredTasks = useMemo(() => {
    switch (state.filter) {
      case 'done': return state.tasks.filter(t => t.done);
      case 'active': return state.tasks.filter(t => !t.done);
      default: return state.tasks;
    }
  }, [state.tasks, state.filter]);

  // Hitung statistik
  const stats = useMemo(() => ({
    total: state.tasks.length,
    done: state.tasks.filter(t => t.done).length,
    active: state.tasks.filter(t => !t.done).length,
  }), [state.tasks]);

  const handleAdd = (e) => {
    e.preventDefault();
    const text = inputRef.current.value.trim();
    if (!text) return;
    dispatch({ type: 'ADD_TASK', payload: text });
    inputRef.current.value = '';
    inputRef.current.focus(); // auto-focus setelah tambah task
  };

  return (
    <div className="p-6 max-w-lg">
      <h1>Task Manager</h1>
      <p>Total: {stats.total} | Done: {stats.done} | Active: {stats.active}</p>

      <form onSubmit={handleAdd}>
        <input ref={inputRef} placeholder="Tambah task..." />
        <button type="submit">Tambah</button>
      </form>

      <div className="flex gap-2">
        {['all', 'active', 'done'].map(f => (
          <button key={f}
            onClick={() => dispatch({ type: 'SET_FILTER', payload: f })}
            className={state.filter === f ? 'active' : ''}>
            {f}
          </button>
        ))}
      </div>

      <ul>
        {filteredTasks.map(task => (
          <li key={task.id}>
            <input type="checkbox" checked={task.done}
              onChange={() => dispatch({ type: 'TOGGLE_TASK', payload: task.id })} />
            <span className={task.done ? 'line-through' : ''}>{task.text}</span>
            <button onClick={() => dispatch({ type: 'DELETE_TASK', payload: task.id })}>
              ×
            </button>
          </li>
        ))}
      </ul>
    </div>
  );
}

export default TaskManager;

💡 Challenge! Extend Task Manager ini sebagai latihan: (1) Tambahkan edit task inline, (2) Tambahkan drag-and-drop reordering, (3) Tambahkan due date dengan sorting, (4) Tambahkan search/filter by text. Semua ini bisa ditambahin tanpa ngubah reducer yang udah ada — cukup tambah action type baru!

RECAP — YANG UDAH KITA BAHAS 📝

Simpen di kepala ya, ini penting buat bab selanjutnya!

  • ✓useMemo — Cache hasil kalkulasi berat. Biar nggak re-kalkulasi di tiap render. Pakai kalau kalkulasinya mahal, jangan pakai kalau murah — overhead-nya sendiri bisa lebih berat.
  • ✓useCallback — Stabilkan function reference. Biar child component yang di-memo nggak re-render. Sama kayak useMemo tapi buat function. Internally ekivalen dengan useMemo(() => fn, deps).
  • ✓useRef — Kotak penyimpanan rahasia. Nggak trigger re-render, bisa buat akses DOM, simpan timer ID, track previous value, atau mutable storage. Punya satu properti: .current
  • ✓useContext — Stop prop drilling. Share data ke seluruh komponen tree tanpa kirim props berkali-kali. Selalu bikin custom hook buat context biar clean dan aman.
  • ✓useReducer — State management buat logic kompleks. Semua update logic terpusat di satu reducer function. Pakai kalau state saling terkait, useState kalau simple.
  • ✓Custom Hooks — Gabung hooks jadi super hook yang reusable. WAJIB diawali "use". Ini tentang abstraksi logic, bukan UI. Tulis sekali, pakai sepuasnya!
  • ✓useTransition & useDeferredValue — Biar UI tetap smooth pas ngerjain update berat. useTransition untuk manual marking, useDeferredValue untuk auto-defer. Bagian dari React 18 Concurrent Features.
  • ✓Mini Project Task Manager — useReducer + useRef + useMemo + useLocalStorage digabung jadi satu project nyata. Bukti kalo hooks advanced itu powerful kalau dipakai bareng!
