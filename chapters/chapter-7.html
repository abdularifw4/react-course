NEXT.JS — KENAPA BUKAN CREATE REACT APP? 🚀

Pengenalan · 10 menit baca

React itu library — dia cuma handle UI. Buat routing? Kamu harus install React Router. Buat data fetching? Install TanStack Query. Buat SEO? Install react-helmet. Tiap project jadi puzzle yang harus kamu rakit sendiri. Capek kan? 😩

Next.js itu kayak smartphone vs React yang kayak komponen elektronik. Next.js udah jadi produk jadi — tinggal pakai. React kamu harus solder sendiri. Next.js ngasih routing, data fetching, SEO, optimization, deployment — semua dalam satu package. Oh ternyata gitu! 🤯

Yang bikin Next.js spesial: Server Components. Ini fitur revolusioner yang nge-bedain Next.js dari framework lain. Komponen bisa jalan di server (nggak ada JavaScript ke client), yang artinya page load super cepat, SEO auto bagus, dan bundle size kecil. User cuma download JavaScript yang benar-benar butuh interaktivitas.

Versi Next.js sekarang udah 14+ dan sangat mature. Vercel (company di belakang Next.js) actively develop dan punya deployment platform yang seamless. Deploy Next.js ke Vercel cuma satu command: vercel deploy. Auto HTTPS, auto CDN, auto preview per PR. Developer experience-nya top tier.

Tapi Next.js bukan buat semua project. Kalau app kamu pure client-side ( kayak game, canvas editor, atau dashboard yang semua interaktif ), mungkin Vite + React lebih cocok. Next.js paling powerful buat content-heavy app: blog, e-commerce, landing page, SaaS. 🎯

Next.js App Router (v14+) itu paradigma baru — nggak lagi pake pages/ folder, tapi app/ folder dengan file-based routing yang lebih powerful. Layout, loading state, error boundary — semuanya otomatis berdasarkan folder structure. Developer experience-nya next level.

Getting started cuma satu command: npx create-next-app@latest. Next.js bakal nanya beberapa pertanyaan (TypeScript? ESLint? Tailwind? App Router?) — pilih Yes buat semua default. Dalam 1 menit, kamu punya project Next.js lengkap dengan TypeScript, Tailwind, dan hot reload. Nggak perlu konfigurasi manual. 🔥

SERVER vs CLIENT COMPONENTS — DUA DUNIA YANG BERBEDA 🌍

Core Concept · 12 menit baca

Ini konsep paling penting di Next.js — dan yang paling sering bikin bingung. Jadi dengarin baik-baik ya. Di Next.js App Router, ada dua jenis komponen: Server Components (default) dan Client Components (pake "use client").

Analoginya: Server Components itu kayak dapur restoran — masak di belakang, hasilnya udah jadi pas sampai ke meja. Client Components itu kayak barbekyu di meja — kamu masak sendiri di depan customer. Dua-duanya penting, tapi tempatnya beda. Oh ternyata gitu! 🤯

Server Components jalan di server saat request. Nggak ada JavaScript yang dikirim ke browser untuk komponen ini. Cocok buat: fetch data, akses database, read file system, render static HTML. Keuntungannya: page load cepat, SEO bagus, bundle kecil, aman (secret nggak exposed ke client).

Client Components jalan di browser. Butuh JavaScript. Cocok buat: interaktivitas (onClick, onChange), hooks (useState, useEffect), browser API (localStorage, geolocation), event listeners. Keuntungannya: user bisa interact langsung tanpa request ke server.

Satu kesalahan umum: taruh "use client" di layout atau page-level. Ini bikin SEMUA child component jadi Client Component — kehilangan benefit Server Component. Solution? Taruh "use client" di leaf component (komponen paling dalam yang butuh interaktivitas). Parent tetap Server Component. 🎯

Aturan emas: mulai dari Server Component, turun ke Client Component kalau butuh interaktivitas. Jangan taruh "use client" di semua file — itu kebablasan. Server Component bisa render Client Component sebagai child, tapi Client Component nggak bisa import Server Component sebagai prop yang di-render secara dinamis.

Best practice: composition pattern. Server Component fetch data, terus pass ke Client Component sebagai prop (JSX). Client Component fokus handle interaktivitas. Data-nya udah ada — nggak perlu loading state di client. Pattern ini bikin client bundle kecil dan page load cepat. 🎯

Server Component — Default
// ✅ Server Component (default — nggak perlu "use client")
// File: app/page.tsx

import { db } from '@/lib/db';
import { ProductCard } from '@/components/ProductCard';
import { AddToCartButton } from '@/components/AddToCartButton';

// Data fetching langsung di komponen — nggak perlu useEffect!
async function getProducts() {
  const products = await db.product.findMany();
  return products;
}

export default async function HomePage() {
  const products = await getProducts();

  return (
    <main>
      <h1>Produk Terbaru</h1>
      {products.map(product => (
        <div key={product.id}>
          <ProductCard product={product} />        {/* ✅ Server Component */}
          <AddToCartButton productId={product.id} /> {/* ✅ Client Component */}
        </div>
      ))}
    </main>
  );
}
Client Component — Interactive
// ✅ Client Component — WAJIB "use client" di baris pertama
// File: components/AddToCartButton.tsx

'use client';

import { useState } from 'react';

export function AddToCartButton({ productId }: { productId: string }) {
  const [added, setAdded] = useState(false);
  const [loading, setLoading] = useState(false);

  const handleAdd = async () => {
    setLoading(true);
    await fetch('/api/cart', {
      method: 'POST',
      body: JSON.stringify({ productId }),
    });
    setAdded(true);
    setLoading(false);
  };

  return (
    <button onClick={handleAdd} disabled={loading}>
      {loading ? 'Menambahkan...' : added ? '✅ Udah di Keranjang' : 'Tambah ke Keranjang'}
    </button>
  );
}

⚠️ Error paling sering: Lupa "use client" di komponen yang pake hooks atau event handler. Error-nya: useState is not a client component. Solution? Tambah 'use client' di baris pertama file tersebut.

03

ROUTING — FILE SYSTEM YANG CERDAS 🗺️

Routing · 12 menit baca

Di Next.js, kamu nggak perlu konfigurasi router. Cukup bikin folder dan file — Next.js otomatis bikin route berdasarkan struktur folder. app/about/page.tsx = route /about. Sederhana kan? 🎯

Yang bikin powerful: dynamic routes. Pake bracket notation — [slug], [id]. Route-nya jadi dinamis: /blog/hello-world, /product/123. Parameter-nya accessible via params prop di page component.

Layout itu wrapper yang persist di semua child routes. Root layout (app/layout.tsx) wrap SEMUA page. Nested layout (app/blog/layout.tsx) cuma wrap page di folder blog. Layout nggak re-render pas navigasi — jadi sidebar, header, footer tetap ada tanpa flash. Oh ternyata gitu! 🤯

Untuk navigasi antar halaman, Next.js punya <Link> component. Dia handle prefetching otomatis — pas kamu hover link, Next.js udah fetch data di background. Jadi pas klik, page langsung tampil. Nggak ada loading spinner. Ini yang bikin Next.js app terasa secepat native app. ⚡

Yang juga powerful: parallel routes dan intercepting routes. Parallel routes load multiple page di same URL secara bersamaan — cocok buat dashboard dengan panel samping. Intercepting routes buat "modal routing" — buka content di modal tanpa pindah halaman. Ini fitur advanced yang jarang dipake tapi super powerful pas butuh.

Special files yang auto-kerja: loading.tsx (skeleton UI pas loading), error.tsx (error boundary), not-found.tsx (404 page), page.tsx (route handler). Semuanya based on convention — nggak perlu config.

Dynamic Route dengan Params
// File: app/blog/[slug]/page.tsx

type PageProps = {

  // Fetch blog post berdasarkan slug

    .then(res => res.json());

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
      <span>Published: {new Date(post.createdAt).toLocaleDateString('id-ID')}</span>
    </article>
  );
}

04

DATA FETCHING — NGGAK PERLU USEEFFECT LAGI 📡

Data · 12 menit baca

Di React biasa, data fetching pake useEffect + useState. Result? Waterfall loading — browser download JS, render empty UI, fetch data, re-render dengan data. User lihat loading spinner dulu, baru lihat konten. Nggak optimal. 😤

Di Next.js Server Components, kamu bisa await langsung di komponen! Data di-fetch di server, HTML udah lengkap pas sampai ke browser. User langsung lihat konten — no loading spinner, no waterfall. Ini game changer buat UX. Oh ternyata gitu! 🤯

Di real project, Server Component biasanya nggak fetch dari external API — mereka langsung query database via ORM kayak Prisma atau Drizzle. Ini lebih efisien: nggak ada HTTP overhead, type-safe query, dan bisa langsung akses environment variable tanpa expose ke client. Pattern-nya: Prisma client di lib/db.ts, import di Server Component, query langsung. Clean dan efisien. 🔥

Tapi Server Component fetching bukan buat semua kasus. Kalau data-nya real-time (chat, stock price), butuh Client Component + SWR/React Query. Kalau data-nya user-specific (dashboard auth), gabung server fetch + client revalidation.

Next.js juga punya caching strategy built-in: fetch default cache (ISR-like), no-store buat always fresh, revalidate buat periodic refresh. Pattern yang recommended: Server Component fetch buat initial data, Client Component + SWR buat real-time update. Dua pattern ini bisa kombinasikan — Server Component render initial data, Client Component mount dan mulai polling buat update. User langsung lihat data, tapi data tetap fresh. Best of both worlds! ⚡

Nggak perlu library external buat caching — Next.js handle semuanya.

Server Component — Direct Fetch
// ✅ Direct await di Server Component — super clean!
// File: app/products/page.tsx

async function getProducts() {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 3600 } // cache 1 jam, auto-refresh
  });
  if (!res.ok) throw new Error('Failed to fetch');
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return (
    <div>
      <h1>Produk Kami ({products.length} items)</h1>
      <div className="grid grid-cols-3 gap-4">
        {products.map((p: any) => (
          <div key={p.id}>
            <img src={p.image} alt={p.name} />
            <h3>{p.name}</h3>
            <p>Rp {p.price.toLocaleString('id-ID')}</p>
          </div>
        ))}
      </div>
    </div>
  );
}
Client Component — SWR Pattern
// ✅ Client Component + SWR buat real-time data
// File: components/RealtimeStock.tsx

'use client';

import useSWR from 'swr';

const fetcher = (url: string) => fetch(url).then(r => r.json());

export function RealtimeStock({ productId }: { productId: string }) {
  const { data, error, isLoading } = useSWR(
    `/api/stock/${productId}`,
    fetcher,
    { refreshInterval: 5000 } // poll setiap 5 detik
  );

  if (isLoading) return <span>Loading...</span>;
  if (error) return <span>Error</span>;

  return (
    <span className={data.stock < 10 ? 'text-red-500' : 'text-green-500'}>
      Stok: {data.stock} {data.stock < 10 && '— Hampir habis!'}
    </span>
  );
}

💡 Kapan pilih mana: Server fetch → static/semi-static data (blog, product catalog, profile). Client fetch (SWR/React Query) → real-time data, user-specific data, data yang sering berubah. Bisa juga gabung: server fetch buat initial load, client revalidate buat update berikutnya.

METADATA & SEO — AUTO RANKING #1 GOOGLE 🔍

SEO · 10 menit baca

React biasa itu SPA — single HTML file yang isinya kosong <div id="root">. Semua content di-render oleh JavaScript di browser. Google bot bisa baca, tapi butuh waktu — dan kadang skip JavaScript altogether. Result? SEO buruk. 😤

Next.js beda — tiap page udah punya HTML lengkap di server. Google bot langsung baca konten, title, meta description, structured data. Nggak perlu tunggu JavaScript execute. SEO auto bagus tanpa effort extra. Oh ternyata gitu! 🤯

Metadata bisa didefinisikan via generateMetadata() (dynamic, buat halaman berbeda-beda tiap request) atau static metadata export (fixed per page). Next.js otomatis convert ke proper <head> tags.

Contoh nyata: blog post yang share ke LinkedIn. Tanpa Open Graph, preview-nya cuma URL kosong — nggak ada gambar, judul, atau deskripsi. Dengan OG metadata yang bener, preview-nya kaya dan menarik. Ini bisa meningkatkan CTR (click-through rate) secara signifikan. SEO bukan cuma Google — social media juga penting! 📱

Yang sering dilupakan: Open Graph buat social media preview (Facebook, Twitter, LinkedIn), JSON-LD structured data buat Google rich snippets, dan sitemap buat crawl efficiency. Next.js punya semua built-in — tinggal pake.

JSON-LD structured data itu senjata rahasia SEO. Dengan markup yang bener, Google bisa tampilin rich snippets: star rating, price, FAQ accordion, breadcrumb. Ini langsung meningkatkan visibility di search results. Next.js bisa generate JSON-LD dari Server Component — data langsung dari database, nggak perlu manual update. Automation = consistency. 📊

MIDDLEWARE, CACHING & ERROR UI 🛡️

Advanced · 11 menit baca

Tiga fitur Next.js yang sering diabaikan padahal powerful banget. Middleware buat auth redirect dan geo-routing, caching buat performance, error boundary buat graceful failure. Mari bahas satu-satu. 🎯

Middleware jalan sebelum request sampai ke page. Cocok buat: auth check (redirect ke login kalau belum login), A/B testing, geo-based routing, rate limiting. Middleware cuma bisa jadi Server Component — nggak bisa pake hooks atau event handlers. Tapi dia bisa access cookies, headers, dan request URL — cukup buat sebagian besar use case auth dan routing.

Edge Runtime — middleware jalan di edge (CDN), bukan di server Node.js. Artinya super cepat, latency rendah, dan bisa nge-handle jutaan request. Tapi ada limitation: nggak bisa akses Node.js API kayak fs atau native modules. Buat logic yang simple (auth, redirect), ini lebih dari cukup.

Caching di Next.js App Router udah built-in. fetch default-nya cache (dengan revalidate). Kamu bisa override per-request: no-store buat fresh data, revalidate: 60 buat auto-refresh tiap 60 detik. Ini menggantikan getServerSideProps + getStaticProps dari Pages Router. Oh ternyata gitu! 🤯

Error UI lewat error.tsx — komponen khusus yang nampilin pas ada error. Dengan pattern ini, error di satu bagian nggak nge-crash seluruh halaman. Error boundary paling dekat yang handle — jadi error di blog post tetap tampilin header dan sidebar. User bisa navigate ke halaman lain tanpa refresh.

Bisa nested (blog error, dashboard error) atau global. Yang keren: error.tsx bisa jadi Client Component jadi bisa punya "Try Again" button. Satu lagi: loading.tsx. Ini skeleton UI yang auto-tampil pas Next.js fetching data. Nggak perlu state management buat loading — cukup bikin loading.tsx di folder yang sama dengan page.tsx. React Suspense otomatis wrap dan tampilin skeleton. User experience-nya smooth banget — nggak ada blank page, cuma skeleton yang smooth fade in. 🎨

Auto-wrap error boundary tanpa install apapun.

Middleware — Auth Check
// File: middleware.ts (di ROOT project, bukan di app/)

import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const token = request.cookies.get('auth-token');
  const isProtected = request.nextUrl.pathname.startsWith('/dashboard');

  if (isProtected && !token) {
    // Redirect ke login, simpan original URL
    const loginUrl = new URL('/login', request.url);
    loginUrl.searchParams.set('callbackUrl', request.nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }

  return NextResponse.next();
}

// Konfigurasi matcher — middleware cuma jalan di route tertentu
export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*'],
};

SERVER ACTIONS — BIKIN FORM TANPA API 📝

Forms · 11 menit baca

Dulu bikin form di React: handleSubmit → fetch ke API endpoint → handle response. Backend-nya harus bikin route terpisah buat handle POST. Banyak boilerplate. Next.js Server Actions nge-simplify semua ini. Kamu bisa langsung panggil server function dari form — nggak perlu API endpoint. 🔥

Analoginya: Server Actions itu kayak pesan langsung ke dapur vs lewat waiter (API). Lebih cepat, nggak perlu middleware. Form submit → langsung ke server function → return result. Clean, simple, dan type-safe. Oh ternyata gitu! 🤯

Server Actions itu async function yang jalan di server. Bisa didefinisikan di Server Component atau file terpisah. Dipanggil via action prop di <form> atau via formAction di hook. Result bisa di-access via useActionState (React 19).

Yang powerful: Server Actions bisa return data (bukan cuma void), bisa throw error (auto-catch oleh error boundary), bisa revalidate cache (auto-update UI pas data berubah). Ini menggantikan pola lama: API route + fetch + state management. Server Actions juga support progressive enhancement — kalau JavaScript disabled di browser, form tetap bisa submit (dengan full page reload). Ini yang bikin Next.js form lebih resilient dari form React biasa. Accessibility + reliability = win-win.

Satu function, semua handled. ⚡

Server Action — Form Handling
// File: app/actions.ts — Server Actions di file terpisah
'use server';

import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string;
  const content = formData.get('content') as string;

  if (!title || !content) {
    return { error: 'Title dan content wajib diisi!' };
  }

  // Simpan ke database
  await db.post.create({ data: { title, content } });

  // Revalidate cache + redirect
  revalidatePath('/blog');
  redirect('/blog');
}
Form dengan Server Action

⚠️ Security: Server Actions BUKAN client-side function. Mereka jalan di server — jadi secret (API key, database URL) tetap aman. Tapi tetap validasi input! Client bisa manipulasi formData sebelum dikirim. Selalu server-side validation. Plus, Server Actions bisa di-pair dengan useActionState (React 19) buat handle loading, success, dan error state — mirip useState tapi khusus buat form. Tapi tetap validasi input! Client bisa manipulasi formData sebelum dikirim. Selalu server-side validation.

// components/LikeButton.tsx — Client Component
'use client';
import { useState } from 'react';

export function LikeButton({ slug, initialLikes }: { slug: string; initialLikes: number }) {
  const [likes, setLikes] = useState(initialLikes);
  const [liked, setLiked] = useState(false);

  const handleLike = async () => {
    if (liked) return;
    setLikes(l => l + 1);
    setLiked(true);
    await fetch('/api/like', { method: 'POST', body: JSON.stringify({ slug }) });
  };

  return (
    <button onClick={handleLike} className={liked ? 'text-red-500' : 'text-muted'}>
      ❤️ {likes}
    </button>
  );
}

RECAP — YANG UDAH KITA BAHAS 📝

Next.js App Router — framework React yang lengkap dari A sampai Z

  • ✓Kenapa Next.js — Semua dalam satu: routing, data fetching, SEO, optimization, deployment. Lebih efisien dari rakit React sendiri.
  • ✓Server vs Client Components — Default Server, Client kalau butuh interaktivitas. Server = dapur, Client = barbekyu di meja.
  • ✓Routing — File-based routing otomatis. Dynamic route [slug], nested layout, loading/error/not-found built-in.
  • ✓Data Fetching — await langsung di Server Component. Nggak perlu useEffect. Caching built-in.
  • ✓Metadata & SEO — generateMetadata buat dynamic OG tags. Auto HTML lengkap buat Google bot.
  • ✓Middleware, Caching, Error UI — Auth check di middleware, caching strategy, error.tsx buat graceful failure.
  • ✓Server Actions — Form handling tanpa API endpoint. Direct server function dari form action.
  • ✓Mini Project Blog — Gabung semua konsep: Server/Client Components, dynamic routes, metadata, error handling.
