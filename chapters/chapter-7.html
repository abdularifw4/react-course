Next.js Fundamentals

Pernah ngerasa React biasa tuh kayak mobil tanpa mesin? Kita punya komponen keren, tapi gimana cara ngirimnya ke user? Nah, Next.js jawabannya. 🏎️💨

🏃 APP ROUTER — BIKIN ROUTE TANPA PUSING

Oke, bayangin lo mau bikin website punya halaman Home, About, Blog, sama Dashboard. Di React biasa, lo harus pasang React Router, bikin file terpisah, tulis path manual... cape deh. 😥

Di Next.js App Router (mulai versi 13+), lo cuma bikin folder aja. Serius. Itu doang. Nggak ada konfigurasi router. Nggak ada import Route dari mana-mana. Cukup bikin folder, dan boom — route-nya auto jadi. 💯

Analoginya gini: lo punya rumah. Setiap kamar itu satu halaman. Mau tambah kamar? Tinggal bikin folder baru. Mau kamar dalam kamar (nested route)? Bikin folder di dalam folder. Simple kan?

// Struktur folder = routing otomatis!
// app/
//   layout.tsx          → Layout utama (HTML, body tag)
//   page.tsx             → /
//   about/
//     page.tsx           → /about
//   blog/
//     page.tsx           → /blog
//     [slug]/
//       page.tsx         → /blog/:slug (dynamic route!)
//   dashboard/
//     layout.tsx         → Layout khusus dashboard (nested)
//     page.tsx           → /dashboard
//     settings/
//       page.tsx         → /dashboard/settings

// Dynamic route — pake [bracket] buat parameter
interface BlogPostProps {
  params: { slug: string };
}

export default function BlogPost({ params }: BlogPostProps) {
  return 
Baca: {params.slug}
;
}

// Pengen halaman blog di-generate sebelum user buka? (SSG)
// Pake generateStaticParams!
export async function generateStaticParams() {
  const posts = await fetch('https://api.example.com/posts');
  return (await posts.json()).map((p: any) => ({
    slug: p.slug
  }));
}

💡 Oh ternyata gitu! Folder [slug] itu kayak slot kosong di parkiran. Nama slot-nya dinamis — bisa jadi "belajar-react", "tips-nextjs", apapun. Nah, isi dari slot itu nanti bisa lo akses lewat params.slug. Keren nggak? 🎰

⚠️ Jangan lupa: Kalo lo bikin folder tapi nggak ada file page.tsx di dalamnya, route-nya nggak bakal muncul. Next.js butuh page.tsx sebagai "pintu masuk" halaman. Tanpa itu, foldernya cuma jadi dekorasi doang. 😅

👨‍🍳 SERVER COMPONENTS VS 🎮 CLIENT COMPONENTS

Ini konsep yang bikin Next.js beda banget sama React biasa. Jadi gini ya...

Bayangin lo lagi di restoran. Ada chef yang masak di dapur, dan ada meja makan yang interaktif buat tamu. Server Component itu si chef — dia masak (render) di dapur (server), terus kirim hasilnya yang udah matang ke tamu. Nggak perlu tamu repot-repot masak sendiri. 👨‍🍳

Sedangkan Client Component itu kayak meja makan yang punya tombol buat pesan, interaktif, bisa klik-klik. Tamu butuh ini buat interaksi. Tapi chef tetap yang masak makanannya — tamu cuma nerima dan bisa interaksi di meja. 🎮

Kenapa ini penting? Karena Server Component nggak ngirim JavaScript ke browser! Artinya? Website lo lebih cepet, bundle-nya lebih kecil, dan user-nya lebih seneng. 🚀

// SERVER COMPONENT — default di Next.js App Router!
// Nggak usah tulis apa-apa, ini udah otomatis server component

💡 Aturan emas: Mulai dari Server Component dulu. Baru kalau lo butuh useState, useEffect, onClick, atau browser API — baru tambahin 'use client' di paling atas file. Semakin banyak Server Component, semakin cepat web lo. ⚡

⚠️ Trap yang sering kejebak: Kalo lo tulis 'use client' di parent component, semua child-nya otomatis jadi Client Component juga. Jadi hati-hati — jangan asal sembarang taruh 'use client'. Taruh di component yang bener-bener butuh doang! 🎯

📦 DATA FETCHING — GIMANA CARA AMBIL DATA?

Dulu di React, kalo mau ambil data lo pake useEffect + fetch. Nah itu cara lama. Di Next.js, lo bisa langsung async/await di component — karena component-nya jalan di server! 🎉

Tapi wait, ada tiga strategi fetching data di Next.js. Masing-masing punya use case beda. Pahami ini, lo bakal nggak bingung lagi kapan harus pake yang mana.

// === STRATEGI 1: STATIC GENERATION (SSG) ===
// Data di-fetch sekali waktu build. Cocok buat blog, landing page.
async function BlogList() {
  const posts = await fetch('https://api.example.com/posts', {
    cache: 'force-cache' // ← Simpan selamanya sampai build lagi
  }).then(r => r.json());

  return 
{posts.map((p: any) => 
{p.title}
)}
;
}

// === STRATEGI 2: SERVER-SIDE RENDERING (SSR) ===
// Data di-fetch setiap kali user request. Cocok buat data yang sering berubah.
async function LiveScore() {
  const scores = await fetch('https://api.example.com/scores', {
    cache: 'no-store' // ← Selalu fresh, nggak pernah cache
  }).then(r => r.json());

  return 
{scores.map((s: any) => 
{s.team}: {s.score}
)}
;
}

// === STRATEGI 3: INCREMENTAL STATIC REGENERATION (ISR) ===
// Data di-cache TAPI di-update berkala. Best of both worlds!
async function ProductPage() {
  const products = await fetch('https://api.example.com/products', {
    next: { revalidate: 3600 } // ← Update tiap 1 jam
  }).then(r => r.json());

  return 
{products.map((p: any) => 
{p.name}
)}
;
}

💡 Analoginya gini:

  • force-cache = Foto polaroid 📸 — diambil sekali, nggak berubah-ubah
  • no-store = Live TV 📺 — selalu update, realtime
  • revalidate: 3600 = Koran pagi 📰 — terbit tiap jam, tapi di antaranya tetap yang lama

🔌 API ROUTES — BACKEND TANPA SERVER TERPISAH

Nah ini yang paling seru. Lo butuh API buat handle form, autentikasi, atau database operation? Nggak usah bikin server Express terpisah. Next.js udah nyediain API Routes di dalam project lo sendiri! 🎂

Caranya? Tinggal bikin folder api/ di dalam app/, terus buat file route.ts. Setiap file route.ts bisa export fungsi GET, POST, PUT, DELETE — persis kayak REST API yang lo kenal. 😎

// File: app/api/hello/route.ts
import { NextResponse } from 'next/server';

// GET request handler
export async function GET() {
  const data = { message: 'Halo dari Next.js API!' };
  return NextResponse.json(data);
}

// POST request handler
export async function POST(request: Request) {
  const body = await request.json();

  // Misalnya simpan ke database
  // await db.user.create({ data: body });

  return NextResponse.json({
    success: true,
    message: 'Halo ' + body.name + '! Data udah diterima.'
  });
}

// Pake di frontend (Server Component atau Client Component)
async function ContactForm() {
  // Bisa langsung fetch ke API route sendiri!
  const response = await fetch('/api/hello', {
    method: 'POST',
    body: JSON.stringify({ name: 'Arif' })
  });
  const data = await response.json();
  // { success: true, message: "Halo Arif! Data udah diterima." }
}

💡 Oh ternyata gitu! API route itu kayak punya dapur belakang di restoran lo sendiri. Nggak perlu sewa gedung terpisah (server Express/NestJS) cuma buat handle API. Semua dalam satu project. Satu deploy, semuanya jalan. Efisien banget! 🏠🍳

🏗️ LAYOUT & TEMPLATES — SISTEM KERANGKA HALAMAN

Setiap website punya bagian yang tetap ada di semua halaman — header, footer, sidebar. Di Next.js, lo nggak perlu copy-paste itu ke setiap halaman. Ada layout.tsx yang nge-wrap semua halaman secara otomatis. 🧩

Yang lebih keren: layout bisa nested. Artinya dashboard punya layout sendiri, blog punya layout sendiri, tapi tetap punya layout utama (root layout) yang nge-wrap semuanya. Kayak boneka matryoshka — ada dalam ada! 🪖

// app/layout.tsx — Root Layout (WAJIB ada!)

💡 Perbedaan layout vs template: layout.tsx itu persistent — pas lo pindah halaman, layout-nya nggak re-render (state tetap hidup!). template.tsx itu tiap pindah halaman bakal re-render ulang (fresh state). Pilih layout buat kebanyakan kasus! 🎯

📝 RECAP — YANG HARUS LO BAWA PULANG

✅

App Router — Bikin route cuma dengan bikin folder. page.tsx = halaman, [slug] = dynamic route. Gampang banget.

✅

Server vs Client Components — Default Server Component (lebih cepat). Tambah 'use client' cuma kalau butuh interaktivitas.

✅

Data Fetching — force-cache (SSG), no-store (SSR), revalidate (ISR). Tiga strategi, tiga kebutuhan berbeda.

✅

API Routes — Backend langsung di project Next.js. Nggak perlu server terpisah.

✅

Layouts — Wrapper otomatis buat bagian yang tetap ada. Bisa nested, state persistent.

Sekarang lo udah punya mental model Next.js yang solid. Dari sini, lo bisa explore lebih dalam — middleware, image optimization, metadata API, dan lain-lain. Tapi fondasi ini udah lebih dari cukup buat mulai bikin production app! 💪
