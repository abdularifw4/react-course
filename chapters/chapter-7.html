05

Server Actions & Form Handling Tanpa API

Bayangin lo mau bikin form kontak. Dulu, lo harus bikin API route, handle POST request, parse body, save ke database... capek banget. Nah, Server Actions bikin itu semua jadi satu function doang yang jalan di server. Nggak perlu API route!

Server Action — Form Kontak 
// app/actions/contact.ts
"use server";

import { revalidatePath } from "next/cache";

export async function submitContact(formData: FormData) {
  const name = formData.get("name") as string;
  const email = formData.get("email") as string;
  const message = formData.get("message") as string;

  // Validasi
  if (!name || !email) {
    return { error: "Nama dan email wajib diisi!" };
  }

  // Save ke database
  await db.contact.create({ data: { name, email, message } });

  // Revalidate cache
  revalidatePath("/admin/contacts");
  return { success: true };
}
Pakai Server Action di Form 
// app/contact/page.tsx
import { submitContact } from "@/app/actions/contact";

export default function ContactPage() {
  return (
    <form action={submitContact}>
      <input name="name" placeholder="Nama" />
      <input name="email" type="email" placeholder="Email" />
      <textarea name="message" placeholder="Pesan" />
      <button type="submit">Kirim</button>
    </form>
  );
}

// Dengan loading state pakai useFormStatus:
import { useFormStatus } from "react-dom";

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? "Mengirim..." : "Kirim"}
    </button>
  );
}

OH TERNYATA GITU! Server Actions = form handling tanpa bikin API route, tanpa fetch, tanpa JSON parse. Cukup bikin function dengan "use server" dan hubungkan ke form. Next.js urus sisanya!

06

Middleware, Caching & Error UI

Middleware itu kayak satpam di gerbang. Setiap request yang masuk lewati dia dulu sebelum sampe ke page. Bisa dipake buat auth check, redirect, rewrite URL, dan banyak lagi.

Middleware — Auth Protection 
// middleware.ts (di root folder)
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const token = request.cookies.get("token")?.value;

  // Protected routes
  if (request.nextUrl.pathname.startsWith("/dashboard")) {
    if (!token) {
      return NextResponse.redirect(new URL("/login", request.url));
    }
  }

  // Redirect kalau udah login tapi buka /login
  if (request.nextUrl.pathname === "/login" && token) {
    return NextResponse.redirect(new URL("/dashboard", request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/dashboard/:path*", "/login", "/admin/:path*"],
};

export async function generateStaticParams() {

Caching Strategies:

Next.js Fetch Caching 
// SSG: fetch sekali saat build (default)
const data = await fetch("https://api.example.com/data");

// SSR: fetch setiap request
const data = await fetch("https://api.example.com/data", {
  cache: "no-store",
});

// ISR: revalidate tiap 60 detik
const data = await fetch("https://api.example.com/data", {
  next: { revalidate: 60 },
});

// Tag-based: invalidate manual
const data = await fetch("https://api.example.com/data", {
  next: { tags: ["users"] },
});
// Di server action:
import { revalidateTag } from "next/cache";
revalidateTag("users"); // refresh semua fetch dengan tag "users"

OH TERNYATA GITU! Middleware = satpam gerbang. Caching = memory React. generateStaticParams = pre-generate halaman saat build. Error/loading UI = safety net otomatis. Semuanya bekerja sama bikin app lo cepat dan robust!

01

Routing (App Router)

Fundamental · 8 menit baca

Next.js App Router menggunakan folder-based routing. Setiap folder di dalam app/ menjadi route, dan file page.tsx di dalamnya menjadi halaman yang di-render. Ini jauh lebih intuitif dibanding Pages Router yang lama.

Dynamic routes menggunakan folder dengan nama diapit kurung siku: [id]/ untuk parameter dinamis. Catch-all routes menggunakan [...slug]/. Route groups menggunakan (group)/ tanpa mempengaruhi URL.

App Router juga mendukung loading.tsx (automatic loading UI), error.tsx (error boundary), not-found.tsx (404 page), dan layout.tsx (shared layout) secara built-in.

App Router — Struktur Folder
app/
├── layout.tsx          # Root layout (wajib)
├── page.tsx            # Halaman utama (/)
├── loading.tsx         # Loading UI global
├── error.tsx           # Error boundary global
├── not-found.tsx       # 404 page
├── about/
│   └── page.tsx        # /about
├── blog/
│   ├── page.tsx        # /blog
│   ├── [slug]/
│   │   └── page.tsx    # /blog/:slug (dynamic)
│   └── loading.tsx     # Loading untuk /blog/*
└── dashboard/
    ├── layout.tsx      # Layout khusus dashboard
    ├── page.tsx        # /dashboard
    └── settings/
        └── page.tsx    # /dashboard/settings
Navigasi dengan Link & useParams 
// app/layout.tsx — navigasi pakai <Link> (client-side, tanpa reload!)
import Link from 'next/link';

export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <nav className="flex gap-4 p-4 border-b">
          {/* prefetch otomatis — halaman di-load di background */}
          <Link href="/" className="text-ink hover:underline">Home</Link>
          <Link href="/blog" className="text-ink hover:underline">Blog</Link>
          <Link href="/about" className="text-ink hover:underline">About</Link>
        </nav>
        {children}
      </body>
    </html>
  );
}

// app/blog/[slug]/page.tsx — akses parameter dinamis
export default async function BlogPostPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;  // Next.js 15: params adalah Promise
  const post = await getPost(slug);
  return <article><h1>{post.title}</h1></article>;
}

💡 Analogi: Folder-based routing seperti sistem folder di komputer. Setiap folder adalah "alamat" dan file page.tsx adalah "isi halaman" di alamat tersebut. Semakin dalam folder, semakin spesifik URL-nya.

02

Server vs Client Components

Konsep Inti · 8 menit baca

Di Next.js App Router, Server Components adalah default. Mereka di-render di server, tidak mengirim JavaScript ke client, dan bisa langsung mengakses database atau file system. Ini mengurangi bundle size dan meningkatkan performa.

Client Components ditandai dengan 'use client' di baris pertama file. Mereka di-render di browser dan bisa menggunakan useState, useEffect, event handlers, dan browser API.

Peraturan praktis: gunakan Server Components sebanyak mungkin. Hanya gunakan Client Components ketika benar-benar membutuhkan interaktivitas (onClick, onChange, useState, useEffect). Komponen Server bisa me-render Komponen Client, tapi tidak sebaliknya.

Server vs Client Components
// Server Component (default) — no directive needed
// Bisa akses database, file system, env variables langsung
async function UserProfile({ userId }: { userId: string }) {
  // Ini berjalan di server!
  const user = await db.user.findUnique({ where: { id: userId } });
  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.bio}</p>
      <LikeButton postId={userId} />  {/* Client Component */}
    </div>
  );
}

// Client Component — perlu directive
'use client';
import { useState } from 'react';

function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false);
  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? '❤️' : '💖'} Suka
    </button>
  );
}

⚠️ Perhatian: Jangan lupa 'use client' jika komponen menggunakan useState, useEffect, onClick, atau browser API. Tanpa directive ini, kamu akan mendapat error hydration.

Trik penting: kamu bisa mengirim props dari Server Component ke Client Component — tapi props tersebut harus serializable (string, number, array, object). Tidak boleh kirim function atau instance class!

Pola Server → Client Composition 
// app/dashboard/page.tsx (Server Component)
import { db } from '@/lib/db';
import DeleteButton from './DeleteButton';  // Client Component

export default async function Dashboard() {
  const user = await db.user.findFirst();  // query DB di server

  return (
    <main>
      <h1>Halo, {user.name}!</h1>
      {/* ✅ OKE: kirim data serializable ke Client Component */}
      <DeleteButton userId={user.id} />
      {/* ❌ ERROR: tidak boleh kirim function! */}
      {/* <DeleteButton onDelete={() => db.delete()} /> */}
    </main>
  );
}

// app/dashboard/DeleteButton.tsx (Client Component)
'use client';
import { useState } from 'react';

export default function DeleteButton({ userId }: { userId: string }) {
  const [confirming, setConfirming] = useState(false);
  return (
    <button onClick={() => setConfirming(true)}>
      {confirming ? 'Yakin hapus?' : 'Hapus Akun'}
    </button>
  );
}
03

Data Fetching, API Routes & SEO

Praktik · 10 menit baca

Next.js menyediakan Server-side Data Fetching yang powerful. Di Server Components, kamu bisa langsung menggunakan async/await tanpa useEffect. Data di-fetch di server dan HTML yang sudah lengkap dikirim ke client.

API Routes (Route Handlers) memungkinkan kamu membuat API endpoints di dalam proyek Next.js. File route.ts di folder app menjadi endpoint. Sangat berguna untuk webhook, form submissions, atau integrasi pihak ketiga.

Untuk SEO, Next.js menyediakan generateMetadata() untuk mengatur title, description, dan meta tags secara dinamis per halaman. Ini sangat penting untuk performa SEO.

Data Fetching & API Routes
// Server-side data fetching (di Server Component)
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetch(
    `https://api.example.com/posts/${params.slug}`,
    { next: { revalidate: 3600 } }  // ISR: revalidate tiap 1 jam
  ).then(r => r.json());

  return <article>
    <h1>{post.title}</h1>
    <div dangerouslySetInnerHTML={{ __html: post.content }} />
  </article>;
}

// Metadata/SEO
export async function generateMetadata({ params }) {
  const post = await getPost(params.slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: { images: [post.coverImage] },
  };
}

// API Route (app/api/contact/route.ts)
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  const body = await request.json();
  // Proses data...
  return NextResponse.json(
    { success: true, message: 'Pesan terkirim!' },
    { status: 200 }
  );
}

💡 Analogi: Server Components itu seperti restoran dengan chef yang sudah masak sebelum tamu datang. Makanan sudah siap (HTML sudah lengkap), tamu tinggal makan (client tinggal render). Client Components seperti self-service buffet — tamu bisa ambil dan campur sendiri (interaksi dinamis).

Selain fetch() biasa, Next.js punya strategi caching yang bisa kamu atir. Pahami bedanya SSG, SSR, dan ISR biar nggak bingung pas production nanti.

  );
}
