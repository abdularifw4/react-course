01

Routing (App Router)

Fundamental · 8 menit baca

Next.js App Router menggunakan folder-based routing. Setiap folder di dalam app/ menjadi route, dan file page.tsx di dalamnya menjadi halaman yang di-render. Ini jauh lebih intuitif dibanding Pages Router yang lama.

Dynamic routes menggunakan folder dengan nama diapit kurung siku: [id]/ untuk parameter dinamis. Catch-all routes menggunakan [...slug]/. Route groups menggunakan (group)/ tanpa mempengaruhi URL.

App Router juga mendukung loading.tsx (automatic loading UI), error.tsx (error boundary), not-found.tsx (404 page), dan layout.tsx (shared layout) secara built-in.

App Router — Struktur Folder
app/
├── layout.tsx          # Root layout (wajib)
├── page.tsx            # Halaman utama (/)
├── loading.tsx         # Loading UI global
├── error.tsx           # Error boundary global
├── not-found.tsx       # 404 page
├── about/
│   └── page.tsx        # /about
├── blog/
│   ├── page.tsx        # /blog
│   ├── [slug]/
│   │   └── page.tsx    # /blog/:slug (dynamic)
│   └── loading.tsx     # Loading untuk /blog/*
└── dashboard/
    ├── layout.tsx      # Layout khusus dashboard
    ├── page.tsx        # /dashboard
    └── settings/
        └── page.tsx    # /dashboard/settings

💡 Analogi: Folder-based routing seperti sistem folder di komputer. Setiap folder adalah "alamat" dan file page.tsx adalah "isi halaman" di alamat tersebut. Semakin dalam folder, semakin spesifik URL-nya.

02

Server vs Client Components

Konsep Inti · 8 menit baca

Di Next.js App Router, Server Components adalah default. Mereka di-render di server, tidak mengirim JavaScript ke client, dan bisa langsung mengakses database atau file system. Ini mengurangi bundle size dan meningkatkan performa.

Client Components ditandai dengan 'use client' di baris pertama file. Mereka di-render di browser dan bisa menggunakan useState, useEffect, event handlers, dan browser API.

Peraturan praktis: gunakan Server Components sebanyak mungkin. Hanya gunakan Client Components ketika benar-benar membutuhkan interaktivitas (onClick, onChange, useState, useEffect). Komponen Server bisa me-render Komponen Client, tapi tidak sebaliknya.

Server vs Client Components
// Server Component (default) — no directive needed
// Bisa akses database, file system, env variables langsung
async function UserProfile({ userId }: { userId: string }) {
  // Ini berjalan di server!
  const user = await db.user.findUnique({ where: { id: userId } });
  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.bio}</p>
      <LikeButton postId={userId} />  {/* Client Component */}
    </div>
  );
}

// Client Component — perlu directive
'use client';
import { useState } from 'react';

function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false);
  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? '❤️' : '💖'} Suka
    </button>
  );
}

⚠️ Perhatian: Jangan lupa 'use client' jika komponen menggunakan useState, useEffect, onClick, atau browser API. Tanpa directive ini, kamu akan mendapat error hydration.

03

Data Fetching, API Routes & SEO

Praktik · 10 menit baca

Next.js menyediakan Server-side Data Fetching yang powerful. Di Server Components, kamu bisa langsung menggunakan async/await tanpa useEffect. Data di-fetch di server dan HTML yang sudah lengkap dikirim ke client.

API Routes (Route Handlers) memungkinkan kamu membuat API endpoints di dalam proyek Next.js. File route.ts di folder app menjadi endpoint. Sangat berguna untuk webhook, form submissions, atau integrasi pihak ketiga.

Untuk SEO, Next.js menyediakan generateMetadata() untuk mengatur title, description, dan meta tags secara dinamis per halaman. Ini sangat penting untuk performa SEO.

Data Fetching & API Routes
// Server-side data fetching (di Server Component)
async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetch(
    `https://api.example.com/posts/${params.slug}`,
    { next: { revalidate: 3600 } }  // ISR: revalidate tiap 1 jam
  ).then(r => r.json());

  return <article>
    <h1>{post.title}</h1>
    <div dangerouslySetInnerHTML={{ __html: post.content }} />
  </article>;
}

// Metadata/SEO
export async function generateMetadata({ params }) {
  const post = await getPost(params.slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: { images: [post.coverImage] },
  };
}

// API Route (app/api/contact/route.ts)
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  const body = await request.json();
  // Proses data...
  return NextResponse.json(
    { success: true, message: 'Pesan terkirim!' },
    { status: 200 }
  );
}

💡 Analogi: Server Components itu seperti restoran dengan chef yang sudah masak sebelum tamu datang. Makanan sudah siap (HTML sudah lengkap), tamu tinggal makan (client tinggal render). Client Components seperti self-service buffet — tamu bisa ambil dan campur sendiri (interaksi dinamis).
