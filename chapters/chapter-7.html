NEXT.JS — KENAPA BUKAN CREATE REACT APP?

Pengenalan · 10 menit baca

React itu library — dia cuma handle UI. Buat routing? Kamu harus install React Router. Buat data fetching? Install TanStack Query. Buat SEO? Install react-helmet. Tiap project jadi puzzle yang harus kamu rakit sendiri. Capek kan?

Next.js itu kayak smartphone vs React yang kayak komponen elektronik. Next.js udah jadi produk jadi — tinggal pakai. React kamu harus solder sendiri. Next.js ngasih routing, data fetching, SEO, optimization, deployment — semua dalam satu package. Oh ternyata gitu!

Yang bikin Next.js spesial: Server Components. Ini fitur revolusioner yang nge-bedain Next.js dari framework lain. Komponen bisa jalan di server (nggak ada JavaScript ke client), yang artinya page load super cepat, SEO auto bagus, dan bundle size kecil. User cuma download JavaScript yang benar-benar butuh interaktivitas.

Versi Next.js sekarang udah 14+ dan sangat mature. Vercel (company di belakang Next.js) actively develop dan punya deployment platform yang seamless. Deploy Next.js ke Vercel cuma satu command: vercel deploy. Auto HTTPS, auto CDN, auto preview per PR. Developer experience-nya top tier.

Tapi Next.js bukan buat semua project. Kalau app kamu pure client-side ( kayak game, canvas editor, atau dashboard yang semua interaktif ), mungkin Vite + React lebih cocok. Next.js paling powerful buat content-heavy app: blog, e-commerce, landing page, SaaS.

Next.js App Router (v14+) itu paradigma baru — nggak lagi pake pages/ folder, tapi app/ folder dengan file-based routing yang lebih powerful. Layout, loading state, error boundary — semuanya otomatis berdasarkan folder structure. Developer experience-nya next level.

Getting started cuma satu command: npx create-next-app@latest. Next.js bakal nanya beberapa pertanyaan (TypeScript? ESLint? Tailwind? App Router?) — pilih Yes buat semua default. Dalam 1 menit, kamu punya project Next.js lengkap dengan TypeScript, Tailwind, dan hot reload. Nggak perlu konfigurasi manual.

SERVER vs CLIENT COMPONENTS — DUA DUNIA YANG BERBEDA

Core Concept · 12 menit baca

Ini konsep paling penting di Next.js — dan yang paling sering bikin bingung. Jadi dengarin baik-baik ya. Di Next.js App Router, ada dua jenis komponen: Server Components (default) dan Client Components (pake "use client").

Analoginya: Server Components itu kayak dapur restoran — masak di belakang, hasilnya udah jadi pas sampai ke meja. Client Components itu kayak barbekyu di meja — kamu masak sendiri di depan customer. Dua-duanya penting, tapi tempatnya beda. Oh ternyata gitu!

Server Components jalan di server saat request. Nggak ada JavaScript yang dikirim ke browser untuk komponen ini. Cocok buat: fetch data, akses database, read file system, render static HTML. Keuntungannya: page load cepat, SEO bagus, bundle kecil, aman (secret nggak exposed ke client).

Client Components jalan di browser. Butuh JavaScript. Cocok buat: interaktivitas (onClick, onChange), hooks (useState, useEffect), browser API (localStorage, geolocation), event listeners. Keuntungannya: user bisa interact langsung tanpa request ke server.

Satu kesalahan umum: taruh "use client" di layout atau page-level. Ini bikin SEMUA child component jadi Client Component — kehilangan benefit Server Component. Solution? Taruh "use client" di leaf component (komponen paling dalam yang butuh interaktivitas). Parent tetap Server Component.

Aturan emas: mulai dari Server Component, turun ke Client Component kalau butuh interaktivitas. Jangan taruh "use client" di semua file — itu kebablasan. Server Component bisa render Client Component sebagai child. Client Component nggak bisa import Server Component langsung (kalau di-import, si Server Component ikut jadi client). Tapi Server Component bisa dilempar ke Client Component sebagai children/prop dari parent Server Component.

Best practice: composition pattern. Server Component fetch data, terus pass ke Client Component sebagai prop (JSX). Client Component fokus handle interaktivitas. Data-nya udah ada — nggak perlu loading state di client. Pattern ini bikin client bundle kecil dan page load cepat.

Server Component — Default
// ✅ Server Component (default — nggak perlu "use client")
// File: app/page.tsx

import { db } from '@/lib/db';
import { ProductCard } from '@/components/ProductCard';
import { AddToCartButton } from '@/components/AddToCartButton';

// Data fetching langsung di komponen — nggak perlu useEffect!
async function getProducts() {
  const products = await db.product.findMany();
  return products;
}

export default async function HomePage() {
  const products = await getProducts();

  return (
    <main>
      <h1>Produk Terbaru</h1>
      {products.map(product => (
        <div key={product.id}>
          <ProductCard product={product} />        {/* ✅ Server Component */}
          <AddToCartButton productId={product.id} /> {/* ✅ Client Component */}
        </div>
      ))}
    </main>
  );
}
Client Component — Interactive
// ✅ Client Component — WAJIB "use client" di baris pertama
// File: components/AddToCartButton.tsx

'use client';

import { useState } from 'react';

export function AddToCartButton({ productId }: { productId: string }) {
  const [added, setAdded] = useState(false);
  const [loading, setLoading] = useState(false);

  const handleAdd = async () => {
    setLoading(true);
    await fetch('/api/cart', {
      method: 'POST',
      body: JSON.stringify({ productId }),
    });
    setAdded(true);
    setLoading(false);
  };

  return (
    <button onClick={handleAdd} disabled={loading}>
      {loading ? 'Menambahkan...' : added ? '✅ Udah di Keranjang' : 'Tambah ke Keranjang'}
    </button>
  );
}

Error paling sering: Lupa "use client" di komponen yang pake hooks atau event handler. Error-nya: useState is not a client component. Solution? Tambah 'use client' di baris pertama file tersebut.

03

ROUTING — FILE SYSTEM YANG CERDAS

Routing · 12 menit baca

Di Next.js, kamu nggak perlu konfigurasi router. Cukup bikin folder dan file — Next.js otomatis bikin route berdasarkan struktur folder. app/about/page.tsx = route /about. Sederhana kan?

Yang bikin powerful: dynamic routes. Pake bracket notation — [slug], [id]. Route-nya jadi dinamis: /blog/hello-world, /product/123. Parameter-nya accessible via params prop di page component.

Layout itu wrapper yang persist di semua child routes. Root layout (app/layout.tsx) wrap SEMUA page. Nested layout (app/blog/layout.tsx) cuma wrap page di folder blog. Layout nggak re-render pas navigasi — jadi sidebar, header, footer tetap ada tanpa flash. Oh ternyata gitu!

Untuk navigasi antar halaman, Next.js punya <Link> component. Dia handle prefetching otomatis — pas kamu hover link, Next.js udah fetch data di background. Jadi pas klik, page langsung tampil. Nggak ada loading spinner. Ini yang bikin Next.js app terasa secepat native app.

Yang juga powerful: parallel routes dan intercepting routes. Parallel routes load multiple page di same URL secara bersamaan — cocok buat dashboard dengan panel samping. Intercepting routes buat "modal routing" — buka content di modal tanpa pindah halaman. Ini fitur advanced yang jarang dipake tapi super powerful pas butuh.

Special files yang auto-kerja: loading.tsx (skeleton UI pas loading), error.tsx (error boundary), not-found.tsx (404 page), page.tsx (route handler). Semuanya based on convention — nggak perlu config.

Dynamic Route dengan Params

      <p>{post.content}</p>
      <span>Published: {new Date(post.createdAt).toLocaleDateString('id-ID')}</span>
    </article>
  );
}

04

DATA FETCHING — NGGAK PERLU USEEFFECT LAGI

Data · 12 menit baca

Di React biasa, data fetching pake useEffect + useState. Result? Waterfall loading — browser download JS, render empty UI, fetch data, re-render dengan data. User lihat loading spinner dulu, baru lihat konten. Nggak optimal.

Di Next.js Server Components, kamu bisa await langsung di komponen! Data di-fetch di server, HTML udah lengkap pas sampai ke browser. User langsung lihat konten — no loading spinner, no waterfall. Ini game changer buat UX. Oh ternyata gitu!

Di real project, Server Component biasanya nggak fetch dari external API — mereka langsung query database via ORM kayak Prisma atau Drizzle. Ini lebih efisien: nggak ada HTTP overhead, type-safe query, dan bisa langsung akses environment variable tanpa expose ke client. Pattern-nya: Prisma client di lib/db.ts, import di Server Component, query langsung. Clean dan efisien.

Tapi Server Component fetching bukan buat semua kasus. Kalau data-nya real-time (chat, stock price), butuh Client Component + SWR/React Query. Kalau data-nya user-specific (dashboard auth), gabung server fetch + client revalidation.

Next.js juga punya caching strategy built-in: fetch default cache (ISR-like), no-store buat always fresh, revalidate buat periodic refresh. Pattern yang recommended: Server Component fetch buat initial data, Client Component + SWR buat real-time update. Dua pattern ini bisa kombinasikan — Server Component render initial data, Client Component mount dan mulai polling buat update. User langsung lihat data, tapi data tetap fresh. Best of both worlds!

Nggak perlu library external buat caching — Next.js handle semuanya.

Server Component — Direct Fetch
// ✅ Direct await di Server Component — super clean!
// File: app/products/page.tsx

async function getProducts() {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 3600 } // cache 1 jam, auto-refresh
  });
  if (!res.ok) throw new Error('Failed to fetch');
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return (
    <div>
      <h1>Produk Kami ({products.length} items)</h1>
      <div className="grid grid-cols-3 gap-4">
        {products.map((p: any) => (
          <div key={p.id}>
            <img src={p.image} alt={p.name} />
            <h3>{p.name}</h3>
            <p>Rp {p.price.toLocaleString('id-ID')}</p>
          </div>
        ))}
      </div>
    </div>
  );
}
Client Component — SWR Pattern
// ✅ Client Component + SWR buat real-time data
// File: components/RealtimeStock.tsx

'use client';

import useSWR from 'swr';

const fetcher = (url: string) => fetch(url).then(r => r.json());

export function RealtimeStock({ productId }: { productId: string }) {
  const { data, error, isLoading } = useSWR(
    `/api/stock/${productId}`,
    fetcher,
    { refreshInterval: 5000 } // poll setiap 5 detik
  );

  if (isLoading) return <span>Loading...</span>;
  if (error) return <span>Error</span>;

  return (
    <span className={data.stock < 10 ? 'text-red-500' : 'text-green-500'}>
      Stok: {data.stock} {data.stock < 10 && '— Hampir habis!'}
    </span>
  );
}

Kapan pilih mana: Server fetch → static/semi-static data (blog, product catalog, profile). Client fetch (SWR/React Query) → real-time data, user-specific data, data yang sering berubah. Bisa juga gabung: server fetch buat initial load, client revalidate buat update berikutnya.

METADATA & SEO — AUTO RANKING #1 GOOGLE

SEO · 10 menit baca

React biasa itu SPA — single HTML file yang isinya kosong <div id="root">. Semua content di-render oleh JavaScript di browser. Google bot bisa baca, tapi butuh waktu — dan kadang skip JavaScript altogether. Result? SEO buruk.

Next.js beda — tiap page udah punya HTML lengkap di server. Google bot langsung baca konten, title, meta description, structured data. Nggak perlu tunggu JavaScript execute. SEO auto bagus tanpa effort extra. Oh ternyata gitu!

Metadata bisa didefinisikan via generateMetadata() (dynamic, buat halaman berbeda-beda tiap request) atau static metadata export (fixed per page). Next.js otomatis convert ke proper <head> tags.

Contoh nyata: blog post yang share ke LinkedIn. Tanpa Open Graph, preview-nya cuma URL kosong — nggak ada gambar, judul, atau deskripsi. Dengan OG metadata yang bener, preview-nya kaya dan menarik. Ini bisa meningkatkan CTR (click-through rate) secara signifikan. SEO bukan cuma Google — social media juga penting!

Yang sering dilupakan: Open Graph buat social media preview (Facebook, Twitter, LinkedIn), JSON-LD structured data buat Google rich snippets, dan sitemap buat crawl efficiency. Next.js punya semua built-in — tinggal pake.

JSON-LD structured data itu senjata rahasia SEO. Dengan markup yang bener, Google bisa tampilin rich snippets: star rating, price, FAQ accordion, breadcrumb. Ini langsung meningkatkan visibility di search results. Next.js bisa generate JSON-LD dari Server Component — data langsung dari database, nggak perlu manual update. Automation = consistency.

MIDDLEWARE, CACHING & ERROR UI

Advanced · 11 menit baca

Tiga fitur Next.js yang sering diabaikan padahal powerful banget. Middleware buat auth redirect dan geo-routing, caching buat performance, error boundary buat graceful failure. Mari bahas satu-satu.

Middleware jalan sebelum request sampai ke page. Cocok buat: auth check (redirect ke login kalau belum login), A/B testing, geo-based routing, rate limiting. Middleware itu bukan komponen — dia function yang jalan di server (Edge runtime) sebelum request nyampe ke page, jadi emang nggak ada hooks/event handler di situ. Tapi dia bisa access cookies, headers, dan request URL — cukup buat sebagian besar use case auth dan routing.

Edge Runtime — middleware jalan di edge (CDN), bukan di server Node.js. Artinya super cepat, latency rendah, dan bisa nge-handle jutaan request. Tapi ada limitation: nggak bisa akses Node.js API kayak fs atau native modules. Buat logic yang simple (auth, redirect), ini lebih dari cukup.

Caching di Next.js App Router udah built-in. fetch default-nya cache (dengan revalidate). Kamu bisa override per-request: no-store buat fresh data, revalidate: 60 buat auto-refresh tiap 60 detik. Ini menggantikan getServerSideProps + getStaticProps dari Pages Router. Oh ternyata gitu!

Error UI lewat error.tsx — komponen khusus yang nampilin pas ada error. Dengan pattern ini, error di satu bagian nggak nge-crash seluruh halaman. Error boundary paling dekat yang handle — jadi error di blog post tetap tampilin header dan sidebar. User bisa navigate ke halaman lain tanpa refresh.

Bisa nested (blog error, dashboard error) atau global. Yang keren: error.tsx bisa jadi Client Component jadi bisa punya "Try Again" button. Satu lagi: loading.tsx. Ini skeleton UI yang auto-tampil pas Next.js fetching data. Nggak perlu state management buat loading — cukup bikin loading.tsx di folder yang sama dengan page.tsx. React Suspense otomatis wrap dan tampilin skeleton. User experience-nya smooth banget — nggak ada blank page, cuma skeleton yang smooth fade in.

Auto-wrap error boundary tanpa install apapun.

Middleware — Auth Check
// File: middleware.ts (di ROOT project, bukan di app/)

import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const token = request.cookies.get('auth-token');
  const isProtected = request.nextUrl.pathname.startsWith('/dashboard');

  if (isProtected && !token) {
    // Redirect ke login, simpan original URL
    const loginUrl = new URL('/login', request.url);
    loginUrl.searchParams.set('callbackUrl', request.nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }

  return NextResponse.next();
}

// Konfigurasi matcher — middleware cuma jalan di route tertentu
export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*'],
};

SERVER ACTIONS — BIKIN FORM TANPA API

Forms · 11 menit baca

Dulu bikin form di React: handleSubmit → fetch ke API endpoint → handle response. Backend-nya harus bikin route terpisah buat handle POST. Banyak boilerplate. Next.js Server Actions nge-simplify semua ini. Kamu bisa langsung panggil server function dari form — nggak perlu API endpoint.

Analoginya: Server Actions itu kayak pesan langsung ke dapur vs lewat waiter (API). Lebih cepat, nggak perlu middleware. Form submit → langsung ke server function → return result. Clean, simple, dan type-safe. Oh ternyata gitu!

Server Actions itu async function yang jalan di server. Bisa didefinisikan di Server Component atau file terpisah. Dipanggil via action prop di <form> atau via formAction di hook. Result bisa di-access via useActionState (React 19).

Yang powerful: Server Actions bisa return data (bukan cuma void), bisa throw error (auto-catch oleh error boundary), bisa revalidate cache (auto-update UI pas data berubah). Ini menggantikan pola lama: API route + fetch + state management. Server Actions juga support progressive enhancement — kalau JavaScript disabled di browser, form tetap bisa submit (dengan full page reload). Ini yang bikin Next.js form lebih resilient dari form React biasa. Accessibility + reliability = win-win.

Satu function, semua handled.

Server Action — Form Handling
// File: app/actions.ts — Server Actions di file terpisah
'use server';

import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string;
  const content = formData.get('content') as string;

  if (!title || !content) {
    return { error: 'Title dan content wajib diisi!' };
  }

  // Simpan ke database
  await db.post.create({ data: { title, content } });

  // Revalidate cache + redirect
  revalidatePath('/blog');
  redirect('/blog');
}

Security: Server Actions BUKAN client-side function. Mereka jalan di server — jadi secret (API key, database URL) tetap aman. Tapi tetap validasi input! Client bisa manipulasi formData sebelum dikirim. Selalu server-side validation. Plus, Server Actions bisa di-pair dengan useActionState (React 19) buat handle loading, success, dan error state — mirip useState tapi khusus buat form. Tapi tetap validasi input! Client bisa manipulasi formData sebelum dikirim. Selalu server-side validation.

LEVEL SENIOR: RSC, RENDERING STRATEGY & CACHING — MODEL MENTAL YANG BENER

Level Senior · 16 menit baca

Di lesson 2 lo udah kenalan sama Server vs Client Components. Sekarang kita naikin ke level yang misahin developer yang "bisa nempel 'use client'" sama yang ngerti kenapa dan di mana batas itu ditaruh. Ini materi yang paling sering ditanya di interview senior Next.js, dan paling sering bikin bug halus di production. Karena rahasianya: RSC bukan cuma "React di server" — dia model eksekusi yang beda, dengan aturan yang beda.

Pertama, luruskan model mentalnya: di App Router, komponen lo jalan di dua lingkungan yang beda secara fisik. Server Component di-eksekusi di server (saat build atau saat request), hasilnya bukan HTML biasa tapi RSC Payload — deskripsi UI yang di-stream ke browser. Client Component di-hydrate di browser: kode JavaScript-nya beneran dikirim, di-download, di-jalanin ulang biar interaktif. Analoginya kayak katalog cetak vs mesin ATM: katalog (Server Component) udah jadi pas dicetak, isinya diam tapi ringan dan cepet dibagikan; ATM (Client Component) harus ada mesin beneran di lokasi karena user perlu mencet tombol dan dapet respons langsung.

Kedua — dan ini yang bikin bingung — apa yang BISA dan NGGAK BISA di tiap lingkungan. Server Component boleh: async/await di body-nya, query database langsung, baca file, pakai secret dari process.env. Tapi NGGAK boleh: useState, useEffect, event handler (onClick), atau browser API (localStorage, window). Client Component kebalikannya: boleh hooks & interaktivitas, tapi NGGAK boleh async component, dan jangan sekali-sekali akses secret di sana (bakal ke-bundle ke browser). Aturan mati yang gampang diinget: butuh interaktivitas atau browser API → Client; butuh data/secret/berat → Server.

Ketiga, batas 'use client' itu satu arah, dan menular ke bawah. Begitu satu file ditandain 'use client', semua komponen yang di-import dari situ ikut masuk client bundle. Makanya taruh 'use client' di leaf (daun paling dalam yang butuh interaktivitas), bukan di root. Trik senior: kalau Client Component butuh nampilin sesuatu yang berat/statis, jangan di-import — terima dia sebagai children prop. Server Component render children-nya di server, Client Component cuma "membungkus" tanpa perlu tau isinya. Ini pola "Client wrapper, Server content" yang bikin bundle tetap kecil.

Keempat, empat strategi rendering — kapan pakai yang mana. Ini keputusan arsitektur, bukan tebak-tebakan:

  • •SSG / Static (default) — halaman di-render sekali saat build, disajikan sebagai HTML statis dari CDN. Paling cepet & murah. Buat konten yang sama untuk semua orang dan jarang berubah: landing page, docs, halaman "Tentang Kami". Pakai generateStaticParams buat pre-render dynamic route.
  • •ISR (Incremental Static Regeneration) — statis, tapi di-refresh di background tiap interval. Set lewat fetch(url, { next: { revalidate: 60 } }) atau export const revalidate = 60. Buat konten yang berubah tapi nggak perlu real-time: katalog produk, daftar artikel blog. Best of both: kecepatan statis + data yang cukup segar.
  • •SSR / Dynamic — di-render ulang di server tiap request. Trigger otomatis kalau lo pakai cookies(), headers(), atau fetch(..., { cache: 'no-store' }). Buat data yang personal atau selalu segar: dashboard user, halaman yang gantung ke sesi login.
  • •CSR (Client-side) — data diambil di browser (Client Component + SWR/React Query). Buat yang beneran real-time atau super-interaktif: chat, harga saham, editor, dashboard yang polling. Sering dikombinasi: Server render kerangka + data awal, Client ambil alih buat update.

Kelima, caching di App Router itu berlapis — dan salah paham di sini bikin bug "kok datanya nggak update?". Ada empat lapis yang perlu lo tau: (1) Request Memoization — fetch yang sama dalam satu render di-dedup otomatis, jadi aman manggil getUser() di banyak komponen. (2) Data Cache — hasil fetch di-simpan lintas request; dikontrol revalidate/no-store dan bisa di-bust manual pakai revalidatePath/revalidateTag. (3) Full Route Cache — HTML + RSC payload halaman statis di-cache di build/server. (4) Router Cache — cache di sisi client biar navigasi antar halaman instan. Catatan penting: di Next 14 fetch default-nya di-cache, tapi di Next 15 default-nya berubah jadi uncached (harus opt-in caching) — makanya selalu cek versi & eksplisit soal caching.

Keenam, streaming & Suspense boundary — senjata buat UX yang mulus. Daripada nunggu SEMUA data siap baru kirim halaman, App Router bisa stream HTML bertahap. File loading.tsx itu sebenarnya cuma <Suspense> boundary otomatis di level segmen route: bagian yang cepet langsung tampil, bagian yang lambat diganti fallback dulu, lalu "diisi" pas datanya nyampe. Lo juga bisa bikin Suspense boundary manual di dalam halaman — bungkus komponen yang lambat pakai <Suspense fallback={...}> biar sisanya nggak ke-block. Ini yang bikin app terasa cepet walau salah satu data-nya lelet. Playground di bawah nge-demo rasa loading→ready ini pakai React biasa.

Pilih strategi rendering — deklaratif per route/fetch
// 1) SSG — statis, di-generate saat build (default kalau nggak ada API dinamis)
export default async function AboutPage() {
  return <h1>Tentang Kami</h1>;
}

// 2) ISR — statis tapi auto-refresh tiap 60 detik
async function getProducts() {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 60 },   // ← inti ISR
  });
  return res.json();
}

// 3) SSR — dinamis, di-render ulang tiap request (karena baca cookies)
import { cookies } from 'next/headers';
export default async function Dashboard() {
  const session = (await cookies()).get('session'); // ← bikin route jadi dynamic
  const data = await fetch('https://api.example.com/me', { cache: 'no-store' });
  // ...
}

// Paksa perilaku di level route (opsional, eksplisit):
export const dynamic = 'force-static';   // atau 'force-dynamic'
export const revalidate = 3600;          // ISR sejam
Streaming + Suspense boundary manual di dalam halaman
// app/dashboard/page.tsx — Server Component

  return (

Daripada cuma baca teori, rasain sendiri. Playground pertama nge-demo pola loading → ready — persis rasa yang dikasih loading.tsx / Suspense fallback, tapi pakai React state biasa (karena Next.js butuh server, nggak bisa jalan di iframe ini). Klik "Muat ulang" buat lihat skeleton muncul lagi.

Playground kedua nge-demo pola komposisi Server/Client: sebuah "server-like" component (statis, cuma nampilin data) yang render sebuah "client" component interaktif (tombol like ber-state). Di Next.js, ProductCard bakal jadi Server Component dan LikeButton bakal ditandai 'use client'. Data-nya ("di-fetch di server") di-pass sebagai prop — client-nya nggak perlu loading state.

OH TERNYATA GITU! Kode Next.js asli (Server Component async, await fetch, cookies()) nggak bisa jalan di playground ini — dia butuh server Node.js & App Router runtime. Playground di atas sengaja pakai React biasa buat nunjukin bagian React-nya: loading state & pola komposisi. Konsep server-nya tetap di code block statis. Jalanin Next.js beneran lewat npm run dev di komputer lo.

JANGAN TERJERAT! "Server Component" bukan berarti "selalu SSR tiap request". Secara default Server Component itu di-render saat build (statis) — dia baru jadi dinamis kalau lo pakai API dinamis (cookies(), headers(), searchParams, atau cache: 'no-store'). Banyak orang ngira semua Server Component jalan tiap request dan panik soal performa — padahal defaultnya justru statis & super cepet.

KESALAHAN UMUM — DOSA NEXT.JS APP ROUTER YANG PALING SERING

Enam kesalahan yang paling sering muncul di code review — pola: kode salah → kenapa → kode benar

1. 'use client' di mana-mana (bunuh benefit RSC). Ini dosa nomor satu. Karena "biar aman dari error", banyak yang nempel 'use client' di root layout atau page-level. Efeknya: SEMUA komponen anak masuk client bundle, data fetching pindah ke useEffect, SEO turun, bundle membengkak — lo kehilangan alasan utama pakai Next.js. Taruh 'use client' di leaf yang beneran interaktif aja.

Kesalahan #1 — 'use client' di page-level
// ❌ SALAH — page ditandai client cuma karena ADA satu tombol
'use client';
import { useState } from 'react';
export default function Page() {
  const [open, setOpen] = useState(false);
  // seluruh list produk ikut ke client bundle, harus fetch via useEffect 😩
  return <div>...ratusan item + satu tombol...</div>;
}

// ✅ BENAR — page tetap Server Component, isolasi interaktivitas ke leaf
// app/page.tsx (Server Component, bisa await data)
export default async function Page() {
  const produk = await getProduk();          // fetch di server, no useEffect
  return (
    <div>
      {produk.map(p => <Card key={p.id} p={p} />)}
      <Toggle />   {/* ← cuma komponen INI yang 'use client' */}
    </div>
  );
}

2. Fetch data pakai useEffect padahal bisa di Server Component. Refleks dari React lama: useEffect + useState buat ambil data awal. Di App Router ini bikin waterfall: browser download JS → render kosong → baru fetch → re-render. Plus butuh 'use client', plus SEO jelek. Data awal? await langsung di Server Component.

Kesalahan #2 — useEffect buat data awal
// ❌ SALAH — waterfall: render kosong dulu, baru fetch di client
'use client';
import { useEffect, useState } from 'react';
export default function Posts() {
  const [posts, setPosts] = useState([]);
  useEffect(() => { fetch('/api/posts').then(r => r.json()).then(setPosts); }, []);
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}

// ✅ BENAR — Server Component, HTML udah lengkap pas sampai browser
export default async function Posts() {
  const posts = await fetch('https://api.example.com/posts',
    { next: { revalidate: 60 } }).then(r => r.json());
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}
// Butuh useEffect fetch cuma buat data real-time/interaktif (chat, polling).

3. Salah pilih strategi data — no-store buat semuanya (atau cache buat data personal). Dua ekstrem yang sama bahayanya. Pukul rata cache: 'no-store' bikin tiap request nge-hit origin → lambat & mahal, padahal datanya jarang berubah. Sebaliknya, nge-cache data personal (profil user, isi keranjang) bikin user A lihat data user B. Cocokin strategi ke sifat datanya.

Kesalahan #3 — strategi cache asal
// ❌ SALAH — katalog yang jarang berubah dipaksa no-store (lambat & mahal)
const produk = await fetch(API + '/produk', { cache: 'no-store' });
// ❌ SALAH — data personal di-cache → bocor antar user
const cart = await fetch(API + '/cart', { next: { revalidate: 3600 } });

// ✅ BENAR — cocokin ke sifat data
const produk = await fetch(API + '/produk', { next: { revalidate: 300 } }); // ISR
const cart   = await fetch(API + '/cart',   { cache: 'no-store' });          // personal
// Rule of thumb: sama-untuk-semua & jarang berubah → cache/ISR;
//                personal / real-time → no-store.

4. Pakai <img> biasa, bukan next/image. <img> mentah nggak di-optimize: gambar 3MB dikirim apa adanya, nggak ada lazy-load, nggak ada WebP/AVIF, dan bikin layout shift (CLS jelek) karena browser nggak tau dimensinya. next/image auto-resize, auto-format modern, lazy-load, dan reserve ruang lewat width/height.

Kesalahan #4 — img mentah
// ❌ SALAH — gambar gede dikirim apa adanya, layout shift, no lazy-load
<img src="/hero.png" alt="Hero" />

// ✅ BENAR — next/image: auto WebP/AVIF, resize, lazy-load, no CLS
import Image from 'next/image';
<Image src="/hero.png" alt="Hero" width={1200} height={600} priority />
// width/height (atau fill) WAJIB → biar Next reserve ruang & hindari layout shift.

5. Bocorin server secret ke Client Component. Variabel tanpa prefix NEXT_PUBLIC_ cuma ada di server. Tapi begitu lo pakai/import-nya di file 'use client', atau lo kasih prefix NEXT_PUBLIC_ ke secret, nilainya ikut ke-bundle & kelihatan di browser. API key, DB URL, token — jangan pernah disentuh di client.

Kesalahan #5 — secret bocor ke client
// ❌ SALAH — secret dipakai di Client Component → ke-bundle ke browser
'use client';
const key = process.env.STRIPE_SECRET_KEY;        // undefined di client / bocor kalau NEXT_PUBLIC_
fetch('https://api.pihak3.com', { headers: { Authorization: key } });

// ❌ SALAH — kasih prefix NEXT_PUBLIC_ ke secret = sengaja mbocorin
// NEXT_PUBLIC_DB_PASSWORD=... ← JANGAN

// ✅ BENAR — secret dipakai di server (Server Component / Server Action / route handler)
'use server';
export async function bayar() {
  const key = process.env.STRIPE_SECRET_KEY;      // aman, cuma di server
  // ...
}
// NEXT_PUBLIC_ cuma buat yang MEMANG boleh publik (mis. URL analytics).

6. Salah pakai dynamic route params (lupa await di Next 15). Di App Router versi baru, params dan searchParams itu Promise — harus di-await. Baca params.slug langsung tanpa await bikin undefined. Salah lain: ngira params ada di komponen sembarang — dia cuma di-inject ke page.tsx/layout.tsx segmen itu.

Kesalahan #6 — params nggak di-await
// ❌ SALAH — params dibaca langsung (Next 15: params itu Promise) → slug undefined
export default function Post({ params }: { params: { slug: string } }) {
  const post = getPost(params.slug);   // 💥 params.slug undefined / warning
  return <h1>{post.title}</h1>;
}

// ✅ BENAR — params/searchParams itu Promise, WAJIB di-await
type Props = { params: Promise<{ slug: string }> };
export default async function Post({ params }: Props) {
  const { slug } = await params;       // ← await dulu
  const post = await getPost(slug);
  return <h1>{post.title}</h1>;
}

LATIHAN — UJI PEMAHAMAN LO

Jawab dulu di kepala sebelum buka solusinya. Ini gaya soal yang beneran keluar di code review & interview Next.js.

Latihan 1: Server atau Client? Ambil keputusan klik untuk lihat solusi

Buat tiap komponen, tentuin: Server Component (default) atau butuh 'use client'? A. Daftar artikel dari database. B. Tombol "like" dengan counter. C. Dropdown menu yang buka-tutup. D. Footer statis berisi link. E. Form pencarian yang nyimpen input ke state. F. Kartu profil yang nampilin nama & foto dari API.

Jawaban: A → Server (fetch/DB, nggak ada interaktivitas). B → Client (state + onClick). C → Client (state buka/tutup). D → Server (statis murni — jangan client-in cuma karena "ada link"). E → Client (nyimpen input ke state + event). F → Server (cuma nampilin data). Pola: ada useState/onClick/browser API → Client. Cuma nampilin data → Server. Kalau ragu, mulai dari Server, turun ke Client cuma di leaf yang butuh.

Latihan 2: Cari bug — kenapa datanya nggak pernah update? klik untuk lihat solusi
Soal — harga produk di admin udah diubah, tapi halaman publik tetap harga lama
// app/produk/page.tsx
export default async function Produk() {
  const res = await fetch('https://api.toko.com/produk'); // default-nya gimana?
  const produk = await res.json();
  return <List produk={produk} />;
}

Bug-nya: di Next 14, fetch default-nya di-cache permanen (statis) — halaman di-generate sekali saat build dan nggak pernah refresh, jadi update harga di admin nggak keliatan. (Di Next 15 defaultnya kebalik: uncached, tapi bisa ke-cache lagi lewat Full Route Cache kalau route-nya statis.) Solusinya: kasih tau Next kapan harus revalidate.

Solusi
// Opsi A — ISR: refresh otomatis tiap 60 detik (cocok buat katalog)
const res = await fetch('https://api.toko.com/produk', {
  next: { revalidate: 60 },
});

// Opsi B — selalu segar tiap request (kalau harga wajib real-time)
const res = await fetch('https://api.toko.com/produk', { cache: 'no-store' });

// Opsi C — on-demand: panggil ini dari admin setelah edit harga
import { revalidatePath } from 'next/cache';
revalidatePath('/produk'); // bust cache TEPAT saat data berubah
Latihan 3: Pilih strategi rendering klik untuk lihat solusi

Tentuin strategi (SSG / ISR / SSR / CSR) yang paling pas: A. Halaman dokumentasi produk. B. Daftar artikel blog yang di-update beberapa kali sehari. C. Dashboard saldo & transaksi user yang login. D. Widget harga crypto yang update tiap detik.

Jawaban: A → SSG (sama untuk semua, jarang berubah — generate saat build, sajikan dari CDN). B → ISR (revalidate mis. 3600 — statis tapi cukup segar). C → SSR (personal & gantung sesi login → cookies()/no-store, render tiap request). D → CSR (real-time, Client Component + polling/WebSocket; boleh Server render kerangka awal). Prinsip: makin personal/sering-berubah datanya, makin geser ke kanan (SSG → ISR → SSR → CSR).

CHALLENGE — MINI "ROUTER" PAKAI STATE

Next.js routing butuh server, jadi di sini kita latih logika di baliknya pakai React biasa: sebuah "router" yang milih konten berdasarkan halaman aktif — mirip cara App Router mencocokkan URL ke page.tsx. Starter baru nangani tab Home; tugas lo: lengkapi biar tab Blog dan About nampilin konten yang bener. Klik ▶ Test buat ngecek — starter-nya pasti gagal di test Blog & About.

Solusi Challenge — buka kalau udah nyerah (atau mau ngecek) klik untuk lihat solusi

Cukup tambahin dua cabang di renderPage() — persis kayak App Router yang mencocokkan tiap segmen URL ke satu page.tsx. Sisa yang nggak cocok jatuh ke 404 (mirip not-found.tsx).

Solusi renderPage()
function renderPage() {
  if (page === 'home')  return <p data-testid="content">Selamat datang di Home</p>;
  if (page === 'blog')  return <p data-testid="content">Daftar artikel Blog</p>;
  if (page === 'about') return <p data-testid="content">Tentang kami</p>;
  return <p data-testid="content">404 — halaman nggak ketemu</p>;
}
