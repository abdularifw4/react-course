Chapter 5

Advanced React Patterns

Pernah ngerasa nulis component yang sama berulang-ulang? 🤯 Nah, chapter ini bakal bikin kamu ngeh “oh ternyata gitu caranya!” — kita bakal bahas pola-pola React yang dipake pro di production.

🧥 HIGHER-ORDER COMPONENTS (HOC)

Bayangin kamu punya jaket anti-apa. Kamu bisa pakai jaket itu ke mana aja — ke kantor, ke mall, ke gym. Jaket-nya nambahin fungsi (perlindungan) tanpa ngubah badan kamu di dalemnya.

Nah, HOC itu persis kayak jaket. Dia itu fungsi yang nge-wrap component kamu, terus ngereturn component baru yang udah ditambahin “kekuatan” — bisa auth, loading state, tracking, atau apapun.

Kapan kita butuh HOC? Pas kamu ngerasa “Eh, ini logic ngecek auth-nya aku copy-paste ke 10 component nih”. Stop! 🛑 Itu tandanya kamu butuh HOC.

// HOC: withAuth — jaket keamanan buat component kamu
function withAuth(Component) {
  return function AuthenticatedComponent(props) {
    const { user, isLoading } = useAuth();

    if (isLoading) return <Spinner />;
    if (!user) return <Navigate to='/login' />;

    return <Component {...props} user={user} />;
  };
}

// Tinggal pakai jaket-nya ke component mana aja!
const ProtectedDashboard = withAuth(Dashboard);
const ProtectedSettings = withAuth(Settings);
const ProtectedProfile = withAuth(Profile);

// HOC: withLoading — jaket loading buat data fetching
function withLoading(Component, fetchDataFn) {
  return function EnhancedComponent(props) {
    const [data, setData] = useState(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);

    useEffect(() => {
      fetchDataFn()
        .then(d => setData(d))
        .catch(e => setError(e))
        .finally(() => setLoading(false));
    }, []);

    if (loading) return <Spinner />;
    if (error) return <Error message={error.message} />;
    return <Component data={data} {...props} />;
  };
}

⚠️ Heads up! Sejak hooks muncul di React 16.8, HOC udah nggak sesering dulu dipake. Tapi jangan salah — library besar kayak react-redux dan react-router masih pake HOC (remember connect() dan withRouter()?). Jadi tetap worth to know!

🎨 RENDER PROPS — KAMU YANG MASAK!

Pernah ke restoran yang menu-nya fix? Mau pesan ayam bakar tapi level pedasnya udah ditentuin chef. Annoying kan? 🤨

Render props itu kebalikannya. Dia ngasih kamu bahan mentah (data), terus kamu sendiri yang masak (render). Jadi kamu punya kontrol penuh soal tampilan, sementara logic-nya diurus sama component yang provide data.

Kapan pake render props? Pas kamu butuh component yang logic-nya sama tapi tampilannya beda-beda. Contoh: mouse tracker yang bisa render cursor custom, atau data fetcher yang bisa nampilin data dalam bentuk card, table, atau list.

// Render Props: MouseTracker — dia cuma kasih data posisi mouse
function MouseTracker({ render }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handler = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', handler);
    return () => window.removeEventListener('mousemove', handler);
  }, []);

  // "Ini bahan-bahannya, silakan masak sendiri ya!"
  return render(position);
}

// Parent yang tentukan rendering — ini yang namanya freedom! 🆓
function App() {
  return (
    <MouseTracker render={({ x, y }) => (
      <h1>Kursor kamu di posisi: {x}, {y}</h1>
    )} />
  );
}

// Versi children prop — lebih natural lagi kayak nulis HTML biasa
function DataFetcher({ url, children }) {
  const [data, setData] = useState(null);
  useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, [url]);

  // Kirim data ke children sebagai fungsi
  return children(data);
}

// Pakai-nya enak banget, kayak compose React biasa!
<DataFetcher url='/api/users'>
  {(users) => (
    users ? users.map(u => <div key={u.id}>{u.name}</div>)
         : <p>Loading...</p>
  )}
</DataFetcher>

💡 Oh ternyata! Kamu udah sering pake render props tanpa sadar. Remember <Route> dari react-router yang pake element prop? Itu render props! Juga children as function pattern yang dipake banyak library modern. Mental model-nya gampang: “aku kasih data, kamu yang decide mau render apa.”

🧱 COMPOUND COMPONENTS — SET LEGO!

Pernah main LEGO? Satu blok sendiri nggak terlalu menarik. Tapi kalau dirangkai bareng blok-blok lain — wuis! Jadi kastil, pesawat, atau robot yang keren. 🏰

Compound components itu kerjaannya sama kayak LEGO. Beberapa component yang keliatan terpisah, tapi sebenarnya mereka berkomunikasi lewat context dan ngerjain satu tugas bareng-bareng.

Contoh paling klasik? <select> dan <option> di HTML. Mereka dua komponen beda, tapi saling ngerti. Nah kita bisa bikin kayak gitun di React!

import { createContext, useContext, useState } from 'react';

// 1. Context buat "secret channel" antar komponen
const TabsContext = createContext();

// 2. Parent (Tabs) — dia bos-nya, dia pegang state
function Tabs({ children, defaultValue }) {
  const [activeTab, setActiveTab] = useState(defaultValue);
  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab }}>
      <div>{children}</div>
    </TabsContext.Provider>
  );
}

// 3. Child components — akses state lewat context
function TabList({ children }) {
  return <div className='flex gap-2'>{children}</div>;
}

function Tab({ value, children }) {
  const { activeTab, setActiveTab } = useContext(TabsContext);
  const isActive = activeTab === value;
  return (
    <button
      onClick={() => setActiveTab(value)}
      className={isActive ? 'bg-accent text-dark-900' : 'bg-dark-700 text-gray-300'}
    >
      {children}
    </button>
  );
}

function TabPanel({ value, children }) {
  const { activeTab } = useContext(TabsContext);
  if (activeTab !== value) return null;
  return <div>{children}</div>;
}

// Pakai-nya super natural — kayak nulis HTML biasa!
function App() {
  return (
    <Tabs defaultValue='profil'>
      <TabList>
        <Tab value='profil'>Profil</Tab>
        <Tab value='settings'>Settings</Tab>
        <Tab value='notifikasi'>Notifikasi</Tab>
      </TabList>
      <TabPanel value='profil'><ProfilForm /></TabPanel>
      <TabPanel value='settings'><SettingsForm /></TabPanel>
      <TabPanel value='notifikasi'><NotifSettings /></TabPanel>
    </Tabs>
  );
}

💡 Kenapa ini keren? Karena API-nya super readable! Siapapun baca kode <Tabs><Tab /><TabPanel /></Tabs> langsung paham maksudnya. Nggak perlu baca dokumentasi panjang. Ini yang bikin library kayak Radix UI dan Headless UI populer — API-nya intuitive banget!

👨‍🍳 CONTAINER vs PRESENTATIONAL — CHEF & PELAYAN

Di restoran ada dua orang penting: chef yang masak di dapur (urus resep, bahan, teknik), dan pelayan yang sajikan makanan ke meja (urus tampilan, kerapian).

Di React, konsepnya sama:

  • 🍳 Container = komponen yang urus logic — fetch data, manage state, handle events
  • 🍽️ Presentational = komponen yang cuma nampilin UI — menerima data dari props, nggak punya state sendiri

Kenapa pisah? Supaya komponen presentational-nya bisa di-reuse. Component list user yang sama bisa dipake di halaman dashboard, halaman admin, atau halaman search — tinggal ganti container-nya!

// 🍽️ Presentational: cuma nampilin, nggak ngurus data dari mana
// Fokusnya: "kalau aku dikasih data begini, aku render gini"
function UserList({ users, onSelect, isLoading }) {
  if (isLoading) return <div>Memuat data...</div>;
  return (
    <ul className='space-y-2'>
      {users.map(user => (
        <li
          key={user.id}
          onClick={() => onSelect(user.id)}
          className='p-3 hover:bg-dark-700 rounded-lg cursor-pointer'
        >
          {user.name} — {user.email}
        </li>
      ))}
    </ul>
  );
}

// 🍳 Container: urus logic & data, terus "sajikan" ke presentational
// Fokusnya: "dari mana data-nya? apa yang terjadi kalau user klik?"
function UserListContainer() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(data => { setUsers(data); setLoading(false); });
  }, []);

  const handleSelect = (id) => {
    console.log('User dipilih:', id);
    // Navigasi, buka modal, dll — logic di sini
  };

  // Sajikan ke presentational component!
  return (
    <UserList
      users={users}
      onSelect={handleSelect}
      isLoading={loading}
    />
  );
}

⚠️ Update dunia modern: Pattern ini sekarang udah banyak digantikan sama custom hooks. Alih-alih bikin container component, kamu tinggal extract logic ke hook (misal useUsers()), terus langsung pake di presentational component. Tapi mental model-nya tetap sama: pisahin logic dan tampilan. Jangan campur aduk!

🔀 KAPAN PAKAI POLA YANG MANA?

Oke, sekarang kamu udah kenal 4 pola. Pertanyaannya: “Gue harus pake yang mana sih?” Tenang, ini cheat sheet-nya:

🧥 HOC

Pas kamu butuh nambahin behavior ke BANYAK komponen yang sama (auth guard, error boundary, analytics). Kayak jaket yang bisa dipake ke mana aja.

🎨 Render Props

Pas logic-nya sama tapi tampilannya harus fleksibel. Component-nya kasih data, parent-nya yang decide render apa. Kayak restoran self-service.

🧱 Compound Components

Pas kamu bikin “suite” komponen yang harus bekerja bareng (tabs, accordion, select). API-nya harus natural kayak nulis HTML biasa. Kayak set LEGO.

👨‍🍳 Container/Presentational

Pas kamu mau pisahin logic dan tampilan. Tapi di era hooks, lebih baik pake custom hooks daripada bikin container component terpisah.

💡 Real talk: Di dunia nyata, kamu nggak harus pake semua pola ini di setiap project. Nggak perlu force! Pake yang make sense buat kasus kamu. Yang penting kamu tahu opsi-nya ada apa aja, jadi pas ketemu masalah yang cocok, kamu langsung “Oh ini bisa pake compound components!” 🎯

📝 KALAU KAMU UDAH SAMPAI SINI...

Selamat! 🎉 Kamu udah kenal 4 pola advanced React yang dipake developer pro di production. Ini bukan teori doang — ini pola yang kamu bakal temuin di library-library besar kayak Radix UI, Headless UI, React Router, dan Redux.

Mental model yang penting: React itu soal komposisi. Semua pola ini punya tujuan yang sama — bikin kode kamu reusable, maintainable, dan enak dibaca. Jangan terlalu fokus ke “nama pattern”-nya, fokus ke problem apa yang dia solve.
