01

HOC (Higher-Order Components)

Pattern Lanjutan · 7 menit baca

Higher-Order Component (HOC) adalah pattern di mana sebuah fungsi menerima komponen sebagai argumen dan mengembalikan komponen baru yang telah ditingkatkan dengan fungsionalitas tambahan. HOC bukan komponen — ia adalah pola arsitektural untuk memisahkan logika reusable.

HOC sangat berguna untuk cross-cutting concerns seperti autentikasi, logging, analytics, atau pemuatan data. Alih-alih menulis logika yang sama berulang-ulang, kamu membungkusnya sekali dalam HOC.

Kelemahan HOC: bisa menyebabkan wrapping hell, debugging sulit, dan props collision. Di era Hooks, banyak kasus HOC bisa digantikan dengan Custom Hooks.

HOC — withAuth & withLogger
// HOC: withAuth — melindungi komponen
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const isLoggedIn = useAuth();
    if (!isLoggedIn) return <Navigate to="/login" />;
    return <WrappedComponent {...props} />;
  };
}

// HOC: withLogger — menambahkan logging
function withLogger(WrappedComponent, name) {
  return function LoggedComponent(props) {
    useEffect(() => {
      console.log(`[${name}] mounted`);
      return () => console.log(`[${name}] unmounted`);
    }, []);
    return <WrappedComponent {...props} />;
  };
}

// Penggunaan:
const ProtectedDashboard = withAuth(Dashboard);
const LoggedProfile = withLogger(Profile, 'Profile');

💡 Analogi: HOC seperti jas hujan. Baju biasa (komponen asli) dibungkus jas hujan (HOC) sehingga mendapat fungsionalitas tambahan tanpa mengubah baju di dalamnya.

02

Render Props

Pattern Lanjutan · 6 menit baca

Render Props adalah pattern di mana komponen menerima fungsi (biasanya sebagai children) yang mengembalikan elemen React. Komponen induk mengelola state/logika, konsumen mengontrol rendering.

Pattern ini lebih deklaratif dan mudah dibaca dibanding HOC. Di era modern, sebagian besar render props bisa digantikan Custom Hooks, namun tetap berguna untuk kontrol rendering penuh.

Library populer yang menggunakan render props: react-query (versi lama) dan Formik.

Render Props — MouseTracker
function MouseTracker({ children }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const handle = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener('mousemove', handle);
    return () => window.removeEventListener('mousemove', handle);
  }, []);
  return children({ x: position.x, y: position.y });
}

function App() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <div>
          <h1>Posisi: ({x}, {y})</h1>
          <div style={{ position:'absolute', left:x, top:y,
            width:20, height:20, background:'#27d890',
            borderRadius:'50%', transform:'translate(-50%,-50%)' }} />
        </div>
      )}
    </MouseTracker>
  );
}

💡 Analogi: Render Props seperti memesan makanan. Koki (komponen induk) menyiapkan bahan (state), kamu (children fungsi) yang tentukan mau diapakan bahan itu.

03

Compound Components

Pattern Lanjutan · 7 menit baca

Compound Components adalah pattern di mana beberapa komponen bekerja sama membentuk satu unit UI koheren. Mereka berbagi state via React Context tanpa manual props passing.

Contoh nyata: <select> dan <option> — tidak perlu dihubungkan manual. Compound Components membawa filosofi sama ke React.

Keuntungan: API deklaratif dan fleksibel. Radix UI dan Headless UI banyak menggunakan pattern ini.

Compound Components — Accordion
const AccordionContext = createContext();

function Accordion({ children, defaultIndex = 0 }) {
  const [activeIndex, setActiveIndex] = useState(defaultIndex);
  const toggleItem = (i) => setActiveIndex(p => p === i ? -1 : i);
  return (
    <AccordionContext.Provider value={{ activeIndex, toggleItem }}>
      <div className="accordion">{children}</div>
    </AccordionContext.Provider>
  );
}

function AccordionItem({ index, children }) {
  const { activeIndex, toggleItem } = useContext(AccordionContext);
  const isOpen = activeIndex === index;
  return (
    <div className="accordion-item">
      {typeof children === 'function'
        ? children({ isOpen, toggle: () => toggleItem(index) })
        : children}
    </div>
  );
}

function FAQ() {
  return (
    <Accordion>
      <AccordionItem index={0}>
        {({ isOpen, toggle }) => (
          <>
            <button onClick={toggle}>Apa itu React? {isOpen ? '▲' : '▼'}</button>
            {isOpen && <p>React adalah library JavaScript untuk UI.</p>}
          </>
        )}
      </AccordionItem>
    </Accordion>
  );
}

💡 Analogi: Compound Components seperti orkestra. Setiap pemain punya peran sendiri, tapi berbagi skor musik (Context) yang sama untuk bermain harmoni.

04

Provider Pattern

Pattern Lanjutan · 6 menit baca

Provider Pattern menggunakan React Context untuk menyebarkan data ke seluruh tree tanpa prop drilling. Berguna untuk data global: tema, autentikasi, bahasa, konfigurasi.

Dua bagian utama: Provider yang membungkus aplikasi dan Custom Hook untuk akses data. Ini membersihkan kode dari prop drilling berantakan.

Pisahkan Provider ke file tersendiri dengan Custom Hook-nya. Provider harus "pasif" — hanya merender children tanpa UI tambahan.

Provider Pattern — ThemeProvider
const ThemeContext = createContext(undefined);

function ThemeProvider({ children, defaultTheme = 'dark' }) {
  const [theme, setTheme] = useState(defaultTheme);
  const toggleTheme = () => setTheme(p => p === 'dark' ? 'light' : 'dark');
  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

function useTheme() {
  const ctx = useContext(ThemeContext);
  if (ctx === undefined) throw new Error('useTheme harus di ThemeProvider');
  return ctx;
}

function Navbar() {
  const { theme, toggleTheme } = useTheme();
  return (
    <nav style={{ background: theme === 'dark' ? '#111' : '#fff' }}>
      <button onClick={toggleTheme}>{theme === 'dark' ? '☀️' : '🌙'}</button>
    </nav>
  );
}

function App() {
  return (
    <ThemeProvider defaultTheme="dark">
      <Navbar /><MainContent />
    </ThemeProvider>
  );
}

⚠️ Perhatian: Jangan gunakan Context untuk data yang berubah sangat sering (posisi mouse). Setiap perubahan Context me-rerender semua konsumen.

05

Container vs Presentational

Arsitektur · 5 menit baca

Pattern Container vs Presentational memisahkan komponen: Container bertanggung jawab logika (fetch, state), Presentational fokus tampilan (menerima data via props).

Container tidak memiliki styling sendiri. Presentational tidak memiliki state dan berfungsi sebagai "template". Komunikasi terjadi melalui props.

Di era Hooks, pattern ini kurang populer karena Custom Hooks bisa memisahkan logika tanpa komponen terpisah. Tapi konsep pemisahan concern tetap relevan.

Container vs Presentational
// Presentational — fokus tampilan
function UserList({ users, isLoading, error }) {
  if (isLoading) return <p>Memuat...</p>;
  if (error) return <p>Error: {error.message}</p>;
  return <ul>{users.map(u =>
    <li key={u.id}>{u.name} — {u.email}</li>
  )}</ul>;
}

// Container — fokus logika
function UserListContainer() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  useEffect(() => {
    fetch('/api/users').then(r => r.json()).then(setUsers)
      .catch(setError).finally(() => setLoading(false));
  }, []);
  return <UserList users={users} isLoading={loading} error={error} />;
}

function App() {
  return <div><h1>Daftar User</h1><UserListContainer /></div>;
}

💡 Analogi: Container adalah chef di dapur yang menyiapkan bahan, Presentational adalah pelayan yang menyajikan makanan. Chef tidak bertemu tamu, pelayan tidak memasak.

06

Props Getters & Control Props

Pattern Lanjutan · 7 menit baca

Props Getters menyediakan fungsi yang mengembalikan objek props yang sudah dikonfigurasi. Pengguna menyebarkan props tersebut ke elemen DOM — mengurangi boilerplate dan memastikan konsistensi.

Control Props melangkah lebih jauh: komponen bisa dikontrol sepenuhnya dari luar (controlled), mengelola state sendiri (uncontrolled), atau campuran. Pattern ini memberikan fleksibilitas maksimal.

Library headless UI seperti Downshift, React Table, dan Radix UI menggunakan kedua pattern ini secara ekstensif.

Props Getters — Toggle
function useToggle({ defaultOn = false, onToggle } = {}) {
  const [isOn, setIsOn] = useState(defaultOn);
  const toggle = () => {
    const next = !isOn;
    setIsOn(next);
    onToggle?.(next);
  };

  // Props Getter — kembalikan semua props yang dibutuhkan
  const getTogglerProps = (props = {}) => ({
    'aria-pressed': isOn,
    onClick: (e) => { props.onClick?.(e); toggle(); },
    ...props,
  });

  return { isOn, toggle, getTogglerProps };
}

// Penggunaan dengan Props Getter:
function App() {
  const { isOn, getTogglerProps } = useToggle();
  return (
    <button {...getTogglerProps({
      style: { background: isOn ? '#27d890' : '#6b7280' },
      className: 'toggle-btn',
    })}>
      {isOn ? 'ON' : 'OFF'}
    </button>
  );
}

💡 Analogi: Props Getters seperti kunci pintu otomatis hotel. Kamu cukup menggesek kartu (memanggil getter), dan pintu terbuka dengan konfigurasi tepat tanpa perlu tahu mekanisme internalnya.
