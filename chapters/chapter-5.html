01

HOC — HIGHER-ORDER COMPONENTS 🧥

Pattern Dasar · 10 menit baca

Pernah denger istilah Higher-Order Component (HOC)? Namanya kedengeran intelektual banget, tapi konsepnya sebenarnya super simpel. Bayangin kamu punya jas hujan — kamu bisa pakai jas itu ke mana aja, di atas baju apa aja. Jas hujannya nambahin fungsi (anti air, hangat) tanpa ngubah baju kamu. Nah, HOC itu jas hujan buat komponen React kamu. 🧥

Secara teknis, HOC adalah fungsi yang nerima komponen dan return komponen baru. Komponen baru itu nggak cuma ngasih ulang komponen lama — dia nambahin behavior, props, atau logic tambahan. Pattern ini udah ada sejak React masih muda, dan dipake di banyak library populer kayak connect() dari Redux dan withRouter() dari React Router v4. Oh ternyata gitu! 🤯

Kenapa sih butuh HOC? Misalnya kamu punya 10 komponen yang masing-masing butuh data dari API, loading state, dan error handling. Tanpa HOC, kamu nulis logic fetch yang sama 10 kali. Dengan HOC, kamu tulis sekali — withData() — dan wrap semua komponen yang butuh. Ini prinsip DRY (Don't Repeat Yourself) yang beneran kerasa dampaknya di codebase besar.

Satu hal penting: HOC itu bukan tentang inheritance atau subclassing. Dia tentang composition — menyusun fungsi-fungsi kecil jadi behavior yang lebih kompleks. Ini filosofi React yang fundamental: composition over inheritance. Jangan kebalik ya!

Tapi HOC juga punya kelemahan yang signifikan. Pertama, dia bikin wrapper hell — komponen ke-wrap berkali-kali jadi susah dibaca. Kedua, dia bisa nimpa props secara silent kalau nggak hati-hati. Ketiga, sulit di-debug karena di DevTools komponennya jadi anonymous. Itulah kenapa di era hooks modern, banyak developer pindah ke custom hooks sebagai alternatif. Tapi tetap penting buat dipahami karena masih banyak dipake di production!

HOC = fungsi yang nerima komponen, return komponen baru
// HOC = fungsi yang nerima komponen, return komponen baru
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const isLoggedIn = useAuth();

    if (!isLoggedIn) {
      return <p>Kamu harus login dulu!</p>;
    }

    // Kirim semua props + tambahin isAuthenticated
    return <WrappedComponent {...props} isAuthenticated={isLoggedIn} />;
  };
}

// Pakainya kayak pakai jas hujan:
const ProtectedDashboard = withAuth(Dashboard);
const ProtectedSettings = withAuth(Settings);
JAVASCRIPT
import { useState, useEffect } from 'react';

// HOC yang nambahin loading & error state ke komponen mana aja
function withLoading(WrappedComponent, url) {
  return function LoadingComponent(props) {
    const [data, setData] = useState(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);

    useEffect(() => {
      setLoading(true);
      fetch(url)
        .then(res => res.json())
        .then(setData)
        .catch(setError)
        .finally(() => setLoading(false));
    }, [url]);

    if (loading) return <div className="spinner">Loading...</div>;
    if (error) return <div className="error">{error.message}</div>;

    return <WrappedComponent data={data} {...props} />;
  };
}

// Wrap komponen mana aja yang butuh fetch data:
const UserProfile = withLoading(ProfileContent, '/api/user');
const ProductList = withLoading(ProductsContent, '/api/products');

💡 HOC vs Custom Hook: Di era hooks modern, banyak use case HOC bisa diganti dengan custom hook. withLoading() jadi useFetch(), withAuth() jadi useAuth(). Custom hooks lebih readable dan nggak bikin wrapper hell. Tapi tetap pelajari HOC karena library lama masih pake pattern ini!

⚠️ Gotcha — Props Collision: Kalau HOC kamu nambahin props yang namanya sama dengan props dari parent, HOC props-nya yang menang (nimpa). Solusinya: selalu kasih prefix kayak hocIsLoading, atau documentasi yang jelas. Ini salah satu alasan kenapa HOC kurang populer sekarang.

02

RENDER PROPS — CHEF YANG KASIH MENU 🍳

Composition · 10 menit baca

Okay, sekarang kita masuk ke pattern yang lebih fleksibel. Kalau HOC itu jas hujan, Render Props itu kayak chef yang menyiapkan bahan, terus ngasih menu ke kamu buat tentuin masak apa. Chef-nya ngurus logic rumit (fetch data, tracking mouse, animate), kamu tinggal tentuin tampilannya. Keren banget kan? 🍳

Secara teknis, Render Props itu pattern di mana komponen nerima sebuah fungsi sebagai prop, dan fungsi itu return JSX. Komponen pemilik data memanggil fungsi tersebut dan ngasih data yang dibutuhkan. Ini bikin siapa pun yang ngontrol rendering — bukan komponen yang punya data, tapi komponen yang pakai data tersebut.

Contoh nyata paling terkenal: <Route> dari React Router v5 yang punya render prop. Atau <List> component dari react-virtualized yang punya rowRenderer prop. Semua ini memakai pattern Render Props! Oh ternyata udah dari dulu ya! 🤯

Keunggulan Render Props dibanding HOC: nggak ada wrapper hell, nggak ada naming conflict, dan kamu bisa langsung liat apa yang di-render. Plus, Render Props bisa share state antar komponen yang sama — misalnya dua komponen berbagi satu <MouseTracker> yang nge-track posisi mouse.

Namun Render Props juga punya kelemahan: bisa bikin callback nesting yang dalam (callback hell) kalau chain-nya panjang. Dan sejak React 16.8 dengan hooks, banyak use case Render Props bisa diganti dengan custom hooks yang lebih simple. Tapi pattern ini tetap powerful dan worth to know!

JAVASCRIPT
import { useState, useEffect } from 'react';

// Komponen yang nge-track mouse position
function MouseTracker({ children }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => {
      setPosition({ x: e.clientX, y: e.clientY });
    };
    window.addEventListener('mousemove', handleMove);
    return () => window.removeEventListener('mousemove', handleMove);
  }, []);

  // children itu fungsi! Kirim data posisi ke parent
  return children(position);
}

// Pakainya — chef kasih bahan, kamu tentuin masaknya:
function App() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <h1>Mouse: {x}, {y}</h1>
      )}
    </MouseTracker>
  );
}

// Bisa juga render hal beda:
function App2() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <div style={{ left: x, top: y, position: 'absolute' }}>
          🎯
        </div>
      )}
    </MouseTracker>
  );
}
JAVASCRIPT
function DataFetcher({ url, children }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(url)
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [url]);

  // Kirim semua state ke consumer via render prop
  return children({ data, loading, error });
}

// Dua komponen beda, satu sumber data:
function UserPage() {
  return (
    <DataFetcher url="/api/users/me">
      {({ data, loading, error }) => {
        if (loading) return <p>Loading profile...</p>;
        if (error) return <p>Error: {error.message}</p>;
        return <h1>Halo, {data.name}!</h1>;
      }}
    </DataFetcher>
  );
}

function ProductPage() {
  return (
    <DataFetcher url="/api/products/featured">
      {({ data, loading, error }) => {
        if (loading) return <div className="skeleton" />;
        if (error) return <p>Gagal load produk</p>;
        return data.map(p => <div key={p.id}>{p.title}</div>);
      }}
    </DataFetcher>
  );
}

💡 Render Props vs Children vs Hook: Render Props bisa pake prop children (kayak di atas) atau prop khusus kayak render={(data) => ...}. Keduanya sama konsepnya. Di era hooks, DataFetcher bisa jadi useFetch() hook yang lebih simple. Tapi kalau kamu butuh share state antar sibling, Render Props masih oke!

⚠️ Performance gotcha: Render Props bikin fungsi baru di setiap render — ini bisa menyebabkan child component re-render yang nggak perlu. Kalau performa jadi masalah, wrap dengan React.memo() atau pertimbangkan pakai custom hook sebagai gantinya.

COMPOUND COMPONENTS — ORKESTRA YANG HARMONIS 🎵

API Design · 11 menit baca

Pernah pake komponen kayak <select><option> di HTML? Dia punya parent (select) dan children (option) yang kerja bareng — padahal masing-masing bisa dipakai berbeda-beda tiap tempat. Nah, Compound Components adalah pola bikin komponen yang sama kayak itu — beberapa komponen yang kerja bareng sebagai satu unit, tapi fleksibel secara individual.

Analoginya kayak orkestra atau band musik 🎵. Ada drummer, gitaris, vokalis, dan bassist — masing-masing punya peran sendiri, tapi semuanya harus harmonis dan nyambung biar musiknya enak didenger. Kalau drummer main tempo beda, musiknya jadi aneh. Di React, <Tabs>, <TabList>, <TabPanel> — mereka itu band-nya!

Keunggulan utama Compound Components: API yang expressif dan clean. User komponen kamu bisa baca kode dan langsung paham hubungannya. <Tabs><Tab>Home</Tab><Tab>About</Tab></Tabs> — nggak perlu documentation panjang lebar, strukturnya sendiri udah nunjukkan hierarki. Oh ternyata gitu! 🤯

Implementasinya biasanya pake dua teknik: React.Children.map() buat manipulate children, atau Context buat share state antar komponen. Context lebih direkomendasi karena nggak bergantung pada struktur DOM dan lebih fleksibel. Kita bakal liat contohnya pake Context.

Contoh compound component yang paling populer di library: <Accordion> dari Radix UI, <Menu> dari Headless UI, dan <RadioGroup> dari Reach UI. Semua mengikuti pattern ini karena emang UX-nya yang paling natural buat developer!

JAVASCRIPT
import { createContext, useContext, useState } from 'react';

// 1. Context buat share state antar komponen
const AccordionContext = createContext();

// 2. Parent component — kayak leader band
function Accordion({ children, defaultIndex = 0 }) {
  const [openIndex, setOpenIndex] = useState(defaultIndex);

  return (
    <AccordionContext.Provider value={{ openIndex, setOpenIndex }}>
      {children}
    </AccordionContext.Provider>
  );
}

// 3. Child item — terima klik, toggle open/close
function AccordionItem({ index, title, children }) {
  const { openIndex, setOpenIndex } = useContext(AccordionContext);
  const isOpen = openIndex === index;

  return (
    <div>
      <button onClick={() => setOpenIndex(isOpen ? null : index)}>
        {title} {isOpen ? '▲' : '▼'}
      </button>
      {isOpen && <div>{children}</div>}
    </div>
  );
}

// 4. Pakainya — clean dan expressive!
function FAQ() {
  return (
    <Accordion>
      <AccordionItem index={0} title="Apa itu React?">
        React adalah library JavaScript untuk bikin UI.
      </AccordionItem>
      <AccordionItem index={1} title="Kenapa pake React?">
        Karena component-based, declarative, dan ecosystem-nya besar!
      </AccordionItem>
    </Accordion>
  );
}
JAVASCRIPT
const SelectContext = createContext();

function Select({ children, value, onChange }) {
  const [label, setLabel] = useState('');

  return (
    <SelectContext.Provider value={{ value, onChange, label, setLabel }}>
      <div className="select-wrapper">{children}</div>
    </SelectContext.Provider>
  );
}

function SelectTrigger({ placeholder = 'Pilih...' }) {
  const { label } = useContext(SelectContext);
  return <button>{label || placeholder}</button>;
}

function SelectOption({ children, value: optValue }) {
  const { value, onChange, setLabel } = useContext(SelectContext);

  return (
    <button
      className={value === optValue ? 'active' : ''}
      onClick={() => { onChange(optValue); setLabel(children); }}
    >
      {children}
    </button>
  );
}

// Pakainya — API-nya mirip HTML select:
function App() {
  const [role, setRole] = useState('');
  return (
    <Select value={role} onChange={setRole}>
      <SelectTrigger />
      <SelectOption value="admin">Admin</SelectOption>
      <SelectOption value="user">User</SelectOption>
      <SelectOption value="guest">Guest</SelectOption>
    </Select>
  );
}

💡 Best practice: Selalu pakai Context (bukan React.Children.map()) untuk share state di Compound Components. Context lebih fleksibel karena nggak bergantung pada urutan children — kamu bisa naro intermediate komponen di antara parent dan child tanpa nge-break logic.

⚠️ Implicit dependency: Kelemahan Compound Components adalah dependency-nya implicit. AccordionItem harus di-dalem Accordion — kalau lupa, error-nya nggak jelas. Solusinya: tambahin check di Context consumer dan kasih error message yang helpful.

PROVIDER PATTERN — EVOLUSI DARI USECONTEXT 🌍

State Global · 10 menit baca

Di Bab 4 kita udah bahas useContext sebagai solusi prop drilling. Nah, Provider Pattern itu adalah cara proper ngimplementasiin Context di level architecture. Bukan cuma bikin Context + Provider, tapi bikin API yang encapsulated — user komponen kamu nggak perlu tahu internal-nya.

Bayangin kamu punya listrik di rumah. Kamu nggak perlu tahu dari mana listrik datang, siapa yang nge-generate, atau kabelnya lewat mana. Yang kamu tahu cuma: colok ke stopkontak, dapet listrik. Provider Pattern itu stopkontak-nya 🔌. Developer cukup wrap app dengan provider, dan semua komponen bisa akses data tanpa perlu paham bagaimana data itu disediakan.

Pattern ini dipake hampir di semua library UI modern: React Query punya QueryClientProvider, styled-components punya ThemeProvider, Next.js punya AppProvider. Semua mengikuti pattern yang sama: bikin context, bikin provider, bikin custom hook. Ini udah jadi standar de facto di React ecosystem!

Yang membedakan Provider Pattern dari basic useContext adalah separation of concerns. Logic state management tinggal di dalam provider, komponen consumer cuma manggil custom hook. Ini bikin kode lebih testable, lebih maintainable, dan nggak ada business logic yang bocor ke komponen UI. Oh ternyata pattern ini udah standard ya! 🤯

Satu hal yang penting: pisah context per concern. Jangan bikin satu giant context yang nampung semua state (user + theme + notification + cart). Setiap kali context value berubah, SEMUA consumer re-render. Jadi kalau cart berubah, komponen yang cuma butuh theme ikut re-render. Pisah aja: ThemeProvider, AuthProvider, CartProvider — masing-masing terpisah dan independent.

JAVASCRIPT
import { createContext, useContext, useReducer } from 'react';

// 1. Bikin Context
const NotificationContext = createContext(null);
const NotificationDispatchContext = createContext(null);

// 2. Reducer untuk state management
function notificationReducer(state, action) {
  switch (action.type) {
    case 'ADD':
      return [...state, {
        id: Date.now(),
        message: action.message,
        type: action.typeOf || 'info',
        read: false,
      }];
    case 'REMOVE':
      return state.filter(n => n.id !== action.id);
    case 'MARK_READ':
      return state.map(n =>
        n.id === action.id ? { ...n, read: true } : n
      );
    case 'CLEAR_ALL':
      return [];
    default:
      return state;
  }
}

// 3. Provider component — encapsulate semua logic di sini
function NotificationProvider({ children }) {
  const [notifications, dispatch] = useReducer(notificationReducer, []);

  return (
    <NotificationContext.Provider value={notifications}>
      <NotificationDispatchContext.Provider value={dispatch}>
        {children}
      </NotificationDispatchContext.Provider>
    </NotificationContext.Provider>
  );
}

// 4. Custom hooks — API yang clean buat consumer
function useNotifications() {
  const notifications = useContext(NotificationContext);
  if (notifications === null) {
    throw new Error('useNotifications harus di dalam NotificationProvider!');
  }
  return notifications;
}

function useNotificationActions() {
  const dispatch = useContext(NotificationDispatchContext);
  if (dispatch === null) {
    throw new Error('useNotificationActions harus di dalam NotificationProvider!');
  }
  return {
    add: (message, typeOf) =>
      dispatch({ type: 'ADD', message, typeOf }),
    remove: (id) => dispatch({ type: 'REMOVE', id }),
    markRead: (id) => dispatch({ type: 'MARK_READ', id }),
    clearAll: () => dispatch({ type: 'CLEAR_ALL' }),
  };
}

// 5. Pakai — super clean!
function Header() {
  const notifications = useNotifications();
  const unread = notifications.filter(n => !n.read).length;
  return <span>🔔 {unread} notif</span>;
}
Pisah context per concern — biar re-render minimal
// Pisah context per concern — biar re-render minimal
function App() {
  return (
    <ThemeProvider>          {/* Theme state */}
      <AuthProvider>          {/* User + auth state */}
        <NotificationProvider> {/* Notification state */}
          <CartProvider>       {/* Shopping cart state */}
            <Router>
              <Header />
              <Main />
              <Footer />
            </Router>
          </CartProvider>
        </NotificationProvider>
      </AuthProvider>
    </ThemeProvider>
  );
}

// CartProvider berubah? Header yang cuma butuh theme NGGAK ikut re-render.
// AuthProvider berubah? Komponen yang cuma butuh cart NGGAK ikut re-render.
// Setiap concern terpisah, re-render-nya minimal! 🎯

💡 Pro tip — Separating State & Dispatch: Pisah context untuk state dan dispatch (kayak di atas). Kenapa? Komponen yang cuma butuh dispatch nggak perlu re-render pas state berubah. Ini optimasi kecil tapi berdampak di app besar!

⚠️ Provider nesting: Terlalu banyak nested provider bisa bikin component tree dalam. Kalau udah lebih dari 4-5 level, pertimbangkan bikin composeProviders() utility function yang nggabungin semua provider jadi satu komponen wrapper.

CONTAINER VS PRESENTATIONAL — CHEF & PELAYAN 🍽️

Architecture · 9 menit baca

Di restoran, ada dua peran penting: chef yang masak (urus resep, bahan, proses) dan pelayan yang antar makanan ke meja (urus tampilan, presentasi, interaksi dengan tamu). Keduanya punya tanggung jawab yang beda banget, tapi harus kerja sama. Nah di React, pattern ini disebut Container vs Presentational. 🍽️

Container component (chef) fokus ke bagaimana data bekerja — fetch, state management, business logic. Dia nggak punya style atau markup yang spesifik. Presentational component (pelayan) fokus ke bagaimana data ditampilkan — UI, styling, animasi. Dia nerima data lewat props dan nggak punya state sendiri (kecuali UI state kecil kayak toggle dropdown).

Kenapa ini penting? Karena separation of concerns bikin kode lebih maintainable. Container component bisa di-test dengan unit test (test logic-nya doang), presentational component bisa di-test dengan snapshot test (test tampilannya doang). Plus, presentational component jadi reusable — dia nggak terikat ke data source tertentu. Bisa dipake di halaman berbeda dengan data berbeda!

Di era hooks modern, pattern ini banyak yang diganti dengan custom hooks. Alih-alih bikin Container component, kamu extract logic ke useUserData(), terus pakai di presentational component langsung. Ini lebih simple dan nggak bikin komponen wrapper tambahan. Tapi prinsipnya tetap sama: pisah logic dari presentation! Oh ternyata gitu! 🤯

Yang perlu diwaspadai: jangan terlalu kaku. Nggak semua komponen harus dipisah jadi container + presentational. Untuk komponen kecil yang simple, gabungkan aja jadi satu. Over-engineering juga masalah! Pakai pattern ini kalau komponen-nya emang punya logic yang cukup kompleks dan dipake di beberapa tempat. 🎯

Presentational: fokus tampilan, nerima data via props
// Presentational: fokus tampilan, nerima data via props
// Nggak tahu dari mana data berasal

      <h3>{name}</h3>
      <p>{email}</p>
      <span className="badge">{role}</span>
      <button onClick={onEdit}>Edit Profile</button>
    </div>
  );
}

// ❌ CARA LAMA: Container Component
function UserCardContainer() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('/api/user/me')
      .then(res => res.json())

  if (loading) return <Spinner />;
  return <UserCard {...user} onEdit={handleEdit} />;

// ✅ CARA BARU: Custom Hook (lebih simple, tanpa wrapper)
function useUser() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('/api/user/me')
      .then(res => res.json())
      .then(data => { setUser(data); setLoading(false); });
  }, []);

  return { user, loading };
}

// Pakai langsung tanpa container wrapper:
function ProfilePage() {
  const { user, loading } = useUser();
  if (loading) return <Spinner />;
  return <UserCard {...user} onEdit={handleEdit} />;
}

// Presentational component tetap sama — reusable dan pure!
function UserCard({ name, email, avatar, onEdit }) {
  return (
    <div className="card">
      <img src={avatar} alt={name} />
      <h3>{name}</h3>
      <p>{email}</p>
      <button onClick={onEdit}>Edit</button>
    </div>
  );
}

💡 Cara modern: Alih-alih bikin Container component, extract logic ke custom hook. Komponen presentational tetap sama — dia nerima data lewat props. Tapi sekarang logic-nya ada di hook, bukan di komponen wrapper. Ini lebih simple, lebih testable, dan nggak bikin extra DOM node!

⚠️ Jangan kebablasan: Nggak semua komponen harus dipisah. Kalau komponen-nya kecil dan cuma dipake satu tempat, gabung aja jadi satu. Yang penting prinsipnya: logic dan presentation terpisah secara mental model. Nggak harus selalu berupa file/komponen terpisah!

PROPS GETTERS PATTERN — KUNCI HOTEL YANG FLEKSIBEL 🔑

API Design · 10 menit baca

Bayangin kamu check-in ke hotel dan resepsionis ngasih kamu kunci kamar. Kunci itu bisa langsung dipake buat buka pintu — tinggal colok dan putar. Tapi kamu juga bisa nambahin gantungan kunci sendiri, atau minta kunci cadangan. Kuncinya udah berfungsi, tapi tetap fleksibel. Nah, Props Getters itu konsepnya persis kayak kunci hotel 🔑.

Props Getters adalah fungsi yang return object props yang udah siap pakai. Komponen kamu manggil fungsi tersebut, dapet props yang udah di-set dengan logic internal (onClick, aria attributes, className, dsb), terus kamu bisa langsung spread ke JSX element. Tapi — dan ini yang penting — kamu juga bisa override atau extend props yang dikasih.

Kenapa ini berguna? Bayangin kamu bikin library toggle. User library kamu mau pakai toggle-nya, tapi mereka pengen nambahin onMouseDown sendiri, atau nambahin className custom. Kalau kamu cuma kirim boolean prop kayak isToggled, mereka harus bikin semua handler sendiri. Dengan props getters, mereka cuma spread object yang dikasih — semua udah siap, tapi tetap bisa dikustomisasi!

Pattern ini dipake di Downshift (library autocomplete/headless UI karya Kent C. Dodds), dan juga jadi inspiration buat library headless UI kayak Headless UI dan Radix UI. Konsepnya emang simple tapi bikin API library kamu jadi jauh lebih powerful dan user-friendly. Oh ternyata pattern ini yang dipake library populer! 🤯

Implementasinya: fungsi getter nerima parameter props dari user, lalu merge dengan internal props, dan return hasilnya. User bisa override prop apapun — onClick mereka akan merge dengan onClick internal. Ini bikin kolaborasi antara library developer dan consumer jadi smooth!

JAVASCRIPT
import { useState, useCallback } from 'react';

function useToggle(initialValue = false) {
  const [isToggled, setIsToggled] = useState(initialValue);
  const toggle = useCallback(() => setIsToggled(t => !t), []);

  // ✅ Props Getter — return props object yang siap di-spread
  const getTogglerProps = useCallback((props = {}) => {
    return {
      'aria-pressed': isToggled,
      onClick: (e) => {
        // Jalankan user handler dulu (kalau ada)
        props.onClick?.(e);
        // Terus toggle internal
        toggle();
      },
      role: 'switch',
      tabIndex: 0,
      // Merge semua props dari user — user bisa override!
      ...props,
    };
  }, [isToggled, toggle]);

  return { isToggled, toggle, getTogglerProps };
}

// Pakainya — super fleksibel:
function MyToggle() {
  const { isToggled, getTogglerProps } = useToggle();

  return (
    <div>
      <button {...getTogglerProps()}>
        {isToggled ? 'ON' : 'OFF'}
      </button>
    </div>
  );
}

// User bisa nambahin props sendiri:
function CustomToggle() {
  const { isToggled, getTogglerProps } = useToggle();

  return (
    <button
      {...getTogglerProps({
        className: `toggle-btn ${isToggled ? 'active' : ''}`,
        onMouseDown: () => console.log('mouse down!'),
        onClick: () => console.log('custom click!'), // dijalankan DULU
      })}
    >
      {isToggled ? 'Besar' : 'Kecil'}
    </button>
  );
}
JAVASCRIPT
import { useState, useCallback } from 'react';

function useDisclosure(initial = false) {
  const [isOpen, setIsOpen] = useState(initial);

  const open = useCallback(() => setIsOpen(true), []);
  const close = useCallback(() => setIsOpen(false), []);
  const toggle = useCallback(() => setIsOpen(v => !v), []);

  // Props getter buat trigger (button)
  const getButtonProps = useCallback((props = {}) => ({
    'aria-expanded': isOpen,
    'aria-haspopup': true,
    onClick: (e) => {
      props.onClick?.(e);
      toggle();
    },
    ...props,
  }), [isOpen, toggle]);

  // Props getter buat content (panel)
  const getPanelProps = useCallback((props = {}) => ({
    role: 'region',
    hidden: !isOpen,
    'aria-hidden': !isOpen,
    style: { display: isOpen ? 'block' : 'none' },
    ...props,
  }), [isOpen]);

  return { isOpen, open, close, toggle, getButtonProps, getPanelProps };
}

// Pakai di dropdown/modal/dialog:
function Dropdown() {
  const { isOpen, getButtonProps, getPanelProps } = useDisclosure();

  return (
    <div>
      <button {...getButtonProps({ className: 'dropdown-trigger' })}>
        Menu {isOpen ? '▲' : '▼'}
      </button>
      <div {...getPanelProps({ className: 'dropdown-panel' })}>
        <a href="#">Profile</a>
        <a href="#">Settings</a>
        <a href="#">Logout</a>
      </div>
    </div>
  );
}

💡 Tips merge: Di getter, pastikan user props di-spread setelah internal props, biar user bisa override. Tapi untuk event handler kayak onClick, jangan di-override — merge dua handler biar dua-duanya jalan (user handler dulu, internal handler belakangan).

⚠️ ARIA attributes: Props Getters cocok banget buat nambahin accessibility attributes secara otomatis. Tapi jangan lupa: accessibility attributes yang kamu set via getter bisa di-override user. Pastikan documentasi menjelaskan mana yang sebaiknya nggak di-override.

POLYMORPHIC COMPONENTS — KOMIK YANG BISA JADI APA AJA 🦸

API Design · 9 menit baca

Pernah ngerasa ribet punya komponen <Text> yang kadang harus render sebagai <p>, kadang <h1>, kadang <span>, atau bahkan <button>? Tanpa pattern khusus, kamu harus bikin tiga komponen terpisah atau nambahin if-else yang messy. Nah, Polymorphic Components solusinya — satu komponen, bisa render jadi element HTML mana aja!

Konsepnya gampang: komponen nerima prop as yang tentuin element apa yang harus di-render. Default-nya bisa 'div', tapi user bisa override jadi 'button', 'a', 'section', atau bahkan komponen React lain! Ini kayak superhero yang bisa berubah bentuk jadi apa aja yang dibutuhkan. 🦸

Contoh nyata di library populer: <Box as="section"> dari Chakra UI, <Typography component="h2"> dari MUI, dan <Text as="span"> dari banyak design system. Semua memakai konsep polymorphism yang sama! Oh ternyata library besar semua pake ini ya! 🤯

Implementasinya bisa pake React.createElement() atau dynamic tag. Yang penting: semua props yang nggak kamu pakai internal harus di-forward ke element yang di-render. Ini supaya user bisa tetap nambahin className, id, style, atau props HTML lainnya.

Yang perlu diwaspadai: kalau komponen kamu nerima ref, kamu perlu pake React.forwardRef biar ref bisa diteruskan ke element as-nya. Ini detail teknis yang sering lupa tapi penting buat accessibility dan library kompatibilitas. Oh ternyata ref juga harus di-forward ya! 🤯

JAVASCRIPT
import { forwardRef } from 'react';

// Polymorphic Text component — bisa render jadi element apa aja
const Text = forwardRef(function Text({ as: Element = 'p', variant, size, color, className = '', children, ...rest }, ref) {
  // Map variant ke Tailwind classes
  const baseClasses = 'font-sans leading-relaxed';
  const sizeClasses = {
    xs: 'text-xs', sm: 'text-sm', md: 'text-base',
    lg: 'text-lg', xl: 'text-xl', '2xl': 'text-2xl',
  };
  const colorClasses = {
    ink: 'text-ink', muted: 'text-muted',
    accent: 'text-purple-600', danger: 'text-red-600',
  };

  const classes = [
    baseClasses,
    sizeClasses[size] || '',
    colorClasses[color] || '',
    className,
  ].filter(Boolean).join(' ');

  return (
    <Element ref={ref} className={classes} {...rest}>
      {children}
    </Element>
  );
});

// Pakainya — satu komponen, banyak kemungkinan:
function Page() {
  return (
    <article>
      <Text as="h1" size="2xl" color="ink">
        Judul Artikel
      </Text>

      <Text as="p" size="md" color="muted">
        Paragraf biasa...
      </Text>

      <Text as="span" size="sm" color="accent">
        Highlighted text
      </Text>

      <Text as="label" size="sm" htmlFor="email">
        Email Address
      </Text>

      <Text as="a" href="/about" color="accent">
        Learn more →
      </Text>
    </article>

}

MINI PROJECT — ACCESSIBLE TABS COMPONENT 🚀

Praktik · 20 menit

Sekarang saatnya gabungin tiga pattern yang udah kita pelajari jadi satu project nyata! Kita bakal bikin Accessible Tabs Component yang pake Compound Components buat API-nya, Props Getters buat aksesibilitas props, dan Context buat share state antar komponen. Keren banget kan? 🚀

Kenapa tabs sebagai contoh? Karena tabs itu salah satu UI pattern yang paling umum, tapi paling sering salah implementasi dari sisi accessibility. W3C WAI-ARIA punya spesifikasi khusus buat tabs — role="tablist", role="tab", role="tabpanel", keyboard navigation, dan lain-lain. Dengan Props Getters, semua attributes ini otomatis dikasih — user nggak perlu pusing!

Project ini nge-gabungin 3 pattern sekaligus: Compound Components bikin API-nya expressif (<Tabs><Tab><TabPanel>), Props Getters bikin accessibility otomatis (user nggak perlu paham ARIA), dan Context bikin state sharing antar komponen seamless. Ini contoh nyata gimana pattern-pattern ini bekerja bareng di production! 🔥

JAVASCRIPT
import { createContext, useContext, useState, useCallback, useRef } from 'react';

// 1. Context untuk share state antar compound components
const TabsContext = createContext(null);

// 2. Parent — Tabs (Compound Component Leader)
function Tabs({ defaultIndex = 0, onChange, children }) {
  const [selectedIndex, setSelectedIndex] = useState(defaultIndex);

  const selectTab = useCallback((index) => {
    setSelectedIndex(index);
    onChange?.(index);
  }, [onChange]);

  // Ref buat focus management (keyboard navigation)
  const tabListRef = useRef(null);

  const contextValue = {
    selectedIndex,
    selectTab,
    tabListRef,
    // Props getter buat tab list
    getTabListProps: (props = {}) => ({
      role: 'tablist',
      ref: tabListRef,
      ...props,
    }),
    // Props getter buat tab button
    getTabProps: (index, props = {}) => ({
      role: 'tab',
      id: `tab-${index}`,
      'aria-selected': selectedIndex === index,
      'aria-controls': `panel-${index}`,
      tabIndex: selectedIndex === index ? 0 : -1,
      onClick: (e) => {
        props.onClick?.(e);
        selectTab(index);
      },
      ...props,
    }),
    // Props getter buat tab panel
    getTabPanelProps: (index, props = {}) => ({
      role: 'tabpanel',
      id: `panel-${index}`,
      'aria-labelledby': `tab-${index}`,
      hidden: selectedIndex !== index,
      tabIndex: 0,
      ...props,
    }),
  };

  return (
    <TabsContext.Provider value={contextValue}>
      {children}
    </TabsContext.Provider>
  );
}
3. Child — TabList (wrapper buat tab buttons)
// 3. Child — TabList (wrapper buat tab buttons)
function TabList({ children }) {
  const { getTabListProps, selectedIndex, selectTab, tabListRef } = useContext(TabsContext);

  // Keyboard navigation: Arrow keys, Home, End
  const handleKeyDown = (e) => {
    const tabs = tabListRef.current?.querySelectorAll('[role="tab"]');
    if (!tabs) return;

    const tabCount = tabs.length;
    let newIndex = selectedIndex;

    switch (e.key) {
      case 'ArrowRight': newIndex = (selectedIndex + 1) % tabCount; break;
      case 'ArrowLeft': newIndex = (selectedIndex - 1 + tabCount) % tabCount; break;
      case 'Home': newIndex = 0; break;
      case 'End': newIndex = tabCount - 1; break;
      default: return; // jangan prevent default kalau bukan key kita
    }

    e.preventDefault();
    selectTab(newIndex);
    tabs[newIndex].focus(); // pindah focus ke tab baru
  };

  return (
    <div {...getTabListProps({ onKeyDown: handleKeyDown })}>
      {children}
    </div>
  );
}

// 4. Child — Tab (satu tab button)
function Tab({ index, children }) {
  const { getTabProps, selectedIndex } = useContext(TabsContext);

  return (
    <button {...getTabProps(index)}>
      {children}
    </button>
  );
}

// 5. Child — TabPanel (konten panel)
function TabPanel({ index, children }) {
  const { getTabPanelProps, selectedIndex } = useContext(TabsContext);

  if (selectedIndex !== index) return null; // optional: hide yang inactive

  return (
    <div {...getTabPanelProps(index)}>
      {children}
    </div>
  );
}

// Export semua sebagai compound component
Tabs.TabList = TabList;
Tabs.Tab = Tab;
Tabs.TabPanel = TabPanel;

RECAP — YANG UDAH KITA BAHAS 📝

Simpen di kepala ya, ini penting buat bab selanjutnya!

  • ✓HOC (Higher-Order Components) — Fungsi yang wrap komponen dan nambahin behavior. kayak jas hujan. Masih dipake library lama (Redux connect), tapi di era hooks banyak diganti custom hook. Waspada wrapper hell dan props collision!
  • ✓Render Props — Komponen nerima fungsi sebagai prop yang return JSX. Chef kasih bahan, kamu tentuin masaknya. Fleksibel tapi bisa bikin callback nesting kalau kebanyakan chain.
  • ✓Compound Components — Beberapa komponen yang kerja bareng sebagai satu unit. kayak orkestra. API-nya expressive, implementasi pake Context. Dipake hampir semua headless UI library.
  • ✓Provider Pattern — Evolusi useContext yang proper. Encapsulate state logic di dalam provider, consumer cuma panggil custom hook. Pisah context per concern biar re-render minimal.
  • ✓Container vs Presentational — Pisah logic (chef) dari presentation (pelayan). Di era hooks, container component diganti custom hook. Prinsipnya tetap: logic dan UI terpisah!
  • ✓Props Getters — Fungsi yang return props object siap pakai. kayak kunci hotel yang bisa dikustomisasi. Dipake Downshift dan headless UI library. Sempurna buat accessibility attributes otomatis.
  • ✓Polymorphic Components — Komponen yang bisa render element berbeda via prop as. Superhero yang bisa berubah bentuk. Dipake Chakra UI, MUI, dan design system lainnya.
  • ✓Mini Project: Accessible Tabs — Gabungin Compound Components + Props Getters + Context jadi satu komponen tabs yang accessible. Bukti kalo pattern-pattern ini powerful kalau dipakai bareng!
