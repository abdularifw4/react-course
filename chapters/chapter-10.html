REACT.MEMO & USEMEMO — BERHENTI RE-RENDER YANG NGGAK PERLU

Memoization · 12 menit baca

Performance masalah nomor satu di React: unnecessary re-render. Setiap kali state berubah, React re-render SEMUA komponen di tree — bahkan yang nggak relate ke state yang berubah. Di app kecil, nggak terasa. Di app besar, ini bisa bikin UI jadi choppy dan lambat.

Analoginya: React tanpa memoization itu kayak kantor tanpa pintu — tiap ada pengumuman, semua karyawan berhenti kerja dan denger, termasuk yang nggak terkait. React.memo itu kayak memasang pintu — cuma departemen yang relevan yang dengar. Oh ternyata gitu!

React.memo() nge-wrap komponen supaya cuma re-render kalau props-nya berubah. React bandingin props sebelum dan sesudah pakai shallow comparison. Kalau sama, skip re-render. Cocok buat komponen yang sering receive props yang sama — list items, card components, display-only components.

useMemo memoize computed value. Misal: filtered list dari array 10,000 item — tanpa useMemo, filter jalan setiap render. Dengan useMemo, cuma jalan kalau dependency berubah. Tapi hati-hati: useMemo sendiri punya cost (memory untuk store result). Jangan memoize SEMUA — hanya yang expensive. Gunakan React Profiler untuk identify bottleneck dulu.

useCallback memoize function reference. Berguna kalau kamu pass callback ke child komponen yang di-memo. Tanpa useCallback, function baru tercipta setiap render → child re-render meski React.memo aktif. Tapi sebenarnya useCallback jarang needed — React Compiler (baru!) bisa auto-memoize tanpa manual wrapper.

React.memo — Skip Unnecessary Re-render
// ✅ Tanpa memo — re-render SETIAP kali parent render
function UserCard({ name, role }: { name: string; role: string }) {
  console.log("UserCard rendered:", name);
  return (
    <div className="p-4 bg-white rounded-lg shadow">
      <h3>{name}</h3>
      <p className="text-gray-500">{role}</p>
    </div>
  );
}

// ✅ Dengan memo — cuma re-render kalau props berubah
const MemoUserCard = React.memo(function UserCard({ name, role }) {
  console.log("UserCard rendered:", name);
  return (
    <div className="p-4 bg-white rounded-lg shadow">
      <h3>{name}</h3>
      <p className="text-gray-500">{role}</p>
    </div>
  );
});

// ✅ Custom comparison function
const ProfileCard = React.memo(function ProfileCard({ user }) {
  return <div>{user.name} — {user.bio}</div>;
}, (prev, next) => {
  // Custom: cuma re-render kalau name berubah
  return prev.user.name === next.user.name;
});
useMemo — Expensive Computation
import { useMemo, useState } from "react";

function ProductList({ products }: { products: Product[] }) {
  const [search, setSearch] = useState("");
  const [category, setCategory] = useState("all");

  // ✅ Memoize filter — cuma jalan kalau products/search/category berubah
  const filtered = useMemo(() => {
    console.log("Filtering", products.length, "products...");
    return products
      .filter(p => category === "all" || p.category === category)
      .filter(p => p.name.toLowerCase().includes(search.toLowerCase()));
  }, [products, search, category]);

  // ✅ Memoize sorted result
  const sorted = useMemo(
    () => [...filtered].sort((a, b) => b.price - a.price),
    [filtered]
  );

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <select value={category} onChange={e => setCategory(e.target.value)}>
        <option value="all">All</option>
        <option value="electronics">Electronics</option>
      </select>
      <ul>{sorted.map(p => <li key={p.id}>{p.name} — ${p.price}</li>)}</ul>
    </div>
  );
}

Premature optimization: JANGAN wrap semua komponen dengan React.memo atau useMemo. Memoization punya cost (memory + comparison overhead). Ukur dulu dengan Profiler, baru optimize. Tiga langkah: measure → identify bottleneck → optimize. Jangan optimize secara buta.

USECALLBACK & REACT COMPILER — MEMOIZE OTOMATIS

Advanced · 11 menit baca

useCallback itu memoize function — artinya function reference-nya sama antar render selama dependency nggak berubah. Ini penting kalau kamu pass function sebagai props ke child yang di-memo. Tanpa useCallback, function baru tercipta setiap render → child berpikir props berubah → re-render.

Analoginya: useCallback itu kayak member card — nomor member-mu tetap sama, jadi sistem nggak perlu verifikasi ulang tiap transaksi. Tanpa member card, tiap datang kamu dianggap customer baru. Begitu juga function reference: sama = skip re-render, beda = re-render. Oh ternyata gitu!

Tapi ada yang lebih besar: React Compiler. Ini experimental feature React 19 yang auto-memoize komponen dan hooks. Kamu nggak perlu manual React.memo, useMemo, atau useCallback — compiler analyze kode dan inject memoization secara otomatis. Ini game changer karena menghilangkan cognitive overhead memoization.

React Compiler bekerja dengan analyzing component render: mana value yang "stable" (nggak berubah antar render) dan mana yang "reactive" (berubah berdasarkan state/props). Stable value di-memoize otomatis. Reactive value di-track sebagai dependency. Semua ini terjadi di build time — zero runtime cost.

Status saat ini: React Compiler udah available di React 19 sebagai experimental feature. Beberapa team di Meta udah pake di production. Tapi masih ada edge case — async patterns, closure-heavy code, dan third-party hooks kadang belum compatible. Monitor roadmap dan coba di side project dulu.

useCallback — Callback Stability
import React, { useState, useCallback } from "react";

// ✅ Parent dengan useCallback
function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [filter, setFilter] = useState("all");

  // Tanpa useCallback: fungsi baru setiap render
  // Dengan useCallback: reference sama selama todos nggak berubah
  const addTodo = useCallback((text: string) => {
    setTodos(prev => [...prev, { id: crypto.randomUUID(), text, done: false }]);
  }, []); // deps kosong = fungsi stabil

  const toggleTodo = useCallback((id: string) => {
    setTodos(prev => prev.map(t => t.id === id ? { ...t, done: !t.done } : t));
  }, []);

  return (
    <div>
      <TodoInput onAdd={addTodo} />
      <TodoList todos={todos} onToggle={toggleTodo} />
    </div>
  );
}

// ✅ Child yang di-memo — cuma re-render kalau props berubah
const TodoInput = React.memo(function TodoInput({ onAdd }) {
  console.log("TodoInput rendered");
  const [text, setText] = useState("");
  return (
    <form onSubmit={e => { e.preventDefault(); onAdd(text); setText(""); }}>
      <input value={text} onChange={e => setText(e.target.value)} />
      <button type="submit">Add</button>
    </form>
  );
});

React Compiler setup: npm install -D babel-plugin-react-compiler + tambah plugin di Vite config. Compiler analyze dan auto-memoize. Kamu bisa hapus manual memo di komponen yang compatible. Tapi selalu test performance sebelum dan sesudah.

CODE SPLITTING — JANGAN BAWA SEMUANYA SEKALIGUS

Performance · 12 menit baca

Bundling default: semua JavaScript di satu file. Di app kecil, ok. Di app besar? Bundle bisa 500KB+ JavaScript yang harus di-download dan parse SEBELUM app bisa interaksi. Ini menyebabkan slow Time to Interactive (TTI). User lihat loading spinner yang lama.

Analoginya: code splitting itu kayak koper travel. Tanpa splitting: kamu bawa SEMUA baju di satu koper besar. Berat, susah diangkat. Dengan splitting: kamu bawa cuma yang perlu buat hari ini, sisanya di rumah. Begitu juga dengan kode: load cuma yang perlu buat halaman saat ini, sisanya load on-demand. Oh ternyata gitu!

React.lazy() + Suspense adalah cara native React buat code splitting. React.lazy nge-wrap dynamic import — komponen di-load secara async. Suspense nge-wrap lazy component dan nampilkan fallback (spinner) selama loading. Ketika komponen udah loaded, React render-nya. Seamless.

Pola lanjutan: route-based splitting — tiap halaman di lazy load. Ini paling impactful karena user cuma butuh code halaman yang mereka visit. Dashboard nggak perlu di-load pas user buka login page. Plus component-based splitting — lazy load modal, chart library, rich text editor yang berat. Load cuma pas user butuh.

Di Next.js, code splitting otomatis — tiap page jadi chunk terpisah. Tapi di CRA/Vite, kamu perlu manual setup React.lazy. Plus, Vite support manual chunks di config untuk group common dependencies (React, React DOM) terpisah dari app code. Ini bikin cache lebih efficient — vendor chunk nggak berubah walau app code berubah.

React.lazy + Suspense — Route Splitting
import { lazy, Suspense } from "react";
import { BrowserRouter, Routes, Route } from "react-router-dom";

// ✅ Lazy load halaman — masing-masing jadi chunk terpisah
const Home = lazy(() => import("./pages/Home"));
const Dashboard = lazy(() => import("./pages/Dashboard"));
const Settings = lazy(() => import("./pages/Settings"));
const Profile = lazy(() => import("./pages/Profile"));

// ✅ Shared loading fallback
function LoadingSpinner() {
  return (
    <div className="flex items-center justify-center p-8">
      <div className="animate-spin rounded-full h-8 w-8 border-b-2 border-gray-900" />
      <span className="ml-3 text-gray-500">Loading...</span>
    </div>
  );
}

// ✅ Suspense boundary per-route
export default function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<LoadingSpinner />}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/dashboard" element={<Dashboard />} />
          <Route path="/settings" element={<Settings />} />
          <Route path="/profile/:id" element={<Profile />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}

VIRTUALIZATION — RENDER 10,000 ITEM TANPA NANGIS

Rendering · 11 menit baca

Bayangkan render 10,000 item di DOM sekaligus. Browser harus create 10,000 node, calculate layout, paint, dan composite. Hasilnya? UI freeze, scroll laggy, memory usage tinggi. Ini masalah klasik di tabel data, chat history, dan infinite scroll.

Virtualization (atau windowing) solusinya: cuma render item yang VISIBLE di viewport + sedikit buffer. Item yang nggak visible di-replace dengan placeholder (empty space). Pas user scroll, item baru di-render dan item lama di-destroy. Result: performa konsisten berapapun total item-nya.

Analoginya: virtualization itu kayak teleskop. Kamu nggak lihat semua bintang sekaligus — cuma yang ada di bidang pandang. Begitu kamu geser teleskop, bintang baru muncul, yang lama hilang. Tapi kamu tetap tau semua bintang ada di sana. Oh ternyata gitu!

Library populer: @tanstack/react-virtual (dari TanStack, kompatibel dengan React, Solid, Vue). Ringan, flexible, dan support dynamic size item. Plus react-window (dari Brian Vaughan, mantan React core team) — simple API, performant, tapi support-nya limited. Gue recommend TanStack Virtual untuk project baru.

Pasang virtualization itu trade-off: kompleksitas naik sedikit, tapi performance bisa beda 100x. 10,000 item tanpa virtualization = 2+ detik render. Dengan virtualization = instan, karena cuma render ~20-30 item di viewport. Worth it buat data-heavy app.

TanStack Virtual — Virtualized List
import { useVirtualizer } from "@tanstack/react-virtual";
import { useRef } from "react";

function VirtualList({ items }: { items: Item[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  // ✅ Virtualizer setup — cuma render visible item
  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 48, // estimated height per item
    overscan: 5, // render 5 extra item di atas dan bawah viewport
  });

  return (
    <div ref={parentRef} className="h-[600px] overflow-auto">
      {/* ✅ Total height container — scrollbar jadi akurat */}
      <div style={{ height: `${virtualizer.getTotalSize()}px`, position: "relative" }}>
        {virtualizer.getVirtualItems().map(virtualItem => {
          const item = items[virtualItem.index];
          return (
            <div
              key={virtualItem.key}
              style={{
                position: "absolute",
                top: 0,
                transform: `translateY(${virtualItem.start}px)`,
                height: `${virtualItem.size}px`,
              }}
              className="flex items-center px-4 border-b border-gray-100"
            >
              <span className="font-medium">{item.name}</span>
              <span className="ml-auto text-gray-500">${item.price}</span>
            </div>
          );
        })}
      </div>
    </div>
  );
}

Dynamic size: Untuk item dengan height yang berbeda-beda, pake measureElement callback. Virtualizer measure actual height setelah render, lalu adjust positioning. Lebih akurat tapi sedikit lebih complex. Untuk fixed height, estimateSize sudah cukup.

IMAGE OPTIMIZATION — FILE TERBESAR DI WEB

Assets · 10 menit baca

Image sering jadi penyebab utama web lambat. Satu foto 4MB bisa lebih besar dari SEMUA JavaScript bundle digabung. Image yang nggak dioptimasi = waste bandwidth + slow load + buruk UX. Di mobile, masalahnya makin parah — bandwidth terbatas, data mahal.

Strategi optimasi image: format modern (WebP/AVIF vs JPEG/PNG), responsive sizing (ukuran beda buat mobile vs desktop), lazy loading (load pas mendekati viewport), dan CDN (deliver dari edge server terdekat). Keempatnya kombinasi yang powerful.

WebP rata-rata 25-35% lebih kecil dari JPEG di quality setara. AVIF (next-gen) bisa 50% lebih kecil tapi encoding-nya lambat. Best practice: serve AVIF ke browser yang support, fallback WebP, fallback JPEG. Tools kayak sharp (Node.js) bisa generate multiple format sekaligus.

Di Next.js, next/image handle SEMUA ini otomatis — format conversion, responsive sizing, lazy loading, blur placeholder. Ini salah satu alasan utama pake Next.js untuk production. Di SPA, pake react-optimized-image atau manual implement.

Blur placeholder itu trick yang keren: tampilkan blurred low-res image dulu sementara full image loading. Result: user langsung lihat ada konten (bukan blank), lalu transition ke gambar tajam. Meningkatkan perceived performance secara signifikan. Oh ternyata gitu!

Image Optimization — next/image
// ✅ Next.js Image — otomatis: format, sizing, lazy load
import Image from "next/image";

function ProductCard({ product }) {
  return (
    <div className="card">
      <Image
        src={product.imageUrl}
        alt={product.name}
        width={400}
        height={300}
        placeholder="blur" // blur placeholder otomatis
        blurDataURL={product.blurDataUrl}
        sizes="(max-width: 768px) 100vw, 50vw" // responsive sizing
        quality={80} // default 75, naik untuk foto produk
      />
      <h3>{product.name}</h3>
    </div>
  );
}

// ✅ Native lazy loading (SPA / Vite)
function LazyImage({ src, alt }: { src: string; alt: string }) {
  return (
    <img
      src={src}
      alt={alt}
      loading="lazy" // native browser lazy load
      decoding="async" // decode off main thread
      className="rounded-lg"
    />
  );
}

Sharp CLI: Convert gambar massal: npx sharp -i input.jpg -o output.webp --quality 80. Atau setup script yang auto-convert saat build. Bundler plugin kayak vite-plugin-imagemin bisa otomatis optimasi saat build.

BUNDLE ANALYSIS & PROFILER — UKUR SEBELUM OPTIMIZE

Measurement · 11 menit baca

Performance optimization tanpa measurement itu kayak diet tanpa timbangan — kamu nggak tau apakah usahamu berhasil atau cuma diem-diem aja. Ukur dulu, identifikasi bottleneck, baru optimize. Ini prinsip fundamental yang sering dilupakan developer.

React Profiler (built-in DevTools) nge-capture setiap render: berapa lama, apa yang trigger, mana komponen yang paling sering re-render. Ini starting point untuk optimization. Filter by component, cek "Why did this render?" — langsung tau apa yang trigger re-render. Powerful banget.

Bundle analyzer nge-visualize isi JavaScript bundle — library mana yang paling besar, tree-shaking udah efektif atau belum, ada duplicate dependency atau nggak. Tools: rollup-plugin-visualizer (Vite) atau webpack-bundle-analyzer (Webpack). Result: interactive treemap yang nunjukin proporsi setiap dependency.

Metrik penting yang harus kamu track: LCP (Largest Contentful Paint — waktu konten utama muncul, target < 2.5s), INP (Interaction to Next Paint — responsivitas, target < 200ms; menggantikan FID yang sudah deprecated), CLS (Cumulative Layout Shift — visual stability, target < 0.1). Ini Core Web Vitals dari Google — langsung ngefek SEO ranking. Oh ternyata gitu!

Tools pengukuran: Lighthouse (built-in Chrome DevTools), web-vitals library (field data dari user real), Chrome User Experience Report (agregat data dari semua Chrome user). Jangan cuma rely di dev measurement — production performance bisa beda jauh karena network, device, dan cache.

web-vitals — Track Core Web Vitals
// Catatan: FID (First Input Delay) sudah deprecated dan diganti INP.
// web-vitals v4 menghapus onFID — pakai onINP sebagai metrik responsivitas.
import { onLCP, onCLS, onINP } from "web-vitals";

function reportVitals() {
  onLCP(metric => console.log("LCP:", metric.value)); // target < 2.5s
  onINP(metric => console.log("INP:", metric.value)); // target < 200ms
  onCLS(metric => console.log("CLS:", metric.value)); // target < 0.1

  // Send to analytics endpoint
  onLCP(metric => {
    navigator.sendBeacon("/api/vitals", JSON.stringify({
      name: metric.name,
      value: metric.value,
      rating: metric.rating, // "good" | "needs-improvement" | "poor"
    }));
  });
}

// Call in app entry
reportVitals();

SERVER COMPONENTS & NEXT.JS — FUTURE OF REACT

Architecture · 12 menit baca

React Server Components (RSC) mengubah fundamental cara React bekerja. Sebelumnya: semua komponen render di browser (client). Dengan RSC: komponen bisa render di SERVER — zero JavaScript ke browser. Ini bukan SSR biasa (render HTML di server lalu hydrate). RSC lebih dari itu: komponen server TIDAK perlu hydrate, TIDAK punya state, TIDAK punya event handler. Pure rendering.

Analoginya: Server Components itu kayak kitchen di restoran — masakan disiapkan di dapur (server), cuma makanan final yang dikirim ke meja (browser). Client Components itu kayak BBQ grill di meja — kamu masak sendiri di meja (browser). Dapur lebih efisien untuk masakan yang kompleks. Grill lebih baik untuk interaksi personalisasi. Gabung keduanya = restoran yang efisien. Oh ternyata gitu!

Di Next.js App Router: default = Server Component. Artinya semua komponen secara default render di server — zero JS ke browser. Kalau kamu butuh state, effect, atau event handler, tambah "use client" di top of file. Ini inverted dari Pages Router yang semuanya client. Paradigm shift yang signifikan.

Keuntungan Server Components: bundle size turun drastis (library besar kayak date-fns, markdown parser cuma jalan di server), direct database access (nggak perlu API endpoint, query DB langsung dari komponen), SEO optimal (content langsung di HTML, tanpa JS untuk render). Tapi: nggak bisa pake useState, useEffect, event listener, atau browser API. Trade-off yang worth it untuk halaman content-heavy.

Streaming SSR juga penting: Next.js bisa stream HTML ke browser progressively. User mulai lihat content while sisanya masih di-render. Combined with Suspense, ini bikin perceived performance jauh lebih baik — user interact lebih cepat daripada tunggu SEMUA content selesai render.

Server vs Client Components — Next.js
// ✅ Server Component (default) — render di server, zero JS
// File: app/products/page.tsx (NO "use client")
import { db } from "@/lib/db";

export default async function ProductsPage() {
  // Direct DB access — nggak perlu API endpoint!
  const products = await db.product.findMany({
    orderBy: { createdAt: "desc" },
    take: 20,
  });

  return (
    <div>
      <h1>Products</h1>
      <ProductGrid products={products} />
    </div>
  );
}

// ✅ Client Component — render di browser, punya state/event
"use client";
import { useState } from "react";

export function ProductGrid({ products }) {
  const [filter, setFilter] = useState("all");

  const filtered = filter === "all"
    ? products
    : products.filter(p => p.category === filter);

  return (
    <div>
      <select value={filter} onChange={e => setFilter(e.target.value)}>
        <option value="all">All</option>
        <option value="electronics">Electronics</option>
      </select>
      <ul>
        {filtered.map(p => (
          <li key={p.id}>{p.name} — ${p.price}</li>
        ))}
      </ul>
    </div>
  );
}

Gotcha: Kebalikannya yang benar: Client Components nggak bisa import Server Components (mereka nerima-nya sebagai children/props). Server Component justru BISA import & render Client Component — persis kayak contoh di atas. Aturannya: props yang nyebrang boundary harus serializable — jangan pass function. Untuk interactivity yang butuh server data, pake Server Actions (Next.js) atau API routes.

MINI PROJECT — PERFORMANCE-OPTIMIZED DASHBOARD

Praktik · 20 menit

Waktunya apply semua ilmu performance! Bikin Analytics Dashboard yang menggabungkan: React.memo buat card components, useMemo buat data computation, virtualized list buat data rows, code splitting buat chart library, dan image optimization buat avatar.     estimateSize: () => 40,
    overscan: 5,
  });

  return (
    <div className="space-y-6">
      <div className="grid grid-cols-3 gap-4">
        <MetricCard title="Revenue" value="$12,345" change={12} />
        <MetricCard title="Users" value="1,234" change={8} />
        <MetricCard title="Orders" value="567" change={-3} />
      </div>

      <Suspense fallback={<div className="h-64 bg-gray-100 animate-pulse" />}>
        <Chart data={data} />
      </Suspense>

      <FilterBar filters={["all", "active", "completed"]} active={filter} onChange={setFilter} />

      <div ref={parentRef} className="h-[400px] overflow-auto">
        <div style={{ height: `${virtualizer.getTotalSize()}px`, position: "relative" }}>
          {virtualizer.getVirtualItems().map(row => {
            const item = filtered[row.index];
            return (
              <div key={row.key} className="flex px-4 py-2 border-b"
                style={{ position: "absolute", top: 0, transform: `translateY(${row.start}px)`, height: `${row.size}px` }}>
                <span>{item.name}</span>
                <span className="ml-auto">{item.status}</span>
              </div>
            );
          })}
        </div>
      </div>
    </div>
  );
}

KESALAHAN UMUM — DOSA PERFORMANCE YANG PALING SERING

Enam kesalahan yang paling sering muncul di code review — pola: kode salah → kenapa → kode benar

Ironi terbesar di dunia performance React: lebih banyak app yang jadi lambat GARA-GARA usaha optimasi daripada karena kurang optimasi. Enam kesalahan di bawah ini muncul terus-menerus di code review — dan hampir semuanya lahir dari niat baik yang salah arah. Baca pelan-pelan, karena tiap poin di sini bakal nyelametin lo dari bug yang susah dilacak.

1. Memoize semuanya (premature optimization). Salah: bungkus SETIAP komponen pakai React.memo dan SETIAP nilai pakai useMemo "biar aman". Kenapa salah: memo itu bukan gratis — React harus menyimpan hasil lama + array deps di memory, lalu membandingkan props/deps tiap render. Buat komponen murah (cuma render teks) atau kalkulasi murah (a + b), ongkos perbandingannya lebih besar dari yang dihemat. Benar: tulis kode paling sederhana dulu; tambahkan memo cuma setelah Profiler nunjukin komponen itu beneran mahal.

2. Object / function inline yang bikin memo percuma. Salah: lo bungkus child pakai React.memo, tapi ngasih prop style={{ margin: 8 }} atau onClick={() => ...} yang dibikin inline. Kenapa salah: object dan function literal itu referensi baru tiap render. Shallow comparison React.memo lihat "prop berubah" → child tetap re-render. Memo-nya jadi hiasan doang. Benar: angkat object ke konstanta di luar komponen, atau stabilkan pakai useMemo/useCallback.

3. Dependency array useMemo yang salah. Salah: lupa masukin salah satu variabel yang dipakai di dalam useMemo (atau sengaja dikosongin biar "nggak jalan lagi"). Kenapa salah: hasilnya jadi stale (basi) — variabel berubah tapi memo nahan nilai lama, jadi UI nunjukin data usang. Kebalikannya juga bahaya: masukin object inline ke deps bikin deps selalu berubah → memo nggak pernah nge-cache apa pun. Benar: deps harus berisi SEMUA nilai reaktif yang dibaca di dalam callback — nggak lebih, nggak kurang. Pasang plugin ESLint react-hooks/exhaustive-deps biar ketauan otomatis.

4. Optimasi tanpa mengukur. Salah: nebak "pasti komponen X yang lambat" lalu langsung nambahin memo di mana-mana. Kenapa salah: intuisi soal performance hampir selalu meleset. Sering bottleneck-nya bukan di render React sama sekali — tapi di image 4MB, request API yang lambat, atau layout thrashing. Optimasi buta bikin kode berantakan tanpa perbaikan nyata. Benar: ukur dulu pakai React DevTools Profiler + Lighthouse, temukan angka yang jelek, baru sentuh kode itu.

5. List raksasa tanpa virtualization. Salah: render 5.000 baris tabel sekaligus, lalu nambahin React.memo di tiap baris "biar cepat". Kenapa salah: masalahnya bukan re-render — tapi 5.000 DOM node yang harus dibuat, di-layout, dan di-paint browser. Memo nggak menyelesaikan itu. Benar: pakai virtualization (TanStack Virtual / react-window) supaya cuma ~20-30 baris yang terlihat yang benar-benar ada di DOM.

6. Kerja berat langsung di body render + key churn. Salah: jalanin .sort() / parsing / kalkulasi berat langsung di body komponen, DAN pakai key={index} di list yang bisa berubah urutan. Kenapa salah: body komponen jalan TIAP render, jadi kerja berat itu diulang terus. Dan key={index} bikin React salah mencocokkan elemen saat list disisip/diurut ulang — state & DOM bisa "nyangkut" di item yang salah (ini disebut key churn). Benar: pindahkan kalkulasi berat ke useMemo, dan pakai key ID yang stabil & unik (key={item.id}).

Kesalahan #2 — inline prop bikin React.memo percuma
const Row = React.memo(function Row({ user, style, onSelect }) {
  return <div style={style} onClick={onSelect}>{user.name}</div>;
});

// ❌ SALAH — style & onSelect referensi BARU tiap render → memo sia-sia
function List({ users }) {
  return users.map(u => (
    <Row key={u.id} user={u}
      style={{ padding: 8 }}                 // object baru tiap render
      onSelect={() => console.log(u.id)} />  // function baru tiap render
  ));
}

// ✅ BENAR — referensi stabil, memo baru beneran bekerja
const rowStyle = { padding: 8 };            // konstanta di luar komponen
function List({ users }) {
  const handleSelect = React.useCallback((id) => console.log(id), []);
  return users.map(u => (
    <Row key={u.id} user={u} style={rowStyle}
      onSelect={() => handleSelect(u.id)} />
  ));
}
Kesalahan #6 — key=index bikin state nyangkut di item salah
// ❌ SALAH — key pakai index; hapus item ke-0, checkbox "nyangkut" di baris salah
{todos.map((todo, index) => (
  <TodoItem key={index} todo={todo} />
))}

// ✅ BENAR — key ID stabil; React cocokin elemen ke data yang benar
{todos.map((todo) => (
  <TodoItem key={todo.id} todo={todo} />
))}

Obat diminum saat sakit (setelah Profiler nunjukin masalah), bukan tiap hari "biar sehat". Menaburkan memo di mana-mana bikin kode susah dibaca DAN sering nggak ngefek karena prop-nya inline.

LEVEL SENIOR: MEASURE BEFORE YOU OPTIMIZE — KAPAN MEMO MALAH BIKIN LAMBAT

Level Senior · 15 menit baca

Junior nambahin React.memo karena "kayaknya bakal lambat". Senior nambahin React.memo karena Profiler nunjukin komponen itu render 8ms dan dipanggil 200 kali. Bedanya satu kata: data. Lesson ini soal mindset senior — mengukur dulu, dan tahu kapan optimasi malah jadi beban.

React DevTools Profiler. Ini alat utamamu. Buka tab Profiler → klik Record → interaksi dengan app → Stop. Lo dapat flamegraph: tiap batang = satu komponen, lebarnya = lama render-nya. Klik komponen → panel "Why did this render?" ngasih tau persis apa yang memicu render (props berubah? hook? parent?). Aktifkan setting "Highlight updates when components render" biar lo LIHAT komponen mana yang berkedip tiap interaksi — kalau ada yang berkedip padahal datanya nggak berubah, itu kandidat optimasi. Oh ternyata gitu!

Mengukur ongkos render. Angka yang penting bukan "berapa kali render", tapi "berapa lama TOTAL render itu makan waktu". Komponen yang render 1ms tapi 500 kali (500ms) jauh lebih penting dioptimasi daripada komponen yang render 3ms tapi cuma sekali. Prioritaskan berdasarkan total time, bukan frekuensi. Fokus di top offenders di "Ranked chart" — impact terbesar dengan effort terkecil.

Kapan React.memo / useMemo malah bikin LAMBAT. Ini bagian yang jarang diajarin. Memo punya dua ongkos tetap: (1) menyimpan hasil lama + array deps di memory, dan (2) membandingkan props/deps tiap render. Memo cuma menguntungkan kalau ongkos render yang dihindari > ongkos perbandingan. Dua situasi di mana memo justru rugi:

• Komponen murah. Kalau komponen cuma render sepotong teks, render-nya mungkin 0.05ms. Membandingkan props bisa makan waktu setara atau lebih. Lo membayar ongkos perbandingan untuk menghindari kerja yang nyaris gratis — net loss.

• Props yang hampir selalu berubah. Kalau komponen menerima prop yang berubah tiap render (misal nilai yang di-derive dari state yang sering update), React.memo selalu gagal perbandingan lalu tetap re-render — jadi lo cuma nambah ongkos perbandingan di atas render yang tetap terjadi. Murni pemborosan.

Mental model: perbaiki penyebab, bukan gejala. Kalau komponen re-render terlalu sering, jangan langsung tambal pakai memo. Tanya dulu: kenapa dia re-render? Sering jawabannya adalah masalah struktur yang lebih dalam — state ditaruh terlalu tinggi di tree (angkat ke bawah / colocate), context yang terlalu gemuk (pecah jadi beberapa context), atau parent yang bikin object inline. Memperbaiki penyebab (misal memindahkan state lebih dekat ke tempat pakainya) sering menghapus kebutuhan memo sama sekali. Memo itu solusi terakhir, bukan pertama.

Kemenangan yang realistis. Di dunia nyata, perbaikan performa terbesar hampir nggak pernah datang dari React.memo. Urutan impact dari yang paling besar: (1) virtualize list panjang — bisa 100x; (2) code-split route & library berat — potong bundle awal separuh; (3) optimasi image — sering ini penyebab #1 LCP jelek; (4) defer kerja non-urgent pakai useDeferredValue/useTransition. React.memo/useMemo ada di urutan buncit — berguna, tapi cuma setelah semua di atas beres dan Profiler nunjukin komponen spesifik yang mahal.

Playground 1 — LIHAT SENDIRI React.memo. Klik "Re-render parent" beberapa kali. Perhatikan angka render: PlainChild naik terus tiap klik (padahal prop-nya nggak berubah!), sementara MemoChild diam di 1. Coba hapus React.memo(...) dari MemoChild, klik Jalankan — sekarang dua-duanya naik.

Playground 2 — useMemo & dependency. Angka "kalkulasi berat sudah jalan" cuma naik saat lo klik num + 1 — karena num ada di dependency array. Klik "Toggle theme" sepuluh kali: angkanya nggak naik, karena useMemo pakai hasil cache. Coba ganti baris useMemo jadi slowSquare(num) biasa → sekarang toggle theme pun ikut menjalankan kalkulasi.

OH TERNYATA GITU! Trik dua counter di atas (module-level let yang naik tiap fungsi render dipanggil) adalah cara paling cepat "mengukur" render tanpa buka DevTools. Di kode produksi lo nggak nulis ini — lo pakai Profiler. Tapi buat MEMAHAMI kapan sebuah komponen render, menaruh console.count('NamaKomponen') di body-nya adalah eksperimen 10 detik yang mengajarkan lebih banyak daripada satu jam baca teori.

CHALLENGE — CEGAH RE-RENDER YANG SIA-SIA

Optimalkan komponen sampai semua test lulus. Starter-nya sengaja gagal.

Komponen Badge nampilin prop text yang nggak pernah berubah. Tapi tiap kali tombol "Lain" diklik (yang cuma nambah state nggak berhubungan), Badge ikut re-render — kelihatan dari atribut data-renders yang terus naik. Tugas lo: bungkus Badge pakai React.memo supaya dia skip re-render yang sia-sia. Klik ▶ Test — starter pasti gagal di test terakhir.

Solusi Challenge — buka kalau udah nyerah (atau mau ngecek) klik untuk lihat solusi

Cukup bungkus Badge dengan React.memo. Karena prop text ("STATUS: OK") nggak pernah berubah, shallow comparison React.memo selalu lulus → Badge skip re-render → data-renders tetap "1".

Solusi
const Badge = React.memo(function Badge({ text }) {
  badgeRenders += 1;
  return (
    <span data-testid="badge" data-renders={badgeRenders}
      style={{ padding: '4px 8px', background: '#EEF', borderRadius: 6 }}>
      {text}
    </span>
  );
});

Catatan senior: ini bekerja karena prop-nya stabil. Kalau text-nya justru berubah tiap render, React.memo malah cuma nambah ongkos perbandingan — persis kasus "memo bikin lambat" di Lesson 10.

LATIHAN — ASAH INSTING PERFORMANCE

Kerjain di kepala dulu, baru buka solusinya

Latihan 1: Kenapa memo ini nggak ngefek? klik untuk lihat solusi

Seorang teman bungkus <Chart data={{ points: rows }} /> dan Chart-nya udah pakai React.memo. Tapi Chart tetap re-render tiap parent render. Kenapa, dan gimana benerinnya?

Solusi: Prop data diisi object literal {{ points: rows }} yang dibuat inline — referensi baru tiap render, jadi shallow compare React.memo selalu gagal. Perbaiki dengan menstabilkan object-nya: const data = useMemo(() => ({ points: rows }), [rows]); lalu <Chart data={data} />. Atau lebih baik: kirim rows langsung sebagai prop biasa tanpa membungkusnya dalam object.

Latihan 2: memo atau bukan? klik untuk lihat solusi

Untuk masing-masing komponen, putuskan React.memo layak atau malah rugi: (a) <Avatar url={user.avatar} /> yang cuma render satu img; (b) <DataGrid rows={rows} /> yang render ribuan sel dan sering di-re-render parent dengan rows yang sama; (c) <Clock now={Date.now()} /> yang prop-nya berubah tiap render.

Solusi: (a) Rugi/netral — render-nya murah, ongkos perbandingan setara dengan yang dihemat. Lewati. (b) Layak banget — render mahal (ribuan sel) tapi rows sering sama; memo mencegah re-render besar. Ini kandidat memo ideal. (c) Rugi — prop now berubah tiap render, jadi memo SELALU gagal perbandingan lalu tetap render — cuma nambah ongkos. Aturan main: memo layak saat render mahal DAN props sering stabil.

Latihan 3: Diagnosa scroll yang nge-lag klik untuk lihat solusi

Halaman menampilkan 8.000 komentar dalam satu list. Scroll-nya patah-patah. Seorang junior mengusulkan membungkus tiap <Comment /> dengan React.memo. Apakah itu akan menyelesaikan masalah? Apa solusi yang benar, dan bagaimana urutan langkah senior-nya?

Solusi: React.memo tidak menyelesaikan ini. Masalahnya bukan re-render — melainkan 8.000 DOM node yang harus dibuat, di-layout, dan di-paint browser sekaligus. Memo cuma mencegah render ulang komponen React, bukan mengurangi jumlah node di DOM. Langkah senior: (1) buka Profiler + Performance tab untuk konfirmasi bottleneck ada di layout/paint, bukan di JS; (2) terapkan virtualization (TanStack Virtual / react-window) supaya cuma ~20-30 komentar yang terlihat yang benar-benar dirender; (3) baru ukur ulang. Hasilnya bisa beda 100x — tanpa satu pun React.memo.
