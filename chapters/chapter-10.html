🧠 MEMO — STOP RE-RENDER YANG NGGAK PERLU

Pernah ngalamin yang namanya ngetik di search bar, terus seluruh list di bawahnya re-render? Padahal cuma ngetik huruf "a"? Itu bikin app terasa lambat, bro.

Masalahnya, React itu default-nya re-render SEMUA child component pas parent-nya berubah. Bener-bener semua. Nggak peduli itu perubahan cuma di satu tempat kecil.

Bayangin kamu lagi ngumpet di rumah, terus ibu kamu manggil dari dapur. Semua adik-adik kamu langsung keluar kamar — padahal yang dipanggil cuma kamu doang. 😂 Nah, React.memo itu kayak ngasih tahu adik-adik kamu: "Eits, yang dipanggil cuma kakak. Kalian stay di kamar aja."

import { memo, useMemo, useState } from 'react';

// React.memo: "Aku nggak bakal re-render kalau props-nya masih sama!"
const UserCard = memo(function UserCard({ name, avatar }) {
  console.log('UserCard rendered');
  return (
    <div className='flex items-center gap-3'>
      <img src={avatar} alt={name} className='w-10 h-10 rounded-full' />
      <span>{name}</span>
    </div>
  );
});

function UserList() {
  const [search, setSearch] = useState('');
  // useMemo: "Aku nggak bakal hitung ulang kalau search-nya masih sama"
  const users = useMemo(() => getFilteredUsers(search), [search]);

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      {users.map(u => (
        <UserCard key={u.id} name={u.name} avatar={u.avatar} />
      ))}
    </div>
  );
}

// Custom comparison function
const ExpensiveCard = memo(
  function Card({ data }) { return <div>{data.title}</div>; },
  (prev, next) => prev.data.id === next.data.id // Custom equality
);

💡 Oh ternyata gitu! React.memo bikin komponen "lupa" re-render kalau props-nya nggak berubah. Sedangkan useMemo nge-cache hasil kalkulasi biar nggak dihitung ulang. Tapi hati-hati — jangan memo semuanya ya! Memo itu sendiri juga butuh "biaya". Pake buat komponen yang bener-bener berat aja.

⚠️ Jangan kebablasan! Kalo kamu memo semua komponen, malah bikin app lebih lambat karena React harus ngecek "ini sama atau beda?" di setiap render. Ibarat nge-encrypt pesan "hai" — overkill banget kan? 😅

Eits, tunggu — React.memo aja nggak cukup! Kenapa? Karena kalau kamu kirim function atau object baru sebagai props, memo bakal nganggap "oh, props-nya beda!" dan tetep re-render. Solusinya: pakai useCallback biar function-nya nggak dibikin ulang tiap render.

useCallback + memo = Tim Sempurna 
import { memo, useCallback, useState } from 'react';

// Anak komponen yang "berat" — di-memo biar skip re-render
const TodoItem = memo(function TodoItem({
  todo,
  onToggle,
}: {
  todo: { id: number; text: string; done: boolean };
  onToggle: (id: number) => void;
}) {
  console.log(`TodoItem ${todo.id} rendered`);  // cek kapan dia render
  return (
    <li onClick={() => onToggle(todo.id)}>
      {todo.done ? '✅' : '⬜'} {todo.text}
    </li>
  );
});

// Parent komponen
function TodoList() {
  const [todos, setTodos] = useState(initialTodos);
  const [text, setText] = useState('');

  // ❌ TANPA useCallback — function baru tiap render!
  // const handleToggle = (id) => setTodos(prev => ...)
  // → TodoItem bakal re-render semua walau cuma ngetik di input

  // ✅ PAKAI useCallback — function di-cache, referensi stabil
  const handleToggle = useCallback((id: number) => {
    setTodos((prev) =>
      prev.map((t) => (t.id === id ? { ...t, done: !t.done } : t))
    );
  }, []);  // [] = sekali bikin, selamanya

  return (
    <div>
      {/* Ngetik di sini HANYA re-render input, BUKAN TodoItem! */}
      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="Tambah todo..."
      />
      <ul>
        {todos.map((t) => (
          <TodoItem key={t.id} todo={t} onToggle={handleToggle} />
        ))}
      </ul>
    </div>
  );
}

💡 Ingat triple combo ini: React.memo (skip re-render) + useCallback (stabilin function props) + useMemo (stabilin object props). Tiga-tiganya cuma berguna kalau komponen anaknya berat dan sering re-render tanpa alasan!

📦 CODE SPLITTING — JANGAN BAWA SEMUANYA SEKALIGUS

Coba bayangin kamu mau pindah rumah. Kamu bawa SEMUA barang kamu sekaligus — mulai dari baju, buku, kulkas, sampe koleksi mainan dari SD. Pasti berat banget kan?

Nah, bundle JavaScript kamu juga gitu. Default-nya, React bakal bundle SEMUA code kamu jadi SATU file gede. Padahal user mungkin cuma butuh halaman Home doang, tapi dia harus nunggu download code untuk halaman Settings, Analytics, Profile — yang mungkin dia nggak bakal kunjungi sama sekali.

Oh ternyata gitu! Code splitting itu solusinya — memecah bundle jadi beberapa chunk kecil, terus cuma load yang dibutuhkan. Kayak pindahan rumah yang terorganisir: bawa barang penting dulu, sisanya menyusul. 🚚

import { lazy, Suspense } from 'react';

// Lazy load: "Aku nggak mau load sekarang, nanti aja pas butuh"
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));
const Analytics = lazy(() => import('./pages/Analytics'));

function App() {
  return (
    <Suspense fallback={
      <div className='flex items-center justify-center h-screen'>
        <div className='animate-spin w-8 h-8 border-2 border-ink border-t-transparent rounded-full' />
        <span className='ml-3'>Loading...</span>
      </div>
    }>
      <Routes>
        <Route path='/' element={<Dashboard />} />
        <Route path='/settings' element={<Settings />} />
        <Route path='/analytics' element={<Analytics />} />
      </Routes>
    </Suspense>
  );
}

// Preload on hover (advanced)
function NavItem() {
  const handleMouseEnter = () => {
    // Mulai load sebelum user klik — biar pas diklik udah siap!
    import('./pages/Analytics');
  };
  return <div onMouseEnter={handleMouseEnter}>Analytics</div>;
}

💡 Mental model: lazy() itu kayak kamu bilang ke React, "Eh, code-nya jangan didownload sekarang. Nanti aja pas komponen ini beneran ditampilkan." Terus Suspense itu kayak pesan "Tunggu bentar ya..." yang muncul selama code-nya belum selesai download. Mantap kan? 🎯

🪟 VIRTUALIZATION — RENDER 10,000 ITEM TANPA NANGIS

Pernah diminta bikin list yang isinya 10,000 item? Terus kamu map array-nya langsung di JSX, dan... browser-nya langsung nangis. 💀

Masalahnya bukan datanya — tapi DOM-nya. Setiap item di list itu jadi satu DOM node. Bayangin 10,000 div di dalam satu container. Browser-nya kewalahan render semua itu sekaligus.

Oh ternyata gitu! Virtualization itu konsepnya simpel: kamu cuma render item yang KELIHATAN di layar. Yang di atas dan di bawah viewport? Virtual doang — nggak ada di DOM. Scroll ke bawah? Yang di atas dihapus, yang baru muncul dirender. Hasilnya? 10,000 item tapi cuma ada ~15 DOM node di layar. 🤯

import { FixedSizeList as List } from 'react-window';

// 10,000 items — tapi hanya render yang terlihat!
function BigList({ items }) {
  const Row = ({ index, style }) => (
    <div style={style} className='flex items-center px-4 border-b border-black/10'>
      <span className='text-muted'>{index + 1}.</span>
      <span className='ml-3'>{items[index].name}</span>
      <span className='ml-auto text-ink'>Rp{items[index].price.toLocaleString()}</span>
    </div>
  );

  return (
    <List
      height={600}         // Viewport height
      itemCount={items.length}
      itemSize={56}        // Height per item
      width='100%'
    >
      {Row}
    </List>
  );
}

// Dengan 10,000 items:
// Tanpa virtualization: render 10,000 DOM nodes = LAMBAT
// Dengan virtualization: render ~15 DOM nodes = INSTANT

💡 Oh ternyata gitu! Virtualization itu kayak kaca pembesar — kamu cuma lihat sebagian kecil dari data yang sebenarnya gede banget. Library react-window (atau react-virtuoso) nge-handle semua math-nya buat kamu. Tinggal pasang, dan boom — list apapun jadi smooth. 🚀

Selain virtualization, ada satu hook React 18 yang sering bikin app terasa lebih responsif: useTransition. Hook ini nandain update state tertentu sebagai "low priority" — jadi input user tetap lancar walau re-render-nya berat!

useTransition — Input Lancar, Re-render Berat di Background 
import { useState, useTransition, useMemo } from 'react';

function SearchPage({ allItems }) {  // allItems = 50,000 item
  const [isPending, startTransition] = useTransition();
  const [query, setQuery] = useState('');
  const [highlight, setHighlight] = useState('');

  // Hitung hasil search — INI YANG BERAT (50k item di-filter)
  const results = useMemo(
    () => allItems.filter((item) =>
      item.name.toLowerCase().includes(highlight.toLowerCase())
    ),
    [highlight, allItems]
  );

  return (
    <div>
      {/* Input tetap INSTANT — high priority */}
      <input
        value={query}
        onChange={(e) => {
          setQuery(e.target.value);  // update input langsung (urgent)
          // Filter 50k item di-mark sebagai "low priority transition"
          startTransition(() => {
            setHighlight(e.target.value);  // re-render berat di-background
          });
        }}
        placeholder="Cari di 50,000 item..."
      />

      {/* isPending true selama filter berjalan */}
      {isPending && <span className="animate-pulse">🔄 Filtering...</span>}

      <ul>
        {results.slice(0, 50).map((item) => (
          <li key={item.id}>{item.name}</li>
        ))}
      </ul>
    </div>
  );
}
// Bedanya tanpa useTransition: tiap ketik huruf, React BLOCK
// sampai 50,000 item selesai di-filter. Input kerasa nge-lag! 😵

🔧 PROFILER & BEST PRACTICES — JANGAN OPTIMIZE SECARA BUTA

Oke, sekarang kamu udah tau tool-tool performance-nya. Tapi sebelum kamu mulai memo semua komponen dan lazy load segalanya... berhenti dulu. 🛑

Masalah terbesar developer itu bukan nggak bisa optimize — tapi optimize hal yang salah. Kayak orang diet tapi cuma nggak makan nasi, padahal yang bikin gemuk itu gorengan setiap malam. 😂

Golden rule di dunia performance: "Measure first, optimize second." Pakai React DevTools Profiler buat lihat mana komponen yang benar-bener lambat, baru deh optimize yang itu aja.

// 📋 BEST PRACTICES CHECKLIST — Bookmark ini!
//
// ✅ 1. KEY yang stabil (BUKAN index array!)
// ❌ {items.map((item, index) => <Card key={index} />)}
//    → Kalau list berubah urutannya, React bingung
// ✅ {items.map(item => <Card key={item.id} />)}
//    → ID unik = React tenang, DOM stabil
//
// ✅ 2. Jangan bikin object/function baru di setiap render
// ❌ style={{ color: 'red' }}  → object baru tiap render
// ✅ const style = useMemo(() => ({ color: 'red' }), [])
//

// ❌ onClick={() => doSomething(id)}  → function baru tiap render
// ✅ const handleClick = useCallback(() => doSomething(id), [id])
//
// ✅ 3. Lazy load komponen yang berat
//    → Dashboard boleh eager, tapi Analytics/laporan? Lazy!
// ✅ 4. Pakai production build
//    → npm run build, bukan npm start untuk perform
// ✅ 5. Measure dulu, baru optimize
//    → Buka Profiler → Record → Cari yang lambat → Fix!
// ✅ 6. Jangan update state kalau valuenya sama
//    → React tetap re-render meski state-nya sama value
// ✅ 7. Jaga component tree tetap flat
//    → Semakin dalam nesting-nya, semakin banyak re-render

⚠️ Warning serius nih: Premature optimization is the root of all evil. Kalau app kamu cuma punya 50 komponen, nggak perlu memo-memoan. Fokus dulu ke fitur dan user experience. Performance optimization itu buat pas app kamu udah beneran mulai kerasa lambat — bukan sebelumnya. Nggak percaya? Coba ukur pake Profiler dulu. Data nggak pernah bohong. 📊

Selain Profiler DevTools (UI), React juga punya Profiler API yang bisa kamu pakai langsung di kode. Cocok buat ukur render time di production atau bikin custom performance monitoring:

React Profiler API — Ukur Render di Kode 
import { Profiler, useState, memo } from 'react';

// Callback dipanggil SETIAP KALI komponen di dalam Profiler render
function onRenderCallback(
  id: string,              // id Profiler
  phase: 'mount' | 'update',  // jenis render
  actualDuration: number,  // waktu render sebenarnya (ms)
  baseDuration: number,    // estimasi tanpa memo
  startTime: number,
  commitTime: number
) {
  // Log ke console, atau kirim ke analytics
  if (actualDuration > 16) {  // lebih dari 1 frame (60fps)
    console.warn(`⚠️ ${id} lambat: ${actualDuration.toFixed(2)}ms (${phase})`);
  }
}

function App() {
  const [count, setCount] = useState(0);
  return (
    <Profiler id="Dashboard" onRender={onRenderCallback}>
      <Header />
      <Profiler id="Sidebar" onRender={onRenderCallback}>
        <Sidebar />  {/* bisa nested! */}
      </Profiler>
      <ExpensiveChart data={count} />
    </Profiler>
  );
}

// Hapus Profiler di production biar nggak ada overhead:
// <Profiler id="x" onRender={process.env.NODE_ENV === 'development' ? cb : undefined}>

💡 Cara baca angka: 16ms = budget buat 60fps. Kalau actualDuration lebih gede dari 16ms, berarti render-nya bikin frame drop (layar kedap-kedip). Kalau baseDuration jauh lebih besar dari actualDuration, berarti memo-mu berhasil! Selisihnya = waktu yang di-hemat. 🎯
