01

KENAPA TESTING ITU WAJIB, BUKAN OPSIONAL

Mindset · 10 menit baca

Pernah deploy ke production, trus user report bug yang seharusnya ketahuan di development? Atau refactor kode, ngerasa udah aman, tapi ternyata break fitur lain? Itu karena kamu nggak punya safety net. Testing itu safety net-nya — nggak nge-prevent semua bug, tapi nge-catch yang seharusnya obvious.

Analoginya: testing itu kayak rem kendaraan. Kamu bisa nge-drive tanpa rem (dan mungkin aman di jalan lurus), tapi pas ada yang salah, kamu nggak bisa berhenti. Dengan rem, kamu bisa nge-drive dengan percaya diri — bahkan di jalan yang berbahaya. Begitu juga dengan kode: dengan test, kamu bisa refactor dan add fitur dengan percaya diri. Oh ternyata gitu!

Tapi testing bukan cuma "nulis test". Ini tentang confidence. Confidence buat deploy hari Jumat tanpa takut. Confidence buat refactor tanpa regressi. Confidence buat onboarding developer baru — mereka bisa baca test untuk paham behavior kode. Ini investasi yang balik modalnya cepat.

Yang perlu dipahami: testing itu pyramid, bukan segitiga terbalik. Banyak unit test (cepat, murah), beberapa integration test (medium), sedikit E2E test (lambat, mahal). Jangan fokus di E2E doang — mereka lambat dan flaky. Unit test yang solid jadi fondasi confidence kamu.

Di React ecosystem, tools-nya udah mature: Vitest (unit test, super cepat), React Testing Library (komponen test, behavior-focused), MSW (mock API), Playwright (E2E). Kita bahas semuanya di bab ini.

Testing pyramid itu konsep dari Martin Fowler: base = unit test (banyak, cepat), middle = integration test (sedang), top = E2E test (sedikit, lambat). Rasio ideal: 70-20-10. Unit test nge-cover business logic dan komponen secara isolated. Integration test nge-cover interaksi antar komponen + API. E2E test nge-cover critical user flows end-to-end. Jangan inverted pyramid — banyak E2E, sedikit unit = slow CI dan flaky test.

Setup testing di React udah gampang banget. Vitest sebagai test runner (drop-in Jest replacement, share Vite config), React Testing Library buat render komponen, dan MSW buat mock API. Tinggal install dan setup — kurang dari 5 menit. Kita mulai dari sini dan build up ke patterns yang lebih advanced.

Setup Testing — Vitest + RTL
# Install dependencies
npm install -D vitest @testing-library/react @testing-library/jest-dom
npm install -D @testing-library/user-event jsdom

# vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  test: {
    environment: "jsdom",
    setupFiles: ["./src/test/setup.ts"],

  },
});

Vitest vs Jest: Vitest lebih cepat (native ESM, no transform), config simpler (share vite.config), dan compatible dengan Jest API. Di project baru, pilih Vitest. Di existing Jest project, nggak perlu migrasi — Jest masih solid.

02

REACT TESTING LIBRARY — TEST APA YANG USER LIHAT

Fundamental · 12 menit baca

React Testing Library (RTL) punya filosofi yang beda dari Enzyme. Enzyme test implementation — kamu cek state internal, lifecycle methods, component instance. RTL test behavior — kamu cek apa yang user lihat dan bisa lakukan. Kenapa? Karena user nggak peduli state-nya apa, yang penting UI-nya bekerja.

Analoginya: Enzyme itu kayak nge-test mesin mobil — cek oli, piston, rpm. RTL itu kayak nge-test pengalaman berkendara — apakah rem bekerja? apakah setir responsif? User cuma peduli pengalaman, bukan mesinnya. Oh ternyata gitu!

Query API RTL di-design supaya kamu cari element seperti user. getByRole("button") bukan container.querySelector(".btn"). Priority: getByRole > getByLabelText > getByPlaceholderText > getByText > getByTestId.

Pattern fundamental: arrange-act-assert. Arrange = render komponen + setup. Act = simulate user action (click, type). Assert = cek UI output. Tiga langkah konsisten di semua test.

RTL juga mendorong accessible testing. Karena kamu query element pakai role dan label, komponen yang nggak accessible (misal: button tanpa aria-label, form field tanpa label) langsung susah di-test. Ini positif — testing mendorong accessibility secara natural. Komponen yang mudah di-test = komponen yang accessible. Win-win!

Query variants: getBy* throw kalau element nggak ketemu. queryBy* return null (baik buat cek element TIDAK ada). findBy* auto-wait (baik buat async element). Tiga variant ini penting — pake yang sesuai dengan konteks.

RTL — Login Form Test
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { describe, it, expect } from "vitest";

describe("LoginForm", () => {
  it("calls onSubmit with email and password", async () => {
    const user = userEvent.setup();
    const handleSubmit = vi.fn();

    render(<LoginForm onSubmit={handleSubmit} />);

    await user.type(screen.getByLabelText(/email/i), "arif@mail.com");
    await user.type(screen.getByLabelText(/password/i), "secret123");
    await user.click(screen.getByRole("button", { name: /login/i }));

    expect(handleSubmit).toHaveBeenCalledWith("arif@mail.com", "secret123");
  });

  it("button exists and is enabled", () => {
    render(<LoginForm onSubmit={vi.fn()} />);
    const btn = screen.getByRole("button", { name: /login/i });
    expect(btn).toBeEnabled();
    expect(btn).toBeInTheDocument();
  });
});

Anti-pattern: Jangan test className atau inline style — itu implementation detail. Test behavior: button ada, loading muncul, error tampil. Nggak: button punya class btn-primary.

03

USER EVENT — SIMULASI INTERAKSI USER

Interaction · 11 menit baca

fireEvent vs userEvent — dua cara simulasi interaksi. Tapi BEDA JAUH. fireEvent cuma trigger event DOM. userEvent trigger SEMUA event yang terjadi: focus, keydown, keypress, input, keyup per karakter. Lebih realistis. Oh ternyata gitu!

Best practice: selalu pakai userEvent. Dia muncul error kalau element nggak interactable (disabled button, hidden input). Ini bikin test lebih strict dan menangkap bug UI yang luput.

userEvent.setup() (v14+) adalah recommended API — lebih clean dan predictable. Support delay option buat simulasi real typing speed.

Untuk komponen async, pake waitFor(() => expect(...)) atau findByRole() (auto wait). Jangan pake setTimeout — flaky!

userEvent juga support keyboard (user.keyboard("a{Shift>}B") untuk shift+B), tab navigation (user.tab()), double click (user.dblClick()), dan drag and drop (user.drop()). Semua simulasi real user action — bukan cuma trigger DOM event. Ini yang bikin test kamu mewakili pengalaman user sebenarnya.

Yang sering lupa: cleanup. RTL auto-cleanup setelah setiap test (via afterEach global setup). Tapi kalau kamu customize, pastikan cleanup jalan. Tanpa cleanup, DOM bisa "leak" antar test — state dari test sebelumnya ngefek test selanjutnya. Subtle bug yang susah di-trace.

userEvent — Debounced Search Test
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { vi } from "vitest";

describe("SearchWithDebounce", () => {
  it("debounces search by 500ms", async () => {
    vi.useFakeTimers();
    const onSearch = vi.fn();
    const user = userEvent.setup({ advanceTimers: vi.advanceTimersByTime });

    render(<SearchWithDebounce onSearch={onSearch} />);
    await user.type(screen.getByPlaceholderText("Cari..."), "react");

    expect(onSearch).not.toHaveBeenCalled(); // belum dipanggil
    vi.advanceTimersByTime(500);
    expect(onSearch).toHaveBeenCalledWith("react"); // setelah 500ms

    vi.useRealTimers();
  });
});

findBy vs waitFor: findByRole() = built-in waitFor + query. Return promise. waitFor() = custom assertion yang di-retry. Pilih sesuai kebutuhan.

04

TESTING ASYNC — MOCKING & WAITING ⏳

Async · 12 menit baca

Async component paling tricky di-test. Loading state, error handling, retry logic — semua harus di-cover. Tanpa strategi bener, test bisa jadi flaky (kadang pass, kadang fail).

Dua pendekatan: mock fetch (intercept HTTP) atau mock data layer (inject mock data). Mock fetch lebih realistis. Mock data layer lebih simple.

MSW (Mock Service Worker) intercept HTTP di network level — bukan di kode. Artinya mock-nya berlaku di test, dev, DAN Storybook. Satu mock source, semua environment. Oh ternyata gitu!

Pattern: render → assert loading → await data → assert loaded. Selalu cek loading state dulu buat memastikan UX nggak flash blank screen.

Untuk polling (data yang refresh otomatis), test perlu handle timing. Pakai vi.useFakeTimers() untuk kontrol waktu. Jangan test real polling interval — itu flaky di CI (container bisa slow, timing meleset). Fake timers = deterministic, fast, reliable. Oh ternyata gitu!

Untuk data fetching library kayak TanStack Query, ada opsi lebih simple: QueryClient bisa di-inject langsung di test. Set retry: false dan gcTime: 0 di test client. Nggak perlu tunggu retry — test langsung assert hasil. Plus bisa inject queryFn mock langsung di QueryCache. Nggak perlu MSW untuk test TanStack Query.

Async Test — MSW Setup
import { render, screen, waitFor } from "@testing-library/react";
import { http, HttpResponse } from "msw";
import { setupServer } from "msw/node";
import { afterAll, afterEach, beforeAll, it, expect } from "vitest";

const server = setupServer(
  http.get("/api/users", () => {
    return HttpResponse.json([
      { id: "1", name: "Arif" },
      { id: "2", name: "Budi" },
    ]);
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

it("renders user list after fetching", async () => {
  render(<UserList />);
  expect(screen.getByText("Loading users...")).toBeInTheDocument();

  await waitFor(() => {
    expect(screen.getByText("Arif")).toBeInTheDocument();
  });
});
Error Path Test
it("shows error when API fails", async () => {
  server.use(
    http.get("/api/users", () => new HttpResponse(null, { status: 500 }))
  );
  render(<UserList />);

  await waitFor(() => {
    expect(screen.getByText(/error/i)).toBeInTheDocument();
  });
});

MSW 2.0: Pake http.get() / http.post() API baru. Support async resolver, delay simulation, conditional response berdasarkan request body.

05

MSW — MOCK API YANG REVOLUSIONER

Mocking · 11 menit baca

MSW intercept di network level — kayak proxy server. Mock-nya berlaku buat semua kode yang nge-fetch. Nggak perlu mock per-function. Satu handler, semua environment (Vitest, Playwright, Storybook).

Analoginya: biasa mock itu kayak nggak kasih makanan ke tamu. MSW itu kayak restoran dummy — tamu masuk, pesan, dibalas, pengalaman lengkap. Tamu nggak tau restoran-nya palsu. Oh ternyata gitu!

Keunggulan MSW: mock once, run everywhere. Satu handler dipake di unit test, E2E test, Storybook, bahkan Cypress. Consistent, DRY, maintainable.

MSW 2.0+ pake http.get() API yang clean. Support response resolver async, error simulation, delay, dan cookie. Plus bisa run di browser — dev frontend tanpa backend!

Browser mode MSW juga powerful. Pas development, kamu bisa run MSW di browser via npx msw init public/ untuk generate Service Worker file. Lalu conditional start: cuma aktif di dev mode. Frontend dev bisa independent dari backend — nggak perlu mock server, nggak perlu backend running. Speed up development cycle signifikan.

Yang juga keren: response composition. Kamu bisa chain multiple handlers dan override per-test. server.use(http.get(...)) di individual test override default handler tanpa ngefek test lain. Ini bikin test isolation perfect — setiap test punya mock-nya sendiri.

MSW 2.0 — Handler Variations
import { http, HttpResponse, delay } from "msw";

const handlers = [
  // Success response
  http.get("/api/users", () => {
    return HttpResponse.json([{ id: "1", name: "Arif" }]);
  }),

  // Error response
  http.get("/api/users/:id", ({ params }) => {
    if (params.id === "999") {
      return new HttpResponse(null, { status: 404 });
    }
    return HttpResponse.json({ id: params.id });
  }),

  // Slow network simulation
  http.post("/api/login", async () => {
    await delay("real"); // real network delay
    return HttpResponse.json({ token: "mock-token" });
  }),

  // Conditional response
  http.post("/api/register", async ({ request }) => {
    const body = await request.json();
    if (body.email === "taken@mail.com") {
      return HttpResponse.json({ error: "Email taken" }, { status: 409 });
    }
    return HttpResponse.json({ id: "new" }, { status: 201 });
  }),
];

Setup: MSW Node.js pake setupServer(). Browser pake setupWorker() + npx msw init public/.

PLAYWRIGHT — E2E TEST YANG FAST & RELIABLE

E2E · 12 menit baca

E2E test nge-test app dari perspektif user nyata. Buka browser, navigate, klik, isi form — semua otomatis. Ini safety net terakhir sebelum deploy.

Playwright dari Microsoft: lebih cepat dari Cypress, lebih reliable dari Selenium, support multi browser (Chromium, Firefox, WebKit). Punya auto-wait — nunggu element muncul sebelum interact. Nggak perlu manual setTimeout. Oh ternyata gitu!

Yang spesial: trace viewer. Pas test fail, Playwright bikin recording — screenshot per step, DOM snapshot, network log, console log. Replay dan debug fail test jadi gampang.

Tapi E2E MAHAL — lambat, flaky, maintenance tinggi. Gunakan strategis: test critical user flows doang (login, checkout, signup). Hal sepele = unit test.

Playwright juga punya codegen — record interaksi di browser, generate test code otomatis. Plus UI Mode untuk interactive exploration.

Parallel execution — Playwright run test di multiple worker secara paralel. Satu test suite 50 test bisa selesai dalam hitungan detik. Plus, dia support sharding — split test suite jadi N shards untuk CI parallelism. Ini bikin E2E test yang biasa 10 menit jadi 2 menit.

Setup cepat: npx playwright init generate config + example test + GitHub Actions workflow. Nggak perlu konfigurasi kompleks. Browser di-download otomatis. Zero-config untuk kebanyakan project. Plus visual comparison (screenshot diff) untuk design regression — screenshot sekarang vs baseline, kalau beda jadi fail.

Playwright — Login Flow
import { test, expect } from "@playwright/test";

test.describe("Login Flow", () => {
  test("valid login redirects to dashboard", async ({ page }) => {
    await page.goto("/login");
    await page.getByLabel(/email/i).fill("arif@mail.com");
    await page.getByLabel(/password/i).fill("secret123");
    await page.getByRole("button", { name: /login/i }).click();

    await expect(page).toHaveURL("/dashboard");
    await expect(page.getByText("Selamat datang")).toBeVisible();
  });

  test("wrong password shows error", async ({ page }) => {
    await page.goto("/login");
    await page.getByLabel(/email/i).fill("arif@mail.com");
    await page.getByLabel(/password/i).fill("wrong");
    await page.getByRole("button", { name: /login/i }).click();

    await expect(page.getByText(/password salah/i)).toBeVisible();
  });
});

Playwright + MSW: Gabungkan! MSW bisa handle mock API di E2E test juga. Setup MSW browser worker di public folder, lalu init di Playwright. Satu mock source, semua level test.

BEST PRACTICES — TEST YANG BENER vs WASTE

Strategy · 11 menit baca

Nggak semua test itu good test. Test yang salah jadi liability — lambat, flaky, nggak nangkap bug penting. Test yang bener jadi asset.

Rule #1: Test behavior, bukan implementation. Jangan test internal state atau private methods. Test apa yang user lihat dan lakukan. Refactor internal logic → test tetap pass.

Rule #2: Satu assertion per test concept. Test function cuma test SATU hal. Kalau fail, langsung tau apa yang salah.

Rule #3: Arrange-Act-Assert jelas. Tiga section harus visible. Comment atau whitespace separator. Test yang mudah dibaca = test yang mudah di-maintain.

Rule #4: Test edge cases. Empty input, long string, special characters, network error, concurrent requests. Bug di production sering dari edge case.

Rule #5: CI/CD integration. Pasang Vitest di GitHub Actions. Reject PR yang fail test. Quality secara konsisten, nggak bergantung discipline developer.

Rule #6: Coverage adalah guide, bukan target. 100% coverage nggak berarti bug-free. Bisa 100% coverage tapi semua test cuma cek toBeTruthy(). Fokus di meaningful coverage — test yang nge-catch real bugs. Tool kayak Istanbul/Vitest coverage bisa highlight untested code, tapi jangan kejar angka. Kejar confidence.

Rule #7: Snapshot dengan hati-hati. Jest snapshot test powerful tapi risky. Update snapshot jadi escape route — test fail, update snapshot, test pass, tapi mungkin break sesuatu. Selalu review snapshot diff secara manual. Dan pertimbangkan visual regression test (Playwright screenshot) sebagai alternatif yang lebih reliable.

MINI PROJECT — TEST SUITE KALKULATOR

Praktik · 20 menit

Waktunya apply semua konsep! Bikin test suite lengkap untuk Komponen Calculator — unit test, interaction test, edge case test. Calculator punya: UI interaction (click tombol), state management (display, operator), edge cases (divide by zero, decimal), dan user flow. Semua aspect testing ada di sini.

Calculator Component
import { useState } from "react";

export function Calculator() {
  const [display, setDisplay] = useState("0");
  const [prev, setPrev] = useState<number | null>(null);
  const [operator, setOperator] = useState<string | null>(null);
  const [waiting, setWaiting] = useState(false);

  const inputDigit = (d: string) => {
    if (waiting) { setDisplay(d); setWaiting(false); }
    else { setDisplay(display === "0" ? d : display + d); }
  };

  const calc = (a: number, b: number, op: string): number => {
    if (op === "+") return a + b;
    if (op === "-") return a - b;
    if (op === "*") return a * b;
    if (op === "/") { if (b === 0) throw new Error("Div/0"); return a / b; }
    return b;
  };

  const handleOp = (op: string) => {
    const cur = parseFloat(display);
    if (prev !== null && operator && !waiting) {
      setDisplay(String(calc(prev, cur, operator)));
      setPrev(calc(prev, cur, operator));
    } else { setPrev(cur); }
    setOperator(op); setWaiting(true);
  };

  const handleEquals = () => {
    if (prev === null || !operator) return;
    try {
      const result = calc(prev, parseFloat(display), operator);
      setDisplay(String(result)); setPrev(null); setOperator(null); setWaiting(false);
    } catch { setDisplay("Error"); setPrev(null); setOperator(null); }
  };

  const clear = () => { setDisplay("0"); setPrev(null); setOperator(null); setWaiting(false); };

  const keys = ["7","8","9","+","4","5","6","-","1","2","3","*","0",".","=","/"];
  return (
    <div role="application" aria-label="Calculator">
      <div data-testid="display" aria-live="polite">{display}</div>
      <div className="grid grid-cols-4 gap-2">
        {keys.map(k => (
          <button key={k} aria-label={k} onClick={() => {
            if ("0123456789.".includes(k)) inputDigit(k);
            else if ("+-*/".includes(k)) handleOp(k);
            else if (k === "=") handleEquals();
          }}>{k}</button>
        ))}
        <button onClick={clear} aria-label="Clear" className="col-span-4">C</button>
      </div>
    </div>
  );
}
Calculator — Test Suite
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { describe, it, expect } from "vitest";
import { Calculator } from "./Calculator";

describe("Calculator", () => {
  it("displays 0 initially", () => {
    render(<Calculator />);
    expect(screen.getByTestId("display")).toHaveTextContent("0");
  });

  it("2 + 3 = 5", async () => {
    const user = userEvent.setup();
    render(<Calculator />);
    await user.click(screen.getByLabelText("2"));
    await user.click(screen.getByLabelText("+"));
    await user.click(screen.getByLabelText("3"));
    await user.click(screen.getByLabelText("="));
    expect(screen.getByTestId("display")).toHaveTextContent("5");
  });

  it("divide by zero shows Error", async () => {
    const user = userEvent.setup();
    render(<Calculator />);
    await user.click(screen.getByLabelText("5"));
    await user.click(screen.getByLabelText("/"));
    await user.click(screen.getByLabelText("0"));
    await user.click(screen.getByLabelText("="));
    expect(screen.getByTestId("display")).toHaveTextContent("Error");
  });

  it("clear resets display", async () => {
    const user = userEvent.setup();
    render(<Calculator />);
    await user.click(screen.getByLabelText("9"));
    await user.click(screen.getByLabelText("Clear"));
    expect(screen.getByTestId("display")).toHaveTextContent("0");
  });

  it("chain: 1 + 2 * 3 = 9", async () => {
    const user = userEvent.setup();
    render(<Calculator />);
    await user.click(screen.getByLabelText("1"));
    await user.click(screen.getByLabelText("+"));
    await user.click(screen.getByLabelText("2"));
    await user.click(screen.getByLabelText("*"));
    await user.click(screen.getByLabelText("3"));
    await user.click(screen.getByLabelText("="));
    expect(screen.getByTestId("display")).toHaveTextContent("9");
  });
});

LEVEL SENIOR: TESTING STRATEGY — APA YANG TIDAK PERLU DITES

Level Senior · 14 menit baca

Junior nulis test buat SEMUA hal. Senior tau apa yang NGGAK perlu dites. Bedanya di sini: waktu lo terbatas, dan tiap test punya ongkos maintenance seumur hidupnya. Skill senior bukan "nulis banyak test" — tapi milih test yang ngasih confidence terbanyak per rupiah. Test yang salah pilih malah jadi beban: lambat, rapuh, dan bikin tim males.

Lupakan dulu piramida testing yang lo denger di lesson 1. Ada model yang lebih akurat buat frontend modern: Testing Trophy (Kent C. Dodds). Empat lapisan dari bawah ke atas: (1) Static — TypeScript + ESLint, nangkep typo dan bug sebelum kode jalan (gratis, jalan tiap kali lo ngetik). (2) Unit — function murni, util, custom hook yang di-isolasi. (3) Integration — render beberapa komponen bareng, mock cuma di batas network. (4) E2E — Playwright, alur kritis di browser asli. Bedanya sama piramida: bagian yang digemukin itu integration, bukan unit. Kenapa? Karena integration test paling mirip cara user beneran pakai app. Oh ternyata gitu!

Ini prinsip pegangan yang wajib lo hafal — quote resmi RTL dari Kent C. Dodds: "The more your tests resemble the way your software is used, the more confidence they can give you." (Makin mirip test lo dengan cara user beneran pakai app, makin gede rasa percaya yang dikasih.) Semua keputusan testing lo harusnya bermuara ke kalimat ini. Test yang meniru user = tahan refactor, nangkep bug nyata, dan nggak bohong.

Konsekuensi langsung dari prinsip itu: test behavior, bukan implementation. User nggak peduli komponen lo pakai useState atau useReducer. Mereka peduli: pas diklik, angkanya berubah. Jadi jangan assert state.count === 3 — assert "teks 3 item muncul di layar". Ujian bagusnya: kalau lo refactor internal (misal ganti useState jadi useReducer) tanpa ngubah perilaku, test-nya HARUS tetap hijau. Kalau patah, berarti test lo kegantung implementasi — itu test rapuh.

Sekarang bagian yang bikin lo naik level: daftar yang TIDAK perlu dites. (1) Internal framework & library pihak ketiga — React, React Router, TanStack Query udah dites habis-habisan sama tim mereka; lo nggak perlu nge-test ulang bahwa useState nyimpen nilai. (2) Getter/setter trivial dan konstanta — const PI = 3.14 nggak butuh test. (3) Styling, className, warna — itu tugas visual regression test (screenshot diff), bukan unit test; nge-assert class CSS cuma bikin test patah pas lo ganti nama class padahal app baik-baik aja. (4) Kode generated. (5) Implementation detail apa pun yang user nggak pernah lihat. Nge-test hal-hal ini = effort tinggi, confidence nol.

Yang paling menentukan di level senior: paham cost/confidence tradeoff, dan tau kapan E2E vs kapan integration. E2E itu paling meyakinkan (browser asli, alur lengkap) TAPI paling mahal — lambat, flaky, maintenance berat. Jadi E2E dipakai hemat: cuma buat alur kritis yang kalau rusak = uang hilang (login, checkout, pembayaran). Integration test jauh lebih murah dan confidence-nya tinggi, jadi dia mayoritas. Aturannya: kalau sesuatu bisa lo verifikasi dengan render komponen + mock network (MSW) → integration. Kalau butuh browser beneran, navigasi multi-halaman, cookie, redirect, atau interaksi cross-page → baru E2E.

Rumus saku yang dipakai senior tiap review PR test: tanya dua hal. "Apakah test ini bakal GAGAL kalau app rusak dengan cara yang user notice?" dan "Apakah test ini bakal tetap LULUS setelah refactor yang nggak ngubah perilaku?" Kalau jawabannya "ya" untuk keduanya, itu test bagus. Kalau test gagal cuma karena lo rename variabel internal, buang aja — dia liability, bukan asset.

Apa yang TIDAK Perlu Dites vs yang Perlu
// ❌ NGGAK PERLU — nge-test internal framework / library pihak ketiga
test("useState nyimpen nilai", () => { /* React udah nge-test ini */ });
test("React Router pindah halaman", () => { /* itu tugas tim RR */ });

// ❌ NGGAK PERLU — nge-test styling / className (implementation detail)
expect(button).toHaveClass("btn-primary"); // ganti nama class → test patah, app aman

// ❌ RAPUH — nge-test state internal, bukan yang user lihat
expect(wrapper.state("count")).toBe(3); // refactor ke useReducer → patah tanpa alasan

// ✅ PERLU — nge-test perilaku yang user LIHAT & LAKUKAN
await user.click(screen.getByRole("button", { name: /tambah/i }));
expect(screen.getByText("Keranjang: 1 item")).toBeInTheDocument();
Sweet Spot Trophy — Integration Test
// Integration test — render beberapa komponen bareng, mock CUMA di batas network.
// Ini lapisan paling gemuk di Testing Trophy: confidence tinggi, biaya menengah.
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

it("user nambah produk ke keranjang, badge ikut update", async () => {
  const user = userEvent.setup();
  render(<ProductPage />); // ProductList + Cart + Badge dirender BARENG (bukan di-mock)

  await user.click(screen.getByRole("button", { name: /tambah ke keranjang/i }));
  expect(await screen.findByText("Keranjang (1)")).toBeInTheDocument();
});

OH TERNYATA GITU! "Confident coverage" > "100% coverage". Lo bisa punya 100% coverage yang isinya expect(true).toBe(true) — angkanya cantik, confidence-nya nol. Coverage itu peta area yang belum kesentuh, bukan skor yang dikejar. Senior baca peta itu buat nemu gap penting, bukan buat naikin angka.

JANGAN TERJERAT! Godaan terbesar: nge-E2E semuanya "biar aman". Salah besar. E2E suite yang kegedean bikin CI 20 menit, flaky, dan tim mulai nge-skip yang merah. Lebih baik 5 E2E alur kritis yang solid + banyak integration test, daripada 100 E2E yang sering ngambek. Hemat E2E buat yang beneran mahal kalau rusak.

Cukup teori — rasain sendiri kenapa "test behavior, bukan implementation" itu masuk akal. Playground di bawah ini komponen yang bakal kita test: keranjang sederhana. Klik tombolnya, perhatiin yang berubah di layar ("Keranjang: N item"). Nah, itu persis yang di-assert sama test — bukan nilai count-nya, tapi teks yang user lihat. Kalau lo ganti useState jadi useReducer, teksnya tetap sama → test tetap lulus.

Yang kedua: komponen dengan perilaku async (disimulasiin pakai state + timer). Klik "Muat data user" — muncul "Loading..." dulu, baru "Halo, Arif!". Inilah kenapa lo butuh findBy/waitFor di test async: test harus nunggu teks final muncul, bukan langsung cek. Perhatiin dua state UI yang berbeda — keduanya layak di-assert.

KESALAHAN UMUM — DOSA TESTING YANG PALING SERING

Enam kesalahan yang bikin test jadi liability — pola: kode salah → kenapa → kode benar (cara RTL)

1. Nge-test implementation detail (state/instance), bukan behavior. Ini dosa nomor satu. Lo ngintip state.count atau instance internal komponen. Masalahnya: begitu lo refactor internal tanpa ngubah perilaku (misal useState → useReducer), test-nya patah padahal app baik-baik aja. Test yang bener assert output yang user lihat di layar.

Kesalahan #1 — test state, bukan yang dilihat user
// ❌ SALAH — ngintip state internal (Enzyme-style / akses instance)
expect(wrapper.state("count")).toBe(3);
// refactor internal → test patah walau UI-nya identik

// ✅ BENAR — assert yang user lihat di layar
expect(screen.getByText("Total: 3")).toBeInTheDocument();

2. Query pakai test-id padahal role/teks bisa. getByTestId itu escape hatch, bukan pilihan pertama. Kalau elemen punya role dan teks yang jelas, query lewat itu — karena begitulah user (dan screen reader) menemukannya. Bonus: kalau lo susah query lewat role/label, itu sinyal komponen lo kurang accessible. testId baru dipakai kalau beneran nggak ada cara yang lebih manusiawi.

Kesalahan #2 — testId padahal ada role
// ❌ SALAH — testId buat elemen yang jelas punya role + teks
const btn = screen.getByTestId("submit-button");

// ✅ BENAR — query seperti user: role + nama aksesibel
const btn = screen.getByRole("button", { name: /kirim/i });
// urutan prioritas: getByRole > getByLabelText > getByText > ... > getByTestId (terakhir)

3. Nggak pakai waitFor/findBy buat UI async. Klasik. Data di-fetch async, tapi lo pakai getByText langsung setelah render. Hasilnya: test gagal (elemen belum ada) atau flaky (kadang keburu, kadang enggak). Elemen yang munculnya async WAJIB ditunggu — pakai findBy* (query + auto-retry) atau waitFor. Jangan pernah pakai setTimeout manual.

Kesalahan #3 — getBy buat elemen async
// ❌ SALAH — getBy langsung buat data yang munculnya async
render(<UserList />);
expect(screen.getByText("Arif")).toBeInTheDocument(); // Arif belum ke-fetch → gagal/flaky

// ✅ BENAR — tunggu elemen async muncul
render(<UserList />);
expect(await screen.findByText("Arif")).toBeInTheDocument();
// atau: await waitFor(() => expect(screen.getByText("Arif")).toBeInTheDocument());

4. Snapshot semuanya (snapshot-everything). Snapshot komponen raksasa (300 baris) kelihatan gampang, tapi tiap perubahan kecil bikin diff gede — dan orang cenderung nge-approve update snapshot tanpa baca. Akhirnya snapshot nge-lock markup, bukan behavior, dan nggak ada yang percaya. Assert bagian spesifik yang penting. Snapshot kecil buat output stabil masih oke; snapshot raksasa = liability.

Kesalahan #4 — snapshot raksasa
// ❌ SALAH — snapshot seluruh komponen besar, di-update asal pas gagal
expect(container).toMatchSnapshot(); // 300 baris; diff gede → di-approve tanpa dibaca

// ✅ BENAR — assert bagian spesifik yang bermakna
expect(screen.getByRole("heading")).toHaveTextContent("Profil Arif");
expect(screen.getByRole("button", { name: /simpan/i })).toBeEnabled();

5. Mocking kebanyakan (over-mocking). Kalau lo mock hampir semua yang di-render, test lo cuma nge-test mock — bukan kode asli. Puncak absurditasnya: nge-mock komponen yang justru lagi lo test. Aturan: mock CUMA di batas — network (MSW) atau modul eksternal yang beneran nggak bisa jalan di test. Sisanya render asli, biar test-nya beneran nge-test integrasi.

Kesalahan #5 — mock terlalu banyak
// ❌ SALAH — mock komponen yang justru lagi di-test
vi.mock("./Cart"); // Cart di-mock → test "Cart nampil total" nggak nge-test apa-apa

// ✅ BENAR — mock cuma di batas network, render komponen asli
server.use(http.get("/api/cart", () => HttpResponse.json({ total: 42000 })));
render(<Cart />);
expect(await screen.findByText("Rp 42.000")).toBeInTheDocument();

6. Nge-test internal library, bukan behavior yang keliatan user. Contoh klasik: expect(mockNavigate).toHaveBeenCalledWith("/dashboard"). Itu nge-test React Router (yang udah dites tim-nya), bukan app lo. User nggak lihat "fungsi navigate dipanggil" — mereka lihat halaman dashboard muncul. Assert hasilnya, bukan pemanggilan internal library-nya.

Kesalahan #6 — assert internal library
// ❌ SALAH — nge-test detail library pihak ketiga
expect(mockNavigate).toHaveBeenCalledWith("/dashboard"); // itu tugasnya React Router

// ✅ BENAR — assert hasil yang user LIHAT setelah navigasi
await user.click(screen.getByRole("button", { name: /masuk/i }));
expect(await screen.findByRole("heading", { name: /dashboard/i })).toBeInTheDocument();

LATIHAN — TULIS TEST-NYA SENDIRI

Coba jawab di kepala (atau tulis di editor) sebelum buka solusi. Fokus: query seperti user, assert behavior, tunggu yang async.

Latihan 1: Tulis test RTL buat LikeButton klik untuk lihat solusi

Ada komponen LikeButton: awal tombol bertuliskan "Suka" dan teks "Total suka: 0". Diklik sekali → tombol jadi "Disukai" dan total jadi 1. Tulis satu test yang meng-cover perilaku ini — query lewat role, assert teks yang keliatan.

Solusi
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { LikeButton } from "./LikeButton";

it("toggle teks & naikin hitungan saat diklik", async () => {
  const user = userEvent.setup();
  render(<LikeButton />);

  const btn = screen.getByRole("button", { name: /suka/i });
  expect(btn).toHaveTextContent("Suka");
  expect(screen.getByText(/total suka:/i)).toHaveTextContent("0");

  await user.click(btn);
  expect(screen.getByRole("button")).toHaveTextContent("Disukai");
  expect(screen.getByText(/total suka:/i)).toHaveTextContent("1");
});

Kunci: nggak ada satu pun assertion yang nyentuh state internal. Semua lewat apa yang user lihat (teks tombol + total). Refactor internal LikeButton? Test tetap hijau.

Latihan 2: Perbaiki query yang rapuh klik untuk lihat solusi

Test di bawah pakai getByTestId + nge-assert className, dan pakai fireEvent. Perbaiki jadi versi yang query seperti user dan assert behavior.

Sebelum (rapuh) → Sesudah (bener)
// ❌ Sebelum — testId + assert class + fireEvent
const btn = screen.getByTestId("submit");
expect(btn).toHaveClass("primary");   // implementation detail
fireEvent.click(btn);

// ✅ Sesudah — role + assert hasil yang user lihat
const user = userEvent.setup();
const btn = screen.getByRole("button", { name: /kirim/i });
await user.click(btn);
expect(screen.getByText(/berhasil dikirim/i)).toBeInTheDocument();

Kenapa: class CSS itu implementation detail (ganti nama class → test patah, app aman). Yang penting: tombol ke-klik dan pesan sukses muncul. Plus userEvent lebih realistis dari fireEvent.

Latihan 3: Perbaiki test async yang flaky klik untuk lihat solusi

Komponen Profile fetch nama user, lalu nampilin. Test ini kadang lulus kadang gagal. Kenapa, dan gimana benerinnya?

Solusi
// ❌ Flaky — getBy langsung, data belum datang saat baris ini jalan
render(<Profile />);
expect(screen.getByText("Arif")).toBeInTheDocument();

// ✅ Benar — tunggu elemen async muncul pakai findBy
render(<Profile />);
expect(await screen.findByText("Arif")).toBeInTheDocument();

Kenapa flaky: getByText ngecek SEKALI, langsung. Kalau fetch belum resolve, "Arif" belum ada → gagal. findByText nge-retry sampai muncul (atau timeout). Bonus poin: sekalian assert "Loading..." muncul dulu sebelum data datang.

CHALLENGE — LIKE BUTTON YANG LULUS "TEST BEHAVIOR"

Ini testing-by-behavior in action. Runner di bawah itu efektif harness test DOM: dia nge-klik tombol lo lalu assert apa yang muncul di layar — persis seperti test RTL beneran. Tombol Like harusnya: awal bertuliskan "Suka" (hitungan 0), diklik → jadi "Disukai" (hitungan 1), diklik lagi → balik "Suka" (hitungan 0). Starter-nya handleClick-nya masih kosong, jadi beberapa test bakal gagal. Lengkapin sampai semua hijau. Klik ▶ Test.

Solusi Challenge — buka kalau udah nyerah (atau mau ngecek) klik untuk lihat solusi

Isi handleClick dengan dua updater. Perhatiin liked di dalam setCount itu nilai render saat ini (sebelum toggle) — jadi logikanya konsisten: belum liked → +1, sudah liked → −1.

Solusi lengkap handleClick
function handleClick() {
  setLiked(prev => !prev);
  setCount(prev => (liked ? prev - 1 : prev + 1));
}

Dan ini yang keren: test-nya nggak peduli lo pakai dua useState atau satu objek state atau useReducer — selama teks di layar bener, test lulus. Itulah test behavior, bukan implementation.

RECAP — YANG UDAH KITA BAHAS

Testing bukan opsional — itu safety net buat deploy dengan percaya diri

  • ✓Mindset — Testing = rem kendaraan. Confidence buat deploy dan refactor. Pyramid: unit > integration > E2E.
  • ✓React Testing Library — Test behavior, bukan implementation. getByRole priority query.
  • ✓userEvent — Simulasi user asli. Selalu prefer daripada fireEvent.
  • ✓Async + MSW — Mock API di network level. waitFor + findBy buat async.
  • ✓MSW 2.0 — Mock once, run everywhere. Handler variations: success, error, delay.
  • ✓Playwright — E2E: fast, reliable, multi-browser. Trace viewer, codegen.
  • ✓Best Practices — Test behavior, 1 assertion per concept, edge cases, CI/CD.
  • ✓Mini Project — Calculator test suite: unit, interaction, edge cases, chain ops.
