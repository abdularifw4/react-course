🤔 KENAPA SIH HARUS NGETEST?

Oke, jujur. Dulu aku juga ngerasa testing itu buang-buang waktu. "Kan udah aku cek manual, jalan kok." Eits, kata siapa?

Bayangin gini: kamu punya counter sederhana. Tinggal tombol + dan -. Gampang kan? Terus satu bulan kemudian, kamu refactor, nambah fitur, ubah state management... dan tiba-tiba counter-nya nggak ke-reset pas user logout. User komplen. Kamu panik di jam 12 malam.

Nah, itulah kenapa testing itu penting. Bukan buang waktu — tapi INVESTASI. Sekali tulis test, dia jaga kode kamu 24/7. Kayak security guard yang nggak pernah tidur 🏠

Di dunia React, ada dua tools utama yang bakal jadi best friend kamu:

  • ✦Jest — si "hakim" yang jalanin test dan ngasih tau lulus atau nggak
  • ✦React Testing Library (RTL) — si "pengguna palsu" yang ngetes komponen kamu kayak user asli

💡 Analogi mudahnya: Jest itu kayak ruang ujian + pengawasnya. RTL itu kayak murid yang lagi ujian — dia buka komponen, klik tombol, baca teks, persis kayak user beneran. Kalau hasilnya sesuai ekspektasi, 👏 LULUS!

Sebelum mulai ngetest, kita perlu setup dulu environment-nya. Berikut konfigurasi Jest + RTL yang umum dipake di proyek React (Vite/Create React App):

Setup: Install & Konfigurasi Jest + RTL 
# 1. Install dependencies
npm install --save-dev jest @testing-library/react \
  @testing-library/jest-dom @testing-library/user-event \
  jest-environment-jsdom babel-jest @babel/preset-env \
  @babel/preset-react

# 2. Bikin setup file: src/setupTests.js
echo "import '@testing-library/jest-dom';" > src/setupTests.js

# 3. Jalankan test
npx jest                 # semua test file
npx jest Counter.test    # file spesifik
npx jest --watch         # mode watch (auto re-run)
npx jest --coverage      # dengan laporan coverage
jest.config.js — Konfigurasi Dasar 
// jest.config.js
module.exports = {
  testEnvironment: 'jsdom',          // simulate browser environment
  setupFilesAfterEnv: ['<rootDir>/src/setupTests.js'],
  moduleNameMapper: {
    '\\.(css|less|scss)$': '<rootDir>/__mocks__/styleMock.js',
    '^@/(.*)$': '<rootDir>/src/$1',   // alias @/ -> src/
  },
  transform: {
    '^.+\\.(js|jsx|ts|tsx)$': 'babel-jest',
  },
  testMatch: [
    '**/__tests__/**/*.test.(js|jsx|ts|tsx)',
    '**/?(*.)+(spec|test).(js|jsx|ts|tsx)',
  ],
  collectCoverageFrom: [
    'src/**/*.{js,jsx,ts,tsx}',
    '!src/main.tsx',           // exclude entry point
    '!src/**/*.d.ts',
  ],
};

⚡ JEST + RTL: DYNAMIC DUO TESTIMU

Yuk langsung liat contoh nyata. Misal kita punya komponen Counter yang super simpel — ada angka, tombol "Tambah", tombol "Kurang". Kita mau ngetes: apakah dia beneran berfungsi?

// Counter.test.tsx — test pertamamu! 🎉
import { render, screen, fireEvent } from '@testing-library/react';
import Counter from './Counter';

describe('Counter', () => {
  test('renders initial count', () => {
    render(<Counter />);
    expect(screen.getByText('Count: 0')).toBeInTheDocument();
  });

  test('increments count on click', () => {
    render(<Counter />);
    const button = screen.getByText('Tambah');
    fireEvent.click(button);
    expect(screen.getByText('Count: 1')).toBeInTheDocument();
  });

  test('decrements count on click', () => {
    render(<Counter />);
    fireEvent.click(screen.getByText('Kurang'));
    expect(screen.getByText('Count: -1')).toBeInTheDocument();
  });
});

Gimana, nggak serem kan? 🥳 Mari kita bedah pelan-pelan:

  • ✦render() — kayak nyalain TV. Komponen kamu tampil di "layar virtual" yang bisa dites
  • ✦screen — kayak mata kamu. Bisa cari teks, tombol, input — apa aja yang ada di "layar virtual"
  • ✦fireEvent.click() — kayak jari kamu ngetuk layar hp. Mensimulasikan aksi user
  • ✦expect() — kayak kamu bilang, "Eh, harusnya gini ya..." kalau nggak sesuai, Jest bakal njerimin 💥

⚠️ HAL YANG SERING SALAH: Jangan ngetest implementation detail! Contoh: ngetes apakah useState dipanggil, atau apakah fungsi internal ke-trigger. Itu membleeeeet. Fokusnya: "Apa yang user liat dan lakuin?" Bukan bagaimana kode-nya kerja di dalam.

💡 Mindset RTL: "Test behavior, not implementation." Kamu itu jadi user, bukan developer. User nggak peduli kamu pake useState atau useReducer. Yang penting: klik tombol → angka berubah. Titik. ✅

Sekarang kita naik level dikit — ngetes form dengan validasi. Ini skenario yang sering banget ketemu di app beneran: user isi form, klik submit, muncul error kalau kosong, atau sukses kalau valid.

Ngetes Form dengan Validasi & Submit 
// LoginForm.tsx — komponen yang mau dites
function LoginForm({ onSubmit }) {
  const [email, setEmail] = useState('');
  const [error, setError] = useState('');

  const handleSubmit = (e) => {
    e.preventDefault();
    if (!email.includes('@')) {
      setError('Email nggak valid 😡');
      return;
    }
    onSubmit(email);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        placeholder="email kamu..."
        value={email}
        onChange={(e) => { setEmail(e.target.value); setError(''); }}
      />
      {error && <p role="alert">{error}</p>}
      <button type="submit">Masuk</button>
    </form>
  );
}

// LoginForm.test.tsx
import userEvent from '@testing-library/user-event';

test('menampilkan error kalau email invalid', async () => {
  const user = userEvent.setup();
  const handleSubmit = jest.fn();
  render(<LoginForm onSubmit={handleSubmit} />);

  // Ketik email yang nggak valid
  await user.type(screen.getByPlaceholderText(/email/i), 'arif-bukan-email');
  await user.click(screen.getByRole('button', { name: /masuk/i }));

  // Cek error muncul
  expect(screen.getByRole('alert')).toHaveTextContent('Email nggak valid');
  // Pastikan onSubmit NGGAK dipanggil
  expect(handleSubmit).not.toHaveBeenCalled();
});

test('submit sukses kalau email valid', async () => {
  const user = userEvent.setup();
  const handleSubmit = jest.fn();
  render(<LoginForm onSubmit={handleSubmit} />);

  await user.type(screen.getByPlaceholderText(/email/i), 'arif@example.com');
  await user.click(screen.getByRole('button', { name: /masuk/i }));

  expect(handleSubmit).toHaveBeenCalledWith('arif@example.com');
  expect(screen.queryByRole('alert')).not.toBeInTheDocument(); // no error
});

💡 Pro tip: Pakai userEvent bukan fireEvent kalau bisa! userEvent mensimulasikan interaksi user secara lebih realistis (trigger blur, focus, dll). Cuma pakai fireEvent untuk kasus simpel.

🌐 TEST KOMPONEN ASYNC (YANG FETCH DATA)

Nah, ini yang sering bikin pusing 😵‍💫 Komponen kamu fetch data dari API. Pas dijalanin di test... GAGAL. Karena kan nggak ada API-nya di environment test!

Dulu orang pake mock manual, atau jest.mock. Tapi itu ribet dan fragile. Sekarang ada cara yang lebih keren: MSW (Mock Service Worker).

// UserList.test.tsx — ngetes yang fetch data
import { render, screen, waitFor } from '@testing-library/react';
import { rest } from 'msw';
import { setupServer } from 'msw/node';
import UserList from './UserList';

// Setup "server palsu" yang jawab API request
const server = setupServer(
  rest.get('/api/users', (req, res, ctx) => {
    return res(ctx.json([
      { id: '1', name: 'Arif' },
      { id: '2', name: 'Budi' }
    ]));
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test('renders user list', async () => {
  render(<UserList />);

  // Loading... (user liat ini dulu)
  expect(screen.getByText('Loading...')).toBeInTheDocument();

  // Tunggu data dateng, terus cek nama muncul
  await waitFor(() => {
    expect(screen.getByText('Arif')).toBeInTheDocument();
  });
  expect(screen.getByText('Budi')).toBeInTheDocument();
});

💡 Kenapa MSW keren banget? Bayangin MSW itu kayak prank caller yang pura-pura jadi server API kamu 😂 Dia intercept semua HTTP request di level network — artinya nggak peduli kamu pake axios, fetch, atau library apa pun, mock-nya tetep jalan! Bedanya sama axios mock yang cuma nge-mock library-nya doang.

Perhatikan juga waitFor() — itu kayak kamu bilang, "Tunggu dulu sampe ini muncul, kalau lebih dari 5 detik baru gagal." Karena fetch data itu butuh waktu (meski cuma beberapa milidetik di test), kita perlu kasih dia waktu ⌛

Terus gimana cara ngetes custom hook? Kan hook nggak bisa dipanggil di luar komponen... Tenang, RTL punya senjata rahasia: renderHook!

Ngetes Custom Hook dengan renderHook 
// useLocalStorage.ts — custom hook kita
import { useState, useEffect } from 'react';

function useLocalStorage<T>(key: string, initialValue: T) {
  const [value, setValue] = useState<T>(() => {
    const stored = window.localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initialValue;
  });

  useEffect(() => {
    window.localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue] as const;
}

// useLocalStorage.test.ts
import { renderHook, act } from '@testing-library/react';
import { useLocalStorage } from './useLocalStorage';

test('baca nilai awal dari localStorage', () => {
  window.localStorage.setItem('theme', JSON.stringify('dark'));

  const { result } = renderHook(() => useLocalStorage('theme', 'light'));

  expect(result.current[0]).toBe('dark');  // ambil dari storage
});

test('simpan nilai baru ke localStorage', () => {
  const { result } = renderHook(() => useLocalStorage('count', 0));

  act(() => {
    result.current[1](10);  // setValue(10)
  });

  expect(result.current[0]).toBe(10);
  expect(window.localStorage.getItem('count')).toBe('10');
});

test('fallback ke initialValue kalau storage kosong', () => {
  const { result } = renderHook(() => useLocalStorage('kosong', 'default'));
  expect(result.current[0]).toBe('default');
});

💡 Penting: Selalu bungkus update state di dalam act() saat ngetes hook! React butuh act() untuk tahu kalo ada update state yang harus di-flush. Tanpa itu, kamu bakal dapet warning ⚠️

🎭 E2E TESTING DENGAN PLAYWRIGHT

Oke, unit test sama integration test udah. Tapi ada satu level lagi yang bikin kamu makin pede: End-to-End (E2E) Testing.

Analoginya gini: unit test itu kayak ngetes mesin mobil satu per satu — engine oke? rem oke? lampu oke? E2E test itu... kamu beneran nyetir mobilnya keliling kota 🚗💨

Playwright adalah tool E2E yang lagi hype banget (dan beneran bagus). Dia buka browser beneran, klik, ketik, scroll — persis kayak user. Support Chrome, Firefox, Safari. Dan yang paling asik: kalau test gagal, dia auto-screenshot biar kamu tau salahnya di mana 📸

// tests/user-flow.spec.ts — nyetir "mobil" 🚗
import { test, expect } from '@playwright/test';

test('user login flow', async ({ page }) => {
  // Buka halaman login
  await page.goto('/login');

  // Isi form kayak user beneran
  await page.fill('[name="email"]', 'arif@example.com');
  await page.fill('[name="password"]', 'password123');

  // Klik tombol submit
  await page.click('button[type="submit"]');

  // Cek: udah pindah ke dashboard?
  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1')).toContainText('Dashboard');
});

// Visual regression test — cek tampilan nggak berubah drastis
test('homepage visual', async ({ page }) => {
  await page.goto('/');
  await expect(page).toHaveScreenshot('homepage.png');
});

Yang terakhir itu toHaveScreenshot() — ini fitur visual regression yang sangat berguna. Playwright nyimpen screenshot, terus tiap kali test dijalanin, dia bandingkan sama screenshot sebelumnya. Kalau ada perbedaan (misalnya layout hancur), test gagal dan kamu bisa liat diff-nya. Asik kan? 😎

⚠️ Jangan kebanyakan E2E! E2E test itu lambat dan mahal (resource-nya). Pake unit test + integration test untuk sebagian besar logic, dan E2E cuma untuk critical user flows kayak login, checkout, registrasi. Kira-kira 10-20% dari total test kamu aja.

🧠 TESTING PYRAMID: PETA JALAN TEST KAMU

Sebelum lanjut, ada satu konsep penting yang bakal ngasih kamu clarity soal "test apa yang harus ditulis." Namanya Testing Pyramid.

Bayangin piramida... beneran piramida ya 👇

  • 🧱
    BAGIAN BAWAH (paling banyak): Unit Test
    Test fungsi/helper/hook kecil. Cepat, murah, reliable. Pakai Jest + RTL. Ini fondasi test kamu.
  • 🏲
    BAGIAN TENGAH: Integration Test
    Test beberapa komponen yang kerja bareng. Misalnya form + validation + API call. Masih pakai Jest + RTL + MSW.
  • 🏔️
    PUNCAK (paling sedikit): E2E Test
    Test flow user utama dari awal sampe akhir. Pakai Playwright. Lambat tapi bikin pede banget.

💡 Rule of thumb: Kalau kamu bingung harus nulis test apa, tanya ke diri sendiri: "Kalau bagian ini rusak, user bakal paling kena dampak di mana?" Itu yang prioritas duluan ditest.

Biar tim kamu konsisten, kamu bisa set coverage threshold di Jest. Artinya: kalau coverage turun di bawah angka tertentu, CI bakal gagal. Enak banget buat jaga kualitas 🛡️

Coverage Threshold — Jaga Kualitas Tim 
// jest.config.js (lanjutan)
module.exports = {
  // ... config lainnya
  coverageThreshold: {
    global: {
      branches: 80,    // 80% if/else tertutup
      functions: 80,   // 80% fungsi dipanggil
      lines: 80,       // 80% baris dieksekusi
      statements: 80,  // 80% statements
    },
    // Bisa juga per-file spesifik
    './src/components/**/*.tsx': {
      branches: 90,
      lines: 90,
    },
  },
};

// Output coverage report yang lucu banget dibaca:
// ----------|---------|----------|---------|---------|-------------------
// File      | % Stmts | % Branch | % Funcs | % Lines | Uncovered Line #s
// ----------|---------|----------|---------|---------|-------------------
// All files |   87.5  |   82.1   |   85.0  |   88.2  |
// Counter   |   100   |   100    |   100   |   100   |
// UserList  |   75.0  |   66.6   |   66.6  |   75.0  | 12,18-20

📝 RECAP: YANG HARUS KAMU BAWA PALING

  • ✅Testing itu investasi, bukan buang waktu. Satu jam nulis test bisa selamatin kamu dari bug 3 jam di production
  • ✅Jest + RTL adalah combo utama buat test komponen React. Test behavior, bukan implementation!
  • ✅MSW itu cara modern mock API — intercept di level network, jadi bisa dipake library HTTP apa aja
  • ✅Playwright buat E2E test — buka browser beneran, test critical user flows aja
  • ✅Testing Pyramid — banyak unit test di bawah, sedikit E2E di atas. Jangan kebalik!

Sekarang kamu udah punya senjata buat bikin kode yang nggak cuma jalan, tapi juga terbukti jalan. Selamat tidur nyenyak 💤🛌
