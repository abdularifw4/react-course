CONTEXT API — STATE GLOBAL TANPA LIBRARY 🌐

Fundamental · 10 menit baca

Di Bab 4 kita udah kenal useContext. Sekarang bahas lebih dalam — Context API itu cara React menangani state yang perlu di-share ke banyak komponen tanpa prop drilling. Prop drilling itu kayak numpuk: A → B → C → D, cuma buat kirim data dari A ke D. Capek dan berantakan. 😤

Context API itu kayak pengumuman sekolah — guru nggak perlu kasih tahu satu-satu ke tiap murid. Cuma taruh di papan pengumuman, semua murid bisa baca. Begitu juga Context: data-nya di "publish" satu kali, semua komponen di bawah Provider bisa akses. Oh ternyata gitu! 🤯

Tapi Context punya weakness: re-render cascade. Kalau context value berubah, SEMUA komponen yang consume context itu re-render — bahkan yang cuma butuh sebagian data. Di app besar, ini bisa bikin performance turun drastis. Context bagus buat global state yang jarang berubah (theme, auth, locale). Buat state yang sering berubah (cart, form data, real-time data), library state management lebih cocok.

Best practice: pisah context per concern. Jangan bikin satu "AppContext" yang nampung semua state global. Pisah: AuthContext, ThemeContext, CartContext. Setiap context cuma re-render komponen yang consume-nya. Ini mengurangi unnecessary re-render secara signifikan. 🎯

Pattern yang solid: buat Context → buat Provider component → buat custom hook. Custom hook handle null check dan throw error kalau lupa wrap Provider. Ini bikin API-nya clean dan safe — developer yang lupa wrap Provider langsung dapet error message yang jelas.

Context API — Theme Toggle Pattern
import { createContext, useContext, useState, type ReactNode } from 'react';

// 1. Define context type
interface ThemeCtx {
  theme: 'light' | 'dark';
  toggle: () => void;
  setTheme: (t: 'light' | 'dark') => void;
}

// 2. Create context
const ThemeContext = createContext<ThemeCtx | undefined>(undefined);

// 3. Custom hook — ALWAYS do this
function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be inside ThemeProvider');
  return ctx;
}

// 4. Provider component
function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<'light' | 'dark'>('light');

  const toggle = () => setTheme(t => t === 'light' ? 'dark' : 'light');

  return (
    <ThemeContext.Provider value={{ theme, toggle, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// 5. Pakai di komponen mana aja
function Header() {
  const { theme, toggle } = useTheme();
  return (
    <header className={theme === 'dark' ? 'bg-gray-900 text-white' : 'bg-white'}>
      <h1>Theme: {theme}</h1>
      <button onClick={toggle}>Toggle {theme === 'light' ? '🌙' : '☀️'}</button>
    </header>
  );
}

💡 Performance trick: Pisah context yang sering berubah dari yang jarang. Misal: AuthContext (jarang berubah) terpisah dari CartContext (sering berubah). Pas cart update, komponen yang cuma butuh auth nggak ikut re-render. Ini pattern yang dipake banyak production app.

ZUSTAND — STATE MANAGEMENT MINIMALIS ⚡

Library · 12 menit baca

Zustand (baca: "zoo-stand") adalah state management library yang minimalis. Nggak perlu Provider wrapper, nggak perlu reducer, nggak perlu actions — cuma sebuah store yang bisa di-import di mana aja. API-nya super simple tapi powerful. Ini favorit komunitas React saat ini. 🔥

Analoginya: Zustand itu kayak lemari es — semua orang bisa buka, ambil, dan simpan makanan kapan aja tanpa permission. Context API itu kayak dapur bersama — makanan di-share tapi semua orang di area itu harus dengar kalau ada yang ambil. Zustand lebih simple dan less ceremony. Oh ternyata gitu! 🤯

Kenapa Zustand populer? Pertama: boilerplate minimal. Bandingkan: Context butuh createContext + Provider + useContext + custom hook. Zustand cuma create() satu kali, langsung pakai. Kedua: selective re-render. Komponen cuma re-render kalau bagian state yang dia pilih berubah. Ini beda fundamental dari Context yang re-render semua consumer. Ketiga: devtools built-in. Redux DevTools langsung compatible tanpa config.

Yang perlu diperhatikan: Zustand store itu global by default — bisa diakses dari mana aja. Ini powerful tapi juga berarti komponen bisa mutate state dari mana aja. Di tim besar, disiplin penting: tentukan convention mana komponen boleh update state, mana cuma read-only. Tanpa disiplin, state flow bisa jadi spaghetti. 🎯

Middleware juga powerful: persist buat save ke localStorage, immer buat immutable update yang simple, devtools buat debugging. Masing-masing bisa ditambah sebagai array di konfigurasi store.

Zustand — Counter Store
import { create } from 'zustand';
import { devtools, persist } from 'zustand/middleware';

// ✅ Store setup — cuma satu function!
interface CounterStore {
  count: number;
  increment: () => void;
  decrement: () => void;
  reset: () => void;
  incrementBy: (n: number) => void;
}

const useCounterStore = create<CounterStore>()(
  devtools(
    persist(
      (set) => ({
        count: 0,
        increment: () => set((s) => ({ count: s.count + 1 }), 'increment'),
        decrement: () => set((s) => ({ count: s.count - 1 }), 'decrement'),
        reset: () => set({ count: 0 }, 'reset'),
        incrementBy: (n) => set((s) => ({ count: s.count + n }), 'incrementBy'),
      }),
      { name: 'counter-store' } // persist to localStorage
    )
  )
);

// ✅ Pakai di komponen — selective subscribe!
function Counter() {
  const count = useCounterStore((s) => s.count);
  const increment = useCounterStore((s) => s.increment);
  const decrement = useCounterStore((s) => s.decrement);

  return (
    <div>
      <span>Count: {count}</span>
      <button onClick={increment}>+</button>
      <button onClick={decrement}>-</button>
    </div>
  );
}
Zustand — Shopping Cart
import { create } from 'zustand';

interface CartItem { id: string; name: string; price: number; qty: number; }

interface CartStore {
  items: CartItem[];
  addItem: (item: Omit<CartItem, 'qty'>) => void;
  removeItem: (id: string) => void;
  updateQty: (id: string, qty: number) => void;
  total: () => number;
  clear: () => void;
}

const useCartStore = create<CartStore>()((set, get) => ({
  items: [],
  addItem: (item) => set((s) => {
    const existing = s.items.find(i => i.id === item.id);
    if (existing) {
      return { items: s.items.map(i => i.id === item.id ? { ...i, qty: i.qty + 1 } : i) };
    }
    return { items: [...s.items, { ...item, qty: 1 }] };
  }),
  removeItem: (id) => set((s) => ({ items: s.items.filter(i => i.id !== id) })),
  updateQty: (id, qty) => set((s) => ({
    items: qty <= 0 ? s.items.filter(i => i.id !== id)
      : s.items.map(i => i.id === id ? { ...i, qty } : i)
  })),
  total: () => get().items.reduce((sum, i) => sum + i.price * i.qty, 0),
  clear: () => set({ items: [] }),
}))

⚠️ Gotcha: Gunakan selector untuk selective re-render. Kalau kamu tulis const store = useStore() tanpa selector, komponen re-render setiap KALI state apapun berubah — sama jeleknya kayak Context!

REDUX TOOLKIT — TANK BERAT YANG MODERN 🏋️

Library · 12 menit baca

Redux itu legend — state management pertama yang populer di React. Tapi Redux lama itu banyak boilerplate: actions, reducers, middleware, thunks, selectors — ratusan baris buat hal sederhana. Redux Toolkit (RTK) lahir buat fix semua masalah itu. API-nya modern, concise, dan powerful. 🎯

Analoginya: Redux Toolkit itu kayak tank modern — berat, powerful, tapi udah punya autopilot dan AC. Redux lama itu kayak tank WWII — powerful tapi perlu banyak crew (boilerplate) buat operasikan. RTK menangani complexity buat kamu. Oh ternyata gitu! 🤯

Fitur utama RTK: createSlice (auto-generate action creators + action types), createAsyncThunk (handle loading/success/error state otomatis), configureStore (one-liner store setup), dan RTK Query (data fetching & caching built-in). Dulu butuh library terpisah buat semua ini, sekarang satu package.

Tapi Redux Toolkit bukan selalu jawaban. Buat project kecil-menengah, Zustand atau bahkan Context API cukup. RTK paling cocok buat: app enterprise dengan tim besar, state yang sangat kompleks (multi-step workflows, offline-first, complex business logic), atau project yang sudah existing pakai Redux dan mau modernize. Jangan over-engineer! 🔥

Satu fitur RTK yang sering dilupakan: RTK Query. Ini data fetching library built-in di RTK — bisa replace TanStack Query/SWR kalau kamu udah pake Redux. Auto-generate hooks (useGetUsersQuery, useCreateUserMutation), caching otomatis dengan tag-based invalidation, optimistic update, dan polling. Plus, karena integrated sama Redux store, kamu bisa access fetched data dari mana aja tanpa prop passing. Powerful banget buat CRUD-heavy apps. ⚡

Redux Toolkit — createSlice
import { createSlice, createAsyncThunk, configureStore } from '@reduxjs/toolkit';

// ✅ Async thunk — auto handle loading/success/error
const fetchUsers = createAsyncThunk('users/fetch', async () => {
  const res = await fetch('/api/users');
  return res.json() as User[];
});

// ✅ Slice — actions + reducers satu tempat
const usersSlice = createSlice({
  name: 'users',
  initialState: { list: [] as User[], loading: false, error: null as string | null },
  reducers: {
    addUser: (state, action) => { state.list.push(action.payload); },
    removeUser: (state, action) => { state.list = state.list.filter(u => u.id !== action.payload); },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchUsers.pending, (s) => { s.loading = true; })
      .addCase(fetchUsers.fulfilled, (s, a) => { s.loading = false; s.list = a.payload; })
      .addCase(fetchUsers.rejected, (s) => { s.loading = false; s.error = 'Failed'; });
  },
});

// ✅ Store — one-liner
const store = configureStore({ reducer: { users: usersSlice.reducer } });

// ✅ Auto-generated action creators
store.dispatch(usersSlice.actions.addUser({ id: '1', name: 'Arif' }));
store.dispatch(fetchUsers());

💡 RTK Query: Kalau app kamu banyak data fetching (CRUD), pertimbangkan RTK Query — built-in di Redux Toolkit. Auto-generate hooks (useGetUsersQuery, useCreateUserMutation), caching otomatis, optimistic update, dan polling. Bisa replace TanStack Query/SWR buat project yang udah pakai Redux.

JOTAI — ATOMIC STATE MANAGEMENT ⚛️

Library · 11 menit baca

Jotai punya filosofi berbeda dari state management lain. Dia nggak punya satu "store" besar. Sebagai gantinya, state dipecah jadi atoms — unit state terkecil yang independent. Komponen subscribe ke atom yang dia butuh, dan cuma re-render kalau atom itu berubah. Minimal re-render, maksimal performance. ⚡

Analoginya: Redux/Zustand itu kayak supermarket besar — semua barang di satu tempat, tapi kalau ada orang buka pintu, semua orang yang di dalam terkena dampaknya. Jotai itu kayak vending machine — tiap mesin jual satu jenis barang, cuma orang yang mau barang itu yang dekat mesin. Terisolasi, efficient. Oh ternyata gitu! 🤯

Keunggulan Jotai: true selective re-render. Kalau kamu punya 100 atoms dan satu komponen cuma subscribe ke 1 atom, 99 atom lain berubah pun komponen itu nggak re-render. Ini nggak bisa dicapai Context API (re-render semua consumer) dan bahkan Zustand (selector membantu tapi tetap evaluate seluruh store).

Yang menarik: Jotai punya derived atoms — atom yang nilainya di-compute dari atom lain. Mirip computed property di Vue atau selector di Redux. Plus async atoms buat data fetching. API-nya sangat React-idiomatic — semua feels natural, kayak pake useState tapi global. 🎯

Tapi Jotai kurang cocok buat: state yang saling terkait kompleks (Redux lebih baik), atau tim yang butuh strict architecture pattern (Zustand/Redux punya convention lebih jelas). Jotai paling kuat buat UI state: form state, modal state, filter, pagination, tab — state kecil yang independent.

Jotai — Atoms & Derived State
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// ✅ Primitive atoms — unit state terkecil
const countAtom = atom(0);
const themeAtom = atom<'light' | 'dark'>('light');
const searchAtom = atom('');

// ✅ Derived atom — compute dari atom lain
const doubleCountAtom = atom((get) => get(countAtom) * 2);

// ✅ Read/Write derived atom
const formattedSearchAtom = atom(
  (get) => get(searchAtom).trim().toLowerCase(),      // read
  (_get, set, update: string) => {                     // write
    set(searchAtom, update);
  }
);

// ✅ Pakai di komponen
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const double = useAtomValue(doubleCountAtom); // read-only

  return (
    <div>
      <span>Count: {count} (Double: {double})</span>
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  );
}

// ✅ Performance: useAtomValue = cuma read, nggak re-render pas set
function CountDisplay() {
  const count = useAtomValue(countAtom);
  return <span>{count}</span>;
}

// ✅ Performance: useSetAtom = cuma write, TIDAK subscribe
function CountButtons() {
  const setCount = useSetAtom(countAtom);
  return <button onClick={() => setCount(c => c + 1)}>+</button>;
}

💡 Jotai vs Zustand: Pilih Jotai kalau state-mu banyak tapi independent (atoms). Pilih Zustand kalau state-mu saling terkait dan butuh satu source of truth (store). Keduanya bisa koexist — Jotai buat UI state, Zustand buat client state kompleks. Ingat: server state (data dari API) bukan tugas keduanya — itu ranah query library (TanStack Query/SWR).

05

PERBANDINGAN — MANA YANG PANTAS BUAT PROJECTMU? 🤔

Decision · 10 menit baca

Pilihan state management itu kayak pilih sepatu: nggak ada yang "paling baik" buat semua situasi. Sneakers cocok buat lari, boots buat hiking, formal buat kantor. Masing-masing punya trade-off. Yang penting: pilih yang sesuai dengan kebutuhan projectmu. 🎯

Context API — Start here. Tanpa install apapun, built-in React. Cocok buat: theme toggle, auth state, locale, config. Limitasi: re-render cascade kalau value sering berubah. Max 3-5 contexts sebelum jadi berantakan.

Zustand — Sweet spot. Minimal API, selective re-render, persist middleware. Cocok buat: 80% project React. Cart, filters, form state, UI state, app state. Install: npm i zustand. Learning curve: 15 menit. Ini yang gue recommend buat kebanyakan developer.

Redux Toolkit — Heavy duty. Cocok buat enterprise app, tim besar, state super kompleks. RTK Query bisa replace data fetching library. Trade-off: boilerplate lebih banyak, learning curve lebih tinggi. Tapi kalau udah paham, development speed-nya cepat karena convention-nya jelas.

Jotai — Atomic & minimalist. Cocok buat app dengan banyak independent UI state. Performance-nya top-tier karena true atomic re-render. Trade-off: pattern-nya beda dari convention lain, bisa bingung buat developer yang terbiasa Redux/Zustand.

Decision Matrix — Quick Reference
// ===== KAPAN PAKAI APA =====
//
// Context API:
//   ✅ Theme toggle, auth, locale, config
//   ✅ Project kecil, nggak mau install library
//   ❌ State yang sering berubah (cart, real-time)
//   ❌ Lebih dari 5 global states
//
// Zustand:
//   ✅ Middle-ground — cocok buat 80% project
//   ✅ Cart, filters, form, UI state
//   ✅ Tim kecil-menengah, fast iteration
//   ❌ Enterprise app dengan state super kompleks
//
// Redux Toolkit:
//   ✅ Enterprise app, tim besar (10+ dev)
//   ✅ State kompleks, multi-step workflows
//   ✅ Udah pakai Redux di project existing
//   ❌ Project kecil (overkill)
//
// Jotai:
//   ✅ Banyak independent UI state (form, modal, tabs)
//   ✅ Performance-critical app
//   ✅ Dev yang suka functional/atomic pattern
//   ❌ State yang saling terkait kompleks
//
// ===== RECOMMENDASI GUE =====
// 1. Start dengan Context API
// 2. Kalau mulai "banyak", pindah ke Zustand
// 3. Kalau enterprise/large team, pertimbangkan RTK
// 4. Kalau performance jadi masalah, pertimbangkan Jotai

⚠️ Anti-pattern: JANGAN pake semua library sekaligus di satu project. Pick ONE primary state management dan stick with it. Mixing Context + Zustand + Redux + Jotai = debugging nightmare. Pilih yang paling cocok, dan gunakan consistently.

TESTING STATE MANAGEMENT — JANGAN TRUST, VERIFY 🧪

Testing · 10 menit baca

Nggak peduli library apa yang kamu pake, state management yang nggak di-test itu bom waktu. Bug state itu paling susah di-debug — karena tergantung sequence of actions, timing, dan cross-component dependency. Satu state salah bisa crash seluruh app. Jadi test itu bukan opsional. 🧪

Good news: testing state management itu straightforward. Untuk Context/Zustand/Redux, kamu test di level unit — render komponen, trigger action, cek output. Untuk Jotai, test atom secara terpisah. Tools-nya sama: React Testing Library + Vitest/Jest.

Pattern yang aman: arrange-act-assert. Arrange = setup store + render component. Act = trigger user action (click, type). Assert = cek DOM/ceks store state. Jangan test implementation detail (internals), test behavior (apa yang user lihat). Ini bikin test robust dan nggak fragile. Oh ternyata gitu! 🤯

Untuk async state (data fetching), mock fetch API atau gunakan MSW (Mock Service Worker). MSW intercept HTTP request di network level — jadi nggak perlu mock implementation detail. Test lebih realistic dan nggak fragile terhadap refactoring.

Untuk Redux Toolkit, testing bahkan lebih mudah — redux-mock-store buat mock dispatch, atau langsung test reducer-nya sebagai pure function. Reducer test itu paling cepat karena nggak perlu render — cuma input state + action, assert output state. Unit test par excellence. 🎯

Pattern yang gue recommend: pyramid testing. Banyak unit test (store, reducer, atom), beberapa integration test (komponen + store), sedikit E2E test (user flow). Unit test cepat dan murah. E2E test lambat dan mahal. Balance yang tepat bikin CI/CD cepat tapi tetap confident.

Testing Zustand Store
import { render, screen, fireEvent } from '@testing-library/react';
import { useCounterStore } from './store';

// ✅ Test store directly (unit test)
test('increment increases count', () => {
  const { increment } = useCounterStore.getState();
  expect(useCounterStore.getState().count).toBe(0);
  increment();
  expect(useCounterStore.getState().count).toBe(1);
});

// ✅ Test via component (integration test)
function CounterDisplay() {
  const count = useCounterStore((s) => s.count);
  const increment = useCounterStore((s) => s.increment);
  return <button onClick={increment}>Count: {count}</button>;
}

test('button click updates counter', () => {
  // Reset state before each test
  useCounterStore.setState({ count: 0 });

  render(<CounterDisplay />);
  const btn = screen.getByRole('button');
  expect(btn).toHaveTextContent('Count: 0');

  fireEvent.click(btn);
  expect(btn).toHaveTextContent('Count: 1');
  expect(useCounterStore.getState().count).toBe(1);
});

💡 Testing Context: Untuk test komponen yang pake Context, bungkus render dengan Provider di render() options: render(<MyComp />, { wrapper: ThemeProvider }). Atau bikin custom renderWithProviders() helper buat reuse.

ANTI-PATTERNS — JANGAN LAKUKAN INI! 🚫

Common Mistakes · 10 menit baca

Sudah bahas tools-nya, sekarang bahas kesalahan yang sering bikin state management jadi nightmare. Ini anti-pattern yang gue lihat berulang-ulang di codebase developer — dan cara menghindarinya. Belajar dari kesalahan orang lain lebih murah daripada belajar dari kesalahan sendiri. 😅

Anti-pattern #1: Satu global store raksasa. Semua state di satu tempat — cart, auth, UI, form, theme, semua mixed. Result? Setiap perubahan kecil trigger re-render di seluruh app. Solusi: pisah store per concern, atau pake library yang support selective subscription (Zustand/Jotai).

Anti-pattern #2: State di tempat yang salah. Data yang seharusnya lokal disimpen global. Form input state, modal open/close, dropdown selected — ini contoh state yang seharusnya lokal (useState), bukan global. Global state cuma buat data yang perlu di-share antar komponen yang nggak punya parent-child relationship. 🎯

Anti-pattern #3: Prop drilling yang kebablasan. Daripada prop drill A→B→C→D, lebih baik pake Context atau event bus. Tapi kalau cuma 1-2 level, prop drilling justru lebih simple dan explicit. Nggak semua prop drilling jelek — yang jelek itu yang berlebihan.

Anti-pattern #4: Derived state disimpen. Jangan simpan computed value di state. Kalau fullName = firstName + lastName, nggak perlu simpan fullName di state. Compute di render. Kalau compute-nya mahal, pake useMemo. Simpan derived state = source of truth duplikat = potensi inconsistency. 🔥

Anti-pattern #5: Mutation langsung. Di React, state HARUS immutable. Jangan state.items.push(newItem) — ini mutate langsung. Pakai spread: setState([...state.items, newItem]). Atau pake Immer (built-in di RTK dan Zustand middleware).

Anti-pattern #6: Nggak handle loading/error state. Setiap async state HARUS punya loading, success, dan error state. Nggak handle error = crash di UI. Nggak handle loading = blank screen. Sederhana: { data, loading, error }. Selalu. Tanpa exception.

Anti-pattern #7: Sync state dari URL. Filter, search query, pagination — seharusnya sync dengan URL parameter. Kenapa? Biar user bisa bookmark, share link, dan back-forward button works. Library kayak nuqs (Next.js) atau useSearchParams bisa bantu.

⚠️ Rule of thumb: Sebelum pindah dari useState ke global state, tanya: "Apakah state ini dibutuhkan oleh komponen yang nggak berhubungan parent-child?" Kalau jawabannya tidak, useState sudah cukup. Jangan over-engineer global state!

MINI PROJECT — TASK MANAGER DENGAN ZUSTAND 📋

Praktik · 20 menit

Waktunya apply semua ilmu! Bikin Task Manager dengan Zustand — multi-board (Todo, In Progress, Done), filter, search, persist ke localStorage. Ini project yang real-world banget, nge-gabungin: Zustand store, filter/search pattern, drag-and-drop concept, dan persistence. 🔥

Architecture-nya clean: satu Zustand store handle semua state, komponen presentasi murni receive props, dan UI state (search input, filter dropdown) tetap lokal. Ini pattern yang bisa kamu scale ke app yang lebih besar.

Task Manager — Zustand Store
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

type Board = 'todo' | 'in-progress' | 'done';

interface Task {
  id: string;
  title: string;
  description: string;
  board: Board;
  priority: 'low' | 'medium' | 'high';
  createdAt: number;
}

interface TaskStore {
  tasks: Task[];
  searchQuery: string;
  filterBoard: Board | 'all';
  filterPriority: Task['priority'] | 'all';
  addTask: (task: Omit<Task, 'id' | 'createdAt'>) => void;
  moveTask: (id: string, board: Board) => void;
  deleteTask: (id: string) => void;
  setSearch: (q: string) => void;
  setFilterBoard: (b: Board | 'all') => void;
  setFilterPriority: (p: Task['priority'] | 'all') => void;
  filtered: () => Task[];
}

export const useTaskStore = create<TaskStore>()(
  persist(
    (set, get) => ({
      tasks: [],
      searchQuery: '',
      filterBoard: 'all',
      filterPriority: 'all',

      addTask: (task) => set((s) => ({
        tasks: [...s.tasks, { ...task, id: crypto.randomUUID(), createdAt: Date.now() }]
      })),

      moveTask: (id, board) => set((s) => ({
        tasks: s.tasks.map(t => t.id === id ? { ...t, board } : t)
      })),

      deleteTask: (id) => set((s) => ({
        tasks: s.tasks.filter(t => t.id !== id)
      })),

      setSearch: (q) => set({ searchQuery: q }),
      setFilterBoard: (b) => set({ filterBoard: b }),
      setFilterPriority: (p) => set({ filterPriority: p }),

      filtered: () => {
        const { tasks, searchQuery, filterBoard, filterPriority } = get();
        return tasks.filter(t => {
          if (filterBoard !== 'all' && t.board !== filterBoard) return false;
          if (filterPriority !== 'all' && t.priority !== filterPriority) return false;
          if (searchQuery && !t.title.toLowerCase().includes(searchQuery.toLowerCase())) return false;
          return true;
        });
      },
    }),
    { name: 'task-manager-store' }
  )
);
Task Manager — Board Component
import { useTaskStore } from './store';

function TaskBoard({ board, title, emoji }: { board: Board; title: string; emoji: string }) {

  const moveTask = useTaskStore((s) => s.moveTask);
  const deleteTask = useTaskStore((s) => s.deleteTask);

    'todo': 'in-progress',
    'in-progress': 'done',
    'done': null,
  };

  return (
    <div className="bg-gray-50 rounded-xl p-4">
      <h3 className="font-bold mb-4">{emoji} {title} ({tasks.length})</h3>
      <div className="space-y-3">
        {tasks.map(task => (
          <div key={task.id} className="bg-white p-3 rounded-lg shadow-sm">
            <h4 className="font-medium">{task.title}</h4>
            <p className="text-sm text-gray-500">{task.description}</p>
            <div className="flex gap-2 mt-2">

                task.priority === 'high' ? 'bg-red-100 text-red-700' :
                task.priority === 'medium' ? 'bg-yellow-100 text-yellow-700' :
                'bg-green-100 text-green-700'

              )}

LEVEL SENIOR: KAPAN CONTEXT, KAPAN ZUSTAND, KAPAN SERVER STATE 🧭

Level Senior · 15 menit baca

Ini lesson yang misahin developer yang "hafal 4 library state management" dari yang "tau kapan HARUS pake yang mana". Karena rahasia paling gede di dunia state management itu bukan soal library — tapi soal satu pertanyaan yang jarang ditanyain junior: "state ini asalnya dari mana?" Jawabannya nentuin tool yang tepat, dan salah jawab = ribuan baris kode yang salah tempat. 🎯

Pertama, luruskan model mentalnya. State di aplikasi React itu ada empat kategori, dan tiap kategori punya rumah yang beda:

1. Local UI state — nilai yang cuma dipake satu komponen: input yang lagi diketik, dropdown kebuka/ketutup, tab aktif, hover. Rumahnya: useState/useReducer. Titik. Jangan diangkat ke global. 90% state di app yang sehat itu tetap lokal.

2. Low-frequency global state — data global yang jarang berubah: theme, bahasa/locale, info user yang lagi login, feature flags. Rumahnya: Context API. "Jarang berubah" itu kunci — karena tiap kali value Context ganti, semua consumer-nya re-render. Buat data yang ganti sekali dalam sesi (login sekali, ganti theme sesekali), re-render itu murah dan nggak kerasa.

3. Complex client state — state global yang sering berubah dan saling terkait: isi keranjang belanja, state form wizard multi-step, filter + sorting tabel besar, state editor. Rumahnya: Zustand (atau Redux Toolkit / Jotai). Kenapa bukan Context? Karena Context re-render SEMUA consumer tiap update — buat state yang berubah tiap detik, itu re-render storm. Zustand pake selector: komponen cuma re-render kalau slice yang dia pilih berubah.

4. Server state — dan ini insight paling penting di seluruh bab: sebagian besar yang orang sebut "global state" itu sebenarnya bukan state, tapi cache dari data server. Daftar produk, profil user, list order, komentar — itu semua BUKAN milik client. Itu salinan data yang sumber kebenarannya ada di database, diambil lewat API. Rumahnya: TanStack Query (dulu React Query) atau SWR. Bukan Redux, bukan Zustand, bukan Context.

Analoginya: server state itu kayak foto struk belanja di HP lo 🧾. Struk aslinya (source of truth) ada di kasir/toko. Foto di HP lo cuma salinan — bisa basi (harga udah naik), perlu di-refresh, perlu tau kapan terakhir difoto, perlu di-fetch ulang kalau kebuka lagi. Naruh server state di Redux/useState itu kayak nyoba ngurus "kapan foto ini basi" manual pake tangan — capek dan pasti ada bug. TanStack Query ngurus semua itu otomatis: caching, background refetch, stale time, retry, loading/error, dedup request. Oh ternyata gitu! 🤯

Kesalahan klasik yang bikin app jadi berat: fetch data pake useEffect + useState, terus simpen hasilnya di Redux "biar bisa diakses dari mana aja". Padahal itu ngerjain ulang semua yang udah dikasih gratis sama TanStack Query — dan biasanya versi lo lebih buggy (nggak ada dedup, nggak ada cache invalidation, race condition di mana-mana). Aturan senior: kalau data-nya asalnya dari server, jangan simpen di client store. Pake server-cache library. 🔥

Sekarang, soal Context re-render pitfall — ini pertanyaan interview favorit. Masalahnya: value Context yang berupa object dibikin inline tiap render (value={{ user, setUser }}) itu referensi BARU tiap parent render, jadi semua consumer re-render walaupun isinya sama. Ada tiga cara nge-fix-nya, dari yang paling gampang:

Fix #1 — memoize value-nya. Bungkus value provider pakai useMemo supaya referensinya stabil selama isinya nggak berubah. Fix #2 — pisah context per concern. Jangan satu AppContext raksasa; pecah jadi ThemeContext, AuthContext, CartContext. Update cart nggak nyentuh consumer theme. Fix #3 — pisah state dan dispatch jadi dua Context. Komponen yang cuma butuh nge-dispatch action (tombol) nggak perlu subscribe ke state-nya, jadi nggak re-render pas state berubah. Pola ini (StateContext + DispatchContext) dipake di banyak codebase production.

Dan kalau Context tetep kerasa berat walau udah dipecah? Itu sinyal state-nya udah masuk kategori 3 (complex client state) — pindahin ke Zustand. Zustand pake external store + subscription selektif, jadi masalah "re-render semua consumer" itu hilang by design. Di playground kedua di bawah, lo bakal lihat gimana pola external store itu kerja dari nol — cuma ~15 baris, dan itu literally inti cara Zustand jalan di balik layar.

Decision framework — state ini asalnya dari mana?
// ===== TANYA INI DULU: "STATE INI ASALNYA DARI MANA?" =====
//
// Cuma dipakai 1 komponen (input, toggle, tab)?
//   -> useState / useReducer. JANGAN diangkat ke global.
//
// Global tapi JARANG berubah (theme, locale, user login, flags)?
//   -> Context API. Re-render semua consumer itu murah kalau jarang.
//
// Global, SERING berubah, saling terkait (cart, wizard, filter besar)?
//   -> Zustand (selector = selective re-render). RTK/Jotai juga oke.
//
// Datanya ASALNYA DARI SERVER (list produk, profil, order, komentar)?
//   -> TanStack Query / SWR. Ini CACHE, bukan state kamu.
//      Jangan simpan di Redux/useEffect+useState.
//
// ===== INSIGHT SENIOR =====
// Kebanyakan "global state" di app nyata = server cache yang salah kamar.
// Pindahin ke query library, dan tiba-tiba "kebutuhan Redux" kamu hilang.
Context re-render pitfall + 3 cara fix
// ❌ MASALAH — value object inline = referensi baru tiap render.
// Semua consumer re-render walau isinya nggak berubah.
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [cart, setCart] = useState([]); // cart sering banget berubah!
  return (
    <AppContext.Provider value={{ user, setUser, theme, setTheme, cart, setCart }}>
      {children} {/* update cart -> consumer theme ikut re-render 😱 */}
    </AppContext.Provider>
  );
}

// ✅ FIX #1 — memoize value biar referensinya stabil
const value = useMemo(() => ({ user, setUser }), [user]);

// ✅ FIX #2 — pisah context per concern (yang sering berubah dipisah)
<ThemeContext.Provider value={themeValue}>
  <AuthContext.Provider value={authValue}>
    {children}
  </AuthContext.Provider>
</ThemeContext.Provider>
// cart yang sering berubah? angkat ke Zustand, bukan Context.

// ✅ FIX #3 — pisah STATE dan DISPATCH jadi dua context.
// Komponen yang cuma nge-dispatch nggak subscribe state -> nggak re-render.
const StateCtx = createContext(null);
const DispatchCtx = createContext(null);
function Provider({ children }) {
  const [state, dispatch] = useReducer(reducer, initial);
  return (
    <StateCtx.Provider value={state}>
      <DispatchCtx.Provider value={dispatch}>{children}</DispatchCtx.Provider>
    </StateCtx.Provider>
  );
}
// dispatch dari useReducer itu stabil selamanya -> DispatchCtx nggak pernah
// bikin consumer-nya re-render. Tombol-tombol aman.

Playground pertama: pola production global store yang paling umum — Context + useReducer buat cart. Semua logic update terpusat di reducer, Context nyebarin state + dispatch. Tantangan: tambahin action { type: 'clear' } yang ngosongin keranjang, lalu bikin tombolnya. 👇

Playground kedua: bikin mini "Zustand" dari nol pakai useSyncExternalStore. Store hidup di LUAR React, tiap komponen subscribe ke slice yang dia butuh lewat selector. Perhatiin render counter-nya: klik "count + 1" cuma bikin CountView re-render — NameView DIEM. Ini selective subscription yang nggak bisa dicapai Context. 👇

💡 OH TERNYATA GITU! Urutan default seorang senior pas ketemu kebutuhan state baru: useState → lift state up → Context → Zustand → (server? TanStack Query). Naik satu level cuma kalau level sekarang beneran nyusahin. Kebanyakan bug arsitektur state datang dari lompat kejauhan — langsung Redux buat app to-do list, atau naruh input form di global store. Start small, escalate on evidence.

⚠️ JANGAN TERJERAT! Kalau lo lagi nulis useEffect(() => { fetch(...).then(setData) }, []) lalu naruh data ke Redux/Zustand "biar global" — STOP. Itu server state. Lo lagi bikin ulang caching, dedup, dan revalidation yang udah beres di TanStack Query/SWR — versi lo hampir pasti punya race condition & cache basi. Server state punya kamarnya sendiri.

KESALAHAN UMUM — DOSA STATE MANAGEMENT YANG PALING SERING 🚨

Enam kesalahan yang paling sering muncul di code review — pola: kode salah → kenapa → kode benar

1. Naruh SEMUA state di global. Baru belajar Zustand/Redux terus semua state diangkat ke global "biar rapi" — termasuk input yang lagi diketik, modal kebuka/ketutup, dropdown selected. Padahal state yang cuma dipake satu komponen itu rumahnya useState. Global state bikin komponen susah dipindah/dites, dan tiap ketikan nge-update store yang di-subscribe banyak komponen = re-render sia-sia.

Kesalahan #1 — input form di global store
// ❌ SALAH — input lokal diangkat ke global store
const useStore = create((set) => ({
  searchInput: '',
  setSearchInput: (v) => set({ searchInput: v }), // tiap keystroke update global 😱
}));

// ✅ BENAR — state yang cuma dipakai 1 komponen tetap lokal
function SearchBar() {
  const [q, setQ] = useState(''); // lokal. cepat, terisolasi, gampang dites
  return <input value={q} onChange={e => setQ(e.target.value)} />;
}

2. Pake Context buat update yang high-frequency (re-render storm). Context re-render SEMUA consumer tiap value berubah. Naruh sesuatu yang berubah tiap milidetik — posisi mouse, progress video, nilai form yang diketik — di Context = tiap perubahan nge-trigger re-render seluruh subtree. Buat data high-frequency: pakai state lokal, atau external store dengan selector (Zustand), atau ref kalau nggak perlu dirender.

Kesalahan #2 — high-frequency value di Context
// ❌ SALAH — mouse position (update puluhan kali/detik) di Context
<MouseContext.Provider value={{ x, y }}>
  <WholeApp /> {/* SEMUA consumer re-render tiap mouse gerak 💀 */}
</MouseContext.Provider>

// ✅ BENAR — high-frequency state di external store + selector,
//    atau lokal di komponen yang beneran butuh. Zustand cuma re-render
//    komponen yang subscribe ke slice { x, y } lewat selector.
const usePointer = create((set) => ({ x: 0, y: 0, move: (x, y) => set({ x, y }) }));
const x = usePointer((s) => s.x); // cuma komponen ini yang ikut update

3. Satu Context raksasa (God Context). Semua state global — theme, user, cart, modal, notif — ditumpuk di satu AppContext. Akibatnya: update cart bikin komponen yang cuma butuh theme ikut re-render, karena mereka subscribe ke Context yang sama. Solusi: pecah jadi context per concern. Bonus: value tiap provider di-memoize.

Kesalahan #3 — God Context
// ❌ SALAH — semua digabung, semua consumer sensitif ke semua perubahan
const AppContext = createContext(null);
<AppContext.Provider value={{ theme, user, cart, modal, notifs }}> ...
// update notifs -> komponen yang cuma pakai theme ikut re-render

// ✅ BENAR — pisah per concern; yang jarang berubah dipisah dari yang sering
<ThemeContext.Provider value={themeValue}>
  <AuthContext.Provider value={authValue}>
    <App />
  </AuthContext.Provider>
</ThemeContext.Provider>
// cart & notifs yang sering berubah -> Zustand, bukan Context.

4. Over-globalizing buat ngindarin prop drilling 1-2 level. Prop drilling emang jelek kalau 4-5 level. Tapi ngangkat state ke global cuma buat ngirim prop turun 1-2 level itu obat yang lebih parah dari penyakitnya — lo nambah indirection global buat masalah yang cukup diselesaikan dengan component composition (kirim JSX lewat children) atau ya... oper prop-nya aja. Global state itu ongkosnya nyata: state jadi bisa diubah dari mana aja, susah dilacak.

Kesalahan #4 — global demi hindari 1 level prop
// ❌ BERLEBIHAN — bikin global store cuma buat kirim `user` ke child langsung
const useUserStore = create(() => ({ user: currentUser }));
function Profile() { const user = useUserStore(s => s.user); ... }

// ✅ CUKUP — oper prop aja (1-2 level itu sehat & eksplisit)
function Dashboard({ user }) { return <Profile user={user} />; }

// ✅ ATAU — composition: kirim JSX lewat children, nggak perlu drill
function Layout({ children }) { return <main>{children}</main>; }
<Layout><Profile user={user} /></Layout>

5. Nyimpen server data di client state (bukan server-cache). Ini yang paling mahal. Fetch data API pakai useEffect + useState, simpen di Redux/Zustand "biar global". Lo bakal nulis ulang caching, dedup request, loading/error, dan revalidation — dan versi lo pasti punya race condition. Server state = cache. Pakai TanStack Query/SWR.

Kesalahan #5 — server state di client store
// ❌ SALAH — fetch manual + simpan di store global. Reinventing the wheel.
useEffect(() => {
  fetch('/api/produk').then(r => r.json()).then(setProduk); // race condition,
}, []);                                                     // no cache, no dedup
// ...lalu setProduk ke Zustand/Redux "biar global"

// ✅ BENAR — server state ditangani query library
function useProduk() {
  return useQuery({ queryKey: ['produk'], queryFn: () => fetch('/api/produk').then(r => r.json()) });
  // dapet gratis: cache, dedup, background refetch, loading, error, retry
}
const { data, isLoading, error } = useProduk();

6. Nyimpen derived state (duplikat source of truth). Nyimpen nilai yang bisa DIHITUNG dari state lain — total keranjang, list ter-filter, jumlah item — sebagai field terpisah di store. Sekarang lo punya dua sumber kebenaran yang wajib disinkron manual; lupa update salah satu = data nggak konsisten (total nunjukin angka lama). Derived value itu dihitung saat render (atau di selector), bukan disimpan.

Kesalahan #6 — total disimpan, bukan dihitung
// ❌ SALAH — total disimpan terpisah, wajib disinkron manual tiap mutasi
const useCart = create((set) => ({
  items: [], total: 0,
  add: (item) => set((s) => ({
    items: [...s.items, item],
    total: s.total + item.price, // lupa update di remove/updateQty? total basi 💀
  })),
}));

// ✅ BENAR — total DIHITUNG dari items (satu source of truth)
const useCart = create((set, get) => ({
  items: [],
  add: (item) => set((s) => ({ items: [...s.items, item] })),
}));
// di komponen / selector:
const total = useCart((s) => s.items.reduce((sum, i) => sum + i.price, 0));

✏️ LATIHAN — UJI PEMAHAMAN LO

Jawab dulu di kepala sebelum buka solusinya. Ini gaya soal yang beneran keluar di code review & interview.

✏️ Latihan 1: Taruh di mana? (klasifikasi state) klik untuk lihat solusi

Buat tiap state, tentuin rumahnya: useState / Context / Zustand / TanStack Query.

A. Theme dark/light. B. Daftar produk dari GET /api/products. C. Modal "yakin hapus?" kebuka/ketutup. D. Isi keranjang belanja (add/remove/qty, dipake di navbar & halaman cart). E. Profil user yang lagi login.

Jawaban: A → Context (global, jarang berubah). B → TanStack Query (server state / cache, BUKAN client store!). C → useState (lokal di komponen yang punya modal). D → Zustand (client state kompleks, sering berubah, dipake lintas komponen tak berelasi). E → Context (global, jarang berubah — atau TanStack Query kalau di-fetch dari /api/me). Kunci: tanya "asalnya dari server?" duluan — kalau iya, itu cache.

✏️ Latihan 2: Cari bug — kenapa consumer theme ikut re-render tiap ketik? klik untuk lihat solusi
Soal — ketik di search, seluruh app kedip
function AppProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const [search, setSearch] = useState(''); // diketik tiap huruf
  return (
    <AppContext.Provider value={{ theme, setTheme, search, setSearch }}>
      {children}
    </AppContext.Provider>
  );
}

Bug-nya: dua masalah numpuk. (1) search yang high-frequency dicampur di Context yang sama dengan theme — tiap huruf diketik, value Context berubah, SEMUA consumer (termasuk yang cuma pakai theme) re-render. (2) value-nya object inline, referensi baru tiap render. Fix: keluarin search dari Context (jadikan lokal di komponen search), pisah context per concern, dan memoize value.

Solusi
// search itu LOKAL — nggak ada urusannya sama Context global
function SearchBar() {
  const [search, setSearch] = useState('');
  return <input value={search} onChange={e => setSearch(e.target.value)} />;
}

// theme di context sendiri, value di-memoize
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}
✏️ Latihan 3: Hilangkan derived state klik untuk lihat solusi

Store di bawah nyimpen completedCount terpisah dari todos. Kenapa ini rawan bug, dan gimana benerinnya?

Soal
const useTodos = create((set) => ({
  todos: [],
  completedCount: 0, // ❌ duplikat — harus disinkron manual
  toggle: (id) => set((s) => ({

Kenapa rawan: completedCount bisa dihitung penuh dari todos. Simpan terpisah = dua source of truth yang gampang out-of-sync (toggle lupa update count). Fix: buang field-nya, hitung di selector saat dibutuhkan.

Solusi
const useTodos = create((set) => ({
  todos: [],

    todos: s.todos.map(t => t.id === id ? { ...t, done: !t.done } : t),

🎯 CHALLENGE — LENGKAPI THEME CONTEXT

Di bawah ada ThemeProvider yang nyediain theme dan toggle lewat Context. Consumer udah baca dari context dengan bener — tapi toggle-nya masih kosong, jadi klik tombol nggak ngefek. Tugas lo: bikin toggle beneran ganti antara 'light' ⇄ 'dark' supaya semua test lulus. Klik ▶ Test — starter-nya pasti gagal.

✅ Solusi Challenge — buka kalau udah nyerah (atau mau ngecek) klik untuk lihat solusi

Cukup isi toggle dengan updater function biar nggak gantung ke theme dari closure. Nggak perlu nyentuh consumer sama sekali — itu bukti Context yang dirancang bener: state ganti di provider, semua consumer otomatis ikut update.

Solusi lengkap
function ThemeProvider({ children }) {
  const [theme, setTheme] = React.useState('light');
  const toggle = () => setTheme(t => t === 'light' ? 'dark' : 'light');
  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

RECAP — YANG UDAH KITA BAHAS 📝

State management bukan one-size-fits-all — pilih yang sesuai kebutuhan

  • ✓Context API — Built-in React, cocok buat global state jarang berubah (theme, auth). Pisah context per concern.
  • ✓Zustand — Sweet spot. Minimal API, selective re-render, persist middleware. Cocok buat 80% project.
  • ✓Redux Toolkit — Heavy duty. createSlice, async thunk, RTK Query. Cocok buat enterprise/large team.
  • ✓Jotai — Atomic state. True selective re-render, derived atoms, async atoms. Cocok buat banyak independent state.
  • ✓Perbandingan — Context for simple, Zustand for most, RTK for enterprise, Jotai for performance-critical.
  • ✓Testing — Arrange-act-assert pattern. Test store directly + via component integration.
  • ✓Anti-patterns — Jangan: satu store raksasa, state global yang seharusnya lokal, derived state disimpen, mutation langsung.
  • ✓Mini Project — Task Manager dengan Zustand: multi-board, filter, search, persist.
