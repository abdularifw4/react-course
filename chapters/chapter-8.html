Chapter 8

State Management

Bayangin kamu punya satu lemari buat simpen semua baju. Gampang kan? Nah, bayangin lemari itu harus dibagi ke 50 orang sekaligus. Ribet banget, kan? 🫠 Itulah masalahnya kalau state management kamu berantakan.

🤔 MASALAHNYA APA SIH?

Pernah ngalamin ini? Kamu punya data username di komponen Header. Terus Footer butuh username yang sama. Terus Sidebar juga butuh. Terus ProfilePage juga...

Solusi pertama yang kebayang: "Ya udah props drilling aja!"

Tiba-tiba kamu ngirim props username dari App → Layout → Header → Nav → Avatar. Dan kalau ada perubahan? Wajah langsung 🤯.

⚠️ Props drilling itu kayak kirim paket pos lewat 10 tangan sebelum sampai ke penerima. Bisa aja, tapi capek banget dan error-prone!

Nah, di sinilah state management masuk. Dia itu kayak PA system (pengeras suara) di sekolah — satu kali bicara, semua orang denger. Nggak perlu kirim pesan satu-satu 🎤

📦 CONTEXT API — SENJATA BAWAAN REACT

Context API itu kayak lemari bersama di kost-kostan. Semua penghuni bisa taruh dan ambil barang tanpa harus numpuk di kamar masing-masing. Dan yang paling penting — dia udah built-in di React, nggak perlu install apa-apa!

Bayangin gini: kamu punya data theme (dark/light mode). Data ini butuh diakses oleh Header, Footer, Sidebar, Button, Card — practically semua komponen. Alih-alih kirim props ke tiap komponen, kamu bungkus aja semuanya pake ThemeProvider.

import { createContext, useContext, useReducer, useMemo } from 'react';

// Typed Context — kayak daftar isi lemari bersama
interface Todo {
  id: string; text: string; done: boolean;
}
interface TodoContextType {
  todos: Todo[];
  addTodo: (text: string) => void;
  toggleTodo: (id: string) => void;
}

// 1. Bikin konteksnya dulu
const TodoContext = createContext<TodoContextType | null>(null);

// 2. Bikin Provider — ini "lemarinya"
function TodoProvider({ children }: { children: React.ReactNode }) {
  const [todos, dispatch] = useReducer(todoReducer, []);

  // useMemo biar nggak bikin object baru tiap render
  const value = useMemo(() => ({
    todos,
    addTodo: (text: string) => dispatch({ type: 'ADD', payload: text }),
    toggleTodo: (id: string) => dispatch({ type: 'TOGGLE', payload: id }),
  }), [todos]);

  return <TodoContext.Provider value={value}>{children}</TodoContext.Provider>;
}

// 3. Custom hook — ini "kunci lemari"
function useTodos() {
  const ctx = useContext(TodoContext);
  if (!ctx) throw new Error('useTodos harus dalam TodoProvider bro!');
  return ctx;
}

Oh ternyata gitu! Jadi urutannya: bikin context → bikin provider → wrap komponen → akses pake custom hook. Simpel kan?

💡 Rahasia Context + useReducer: Combo ini itu setup state management yang solid tanpa library eksternal. Tapi hati-hati — setiap kali context value berubah, SEMUA komponen yang make useContext bakal re-render. Kalau datanya besar atau sering berubah (kayak cart belanja), pertimbangkan pake Zustand.

⚠️ Jangan kebablasan bikin Context! Satu context = satu "domain" data. Jangan bikin satu giant context buat semua state. Nanti yang terjadi kayak satu lemari buat semua barang — berantakan! Pisahin: AuthContext, ThemeContext, CartContext, dst.

🐻 ZUSTAND — SI MINIMALIS YANG GEMBUT

Oke, Context API udah keren. Tapi ada kalanya kamu mikir: "Bro, ngapain sih bikin Provider, bungkus komponen, bikin context, bikin hook... capek ah."

Masuk Zustand — library yang konsepnya kayak brankas portabel. Dia di mana aja, bisa diakses dari mana aja. Tanpa Provider. Tanpa boilerplate. Tanpa drama.

import { create } from 'zustand';

// Bikin store — sekali, selesai. Tanpa Provider!
interface CartStore {
  items: CartItem[];
  addItem: (item: CartItem) => void;
  removeItem: (id: string) => void;
  total: () => number;
}

const useCartStore = create<CartStore>((set, get) => ({
  items: [],
  addItem: (item) => set((state) => ({
    items: [...state.items, item]
  })),
  removeItem: (id) => set((state) => ({
    items: state.items.filter(i => i.id !== id)
  })),
  total: () => get().items.reduce((sum, i) => sum + i.price * i.qty, 0),
}));

// Pakai di komponen MANAPUN — nggak perlu bungkus Provider!
function CartButton() {
  // Selector s => s.items → cuma re-render kalau items berubah
  const items = useCartStore(s => s.items);
  const addItem = useCartStore(s => s.addItem);

  return (
    <button onClick={() => addItem({ id: '1', name: 'Produk', price: 50000, qty: 1 })}>
      Tambah ({items.length} item)
    </button>
  );
}

// Bahkan di luar komponen pun bisa!
function getTotalCart() {
  return useCartStore.getState().total();
}

Oh ternyata gitu! Zustand itu bedanya sama Context:

  • ✅ Nggak perlu Provider — tinggal create() langsung jadi
  • ✅ Selector built-in — cuma re-render yang beneran butuh data yang berubah
  • ✅ Bisa diakses di luar komponen — pake getState()
  • ✅ Support middleware — logging, persist ke localStorage, devtools, dll.

💡 Mental model Zustand: Bayangin dia itu kayak variabel global yang "aman". Sama kayak global variabel di JS biasa, tapi dia punya reactivity — komponen yang make selector-nya otomatis update kalau value berubah. It's basically useState tapi global. That's it. That's the whole trick. 🎩

🔨 REDUX TOOLKIT — BOS YANG PEMBERANI

Dulu, Redux itu terkenal karena boilerplate-nya gila. Bikin action, bikin reducer, bikin middleware, bikin store, connect ke komponen — satu fitur bisa butuh 5 file. Developer pada pusing 🤢

Tapi sekarang? Redux Toolkit (RTK) datang sebagai pahlawan. Dia itu Redux versi modern — semua boilerplate disederhanain. Kalau dulu butuh 5 file, sekarang cukup 1.

import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// Async thunk — buat fetch data dari API
const fetchUser = createAsyncThunk('user/fetch', async (id: string) => {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
});

// Slice — semua action + reducer dalam satu tempat
const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, loading: false, error: null },
  reducers: {
    // Sync actions
    logout: (state) => { state.data = null; },
    updateName: (state, action) => { state.data.name = action.payload; },
  },
  extraReducers: (builder) => {
    // Auto-handle loading, success, error dari async thunk
    builder
      .addCase(fetchUser.pending, (state) => { state.loading = true; })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.loading = false;
        state.data = action.payload;
      })
      .addCase(fetchUser.rejected, (state, action) => {
        state.loading = false;
        state.error = action.error.message;
      });
  },
});

export const { logout, updateName } = userSlice.actions;

RTK itu kayak kantor pemerintahan — banyak aturan, banyak prosedur, tapi semuanya jelas dan terstruktur. Cocok buat tim besar yang butuh konsistensi.

💡 Kapan pake Redux Toolkit? Kalau tim kamu lebih dari 5 orang, aplikasinya enterprise-level, dan kamu butuh devtools yang powerful + time-travel debugging. Kalau proyeknya kecil-menengah, Zustand aja udah lebih dari cukup!

🧠 KAPAN PAKAI APA? — CHEAT SHEET GAMPANG

Ini bagian yang paling sering ditanya: "Gue harus pake apa sih?". Tenang, gue bikin flowchart gampang buat kamu:

// 🗺️ PETA PEMILIHAN STATE MANAGEMENT:
//
// ┌─ State cuma dipake 1 komponen?
// │   → useState. Selesai. Pulang. 🏠
// │
// ├─ State dipake 2-3 komponen yang dekat?
// │   → Lift state up + props. Masih gampang. 👍
// │
// ├─ State global tapi jarang berubah (theme, auth, locale)?
// │   → Context API. Built-in, gratis, cukup. 📦
// │
// ├─ State global, sering berubah, butuh performa bagus?
// │   → Zustand. Pilihan terbaik buat kebanyakan kasus. 🐻
// │
// └─ Tim besar, enterprise, butuh devtools + time-travel?
//     → Redux Toolkit. Bos yang terstruktur. 🔨
//
// Hint: 80% kasus di dunia nyata bisa di-handle
//       sama useState + Context + Zustand. 🎯

// Contoh real: Mix semua di satu app
function App() {
  return (
    <ThemeProvider>         {/* Context → jarang berubah */}
      <AuthProvider>        {/* Context → jarang berubah */}
        <Router>
          <CartPage />       {/* Zustand → sering berubah */}
          <UserPage />        {/* Zustand → sering berubah */}
          <Counter />         {/* useState → cuma lokal */}
        </Router>
      </AuthProvider>
    </ThemeProvider>
  );
}

💡 Aturan emas: Mulai dari yang paling sederhana (useState). Naik level kalau udah beneran butuh. Nggak usah over-engineer dari awal. Pernah dengar quote ini? "Premature optimization is the root of all evil" — berlaku juga buat state management!

📝 RINGKASAN — YANG HARUS KAMU BAWA PULANG

  • 📌 Props drilling itu musuh — kalau udah lebih dari 3 level, waktunya cari solusi lain
  • 📌 Context API itu gratis dan built-in — cocok buat theme, auth, locale
  • 📌 Zustand itu minimalis dan powerful — pilihan terbaik buat 80% kasus
  • 📌 Redux Toolkit itu lengkap dan terstruktur — buat tim besar dan enterprise
  • 📌 Mulai dari simple — useState → Context → Zustand → Redux. Jangan lompat!
