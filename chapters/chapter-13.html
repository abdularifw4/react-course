MILESTONE 2 — FORM TAMBAH HABIT + RENDER LIST

Milestone · ±45 menit ngoding

Target: user bisa ngetik nama habit di form, submit, dan habit-nya muncul di list. Ini pola paling fundamental di React — controlled input → update state → re-render list — yang bakal lo pakai di hampir setiap app seumur karir lo.

Yang penting di sini adalah desain bentuk data-nya. Habit kita bukan sekadar string — dia butuh id unik (buat key dan toggle nanti) dan tempat nyimpen catatan centang harian. Kita pakai object checks dengan key tanggal: { '2026-07-02': true }. Percaya deh, mikirin bentuk data sebelum ngoding itu kebiasaan senior nomor satu.

Acceptance criteria:

  • Input controlled — nilainya dari state, bukan dari DOM
  • Submit via Enter ataupun tombol; input kosong / cuma spasi ditolak
  • Setelah submit, input dikosongkan lagi
  • Habit baru punya id unik dan checks: {}
  • List kosong menampilkan pesan ajakan, bukan halaman blank

Hint: State habits tinggal di App (bukan di form!) karena list juga butuh baca dia — ini lifting state up dari Bab 3. Form cukup nerima callback onAdd(name). Buat id unik, browser modern punya crypto.randomUUID() — nggak perlu library.

MILESTONE 3 — TOGGLE CHECK HARIAN

Milestone · ±60 menit ngoding

Target: setiap habit punya tombol centang untuk "hari ini". Klik sekali → tercentang, klik lagi → batal. State-nya tetap di App (lift-up), dan yang di-update adalah checks[tanggalHariIni] di dalam habit yang diklik — secara immutable, dua level dalamnya. Ini latihan update nested state yang paling sering keluar di interview.

Acceptance criteria:

  • Tiap item punya tombol toggle dengan indikator visual jelas ( / )
  • Toggle cuma ngubah habit yang diklik — habit lain nggak berubah
  • Key tanggal pakai waktu lokal, format YYYY-MM-DD
  • Nggak ada mutasi: map + spread, bukan assignment langsung

JANGAN TERJERAT! Jangan pakai new Date().toISOString().slice(0, 10) buat key tanggal! toISOString() itu UTC — kalau user di Jakarta (UTC+7) nyentang jam 6 pagi, key-nya masih tanggal kemarin. Bug timezone kayak gini yang bikin app "kadang bener kadang salah" dan susah banget di-debug. Pakai getFullYear/getMonth/getDate yang lokal.

Hint: Bikin helper todayKey() di src/utils/date.js. Pola update-nya: habits.map(h => h.id === id ? {...h, checks: {...h.checks, [today]: !h.checks[today]}} : h). Pecah HabitItem jadi komponen sendiri biar list-nya bersih.

// Format Date object jadi key 'YYYY-MM-DD' pakai waktu LOKAL.
// keyOf dipisah karena milestone 5 butuh format tanggal selain hari ini.
export function keyOf(d) {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, '0');
  const day = String(d.getDate()).padStart(2, '0');
  return `${y}-${m}-${day}`;
}

export function todayKey() {
  return keyOf(new Date());
}

MILESTONE 4 — PERSIST KE LOCALSTORAGE

Milestone · ±45 menit ngoding

Target: refresh browser, data habit tetap ada. Analoginya: selama ini state lo kayak nulis di papan tulis — begitu kelas bubar (tab ditutup), semuanya kehapus. localStorage itu lemari arsip di rumah user: apa yang lo simpen di situ tetap ada besok, minggu depan, sampai user sendiri yang buang.

Cara paling elegan: bikin custom hook useLocalStorage yang API-nya persis useState — jadi di App.jsx lo cuma ganti SATU baris. Ini contoh nyata kenapa custom hook disebut "unit arsitektur" di Bab 12: logic persist dibungkus rapi, komponen nggak perlu tahu detailnya.

Acceptance criteria:

  • Tambah habit + centang, refresh halaman → data masih utuh
  • Baca localStorage cuma sekali di render pertama (lazy initializer)
  • JSON korup / storage error nggak bikin app crash (try–catch)
  • Hook-nya generik — bisa dipakai buat state apa pun, bukan cuma habits

Hint: Dua bahan: (1) useState(() => ...) dengan fungsi — dieksekusi sekali doang buat baca storage, dan (2) useEffect dengan deps [key, value] buat nulis setiap kali value berubah. Kasih nama key yang berversi kayak 'habits-v1' — kalau bentuk datanya berubah nanti, lo tinggal ganti versi tanpa nabrak data lama.

src/hooks/useLocalStorage.js
import { useEffect, useState } from 'react';

export default function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    // lazy initializer: cuma jalan di render PERTAMA
    try {
      const stored = localStorage.getItem(key);
      return stored !== null ? JSON.parse(stored) : initialValue;
    } catch {
      // JSON korup / storage diblokir — fallback ke initial
      return initialValue;
    }
  });

  useEffect(() => {
    try {
      localStorage.setItem(key, JSON.stringify(value));
    } catch {
      // storage penuh / private mode — app tetap jalan tanpa persist
    }
  }, [key, value]);

  return [value, setValue];
}

MILESTONE 5 — STREAK COUNTER + STATS

Milestone · ±60 menit ngoding

Target: hitung streak — berapa hari berturut-turut habit dicentang — plus bar statistik: total habit, berapa yang udah dicentang hari ini, dan streak terbaik. Ini fitur yang bikin app lo dari "todo list biasa" jadi "app yang bikin nagih" ala Duolingo.

Poin arsitekturnya: streak itu derived state — dihitung dari checks, BUKAN disimpen sebagai state terpisah. Kalau lo nyimpen streak di state, lo harus sinkronin dia manual setiap toggle — dan suatu saat pasti lupa, terus datanya nggak konsisten. Rumus emasnya: kalau bisa dihitung dari state lain, jangan jadi state.

Acceptance criteria:

  • Fungsi calcStreak(checks) murni (pure) di utils/ — gampang dites
  • Hari ini belum dicentang ≠ streak putus — hitungan mulai dari kemarin
  • Tiap habit menampilkan streak-nya (kalau > 0)
  • StatsBar: jumlah habit, selesai hari ini, streak terbaik

Hint: Mundur satu-satu dari hari ini: bikin Date, cek checks[keyOf(d)], kalau true tambah counter lalu d.setDate(d.getDate() - 1), ulangi sampai ketemu hari bolong. Spesial case: kalau hari ini belum dicentang, mulai ngecek dari kemarin — user belum tentu udah "gagal" hari ini, mungkin cuma belum sempet.

Solusi Milestone 5 — buka kalau udah nyoba sendiriklik untuk lihat solusi
src/utils/streak.js
import { keyOf } from './date';

// Hitung berapa hari BERTURUT-TURUT checks bernilai true,
// mundur dari hari ini (atau kemarin kalau hari ini belum dicentang).
export function calcStreak(checks) {
  let streak = 0;
  const d = new Date();

  // hari ini belum dicentang? bukan berarti putus — mulai dari kemarin
  if (!checks[keyOf(d)]) {
    d.setDate(d.getDate() - 1);
  }

  while (checks[keyOf(d)]) {
    streak++;
    d.setDate(d.getDate() - 1);
  }

  return streak;
}

import { calcStreak } from '../utils/streak';

export default function StatsBar({ habits }) {
  const today = todayKey();

import { useEffect, useState } from 'react';

export default function useProducts() {
  const [products, setProducts] = useState([]);
  const [status, setStatus] = useState('loading'); // 'loading' | 'success' | 'error'

  useEffect(() => {
    const controller = new AbortController();
    setStatus('loading');

    fetch('https://fakestoreapi.com/products', { signal: controller.signal })
      .then((res) => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then((data) => {
        setProducts(data);
        setStatus('success');
      })
      .catch((err) => {
        // AbortError = fetch dibatalkan (user pindah halaman) — BUKAN error UI
        if (err.name !== 'AbortError') setStatus('error');
      });

    return () => controller.abort(); // cleanup: batalkan fetch yang nggantung
  }, []);

  return { products, status };
}

MILESTONE 9 — CART GLOBAL: CONTEXT + USEREDUCER

Milestone · ±90 menit ngoding — ini jantungnya project

Target: keranjang belanja yang bisa diakses dari halaman mana pun: tambah dari halaman detail, badge jumlah item di navbar, dan halaman /cart buat ubah qty/hapus item. Analoginya: cart itu tas belanja yang nempel di badan lo ke mana pun lo jalan di mall — pindah toko (halaman), isi tasnya ikut. State kayak gini nggak bisa hidup di satu halaman; dia harus global.

Keputusan arsitektur: Context + useReducer, bukan Zustand. Kenapa? Dua-duanya valid (Bab 8), tapi di sini gue pilih Context + useReducer karena: (1) zero dependency — cart logic-nya murni React, nggak nambah package; (2) operasi cart itu state transition klasik (ADD/REMOVE/SET_QTY/CLEAR) yang emang jodohnya reducer — semua aturan bisnis ngumpul di satu fungsi pure yang gampang dites; (3) skala state-nya kecil dan update-nya jarang, jadi kelemahan Context (re-render semua consumer) nggak kerasa. Kalau app lo nanti punya banyak state global yang update-nya sering dan saling silang — baru Zustand jadi pilihan yang lebih enak. Yang penting: lo bisa jelasin alasannya, bukan cuma ikut-ikutan.

Acceptance criteria:

  • Reducer handle ADD (produk sama → qty naik, bukan duplikat), REMOVE, SET_QTY (qty 0 → item hilang), CLEAR
  • Custom hook useCart() — komponen nggak pernah nyentuh dispatch mentah
  • Badge di navbar update realtime; total harga dihitung, bukan disimpen
  • Cart kosong → empty state dengan ajakan balik ke katalog

src/features/cart/CartContext.jsx
import { createContext, useContext, useReducer } from 'react';

const CartContext = createContext(null);

function cartReducer(items, action) {
  switch (action.type) {
    case 'ADD': {
      const existing = items.find((i) => i.id === action.product.id);
      if (existing) {
        // produk sama → naikin qty, jangan bikin baris duplikat
        return items.map((i) =>
          i.id === action.product.id ? { ...i, qty: i.qty + 1 } : i
        );
      }
      const { id, title, price, image } = action.product;
      return [...items, { id, title, price, image, qty: 1 }];
    }
    case 'REMOVE':
      return items.filter((i) => i.id !== action.id);
    case 'SET_QTY':
      if (action.qty < 1) return items.filter((i) => i.id !== action.id);
      return items.map((i) =>
        i.id === action.id ? { ...i, qty: action.qty } : i
      );
    case 'CLEAR':
      return [];
    default:
      return items;
  }
}

export function CartProvider({ children }) {
  const [items, dispatch] = useReducer(cartReducer, []);
  return (
    <CartContext.Provider value={{ items, dispatch }}>
      {children}
    </CartContext.Provider>
  );
}

// Satu-satunya pintu masuk ke cart — komponen nggak nyentuh dispatch mentah
export function useCart() {
  const ctx = useContext(CartContext);
  if (!ctx) throw new Error('useCart harus dipakai di dalam <CartProvider>');

  const { items, dispatch } = ctx;
  const totalQty = items.reduce((sum, i) => sum + i.qty, 0);
  const totalPrice = items.reduce((sum, i) => sum + i.qty * i.price, 0);

  return {
    items,
    totalQty,
    totalPrice, // derived — dihitung, bukan disimpen!
    add: (product) => dispatch({ type: 'ADD', product }),
    remove: (id) => dispatch({ type: 'REMOVE', id }),
    setQty: (id, qty) => dispatch({ type: 'SET_QTY', id, qty }),
    clear: () => dispatch({ type: 'CLEAR' }),
  };
}

// Pure function — bisa dites tanpa render komponen sama sekali
function validate(form) {
  const errors = {};
  if (form.name.trim().length < 3) errors.name = 'Nama minimal 3 karakter';
  if (!/^\S+@\S+\.\S+$/.test(form.email)) errors.email = 'Format email nggak valid';
  if (form.address.trim().length < 10) errors.address = 'Alamat minimal 10 karakter';
  return errors;
}

export default function CheckoutPage() {
  const { items, totalPrice, clear } = useCart();
  const [form, setForm] = useState(initialForm);
  const [errors, setErrors] = useState({});
  const [submitted, setSubmitted] = useState(false);
  const navigate = useNavigate();

  function handleChange(e) {
    // satu handler untuk semua field — kuncinya atribut name
    setForm({ ...form, [e.target.name]: e.target.value });
  }

  function handleSubmit(e) {
    e.preventDefault();
    const nextErrors = validate(form);
    setErrors(nextErrors);
    if (Object.keys(nextErrors).length > 0) return;

    setSubmitted(true);
    clear(); // kosongkan cart — "pesanan terkirim"
  }

  // Redirect 2,5 detik setelah sukses. Kenapa useEffect, bukan setTimeout
  // telanjang di handler? Biar timer-nya DIBERSIHKAN kalau user keburu
  // pindah halaman sendiri — tanpa cleanup, navigate() tetap nyala dari
  // komponen yang udah unmount dan nyeret user pindah halaman tiba-tiba.
  useEffect(() => {
    if (!submitted) return;
    const id = setTimeout(() => navigate('/'), 2500);
    return () => clearTimeout(id);
  }, [submitted, navigate]);

  if (submitted) {
    return <p className="success">🎉 Pesanan diterima! Makasih udah belanja di TokoMini.</p>;
  }
  if (items.length === 0) {
    return <p className="info">Keranjang kosong — nggak ada yang bisa di-checkout.</p>;
  }

  return (
    <form onSubmit={handleSubmit} noValidate className="checkout-form">
      <h1>Checkout — total ${totalPrice.toFixed(2)}</h1>

      <label>
        Nama
        <input name="name" value={form.name} onChange={handleChange} />
        {errors.name && <span className="field-error">{errors.name}</span>}
      </label>

      <label>
        Email
        <input name="email" type="email" value={form.email} onChange={handleChange} />
        {errors.email && <span className="field-error">{errors.email}</span>}
      </label>

      <label>
        Alamat pengiriman
        <textarea name="address" rows="3" value={form.address} onChange={handleChange} />

    return () => controller.abort();
  }, [reloadKey]); // reloadKey naik → effect jalan ulang → refetch

  return {
    products,
    status,
    retry: () => setReloadKey((k) => k + 1),
  };
}

MILESTONE 13 — SIMULASI DATA STREAM: useLiveData

Milestone · ±60 menit ngoding

Target: custom hook useLiveData yang tiap detik ngeluarin titik harga baru pakai algoritma random walk: harga berikutnya = harga sekarang ± perubahan acak kecil. Hasilnya grafik yang naik-turun natural kayak harga saham beneran — bayangin orang mabuk jalan pulang: tiap langkah arahnya acak, tapi selalu berangkat dari posisi terakhir, nggak teleport. Itulah kenapa random walk keliatan "hidup", beda sama Math.random() murni yang loncat-loncat kayak sinyal rusak.

Acceptance criteria:

  • Hook return array points: { time, price }, maksimal 60 titik terakhir
  • Interval dibersihkan di cleanup — nggak ada interval zombie saat unmount
  • Update state pakai functional updater (bukan closure basi)
  • Panel statistik menampilkan harga terakhir + arah naik/turun

JANGAN TERJERAT! Di dalam callback setInterval, JANGAN baca state langsung (setPoints([...points, baru])) — callback-nya dibuat sekali di mount, jadi points yang dia lihat selamanya array kosong. Ini stale closure, bug useEffect paling legendaris (Bab 3 & 11). Solusinya functional updater: setPoints((prev) => ...), plus useRef buat nyimpen harga terakhir.

Solusi Milestone 13 — buka kalau udah nyoba sendiriklik untuk lihat solusi
src/hooks/useLiveData.js
import { useEffect, useRef, useState } from 'react';

const MAX_POINTS = 60;

export default function useLiveData({ start = 100, intervalMs = 1000 } = {}) {
  const [points, setPoints] = useState([]);
  const lastPrice = useRef(start); // harga terakhir hidup di ref, bukan state

  useEffect(() => {
    const id = setInterval(() => {
      // random walk: harga baru = harga lama ± perubahan kecil (-1 s/d +1)
      const change = (Math.random() - 0.5) * 2;
      const price = Math.max(1, lastPrice.current + change);
      lastPrice.current = price;

      const point = {
        time: new Date().toLocaleTimeString('id-ID'),
        price: Number(price.toFixed(2)),
      };

      // functional updater — anti stale closure
      // slice(-MAX_POINTS): buang titik lama, memori nggak bocor
      setPoints((prev) => [...prev, point].slice(-MAX_POINTS));
    }, intervalMs);

    return () => clearInterval(id); // WAJIB: tanpa ini interval jadi zombie
  }, [intervalMs]);

  return points;
}

    id: nextId++,
    user: NAMES[Math.floor(Math.random() * NAMES.length)],
    amount: Number((Math.random() * 500 + 5).toFixed(2)),

import { useState } from 'react';

const BUFFER = 5; // baris ekstra di atas & bawah biar scroll ngebut nggak blank

export default function VirtualList({ items, rowHeight = 36, height = 420, renderRow }) {
  const [scrollTop, setScrollTop] = useState(0);

  const total = items.length;
  const startIndex = Math.max(0, Math.floor(scrollTop / rowHeight) - BUFFER);
  const visibleCount = Math.ceil(height / rowHeight) + BUFFER * 2;
  const endIndex = Math.min(total, startIndex + visibleCount);
  const visibleItems = items.slice(startIndex, endIndex);

  return (
    <div
      style={{ height, overflowY: 'auto' }}
      onScroll={(e) => setScrollTop(e.currentTarget.scrollTop)}
    >
      {/* div "hantu" setinggi SEMUA item — scrollbar-nya jadi jujur */}
      <div style={{ height: total * rowHeight, position: 'relative' }}>
        {visibleItems.map((item, i) => (
          <div
            key={item.id} // kontrak VirtualList: tiap item WAJIB punya id unik & stabil
            style={{
              position: 'absolute',
              top: (startIndex + i) * rowHeight, // posisi absolut sesuai index asli
              height: rowHeight,
              left: 0,
              right: 0,
            }}
          >
            {renderRow(item)}
          </div>
        ))}
      </div>
    </div>
  );
}

MILESTONE 17 — DARK MODE: CONTEXT + CSS VARIABLES

Milestone · ±45 menit ngoding — milestone terakhir!

Target: tombol / di topbar yang nge-toggle seluruh dashboard antara light dan dark, pilihannya keinget setelah refresh. Arsitekturnya elegan: React cuma ngurus SATU hal — atribut data-theme di <html>. Sisanya kerjaan CSS: karena semua warna dari awal pakai var(--...), satu blok [data-theme='dark'] mengganti seluruh palet — panel, teks, border, bahkan chart — tanpa satu komponen pun re-render.

Acceptance criteria:

  • ThemeProvider + hook useTheme() dengan guard error — dan cuma komponen ThemeToggle yang manggil hook-nya
  • Pilihan tema persist di localStorage, dibaca lagi saat load
  • Seluruh UI ganti warna, termasuk grid & garis chart
  • Transisi warna halus (CSS transition), nggak "njepret"

Hint: Di dalam useEffect provider: document.documentElement.dataset.theme = theme. Ini side effect DOM di luar React — dan justru itu tempatnya useEffect. Initial state pakai lazy initializer yang baca localStorage (pola yang sama kayak useLocalStorage di Project 1 — semuanya nyambung kan?).

src/theme/ThemeContext.jsx
import { createContext, useContext, useEffect, useState } from 'react';

const ThemeContext = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState(() => {
    try { return localStorage.getItem('theme') || 'light'; }
    catch { return 'light'; }
  });

  useEffect(() => {
    // React cuma nyetel satu atribut — CSS yang kerja keras
    document.documentElement.dataset.theme = theme;
    try { localStorage.setItem('theme', theme); } catch { /* private mode */ }
  }, [theme]);

  const toggle = () => setTheme((t) => (t === 'light' ? 'dark' : 'light'));

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme harus di dalam <ThemeProvider>');
  return ctx;
}
