TYPESCRIPT & REACT — KENAPA HARUS PAKAI TS? 🛡️

Pengenalan · 10 menit baca

Pernah ngalamin bug di production cuma karena typo? Misalnya nulis props.nam bukan props.nama. Di JavaScript, errornya cuma muncul pas runtime — artinya user yang nemu bug-nya dulu, baru kamu. Nggak enak banget kan? 😤

Nah, sekarang bayangin kamu kerja di tim 10 developer. Tanpa TypeScript, satu developer nulis komponen, developer lain pake — dan nggak tau props apa yang dibutuhkan. Doku? Kadang nggak update. Result? Bug di production karena props salah. Dengan TypeScript, props interface = living documentation yang selalu akurat. Nggak mungkin beda antara kode dan docs. 📚

TypeScript itu kayak helm safety waktu naik motor. Kamu tetap bisa naik tanpa helm (JavaScript), tapi kalau jatuh (bug), kepala kamu nggak terlindungi. Dengan helm, kalau ada yang salah, kamu dikasih tahu sebelum jalan — bukan pas sudah kejebak di tengah jalan. Oh ternyata gitu! 🤯

Yang bikin TypeScript spesial buat React: dia nge-catch error di waktu nulis kode, bukan pas runtime. IDE kamu langsung kasih merah underline kalau props salah, state type bermasalah, atau function return value nggak sesuai. Ini game changer banget buat productivity — apalagi di tim besar.

Tapi TypeScript BUKAN magic bullet. Dia punya learning curve, bikin kode lebih verbose, dan kadang nge-block kamu yang mau cepat prototype. Jadi jangan kebablasan — pakai TypeScript di level yang masuk akal buat projectmu. Small side project? Mungkin JS cukup. Production app dengan tim? TypeScript wajib. Kalau masih ragu, coba dulu di satu file — rasain sendiri bedanya. 🎯 Small side project? Mungkin JS cukup. Production app dengan tim? TypeScript wajib. 🎯

JavaScript vs TypeScript — Beda Apa?
// ❌ JavaScript — error baru ketahuan pas runtime
function Greeting({ name }) {
  return <h1>Hello, {name.tolowerCase()}</h1>; // TypeError: name.tolowerCase is not a function
}

// ✅ TypeScript — error ketahuan langsung di editor
interface GreetingProps {
  name: string;
}

function Greeting({ name }: GreetingProps) {
  return <h1>Hello, {name.toLowerCase()}</h1>; // ✅ aman!
}

💡 Fact: TypeScript diciptakan oleh Microsoft tahun 2012, dan sekarang dipake hampir semua library React besar (React sendiri, Next.js, Zustand, TanStack Query). Jadi belajar TS itu investasi yang pasti balik modalnya.

⚠️ Perhatian: TypeScript itu superset JavaScript — artinya semua kode JS valid itu juga TS valid. Kamu bisa migrasi pelan-pelan, file per file, nggak harus sekaligus. Ini yang bikin adopt TS di project existing jadi smooth.

TYPING PROPS & STATE — INTERFACE VS TYPE 📝

Fundamental · 12 menit baca

Kamu pasti pernah bingung: pake interface atau type? Di TypeScript, keduanya hampir sama — kayak memilih antara pensil dan ballpoint. Bisa nulis huruf yang sama, tapi ada perbedaan halus. Interface lebih cocok buat object shape (props, API response), type lebih fleksibel (union, intersection, mapped types).

Satu hal yang sering bikin pusing: kalau interface dan type bisa buat hal yang sama, mana yang harus dipilih? Jawabannya: konsistensi di tim. Pilih satu dan patuhi. Di React community, interface lebih populer buat props — tapi ini preferensi, bukan aturan mutlak.

Yang perlu tau: interface bisa extends interface lain, type bisa gabung pake & (intersection). Di React, kedua pattern ini equally useful — pilih yang lebih readable buat kasusmu.

Analoginya: interface itu kayak kontrak kerja — jelas, formal, bisa di-extend. type itu kayak notes personal — lebih flexible, bisa berbagai format. Di dunia React, kebanyakan developer pilih interface untuk props/state, type untuk yang lebih kompleks.

Analoginya: interface itu kayak kontrak kerja — jelas, formal, bisa di-extend. type itu kayak notes personal — lebih flexible, bisa berbagai format. Di dunia React, kebanyakan developer pilih interface untuk props/state, type untuk yang lebih kompleks.

Best practice di React: buat interface untuk setiap komponen props. Ini bikin kode self-documenting — tanpa baca implementation, kamu udah tau komponen butuh apa. Plus, pas ada breaking change, TypeScript langsung kasih error di semua tempat yang perlu diperbaiki. Nggak ada lagi "tiba-tiba app crash karena nambah field baru di API". 🔥

Yang sering terlewat: TypeScript juga bantu refactoring. Mau rename field nama jadi fullName? Tanpa TS, kamu harus manual find-and-replace DAN berharap nggak ada yang kelewat. Dengan TS, langsung compile error di semua tempat yang perlu di-fix. Satu rename, semua ter-update. Efisiensi ×10. ⚡

Optional props pake tanda ? — artinya field itu boleh ada boleh nggak. Default values bisa didefinisikan langsung di destructuring. Children props udah built-in dari React, tinggal extend ReactNode.

GENERIC COMPONENTS — CETAKAN KUE FLEKSIBEL 🧁

Advanced · 12 menit baca

Pernah bikin komponen list yang harus bisa nerima berbagai tipe data? Misalnya List<User>, List<Product>, List<Article>. Tanpa generic, kamu harus bikin tiga komponen yang logikanya sama. Dengan generic? Satu komponen, semua tipe data. 🎯

Generics itu kayak cetakan kue — bentuk cetakannya sama, tapi adonannya bisa beda-beda. Kamu bisa cetak kue cokelat, kue vanilla, atau kue stroberi dari cetakan yang sama. Di TypeScript, <T> itu "adonannya" — tipe data yang nanti diisi pas pemakaian.

Ini powerful banget buat library dan reusable components. Bayangin kamu bikin DataTable<T> yang bisa nerapa apa aja — columns-nya otomatis type-safe berdasarkan T. Nggak perlu bikin table component buat setiap entity. Satu komponen, ratusan use case. 🔥

Tapi jangan over-generic! Kalau komponen cuma dipake buat satu tipe data, nggak perlu generic. Generic berguna kalau komponen emang reusable across different types. Komponen yang too generic jadi susah dipahami — sama kayak fungsi yang parameter-nya terlalu abstract.

Generic Component — Reusable List
import { useState, type ReactNode } from 'react';

// Generic type T — fleksibel buat data apa aja
interface ListProps<T> {
  items: T[];
  renderItem: (item: T, index: number) => ReactNode;
  keyExtractor: (item: T) => string;
  emptyMessage?: string;
  className?: string;
}

// ✅ Satu komponen, bisa handle User, Product, apa aja
function GenericList<T>({ items, renderItem, keyExtractor, emptyMessage = 'Kosong' }: ListProps<T>) {
  if (items.length === 0) return <p>{emptyMessage}</p>;

  return (
    <ul className={className}>
      {items.map((item, i) => (
        <li key={keyExtractor(item)}>{renderItem(item, i)}</li>
      ))}
    </ul>
  );
}

// Pakai untuk User
interface User { id: string; name: string; }
const users: User[] = [{ id: '1', name: 'Arif' }];

function UserList() {
  return (
    <GenericList
      items={users}
      renderItem={(user) => <span>{user.name}</span>}
      keyExtractor={(user) => user.id}
    />
  );
}

// Pakai untuk Product — komponen yang SAMA
interface Product { sku: string; title: string; price: number; }
const products: Product[] = [{ sku: 'P1', title: 'Kaos', price: 150000 }];

function ProductList() {
  return (
    <GenericList
      items={products}
      renderItem={(p) => <span>{p.title} — Rp{p.price.toLocaleString()}</span>}
      keyExtractor={(p) => p.sku}
      emptyMessage="Belum ada produk"
    />
  );
}
Generic Hook — useFetch
import { useState, useEffect, useCallback } from 'react';

// Generic hook — return type otomatis sesuai T
interface UseFetchResult<T> {
  data: T | null;
  loading: boolean;
  error: string | null;
  refetch: () => void;
}

function useFetch<T>(url: string): UseFetchResult<T> {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  const fetchData = useCallback(async () => {
    try {
      setLoading(true);
      setError(null);
      const res = await fetch(url);
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      const json = await res.json() as T;
      setData(json);
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Unknown error');
    } finally {
      setLoading(false);
    }
  }, [url]);

  useEffect(() => { fetchData(); }, [fetchData]);
  return { data, loading, error, refetch: fetchData };
}

// ✅ TypeScript tau return type-nya User[] secara otomatis
const { data: users, loading } = useFetch<User[]>('/api/users');
// users inferred as User[] | null — auto-complete bekerja!

💡 Naming convention: Biasakan pake single letter uppercase buat generic: T, K, V. Kalau ada beberapa generic, pake nama deskriptif: TData, TError.

TYPING EVENTS — REMOTE TV YANG BEDA-BEDA 📺

Events · 10 menit baca

React events itu kayak remote TV — banyak tombol, tiap tombol fungsi beda. Pas nge-klik, gunakan MouseEvent. Pas ngetik di input, ChangeEvent. Pas submit form, FormEvent. Masing-masing punya properti yang berbeda sesuai konteksnya.

Di TypeScript, tiap event handler harus tipe event-nya sesuai. Nggak bisa sembarangan — ini yang bikin kode lebih safe. onClick handler harus terima MouseEvent, bukan KeyboardEvent. TypeScript pastikan kamu nggak salah pakai event. Oh ternyata gitu! 🤯

Pattern yang sering dipake: eksplisit type handler function, lalu pas ke JSX. Ini lebih clean daripada inline type di JSX. Plus, reusable — bisa dipake di beberapa event yang sama.

Yang bikin TypeScript + React events powerful: auto-complete di JSX. Pas kamu nulis onChange={}, TypeScript langsung kasih tahu handler harus nerima ChangeEvent. Nggak perlu buka docs lagi — IDE jadi teacher kamu. 🔥

Satu lagi yang powerful: event handler sebagai prop. Komponen parent bisa nge-pass typed handler ke child, dan TypeScript pastikan signature-nya cocok. Ini bikin component API lebih explicit dan safe.

Di real project, event typing sering dikombinasi dengan generic. Misalnya bikin useForm<T> hook yang return type-safe handlers buat setiap field. Ini pattern yang dipake React Hook Form, Formik, dan library form lainnya.

Berikutnya ada KeyboardEvent — penting banget buat accessibility. Pas user tekan Enter di input, Escape di modal, atau Tab untuk navigasi, semuanya KeyboardEvent. Type-nya udah include .key, .code, dan .metaKey (Cmd/Ctrl) yang type-safe.

Yang sering bikin pusing: event.target vs event.currentTarget. target itu element yang sebenarnya di-klik (bisa child), currentTarget itu element yang punya handler. Di React, hampir selalu pake currentTarget atau cast target ke tipe spesifik.

Typing React Events
import { useState, type FormEvent, type ChangeEvent, type MouseEvent } from 'react';

// ✅ ChangeEvent — buat input/select/textarea
function SearchBar() {
  const [query, setQuery] = useState('');

  const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
    setQuery(e.target.value); // e.target inferred as HTMLInputElement
  };

  return <input value={query} onChange={handleChange} placeholder="Cari..." />;
}

// ✅ FormEvent — buat form submit
function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');

  const handleSubmit = (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    console.log({ email, password }); // ✅ type safe
  };

  return (
    <form onSubmit={handleSubmit}>
      <input onChange={(e) => setEmail(e.target.value)} />
      <input onChange={(e) => setPassword(e.target.value)} type="password" />
      <button type="submit">Login</button>
    </form>
  );
}

// ✅ MouseEvent — buat klik
function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = (e: MouseEvent<HTMLButtonElement>) => {
    e.stopPropagation();
    setCount(c => c + 1);
  };

  return <button onClick={handleClick}>Count: {count}</button>;
}

⚠️ Gotcha: Pas pake e.target, TypeScript nggak otomatis tau element-nya apa. Kamu harus cast: (e.target as HTMLInputElement).value. Atau lebih safe: pake currentTarget yang udah type-safe.

TYPING HOOKS — LANJUTAN BAB 4 🔗

Hooks · 11 menit baca

Di Bab 4 kita udah bahas hooks advanced — sekarang waktunya kasih mereka "helm safety" TypeScript. useRef, useContext, useReducer — semua bisa di-type dengan aman. Dan hasilnya? Bug yang sebelumnya baru ketahuan di runtime, sekarang ketahuan langsung di editor. 🛡️

useRef punya dua use case utama di TypeScript: DOM ref (generic-nya element type) dan mutable value (generic-nya value type). Bedanya jelas — DOM ref itu buat akses .current yang berupa DOM element, mutable value buat simpen data biasa.

useContext tinggal pass type ke createContext. Kalau kamu bikin custom hook, return type-nya otomatis inferred — tapi lebih baik eksplisit buat kejelasan. Oh ternyata gitu! 🤯

useReducer lebih menarik — kamu harus type state, action, dan dispatch. Dengan discriminated union untuk action type, TypeScript bisa narrow otomatis di switch case. Ini pattern yang dipake Redux Toolkit juga — powerful banget.

Best practice: selalu bikin custom hook buat context (kayak useTheme() di contoh atas). Custom hook bisa throw error kalau context undefined — ini ngasih error message yang jelas pas developer lupa wrap Provider. Nggak ada lagi bug misterius "undefined is not a function" karena context null. 🔥

useRef, useContext, useReducer — TypeScript
import { useRef, useContext, useReducer, createContext } from 'react';

// ✅ useRef — DOM ref
function AutoFocusInput() {
  const inputRef = useRef<HTMLInputElement>(null); // typed!
  useEffect(() => { inputRef.current?.focus(); }, []);
  return <input ref={inputRef} placeholder="Auto focus!" />;
}

// ✅ useRef — mutable value
function Timer() {
  const timerId = useRef<number | null>(null);
  const start = () => { timerId.current = window.setInterval(() => {}, 1000); };
  const stop = () => { if (timerId.current) clearInterval(timerId.current); };
  return <button onClick={start}>Start</button>;
}

// ✅ useContext dengan typed context
interface ThemeCtx { theme: 'light' | 'dark'; toggle: () => void; }
const ThemeContext = createContext<ThemeCtx | undefined>(undefined);

function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be in ThemeProvider');
  return ctx; // inferred as ThemeCtx — ✅ type safe!
}

// ✅ useReducer — discriminated union action
interface Todo { id: number; text: string; done: boolean; }
type TodoAction =
  | { type: 'ADD'; text: string }
  | { type: 'TOGGLE'; id: number }
  | { type: 'DELETE'; id: number };

function todoReducer(state: Todo[], action: TodoAction): Todo[] {
  switch (action.type) {
    case 'ADD': return [...state, { id: Date.now(), text: action.text, done: false }];
    case 'TOGGLE': return state.map(t => t.id === action.id ? { ...t, done: !t.done } : t);
    case 'DELETE': return state.filter(t => t.id !== action.id);
    default: return state; // TypeScript tau action di sini "never"
  }
}

function TodoApp() {
  const [todos, dispatch] = useReducer(todoReducer, []);
  // dispatch inferred: (action: TodoAction) => void — ✅ auto-complete action types!
}

💡 Discriminated union: Perhatikan TodoAction — setiap action punya type field yang unik. TypeScript bisa narrow type otomatis di switch case. Jadi di case 'ADD', action.text exist. Di case 'TOGGLE', action.id exist. Auto-complete bekerja dengan sempurna!

UTILITY TYPES — TRANSFORMER ROBOT 🤖

Advanced · 12 menit baca

Utility Types itu kayak transformer robot — mereka nge-transform tipe yang udah ada jadi tipe baru tanpa kamu nulis ulang dari nol. Mau bikin semua field optional? Partial. Mau hapus field tertentu? Omit. Mau ambil field tertentu? Pick. Powerful dan hemat waktu! 🔥

Ini yang bikin TypeScript berbeda dari bahasa typed lain. Di Java/C#, kalau mau variant dari tipe, kamu harus bikin class baru. Di TypeScript, cukup "transform" tipe yang sudah ada. Ini bikin kode lebih DRY dan konsisten.

Yang paling sering dipake di React: Partial (form state yang bertahap), Omit (exclude field dari props), Record (dictionary/object map). Ketiganya muncul di hampir setiap project React + TypeScript. Oh ternyata gitu! 🤯

Utility Types — Partial, Omit, Pick, Record
interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'user';
  avatar: string;
}

// ✅ Partial — semua field jadi optional (buat form step-by-step)
type UserFormData = Partial<User>;
const formData: UserFormData = { name: 'Arif' }; // ✅ hanya name, sisanya optional

// ✅ Required — semua field WAJIB (override optional)
type StrictUser = Required<User>;

// ✅ Omit — hapus field tertentu (buat update tanpa id)
type UpdateUserDTO = Omit<User, 'id'>;
const update: UpdateUserDTO = { name: 'Arif', email: 'a@b.com', role: 'user', avatar: 'x' };

// ✅ Pick — ambil field tertentu saja
type UserPreview = Pick<User, 'id' | 'name'>;
const preview: UserPreview = { id: '1', name: 'Arif' };

// ✅ Record — bikin object dengan key-value type
type RolePermissions = Record<User['role'], string[]>;
const perms: RolePermissions = {
  admin: ['read', 'write', 'delete'],
  user: ['read', 'write'],
};

// ✅ Combinable — Omit + Partial
type CreateUserDTO = Partial<Omit<User, 'id'>>;
const newUser: CreateUserDTO = { name: 'New User', role: 'user' };
Utility Types di React — Real World
import { useState } from 'react';

// Partial buat form yang diisi bertahap
interface ProfileForm {
  name: string;
  bio: string;
  website: string;
  location: string;
}

function ProfileEditor() {
  // Semua field optional — user isi pelan-pelan
  const [form, setForm] = useState<Partial<ProfileForm>>({});

  const updateField = <K extends keyof ProfileForm>(
    field: K,
    value: ProfileForm[K]
  ) => {
    setForm(prev => ({ ...prev, [field]: value }));
  };

  // ✅ type safe — field name dan value type di-check
  return (
    <div>
      <input onChange={e => updateField('name', e.target.value)} />
      <textarea onChange={e => updateField('bio', e.target.value)} />
      <button disabled={Object.keys(form).length === 0}>Save</button>
    </div>
  );
}

// Record buat dictionary
type LabelMap = Record<string, string>;
const labels: LabelMap = { name: 'Nama', email: 'Email', role: 'Peran' };

COMMON MISTAKES & BEST PRACTICES 🚨

Tips · 10 menit baca

Sebelum masuk ke jebakan, penting bahas organisasi type. Di project besar, type definitions bisa jadi ratusan. Mana yang harus di-file terpisah? Rule of thumb: kalau type dipake di lebih dari 2 file, extract ke types.ts atau types/ folder. Kalau cuma dipake satu komponen, taruh di file komponen itu sendiri.

Banyak developer yang baru migrasi ke TypeScript bikin error yang sama berulang-ulang. Ini jebakan yang paling sering kena — dan cara menghindarinya. Biar nggak buang-buang waktu debug type error yang sebenarnya bisa dicegah. 🎯

Jebakan #1: any everywhere. Paling sering, paling berbahaya. any itu kayak bilang ke TypeScript "aku nggak butuh bantuan kamu" — padahal itu alasan utama pakai TS! Kalau bingung type-nya apa, pake unknown dulu — dia force kamu handle sebelum pakai.

Jebakan #2: Type assertion berlebihan. as itu override TypeScript — kamu bilang "percaya aku, ini tipe-nya X". Kadang perlu, tapi kalau kebanyakan, berarti type system-mu kurang ketat. Tiap as itu potensi runtime error.

Jebakan #3: Non-null assertion abuse. ! (non-null assertion) ngasih tahu TS "ini pasti bukan null". Tapi kalau ternyata null? Crash di runtime. Lebih baik handle null properly dengan optional chaining dan nullish coalescing.

Jebakan #4: Nggak type export. Kalau interface kamu cuma dipake internal, nggak apa-apa. Tapi kalau jadi props/component type yang di-import tempat lain, selalu export. Nggak export = nggak bisa di-reuse = duplikasi type di banyak file. 🔁

Common Mistakes — Bukan vs Benar
// ❌ Jebakan 1: any everywhere
function processData(data: any) {
  return data.name.tolowerCase(); // Nggak ada error, tapi bisa crash!
}

// ✅ Fix: gunakan unknown + type guard
function processDataSafe(data: unknown) {
  if (typeof data === 'object' && data !== null && 'name' in data) {
    return (data as { name: string }).name.toLowerCase();
  }
  throw new Error('Invalid data');
}

// ❌ Jebakan 2: type assertion berlebihan
const user = fetchData() as User; // kalau fetchData return object lain? crash!

// ✅ Fix: type guard atau zod validation
const result = fetchData();
if (isUser(result)) {
  console.log(result.name); // ✅ TypeScript tau ini User
}

// ❌ Jebakan 3: non-null assertion
const el = document.getElementById('app')!;
el.innerHTML = 'hello'; // kalau element nggak ada? crash!

// ✅ Fix: optional chaining
const el = document.getElementById('app');
el?.innerHTML = 'hello'; // ✅ aman, cuma jalan kalau element ada

// ❌ Jebakan 4: inline type tanpa export
function Card({ user }: { user: { name: string; email: string } }) {}

// ✅ Fix: extract & export interface
interface CardProps { user: Pick<User, 'name' | 'email'>; }
function Card({ user }: CardProps) {}

⚠️ ESLint rule: Pasang @typescript-eslint/no-explicit-any dan @typescript-eslint/no-non-null-assertion sebagai error di projectmu. Ini bikin tim nggak bisa pakai any dan ! sembarangan. Discipline from the start!

💡 Best practices ringkas: (1) Selalu type props dan state, (2) Export reusable interfaces, (3) Pake unknown bukan any, (4) Minimalisir type assertion, (5) Pake discriminated union buat action types, (6) Pasang strict mode di tsconfig.json.

MINI PROJECT — TYPE-SAFE TODO APP 🚀

Praktik · 20 menit

Sekarang saatnya gabungin semua yang udah kita pelajari jadi satu project! Bikin Todo App yang fully type-safe — interface, generics, utility types, discriminated union action, semuanya ada. Ini contoh nyata gimana TypeScript bikin React app lebih robust. 🔥

Todo App — Types & Reducer
import { useReducer, useRef, useMemo, type FormEvent } from 'react';

// ===== DATA MODEL =====
interface Todo {
  id: string;
  text: string;
  done: boolean;
  priority: 'low' | 'medium' | 'high';
  createdAt: number;
}

// ===== UTILITY TYPES =====
type CreateTodoDTO = Omit<Todo, 'id' | 'done' | 'createdAt'>;
type TodoFilter = Pick<Todo, 'done'> & { priority?: Todo['priority'] };

// ===== DISCRIMINATED UNION ACTIONS =====
type TodoAction =
  | { type: 'ADD'; payload: CreateTodoDTO }
  | { type: 'TOGGLE'; payload: string }
  | { type: 'DELETE'; payload: string }
  | { type: 'UPDATE'; payload: { id: string; text: string } }
  | { type: 'SET_FILTER'; payload: TodoFilter };

// ===== REDUCER =====
const initialState: Todo[] = [];
function todoReducer(state: Todo[], action: TodoAction): Todo[] {
  switch (action.type) {
    case 'ADD':
      return [...state, {
        id: crypto.randomUUID(),
        ...action.payload,
        done: false,
        createdAt: Date.now(),
      }];
    case 'TOGGLE':
      return state.map(t => t.id === action.payload ? { ...t, done: !t.done } : t);
    case 'DELETE':
      return state.filter(t => t.id !== action.payload);
    case 'UPDATE':
      return state.map(t => t.id === action.payload.id ? { ...t, text: action.payload.text } : t);
    case 'SET_FILTER':
      return state; // filter handled in component via useMemo
    default:
      const _: never = action;
      return state;
  }
}
Todo App — Component
function TodoApp() {
  const [todos, dispatch] = useReducer(todoReducer, initialState);
  const [form, setForm] = useState<Partial<CreateTodoDTO>>({});
  const inputRef = useRef<HTMLInputElement>(null);

  // Type-safe filter
  const [filter, setFilter] = useState<TodoFilter>({ done: undefined });

  const filtered = useMemo(() => {
    return todos.filter(t => {
      if (filter.done !== undefined && t.done !== filter.done) return false;
      if (filter.priority && t.priority !== filter.priority) return false;
      return true;
    });
  }, [todos, filter]);

  const handleSubmit = (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    if (!form.text || !form.priority) return;
    dispatch({ type: 'ADD', payload: { text: form.text, priority: form.priority } });
    setForm({});
    inputRef.current?.focus();
  };

  return (
    <div className="max-w-md mx-auto p-6">
      <h1>Todo App ✅</h1>
      <form onSubmit={handleSubmit}>
        <input ref={inputRef} value={form.text ?? ''}
          onChange={e => setForm(f => ({ ...f, text: e.target.value }))} placeholder="Todo..." />
        <select value={form.priority ?? 'medium'}
          onChange={e => setForm(f => ({ ...f, priority: e.target.value as Todo['priority'] }))}>
          <option value="low">Low</option>
          <option value="medium">Medium</option>
          <option value="high">High</option>
        </select>
        <button type="submit">Tambah</button>
      </form>
      <div className="flex gap-2 my-4">
        {(['all','active','done'] as const).map(f => (
          <button key={f} onClick={() => setFilter({ done: f === 'all' ? undefined : f === 'done' })}>
            {f}
          </button>
        ))}
      </div>
      <ul>{filtered.map(t => (
        <li key={t.id}>
          <input type="checkbox" checked={t.done}
            onChange={() => dispatch({ type: 'TOGGLE', payload: t.id })} />
          <span className={t.done ? 'line-through' : ''}>{t.text}</span>
          <span className="text-xs">{t.priority}</span>
          <button onClick={() => dispatch({ type: 'DELETE', payload: t.id })}>×</button>
        </li>
      ))}</ul>
    </div>
  );
}

TypeScript di React bukan opsional — ini standar industry 2024+

  • ✓TypeScript + React — Helm safety buat kode kamu. Catch error di editor, bukan di runtime. Migrasi pelan-pelan, file per file.
  • ✓Typing Props & State — Interface buat object shape, type buat union/complex types. Selalu buat interface untuk setiap komponen props.
  • ✓Generic Components — Satu komponen, semua tipe data. Pake <T> buat reusable components dan hooks. Jangan over-generic!
  • ✓Typing Events — MouseEvent, ChangeEvent, FormEvent. Tiap event handler harus tipe event-nya sesuai. Pake currentTarget, bukan target.
  • ✓Typing Hooks — useRef<HTMLInputElement>, typed context, discriminated union buat useReducer action types.
  • ✓Utility Types — Partial, Omit, Pick, Record. Transformer tipe yang hemat waktu dan bikin kode lebih DRY.
  • ✓Common Mistakes — Hindari any, minimasi assertion, handle null properly, export reusable types.
  • ✓Mini Project Todo App — Full type-safe app dengan interface, utility types, discriminated union, dan typed hooks.
