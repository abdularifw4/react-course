Pernah ngerasa kode React kamu tuh kayak main tebak-tebakan? 🤔 Props-nya bener nggak ya? State-nya string atau number? Yuk kita selesain masalah itu buat selamanya.

🚀 SETUP REACT + TYPESCRIPT — Nggak Seribet yang Kamu Kira!

Oke, jujur. Dulu pas pertama denger TypeScript, aku juga mikir: “Ribet ah, JavaScript aja udah pusing.” Tapi percaya deh, setelah coba, rasanya kayak naik kelas dari naik angkot ke naik MRT. 🚇 Lebih aman, lebih cepat, dan nggak bikin pusing.

TypeScript itu bodyguard buat kode kamu. Dia ngecek semuanya sebelum jalan — jadi bug yang dulu baru ketemu di produksi jam 2 malam, sekarang langsung ketangkep pas kamu ngetik di editor. AUTO-COMPLETE JADI SUPER POWERS! 🦸‍♂️

# Cara paling gampang: pake Vite (recommended banget!)
npm create vite@latest my-app -- --template react-ts

# Atau kalau kamu masih suka CRA (legacy, tapi masih jalan):
npx create-react-app my-app --template typescript

# Setelah itu, struktur foldernya biasanya kayak gini:
# src/
#   components/
#     Button.tsx      <-- perhatikan .tsx bukan .js!
#     Card.tsx
#   types/
#     index.ts         <-- tempat simpan semua tipe reusable
#   App.tsx
#   main.tsx

Satu hal yang wajib kamu lakuin dari hari pertama: buka file tsconfig.json, terus pastikan “strict”: true. Kenapa? Karena kalau nggak, sama aja kayak beli helm mahal tapi nggak dipake. 🪖 Helm-nya ada, tapi kepala tetep bahaya.

💡 Real talk: Strict mode emang bikin lebih banyak merah di editor pas awal-awal. Tapi itu merah yang berguna — kayak alarm kebakaran yang emang nyala pas ada api, bukan alarm palsu. Lebih baik ribet 5 menit di awal daripada debugging 5 jam di produksi. Trust me. 😤

📝 TYPING PROPS & STATE — Ngasih Label di Semua Kotak

Bayangin kamu pindah rumah. Kalau semua kardus nggak ada label, pas mau cari baju kerja... kepalang tanggung. 😫 Nah, interface di TypeScript itu kayak label kardus kamu. Biar pas butuh apa, langsung ketemu. Ga perlu buka-buka satu-satu.

// Interface itu kayak “kontrak” komponen kamu
interface UserCardProps {
  name: string;          // wajib isi!
  email: string;          // wajib isi!
  age?: number;           // pakai ? = optional, boleh nggak diisi
  role: ‘admin’ | ‘user’ | ‘guest’;  // union type — cuma 3 ini yang boleh!
  onEdit: (id: string) => void;       // function yang return void
  children: React.ReactNode;           // buat ngewrap anak komponen
}

// Sekarang function signature-nya langsung rapi
function UserCard({ name, email, age, role, onEdit, children }: UserCardProps) {
  return (
    <div>
      <h3>{name}</h3>
      <p>{email}</p>
      {age && <p>Umur: {age}</p>}
      <span>{role}</span>
      <button onClick={() => onEdit(‘123’)}>Edit</button>
      {children}
    </div>
  );
}

Nah, yang sering bikin bingung tuh ngetik state. Kan di React state itu biasanya objek ya. Nah di TypeScript, kita juga harus ngasih tahu bentuk objek-nya kayak apa:

// State juga butuh label!
interface FormState {
  username: string;
  password: string;
  errors: Record<string, string>;  // key string, value string
  isLoading: boolean;
}

function LoginForm() {
  const [state, setState] = useState<FormState>({
    username: ‘’,
    password: ‘’,
    errors: {},
    isLoading: false
  });
  
  // Sekarang kalau kamu salah ketik state.userename,
  // TypeScript LANGSUNG nyalakan alarm! 🚨
}

💡 Oh ternyata gitu! Interface vs type alias — buat most cases di React, interface aja udah cukup. Lebih readable dan extensible. Type alias baru dipake kalau kamu butuh union types atau mapped types. Jangan overthink ini, yang penting jalan dulu! 🏃

🔄 GENERICS — Satu Komponen, Bisa Nangani Semuanya

Pernah bikin komponen List yang bisa nampilin User, Product, Article... tapi tiap ganti data, kamu copy-paste komponennya dan cuma ganti tipe-nya doang? 🤦‍♂️ Stop doing that! Generics bikin kamu cukup bikin sekali, pakai berkali-kali — dan tetap type-safe!

Analoginya gini: generics itu kayak cetakan kue 🥰. Cetakan-nya sama, tapi adonan-nya bisa beda-beda — bisa coklat, bisa pandan, bisa keju. Bentuk kue-nya tetap konsisten, tapi isinya fleksibel.

// List component yang “universal” — bisa nangani tipe apapun!
interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
  keyExtractor: (item: T) => string;
  onItemSelect?: (item: T) => void;   // optional callback
}

function List<T>({ items, renderItem, keyExtractor, onItemSelect }: ListProps<T>) {
  return (
    <ul>
      {items.map(item => (
        <li key={keyExtractor(item)} onClick={() => onItemSelect?.(item)}>
          {renderItem(item)}
        </li>
      ))}
    </ul>
  );
}

// ——— Pakai buat nampilin User ———
interface User { id: string; name: string; }
const users: User[] = [{ id: ‘1’, name: ‘Arif’ }];

<List<User>
  items={users}
  keyExtractor={u => u.id}
  renderItem={u => <span>{u.name}</span>}
  onItemSelect={u => console.log(u)}  // TypeScript tau u itu User!
/>

// ——— Pakai buat nampilin Product ———
interface Product { sku: string; title: string; price: number; }

<List<Product>
  items={products}
  keyExtractor={p => p.sku}
  renderItem={p => <span>{p.title} - Rp{p.price}</span>}
  // onItemSelect-nya juga otomatis terima Product!
/>

💡 Pro tip: Pas pertama kali lihat generics, wajar kalau mikir “ini terlalu abstrak”. Tapi coba bayangin: kalau nggak pake generics, kamu bisa aja punya 5 versi List yang beda cuma di tipe data-nya. Generics mencegah itu. Satu komponen, lima kegunaan. Efisiensi maksimal! 🎯

🧠 UTILITY TYPES — Swiss Army Knife Buat Tipe Data

Oke, ini bagian yang sering banget dipake di dunia nyata tapi jarang diajarin dengan baik. TypeScript punya utility types bawaan yang kayak Swiss army knife — kecil tapi multitool banget! 🔪

Bayangin kamu punya tipe User yang lengkap banget — ada id, name, email, age, role. Tapi pas mau bikin form edit, kamu nggak butuh semuanya kan? Cuma name dan email aja. Nah, daripada bikin interface baru dari nol, utility types nolong kamu “potong” tipe yang udah ada.

interface User {
  id: string;
  name: string;
  email: string;
  age: number;
  role: ‘admin’ | ‘user’;
}

// 🔧 Partial — semua properti jadi optional (bohong banget buat update!)
function updateUser(id: string, data: Partial<User>) {
  // data bisa cuma { name: ‘Arif’ } — nggak perlu kirim semua field!
}

// 🔧 Pick — ambil CUMA properti yang kamu mau
type UserPreview = Pick<User, ‘id’ | ‘name’>;
// hasilnya: { id: string; name: string }
// Pas banget buat card preview di list!

// 🔧 Omit — KECUALIKAN properti yang nggak kamu butuh
type CreateUser = Omit<User, ‘id’>;
// hasilnya: { name: string; email: string; age: number; role: ... }
// Karena pas create, id-nya otomatis dari server kan?

// 🔧 Record — bikin tipe objek secara cepat
type ThemeColors = Record<string, string>;
const colors: ThemeColors = {
  primary: ‘#27d890’,
  secondary: ‘#111827’,
  danger: ‘#ef4444’
};

// 🔧 Required — kebalikan Partial, semua jadi WAJIB
type RequiredUser = Required<Partial<User>>;

// 🔧 Readonly — objek jadi immutable, nggak bisa diubah
type FrozenUser = Readonly<User>;

💡 Cheat sheet mental: Partial = “santai dulu aja, semua optional” · Pick = “mau ini sama ini aja” · Omit = “semua boleh, KECUALI ini” · Record = “dictionary dari A ke B”. Ingat 4 ini, kamu udah cover 90% use case! 🧠

✅ RINGKASAN — Yang Perlu Kamu Bawa Pulang

Oke, kita udah bahas banyak. Biar nggak mabok, ini poin-poin penting yang harus kamu inget:

TypeScript bukan buat bikin kode kamu lebih ribet — dia bikin kode kamu lebih predictable. Pas kamu kerja di tim, TypeScript itu kayak dokumentasi yang selalu up-to-date. Nggak perlu bertanya “ini props-nya apa aja?” — tinggal liat interface-nya. Done. 🎉
